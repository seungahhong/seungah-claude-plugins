# backend-harness

백엔드/API를 **실행 기반 검증**으로 구현하는 멀티 에이전트 하네스입니다.
"됐다"는 *말*이 아니라 **빌드·테스트가 실제로 통과**해야만 통과로 인정하고, **환경 구성을 1급 시민**으로 끌어올려
구현 전에 빌드·실행 가능한 상태부터 독립 단계로 확보합니다.

## 두 가지 차별점

- **실행 기반 검증(에이전트 자기보고 불신)** — 통과 판정은 만든 에이전트와 분리된 `be-verifier`가 빌드·마이그레이션·테스트를 *직접 다시 실행*해 내립니다. commit·"통과했습니다" 같은 자기보고는 증거가 아닙니다. 통과를 위해 테스트를 무력화·스킵·우회한 흔적(reward hacking)을 적대적으로 잡고, 고커버리지로 게이트합니다.
- **환경 구성을 1급 시민으로** — 저장소 단위 백엔드 작업에서는 *실행 가능한 환경을 세우는 것 자체가 최대 병목*이라는 근거에 따라, 환경 구성을 구현에 묻지 않고 `env-provisioner` 독립 단계로 둡니다. 환경이 안 서면 이후 구현·검증 신호가 모두 거짓이 되기 때문입니다.

## 왜 이렇게 하나 (근거)

저장소 단위 백엔드 작업은 최신 모델로도 절반가량 실패합니다 — FreshBrew 52.3%(arXiv:2510.04852)[GOLD],
MigrationBench 71.67%/53.33%(arXiv:2505.09569)[GOLD], Web-Bench Pass@1 25.1%(arXiv:2505.07473)[GOLD].
환경 구축 F2P는 ≤37.7%로 최저 병목이고, 멀티에이전트가 단일에이전트보다 낫습니다(arXiv:2512.06915)[GOLD].
에이전트는 과신합니다 — 47.41% commit(성공 주장)하나 실제 F2P 35.53%(같은 출처)[GOLD]. 그래서 *자기보고가
아니라 실제 실행*으로만 통과를 인정합니다.

> 수치는 출처·날짜·등급과 함께만 인용하며, 이 하네스가 점수를 'N% 올린다'고 약속하지 않습니다 — 위 수치는 현재 한계를 보여줄 뿐입니다.

## 설치

이 저장소를 Claude Code 플러그인 마켓플레이스로 추가한 뒤 `backend-harness` 플러그인을 활성화하면,
`backend-harness` 오케스트레이터 스킬이 자동 트리거되거나 직접 호출할 수 있고, `test-generator` 스킬도 함께 쓸 수 있습니다.

## 스킬

| 스킬 | 역할 |
|------|------|
| `backend-harness` | 오케스트레이터(진입점). 범위/계약 → 설계 → 환경 구성 → 구현 → 실행 검증의 5단계를 매 단계 승인 게이트로 끌고 가며 전용 에이전트(be-architect / env-provisioner / be-implementer / be-verifier)를 호출한다. |
| `test-generator` | 기존 코드에 실행기반 테스트를 generate→compile→execute→repair 공진화 루프로 생성·수리한다. 자유형 재프롬프트 대신 오류 유형별 결정론적 수리 템플릿을 쓰고 커버리지 피드백으로 반복한다(QA 탈-FE편중). |

## 에이전트 팀 (모두 `model: opus`)

| Phase | 에이전트 | 역할 |
|-------|----------|------|
| 설계 | `be-architect` | 서비스 경계·API 계약·데이터 모델/마이그레이션 + 검증 후크·환경 요구사항 |
| 환경 | `env-provisioner` | 빌드·실행·테스트 가능 상태 확보(런타임·의존성·외부 서비스·헬스체크) — 독립 1급 단계 |
| 구현 | `be-implementer` | API 핸들러·서비스 로직·DB 스키마/마이그레이션 구현(계약 준수, 자기보고≠통과) |
| 검증 | `be-verifier` | 실행 기반 PASS/FAIL — 빌드·테스트 직접 재실행, 고커버리지·reward-hacking 가드 |

## 언제 쓰나 / 언제 다른 도구를 쓰나

**이 하네스를 쓰세요**
- 백엔드/API를 **설계부터 실행 검증까지 구현**하고 싶을 때
- 에이전트가 "통과했다"는 *말*이 아니라 **빌드·테스트의 실제 통과**로만 끝내고 싶을 때
- 환경 구성이 막히는 게 가장 큰 리스크라, **환경부터 안정적으로 세우고** 구현하고 싶을 때

**범위 밖 (다른 워크플로로 처리)**
- 완성된 API spec *검수*·breaking change 점검 — 이 하네스는 계약을 받아 *구현*한다(완성 spec 검수가 아님).
- 프론트엔드 화면·컴포넌트 구현 — 이 하네스는 서버/API다.
- 기획문서(PRD)·사용자 스토리 작성.
- 커밋 메시지·PR 리뷰 — 커밋/PR이 필요하면 별도 커밋 워크플로를 쓴다.
- 하네스(CLAUDE.md/SKILL.md/agents) 자체 진단·개선.
- 검증 가능한 목표를 향한 *단발 자율 반복 루프*. 이 하네스는 *BE 구현 워크플로 오케스트레이션*(설계→환경→구현→검증)이다.
- 새 기능을 **TDD(Red-Green-Refactor)**로 — 테스트 먼저, 코드 나중(test-first). test-generator는 반대로 *기존 코드*의 실행기반 테스트 생성·수리(test-after)다.

## 진행 방식

| Phase | 단계 | 핵심 산출물 |
|-------|------|-------------|
| 0 | 범위/계약 입력 | 기능 범위·주어진 계약·기술 제약 — 승인 게이트 |
| 1 | 설계 (be-architect) | Design Spec(서비스 경계·API 계약·데이터 모델·검증 후크·환경 요구사항) |
| 2 | 환경 구성 (env-provisioner) | Environment Readiness(재현 가능 절차 + 실제 통과 증거) — 1급 독립 단계 |
| 3 | 구현 (be-implementer) | Implementation 리포트(변경·계약 대비·로컬 확인 증거) |
| 4 | 실행 검증 (be-verifier) | Verdict(빌드·마이그레이션·테스트·커버리지·계약 인수·reward-hacking 점검) — PASS는 실제 실행으로만 |

- 매 Phase 후 `[Phase N] {핵심결정/증거} — 다음: {다음}. 진행할까요?` 1줄 보고로 승인을 받아 진행합니다.
- Phase 4가 **FAIL**이면 어느 항목/어떤 reward hacking인지에 따라 구현(또는 설계·환경) 단계로 되돌리고, **BLOCKED**(환경 문제로 검증 불가)면 거짓 FAIL로 세지 않고 환경 보정으로 되돌립니다.

## evals

`evals/trigger-eval.json`은 이 하네스가 발동해야 하는 경우(should-trigger)와 발동하면 안 되는 경우
(should-not-trigger — 계약 검수·FE·PRD·커밋/리뷰·하네스 진단·자율 반복 루프·FE TDD·설정 변경)를 정의해 트리거 정확도를 점검합니다.
