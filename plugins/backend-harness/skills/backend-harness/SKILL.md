# Backend Harness — 실행 기반 검증으로 끌고 가는 백엔드 구현 오케스트레이터

백엔드/API 작업을 **범위/계약 → 설계 → 환경 구성 → 구현 → 실행 검증**의 5단계로 끌고 간다. 이 하네스의 두 차별점은:

1. **실행 기반 검증(에이전트 자기보고 불신)** — "됐다"는 *말*이나 commit이 아니라 **빌드·테스트가 실제로 통과**해야만 통과로 인정한다. 검증은 만든 에이전트와 분리된 be-verifier가 직접 재실행해 판정하고, 테스트를 무력화·우회한 흔적(reward hacking)을 적대적으로 잡는다.
2. **환경 구성을 1급 시민으로** — 저장소 단위 BE 작업에서 *실행 가능한 환경을 세우는 것 자체가 최대 병목*이라는 근거에 따라, env-provisioner를 구현에 묻지 않고 **독립 Phase**로 둔다.

## 왜 이 구조인가 (근거 — 등급·CAVEAT 동반)

- **저장소 단위 BE 작업은 SOTA도 절반가량 실패**: FreshBrew JDK17 마이그레이션 최고 Gemini 2.5 Flash 52.3%(arXiv:2510.04852)[GOLD]; MigrationBench Claude-4.5-Sonnet 71.67%(minimal)/53.33%(maximal)(arXiv:2505.09569)[GOLD]; Web-Bench의 Web-Agent에서 SOTA Claude 3.7 Pass@1 25.1%(SWE-bench Verified 65.4% 대비 급락)(arXiv:2505.07473)[GOLD]. → 단발 자동완성이 아니라 *구조화된 멀티에이전트 + 실행 검증*이 정당화된다.
- **환경 구성이 최대 병목**: 실행가능 Docker 환경 구축의 F2P ≤37.7%, 멀티에이전트(SWE-Builder 4-agent)가 단일에이전트(RepoLaunch)보다 우월(arXiv:2512.06915, 2025-12)[GOLD]. → env-provisioner를 **독립 Phase**로 둔 근거.
- **에이전트 과신 → 실행 검증 필수**: Claude-Sonnet-4가 47.41% commit(성공 주장)하나 실제 F2P 35.53%(arXiv:2512.06915)[GOLD]. 자기평가는 신뢰 불가하다. reward hacking 방지엔 고커버리지 테스트가 필요(arXiv:2510.04852)[GOLD]. → 통과는 *실제 실행*으로만, 커버리지는 게이트로.

> 수치는 위 출처·날짜·등급과 함께만 인용한다. '개선 N%' 같은 약속은 하지 않는다 — baseline 수치는 현재 한계를 보여줄 뿐, 이 하네스가 그만큼 올린다는 보장이 아니다.

근거 dossier 전문: [references/backend-harness-research.md](./references/backend-harness-research.md). 원칙·anti-pattern: [references/backend-harness-principles.md](./references/backend-harness-principles.md).

## 경계 (먼저 읽고 발동 여부를 판단하라)

이 하네스는 **'백엔드/API를 실행 검증까지 구현한다'**. 다음은 명시적으로 범위 밖이다.

- **API 계약 *검수*** — 완성된 외부 spec의 일관성·breaking change·하위호환 점검은 범위 밖이다. backend-harness는 *구현*이다(계약을 받아 코드로 만든다).
- **FE 화면·컴포넌트 구현** — 이 하네스는 서버/API다.
- **PRD·사용자 스토리 작성**.
- **커밋 메시지·PR 리뷰** — 커밋/PR이 필요하면 별도 커밋 워크플로를 쓴다.
- **하네스 자체 진단·개선** — 루트 CLAUDE.md/SKILL.md/agents의 결함을 trace로 고도화하는 것은 범위 밖이다.
- **검증 가능한 목표를 향한 단발 자율 반복 루프** — 목표 기반 자가수정 반복은 범위 밖이다. backend-harness는 *BE 구현 워크플로 오케스트레이션*(설계→환경→구현→검증)이다.
- **기존 코드의 단위/통합 테스트 생성·수리만** — 같은 플러그인의 `test-generator` 스킬을 직접 쓴다(전체 구현 워크플로가 아니라 테스트만 필요할 때).

경계가 모호하면 한 질문으로 확인한다 — "백엔드를 *설계부터 실행 검증까지 구현*하는 건가요, 아니면 완성 spec *검수*·기존 코드 *테스트 생성*·FE만 필요한가요?"

## 내재화 원칙 (모든 Phase가 따른다)

- **실행 기반 검증** — 통과는 자기보고가 아니라 빌드·테스트의 실제 통과로만. 증거(명령·종료코드·카운트) 없는 PASS 금지.
- **환경 우선(1급 시민)** — 환경이 안 서면 이후 신호가 모두 거짓이 된다. 구현 전에 빌드·실행 가능 상태를 독립 Phase로 확보한다.
- **generator/checker 분리** — 구현(be-implementer)과 검증(be-verifier)을 분리한다. 검증자는 코드를 고치지 않는다.
- **reward-hacking 가드** — 테스트 무력화·스킵·우회, 마이그레이션 우회를 적대적으로 점검한다. 고커버리지로 게이트한다.
- **계약 준수** — be-architect의 API 계약·데이터 모델을 구현으로 임의 우회하지 않는다. 못 맞추면 설계로 되먹임한다.
- **승인 게이트·관찰성** — 매 Phase 산출물 미리보기 후 1줄 보고로 승인을 받아 진행한다. 요청되지 않은 사이드 에이전트나 중복 실행을 만들지 않는다.

## 에이전트 팀

| Phase | 에이전트 | 역할 |
|-------|----------|------|
| 1 설계 | `be-architect` | 서비스 경계·API 계약·데이터 모델/마이그레이션 전략 + 검증 후크·환경 요구사항 |
| 2 환경 | `env-provisioner` | 빌드·실행·테스트 가능한 상태 확보(런타임·의존성·외부 서비스·헬스체크) — **독립 1급 단계** |
| 3 구현 | `be-implementer` | API 핸들러·서비스 로직·DB 스키마/마이그레이션 구현(계약 준수, 작은 단위 전진) |
| 4 검증 | `be-verifier` | 실행 기반 PASS/FAIL — 빌드·테스트 직접 재실행, 고커버리지·reward-hacking 가드 |

각 에이전트 정의는 `../../agents/{name}.md`에 있다. **모든 Agent 호출은 `model: "opus"`를 명시한다** — 설계·검증 추론 품질이 거짓 PASS 여부를 좌우한다.

테스트가 더 필요하면 Phase 3~4 사이에 같은 플러그인의 [test-generator](../test-generator/SKILL.md) 스킬로 실행기반 테스트를 생성·수리할 수 있다(별개 스킬, 본 오케스트레이터가 강제하지 않음).

# 인터랙티브 플로우

## Phase 0 — 범위/계약 입력 · 승인 게이트

사용자에게서 다음을 모은다(없으면 한 번에 질문):
- 구현할 기능 범위(엔드포인트·유스케이스).
- 주어진 API 계약·데이터 요건(있으면 — OpenAPI/스키마/문서). 없으면 Phase 1에서 be-architect가 설계한다.
- 런타임·프레임워크·DB 등 기술 제약, 대상 코드베이스(신규/기존).

`[Phase 0] 범위·계약·제약 확정 — 다음: 설계(be-architect). 진행할까요?` 로 승인을 받는다.

## Phase 1 — 설계 (be-architect) · 승인 게이트

```
Agent(
  subagent_type="be-architect", model="opus",
  prompt="""
  [역할] 범위/계약을 구현 가능·검증 가능한 설계로 변환한다.
  [입력] 범위/계약/제약: {Phase 0 입력}, 기존 코드/스키마: {있으면}
  [규칙] 서비스 경계 → API 계약(엔드포인트·요청/응답 스키마·상태코드·에러·인증, 가능하면 OpenAPI/JSON Schema) →
         데이터 모델(엔티티·인덱스·마이그레이션 전략·불변식)을 설계하라. 각 결정에 *검증 후크*(인수 조건/계약 테스트)를 붙이고,
         이 설계가 요구하는 런타임·외부 의존성·환경변수를 'env-provisioner 요구사항'으로 목록화하라. 코드는 작성하지 마라.
  [출력] Design Spec(서비스 경계·API 계약·데이터 모델·환경 요구사항·미해결/가정).
  """
)
```

Design Spec을 미리보기로 보여주고: `[Phase 1] 서비스 경계·API 계약·데이터 모델 설계 완료 — 다음: 환경 구성(env-provisioner). 진행할까요?`

## Phase 2 — 환경 구성 (env-provisioner) · 승인 게이트 · **독립 1급 단계**

> 구현 전에 분리하는 이유: 실행 가능한 환경 구축이 BE 작업의 최대 병목이며(arXiv:2512.06915)[GOLD], 환경이 안 서면 이후 구현·검증이 모두 거짓 신호가 된다.

```
Agent(
  subagent_type="env-provisioner", model="opus",
  prompt="""
  [역할] 설계가 요구하는 환경에서 빌드·실행·테스트가 *실제로 가능한 상태*를 확보한다.
  [입력] 환경 요구사항: {Design Spec의 env 요구}, 기존 빌드/실행 설정: {있으면}
  [규칙] 런타임·의존성 설치, 외부 서비스(DB·브로커 등) 기동, 마이그레이션·시드 적용 가능 여부, 환경변수/시크릿 배선(값 아님),
         빌드·헬스체크·스모크 테스트를 *실제로 돌려* 통과를 증명하라. 버전을 고정하고 가능하면 환경을 코드화/컨테이너화하라.
         '준비됨'은 자기보고가 아니라 실제 통과 출력(명령·종료코드)으로만 선언하라. 기능 코드는 만들지 마라.
  [출력] Environment Readiness(준비 절차 + 실제 통과 증거 + 미해결 병목).
  """
)
```

`[Phase 2] 빌드·실행 가능 상태 확보(실제 통과 증거 동반) — 다음: 구현(be-implementer). 진행할까요?` 환경 구축이 막히면(BLOCKED) 구현으로 넘어가지 말고 범위·설계로 되먹임한다.

## Phase 3 — 구현 (be-implementer) · 승인 게이트

```
Agent(
  subagent_type="be-implementer", model="opus",
  prompt="""
  [역할] 설계와 준비된 환경 위에서 API·서비스 로직·DB 스키마/마이그레이션을 구현한다.
  [입력] Design Spec: {Phase 1}, Environment Readiness: {Phase 2}
  [규칙] API 계약·데이터 모델을 그대로 따르고, 엔드포인트/마이그레이션 단위로 쪼개 각 단계가 빌드·기동되는지 로컬에서 확인하라.
         통과를 위해 테스트를 약화/스킵/우회하지 마라(reward hacking). 최종 PASS는 선언하지 말고(be-verifier가 독립 판정),
         로컬 확인 증거(명령·종료코드)만 남겨라. 환경이 부족하면 직접 갈아엎지 말고 env-provisioner 되먹임으로 남겨라.
  [출력] Implementation 리포트(변경 요약·설계 계약 대비·로컬 확인 증거·환경 되먹임).
  """
)
```

`[Phase 3] API·스키마/마이그레이션 구현 완료(로컬 확인 증거 동반) — 다음: 실행 검증(be-verifier). 진행할까요?`

## Phase 4 — 실행 검증 (be-verifier) · 승인 게이트 · **핵심 게이트**

```
Agent(
  subagent_type="be-verifier", model="opus",
  prompt="""
  [역할] 구현이 설계 계약을 *실제로* 충족하는지 빌드·테스트를 직접 재실행해 판정한다(자기보고 불신).
  [입력] Design Spec: {Phase 1}, Environment Readiness: {Phase 2}, Implementation 리포트 + diff: {Phase 3}
  [규칙] be-implementer의 결과를 신뢰하지 말고 빌드·마이그레이션·테스트를 직접 다시 실행하라. PASS는 실제 통과 출력으로만.
         계약을 인수 조건으로 엔드포인트/스키마별 개별 판정(하나라도 미충족이면 FAIL). 고커버리지(계약·에러·경계 경로)를 게이트하라.
         reward hacking(테스트 스킵/삭제/약화, assert 제거, 환경 분기 우회, 마이그레이션 우회)을 diff로 적대적으로 점검하라.
         환경 문제로 검증 불가면 FAIL이 아니라 BLOCKED로 분리해 env-provisioner 보정을 요청하라. 코드는 고치지 마라.
  [출력] Verdict(빌드·마이그레이션·테스트·커버리지·계약 인수·reward-hacking 점검 + 종합 PASS/FAIL/BLOCKED).
  """
)
```

분기:
- **PASS**(실제 실행 증거 동반) → 마무리.
- **FAIL** → 어느 항목/어떤 reward hacking인지에 따라 be-implementer(또는 설계가 틀렸으면 Phase 1, 환경이면 Phase 2)로 되돌린다. 같은 표적이 반복되면 사용자에게 구조 변경(설계 재정의·범위 축소)을 권고한다.
- **BLOCKED** → env-provisioner 보정 게이트로(거짓 FAIL로 카운트하지 않는다).

`[Phase 4] {PASS|FAIL|BLOCKED}: {빌드·테스트·커버리지·계약 증거 요약} — {마무리 | 되돌릴 Phase}. 진행할까요?`

## 마무리 — 결과 보고

- **결과**: PASS(빌드·테스트 실제 통과·계약 충족·reward-hacking 없음) / 미완(되돌린 Phase·사유).
- **산출물**: 설계(API 계약·데이터 모델)·환경 준비 절차·구현 변경·검증 증거(명령·종료코드·커버리지) 경로.
- **남은 위험**: 미충족 계약·미측정 커버리지·flaky·환경 병목.

최종 보고 형식: `[Backend 종료] {PASS|미완:되돌린 Phase} — 엔드포인트 {구현/검증 수}, 테스트 {통과/전체}, 커버리지 {수치 또는 미측정}.`
