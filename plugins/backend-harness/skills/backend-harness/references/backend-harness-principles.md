# Backend Harness — 원리 · 실행 검증 · 환경 1급 시민 · anti-pattern

> 이 문서는 backend-harness 하네스가 따르는 설계 원리의 근거다. 오케스트레이터(SKILL.md)와 각 에이전트가 참조한다.
> 정량 수치는 [backend-harness-research.md](./backend-harness-research.md)에 출처·등급·CAVEAT와 함께 둔다.

## 1. 무엇이 backend-harness인가

backend-harness는 백엔드/API를 **범위/계약 → 설계 → 환경 구성 → 구현 → 실행 검증**으로 끌고 가는 멀티 에이전트
오케스트레이션이다. 단발 자동완성이 아니라, 저장소 단위 BE 작업이 SOTA 모델로도 절반가량 실패하는 난도라는
관측(§research)에 대응해 *구조화 + 실행 검증*으로 신뢰도를 확보한다.

두 차별점:
1. **실행 기반 검증(자기보고 불신)** — 통과는 "에이전트가 통과했다고 말함"이 아니라 *빌드·테스트가 실제로 통과*.
2. **환경 구성을 1급 시민으로** — 환경 구축이 최대 병목이므로 구현에 묻지 않고 독립 Phase로 분리.

## 2. 6원칙

1. **실행 기반 검증(execution-based)** — PASS는 자기보고·commit·"됐다는 말"이 아니라 빌드 성공 + 마이그레이션 적용 + 테스트 실제 통과의 raw 출력으로만. 증거(명령·종료코드·카운트) 없는 PASS는 무효다.
2. **환경 우선(environment-first, 1급 시민)** — 환경이 안 서면 이후 구현·검증 신호가 모두 거짓이 된다. 구현 전에 빌드·실행·테스트 가능 상태를 *독립 Phase*로 확보하고, 준비 완료는 헬스체크·스모크의 실제 통과로만 선언한다.
3. **generator/checker 분리** — 구현(be-implementer)과 검증(be-verifier)을 다른 에이전트로 분리한다. 같은 컨텍스트에서 자기 산출물을 평가하면 자기 작업을 칭찬하는 편향이 생긴다. 검증자는 코드를 고치지 않는다(고치면 다음 검증의 독립성이 깨진다).
4. **reward-hacking 가드** — 통과를 위해 테스트를 스킵/삭제/약화(assert 제거·`skip`/`xfail`·환경 분기 우회)했는지, 목/스텁만 검증하는지, 마이그레이션을 우회했는지 diff로 적대적으로 점검한다. 고커버리지(계약·에러·경계 경로)로 게이트한다.
5. **계약 준수(contract-fidelity)** — be-architect의 API 계약·데이터 모델을 구현으로 임의 우회하지 않는다. 못 맞추면 우회하지 말고 설계로 되먹임한다.
6. **승인 게이트·최소 사이드이펙트** — 매 Phase 산출물 미리보기 후 1줄 보고로 승인을 받아 진행한다. 요청되지 않은 사이드 에이전트나 중복 실행을 만들지 않는다.

## 3. 왜 env-provisioner가 독립 Phase인가

환경 구성을 구현에 섞으면 두 가지가 망가진다.
- **거짓 신호**: 환경이 절반만 선 채 구현하면 "빌드 실패"가 코드 결함인지 환경 결함인지 구분되지 않아, be-verifier의 FAIL이 오염된다.
- **병목 은폐**: BE 작업에서 *실행 가능한 환경을 만드는 것 자체가 최대 병목*인데(F2P ≤37.7%, §research), 이를 구현 단계에 숨기면 가장 비싼 실패가 가장 늦게 터진다.

그래서 환경을 1급 Phase로 끌어내 ① 재현 가능한 절차로 코드화하고 ② 실제 통과 증거로 닫고 ③ 막히면 즉시 범위·설계로 되먹인다. be-verifier가 환경 문제를 만나면 FAIL이 아니라 **BLOCKED**로 분리해 이 Phase로 되돌린다(거짓 FAIL 방지).

## 4. 왜 자기보고를 믿지 않는가

에이전트의 자기평가는 과신한다 — commit/성공 주장 비율이 실제 통과율(F2P)보다 높다(§research). 따라서:
- be-implementer는 로컬 확인 증거만 남기고 *최종 PASS를 선언하지 않는다*.
- be-verifier는 그 결과를 신뢰하지 않고 *직접 빌드·테스트를 다시 실행*해 판정한다(독립 재현).
- 통과는 실제 실행 출력으로만, 커버리지는 게이트로 — 그래야 "그린이라 주장하지만 실제로는 빨강"이거나 "테스트를 무력화해 초록"인 reward hacking을 막는다.

## 5. anti-pattern (이렇게 하지 마라)

- **자기보고를 통과로 반올림** — "테스트 통과했습니다"를 증거 없이 PASS로 인정. → 실제 실행 출력 요구.
- **환경을 구현에 묻기** — 환경 셋업을 구현 중에 임기응변. → 독립 Phase + 헬스체크로 닫기.
- **그린을 위해 테스트 약화** — assert 제거·skip·환경 분기로 우회해 초록 만들기. → reward-hacking 가드 + 커버리지 게이트.
- **커버리지 없는 그린을 통과로** — 행복 경로만 덮은 테스트. → 계약·에러·경계 경로 커버 요구.
- **계약을 구현으로 우회** — 설계가 불편하면 코드로 임의 변경. → 설계 게이트로 되먹임.
- **flaky를 통과로 반올림** — 가끔 초록을 PASS로. → 재실행 재현성 확인, 비결정성을 결함으로 보고.
- **마이그레이션 미검증** — 적용/롤백을 실제로 안 돌려보고 통과. → 마이그레이션 적용·롤백 실제 실행.

## 6. 범위 경계 (이 플러그인이 하지 않는 것)

- **완성 spec *검수*·breaking change 점검** — 범위 밖. backend-harness는 계약을 받아 *구현*한다.
- **FE 화면·컴포넌트** — 범위 밖. backend-harness는 서버/API.
- **PRD·사용자 스토리 작성** — 범위 밖. backend-harness는 코드 구현.
- **커밋/PR 리뷰** — 범위 밖. 커밋/PR이 필요하면 별도 커밋 워크플로를 쓴다.
- **하네스 자체 진단·개선** — 범위 밖.
- **검증 가능한 목표를 향한 *단발 자율 반복 루프*** — 범위 밖. backend-harness는 *BE 구현 워크플로 오케스트레이션*(설계→환경→구현→검증)이다. be-verifier의 generator/checker 분리·증거 없는 PASS 금지 원칙은 일반적인 독립 검증자 원칙과 정합하나 어떤 외부 워크플로에도 *의존하지 않는다*.
- **test-generator(같은 플러그인)** — 전체 구현 워크플로가 아니라 기존 코드의 실행기반 테스트 생성·수리만 필요할 때 직접 쓰는 스킬. test-first(Red-Green-Refactor) TDD와 구분(이 스킬은 test-after).
