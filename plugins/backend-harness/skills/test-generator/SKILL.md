# Test Generator — 실행기반 테스트 생성 + 경험적 수리 루프

기존 코드(주로 백엔드·통합/단위)에 **실제로 컴파일되고 실행되며 의미 있는 커버리지를 내는** 테스트를 만든다.
raw LLM이 생성한 테스트는 환각으로 컴파일/런타임 오류가 잦고 커버리지가 낮다는 관측에 대응해, 자유형 재프롬프트가
아니라 **오류 유형별 결정론적 수리 템플릿**과 **generate→compile→execute→repair 공진화 루프**로 신뢰도를 확보한다.

## 근거 (등급·CAVEAT 동반)

- raw LLM 테스트는 컴파일/런타임 오류·낮은 커버리지 → **5개 경험적 수리 템플릿**(컴파일 ~50%/런타임 ~75% 수리)과 **generate→compile→execute→repair 공진화 루프**로 보완, baseline(ChatGPT/EvoSuite/A3Test) 대비 pass +18%/coverage +20% 보고. 출처 TestART, arXiv:2408.03095 (ACM TOSEM 2025) **[SILVER]**.
> **CAVEAT(필수)**: 2024-08 데이터라 최신 모델엔 수치가 안 맞을 수 있다. **효과크기를 단정하지 말 것** — 차용하는 것은 *패턴*(루프 + 결정론적 수리 템플릿 + 커버리지 피드백 반복)이지 +18%/+20% 같은 수치 약속이 아니다.

자세한 수리 템플릿·루프 운용은 [references/test-generator-guide.md](./references/test-generator-guide.md), 하네스 차원 근거는 [../backend-harness/references/backend-harness-research.md](../backend-harness/references/backend-harness-research.md) 참조.

## 경계 (먼저 읽고 발동 여부를 판단하라)

- **test-first(Red-Green-Refactor) TDD와 구분** — TDD는 *테스트를 먼저 쓰고 그 테스트를 통과하도록 코드를 만든다*(test-first). 이 스킬은 반대로 *이미 있는 코드*에 실행기반 테스트를 생성·수리해 커버리지를 확보한다(test-after, 주로 BE/통합·단위). 새 기능을 test-first로 만들 거면 이 스킬이 아니라 TDD 워크플로를 쓴다.
- **전체 BE 구현 워크플로가 필요하면** — 설계→환경→구현→검증은 같은 플러그인의 [backend-harness](../backend-harness/SKILL.md) 오케스트레이터다. 이 스킬은 그중 *테스트 생성·수리*만 떼어 쓰는 model-invocable 스킬이다.
- **검증 가능한 목표를 향한 단발 자율 반복 루프** — 범위 밖. 이 스킬의 루프 토폴로지는 그런 자율 반복 루프와 정합하나(아래) 어떤 외부 워크플로에도 의존하지 않는다.
- **완성 spec 검수**, **커밋/PR**은 범위 밖.

## 공진화 루프 (4단계)

```
① generate  — 대상 코드(시그니처·동작·계약)를 읽고 단위/통합 테스트 초안을 생성
② compile   — 실제로 컴파일/로드한다. 실패하면 오류 유형 분류 → 결정론적 '수리 템플릿' 적용 (자유형 재프롬프트 금지)
③ execute   — 실제로 실행한다. 런타임 실패(픽스처·의존성·단언 오류) → 유형별 수리 템플릿 적용
④ repair    — ②③의 수리를 반영해 재시도. 커버리지 피드백(미커버 경로)으로 ①로 되돌아가 테스트 보강
```

- 통과(컴파일·실행 그린) + 목표 커버리지 도달 / 최대 반복 / 무진전(같은 오류 반복) 시 멈춘다(무한 루프 방지).
- **수리는 자유형 재프롬프트가 아니라 결정론적 템플릿으로** — 같은 오류에 같은 수리를 적용해 재현성을 확보한다(가이드 §수리 템플릿).

## 내재화 원칙

- **실행 기반** — "테스트가 맞다"는 자기보고가 아니라 *실제 컴파일·실행 통과*로만 인정한다. 증거(컴파일 출력·실행 결과·커버리지 수치) 없는 완료 금지.
- **결정론적 수리** — 컴파일/런타임 오류는 유형별 수리 템플릿으로 고친다(자유형 재프롬프트는 비결정·환각 재발 위험).
- **커버리지 피드백 반복** — 단순 그린이 아니라 계약·에러·경계 경로가 덮였는지 커버리지로 게이트하고, 미커버 경로를 다음 반복의 표적으로 삼는다.
- **테스트를 코드에 맞춰 무력화하지 않는다** — 코드 버그를 숨기려고 단언을 약화/삭제하지 않는다(이는 backend-harness/be-verifier의 reward-hacking 가드 대상). 코드 결함으로 의심되면 단언을 약화하지 말고 *그 사실을 보고*한다.
- **judge 캘리브레이션 인지** — 자동 판정(테스트 적절성에 대한 LLM 판단)은 *보조 신호*다. 통과/실패는 실제 실행으로, 의미적 적절성은 **사람 검토로** 닫는다(자동 judge 단독 승인 금지).

## 자율 반복 루프와의 정합 (설계 원칙)

이 4단계 루프는 일반적인 목표 지향 실행 루프(Execute→Verify→Diagnose→Improve)와 토폴로지가 정합한다 —
generate=Execute, compile/execute=Verify, 오류 유형 분류=Diagnose, 수리 템플릿=Improve. 검증 가능한 목표를 향해
*자율 반복*으로 끌고 가고 싶을 때도 같은 토폴로지를 그대로 쓸 수 있다.
다만 이 스킬은 그 자체로 동작하는 자족 스킬이며 어떤 외부 워크플로에도 의존하지 않는다.

## 산출

- 생성·수리된 테스트 파일 경로.
- 실행 증거: 컴파일 결과·실행 통과/실패 카운트·커버리지 수치(미측정이면 명시).
- 사람 검토 요청: 자동으로 확신 못 하는 의미적 적절성·코드 버그 의심 지점.
- 1줄 보고: `[test-generator] 테스트 {생성/통과 수}, 커버리지 {수치 또는 미측정}, 수리 {적용 템플릿 수} — 사람 검토 필요: {N건}.`
