# Test Generator — 운용 가이드 (공진화 루프 · 경험적 수리 템플릿)

> 이 문서는 `test-generator` 스킬이 따르는 실행 기준이다. SKILL.md가 참조한다.
> 정량 효과크기는 단정하지 않는다(아래 근거의 CAVEAT 참조) — 차용하는 것은 *패턴*이다.

## 1. 왜 raw LLM 테스트로는 부족한가

raw LLM이 생성한 테스트는 환각으로 ① 존재하지 않는 심볼/시그니처를 호출해 **컴파일 오류**가 나거나, ② 픽스처·의존성
누락으로 **런타임 오류**가 나거나, ③ 행복 경로만 덮어 **커버리지가 낮다**. 자유형 재프롬프트("고쳐줘")로 다시 시키면
비결정적으로 또 다른 환각을 만들기 쉽다. 그래서 *오류 유형별 결정론적 수리 + 커버리지 피드백 반복*이 필요하다.

> 근거: TestART(arXiv:2408.03095, ACM TOSEM 2025) **[SILVER]** — 5개 경험적 수리 템플릿(컴파일 ~50%/런타임 ~75% 수리),
> generate→compile→execute→repair 공진화 루프, baseline 대비 pass +18%/coverage +20% 보고.
> **CAVEAT(필수)**: 2024-08 데이터라 최신 모델엔 수치가 안 맞을 수 있다 → 효과크기 단정 금지, 패턴(루프·템플릿)만 차용.

## 2. 공진화 루프 (generate → compile → execute → repair)

```
① generate
   - 대상 코드의 시그니처·동작·계약(입출력·에러·상태)을 읽는다.
   - 단위 테스트(순수 로직) + 통합 테스트(DB·외부 의존성 경계)를 구분해 초안 생성.
   - 행복 경로만이 아니라 에러·경계 케이스를 명시적으로 포함한다.

② compile  (실제로 컴파일/로드 — 자기보고 아님)
   - 실패 시 오류를 §3 유형으로 분류 → 해당 결정론적 수리 템플릿 적용.
   - 자유형 "다시 생성"으로 도망치지 않는다(환각 재발).

③ execute  (실제로 실행)
   - 런타임 실패(픽스처·의존성·단언)를 §3 유형으로 분류 → 수리 템플릿 적용.
   - flaky(비결정)면 재실행으로 재현성을 확인하고 비결정성 자체를 결함으로 보고.

④ repair → 재시도 + 커버리지 피드백
   - ②③의 수리를 반영해 재실행.
   - 커버리지 리포트에서 미커버 경로(특히 에러·분기)를 다음 반복의 ① 표적으로 되돌린다.
```

**중단조건(무한 루프 방지)**: 컴파일·실행 그린 + 목표 커버리지 도달 / 최대 반복 도달 / 무진전(같은 오류·미커버가 임계 회 반복). 무진전이면 자동 재시도를 멈추고 사람 검토로 에스컬레이트한다.

## 3. 경험적 수리 템플릿 (오류 유형 → 결정론적 수리)

자유형 재프롬프트 대신, 오류를 유형으로 분류해 같은 수리를 적용한다(재현성).

| # | 오류 유형 | 전형 증상 | 결정론적 수리 |
|---|-----------|-----------|---------------|
| T1 | 심볼/시그니처 불일치 (compile) | 없는 메서드·클래스 호출, 인자 수/타입 불일치 | 대상 코드의 실제 시그니처를 다시 읽어 호출부를 그에 맞춰 정정(추측 금지) |
| T2 | import/의존성 누락 (compile) | 미해결 import, 모듈 not found | 실제 패키지 경로로 import 정정, 테스트 의존성 부재면 env 보강 요청 |
| T3 | 픽스처/셋업 누락 (runtime) | NPE/None·연결 거부·초기화 안 됨 | 필요한 픽스처·셋업/티어다운·DB 연결·시드를 명시적으로 추가 |
| T4 | 단언 불일치 (runtime) | 기대값이 실제 계약과 어긋남 | *코드 동작이 옳으면* 기대값을 실제 계약에 맞춰 정정; *코드 버그 의심*이면 단언을 약화하지 말고 사람 검토로 보고 |
| T5 | 환경/격리 문제 (runtime) | 외부 호출·시간·랜덤·전역 상태 의존 | 경계를 목/스텁/고정 시드로 격리(단, 핵심 동작은 목으로 대체하지 말고 실제 검증 유지) |

> T4·T5의 핵심: 테스트를 "초록"으로 만들려고 *코드 버그를 숨기거나 핵심 동작을 무력화*하지 않는다. 이는 backend-harness/be-verifier의 reward-hacking 가드 대상이며, 의심되면 단언을 약화하는 대신 **그 사실을 사람 검토로 보고**한다.

## 4. 커버리지 게이트

- **단순 그린 ≠ 충분** — 계약 경로·에러 경로·경계 케이스가 덮였는지 커버리지로 게이트한다.
- 미커버 경로(특히 분기·예외)를 다음 반복의 생성 표적으로 명시한다.
- 커버리지 도구가 없으면 미측정을 PASS로 간주하지 않고, 측정 가능 환경을 (backend-harness 맥락이면 env-provisioner에) 요청하거나 미측정을 명시적 위험으로 보고한다.

## 5. judge 캘리브레이션 — 자동 판정은 보조, 사람 검토 필수

- 테스트의 *의미적 적절성*(이 테스트가 정말 의도한 계약을 검증하는가)에 대한 자동 LLM 판정은 캘리브레이션이 흔들릴 수 있다 → **보조 신호로만** 쓴다.
- **기계검증 가능한 신호 우선**: 컴파일 결과·실행 통과/실패·커버리지 수치는 실제 실행으로 확정한다.
- **사람 검토로 닫기**: 자동 judge 단독으로 "완료"를 승인하지 않는다. 의미적 적절성·코드 버그 의심 지점은 사람 검토 요청으로 surface한다.

## 6. 경계 재확인

- **test-first(Red-Green-Refactor) TDD** = 테스트를 먼저 쓰고 코드를 만든다. 이 스킬 = test-after: *기존 코드*에 실행기반 테스트를 생성·수리한다(주로 BE/통합·단위).
- **backend-harness 오케스트레이터(같은 플러그인)** = 설계→환경→구현→검증 전체. 이 스킬은 그중 테스트 생성·수리만 떼어 쓰는 스킬.
- **검증 가능한 목표를 향한 단발 자율 반복 루프** = 범위 밖. 토폴로지는 정합하나 어떤 외부 워크플로에도 비의존(자족 스킬).
