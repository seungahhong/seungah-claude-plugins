# ai-readable-codebase

코드베이스가 **AI 코딩 에이전트가 읽고 안전하게 기여할 수 있는 구조**인지를 진단하고 개선하는 도메인 무관 멀티 에이전트 하네스입니다.

핵심 전제는 두 가지입니다.

1. **구조가 프롬프트보다 먼저다(structure before prompt)** — 팀이 뒤처지는 진짜 이유는 프롬프트 엔지니어링 실력이 아니라 *에이전트가 기여할 수 있는 코드베이스 구조를 갖추지 못해서*입니다.
2. **코드 품질(Q축)과 AI 접근성(A축)은 다른 차원이다** — 사람에게 깨끗한 코드라도 패턴이 들쭉날쭉하고 빌드 피드백이 약하며 모듈 경계가 모호하면 에이전트에게는 적대적일 수 있습니다.

> 이 하네스는 *애플리케이션 코드베이스 자체*의 구조적 AI 접근성을 진단·개선합니다.
> 한 기능을 구현·검증하거나(backend), 상류 산출물을 핸드오프 게이트로 검수하거나(review), 하네스 자체를 진단(meta)하는 것은 범위 밖입니다.

## 왜 "AI 접근성(A축)"인가

빌드 실패는 즉각적이고 모호하지 않은 신호입니다. 반면 자연어 가이드(CLAUDE.md)는 세 가지 이유로 단독으로는 실패합니다.

- **해석 모호** — 경계가 회색지대로 남습니다(예: Jackson `@JsonProperty`가 "인프라" 관심사인가?).
- **컨텍스트 경쟁** — 상세한 가이드일수록 생성 시점에 코드와 어텐션을 다투다 후순위로 밀립니다.
- **피드백 부재** — 위반이 사람 PR 리뷰 전까지 조용히 누적됩니다.

그래서 이 하네스는 **빌드가 강제하고(무엇이 불가능한지) 문서가 설명한다(무엇이 바람직한지/왜)**를 원칙으로, 가장 중요한 아키텍처 규칙을 *가장 빠른 피드백 계층(컴파일)*에서 잡도록 설계합니다.

## 설치

이 저장소를 Claude Code 플러그인 마켓플레이스로 추가한 뒤 `ai-readable-codebase` 플러그인을 활성화하면,
`ai-readable-codebase` 스킬이 자동 트리거되거나 직접 호출할 수 있습니다.

## 스킬

| 스킬 | 역할 |
|------|------|
| `ai-readable-codebase` | 오케스트레이터(진입점). 진단(Assess) → 구조·가드레일(Guardrails) → 독립 실행(Standalone) → 수용 증명·재측정(Acceptance & Re-grade)의 4단계를 진행하며, 각 단계에서 전용 에이전트를 호출한다. |

## 에이전트 팀 (모두 `model: opus`)

| Phase | 에이전트 | 역할 |
|-------|----------|------|
| 0 Assess | `accessibility-assessor` | 2축(Q/A)으로 진단하고 L1~L5 AI 접근성 등급을 *증거 기반*으로 매긴 뒤 A축 격차·목표 등급·우선순위 백로그를 낸다 |
| 1 Guardrails | `guardrail-architect` | 핵심 아키텍처 규칙을 빌드/컴파일 계층으로 옮겨 의존 방향을 물리적으로 강제하고, 빌드 피드백을 3차원으로 '가르치게' 설계하며 빌드 vs CLAUDE.md 역할을 분담한다 |
| 2 Standalone | `standalone-designer` | 도메인 슬라이스를 전체 시스템 없이 독립 실행 가능하게(port/adapter 치환·use-case 경유 seed) 설계한다 |
| 3 Acceptance & Re-grade | `acceptance-verifier` | 수용 증명(자동 E2E) 인프라를 설계하고, 구조가 실제로 바뀌어 등급이 올랐는지 적대적으로 재측정한다(reward-hacking 가드) |

## L1~L5 AI 접근성 등급

| 등급 | 이름 | 특징(요약) |
|------|------|------------|
| L1 | AI-Hostile | 안정적 빌드·모듈 구조·테스트 부재. 에이전트가 깨진 코드를 양산하고 피드백 루프가 없다. |
| L2 | AI-Tolerant | 빌드는 있으나 가드레일 약함. 단순 변경은 되지만 모듈 횡단·신규 결정에서 실패. |
| L3 | AI-Guided | 일관된 빌드/CI·예측 가능한 모듈·적정 테스트. 가이드 따르기는 되나 경계 횡단은 사람 개입. |
| L4 | AI-Partnered | 빌드가 아키텍처를 물리적으로 강제·standalone·수용 증명 인프라. 에이전트가 자율 생성·검증·PR. |
| L5 | AI-Native | 에이전트 워크플로가 빌드에 네이티브 통합·자동 수용 증명·에이전트 간 위임. (현재는 지향점) |

## 5단계 사용법

1. **대상 지정** — 진단할 코드베이스(또는 서비스/모듈)를 지정합니다. 모호하면 `accessibility-assessor`가 진단에 필요한 최소 질문을 합니다.
2. **진단 승인 게이트** — Phase 0에서 2축 진단·L1~L5 등급(증거 인용)·A축 격차·목표 등급·우선순위 백로그를 받아 **검토·승인**합니다. 승인 전에는 개선 설계를 시작하지 않습니다.
3. **빌드 가드레일 설계** — Phase 1에서 어떤 규칙을 빌드 계층에서 물리적으로 강제하고, 무엇을 CLAUDE.md로 설명할지, 빌드 피드백 메시지를 어떻게 '가르치게' 바꿀지 설계합니다.
4. **독립 실행 설계** — Phase 2에서 도메인 슬라이스를 전체 시스템 없이 검증 가능하게(port/adapter 치환·use-case 경유 seed) 설계합니다.
5. **수용 증명 + 등급 재측정** — Phase 3에서 자동 수용 증명 인프라를 설계하고, 구조가 *실제로* 바뀌어 등급이 올랐는지 증거로 재측정합니다(green ≠ 개선, 자기보고 불신).

## 언제 쓰나 / 언제 다른 도구를 쓰나

**이 하네스를 쓰세요**
- 코드베이스가 **AI 에이전트가 읽고 기여하기 쉬운 구조인지 진단·등급(L1~L5)**을 매기고 싶을 때
- **Q축(코드 품질)은 괜찮은데 A축(AI 접근성)이 낮은** 것 같아 그 격차를 찾고 싶을 때
- 아키텍처 규칙을 **빌드/컴파일 계층에서 물리적으로 강제**하고 빌드 피드백을 에이전트가 학습하게 설계하고 싶을 때
- 도메인 슬라이스를 **전체 시스템 없이 독립 실행**하게 만들고 싶을 때
- AI가 만든 PR에 **자동 수용 증명**을 붙여 리뷰를 설계 중심으로 옮기고 싶을 때

**이 하네스를 쓰지 마세요 (범위 밖)**
- 한 **기능을 실행 기반으로 구현·검증** — 그건 백엔드 구현 도메인입니다(이 하네스는 *구조 진단·개선*이지 기능 구현이 아님)
- 코드 착수 *전* **상류 산출물(기획·디자인·계약·AC) 핸드오프 게이트 검수**
- **하네스(CLAUDE.md/SKILL.md/agents) 자체**를 trace로 진단·개선
- 커밋→프로덕션 **전달 파이프라인(CI/CD·릴리스·IaC)** 설계
- 에이전트가 구현할 **실행 가능 명세(spec)** 작성
- 모델에 넣을 **컨텍스트 페이로드 조립·압축·관리**
- 이미 **완성된 코드의 리뷰·커밋 메시지·PR 리뷰**

경계가 모호하면 한 질문으로 확인합니다 — "*코드베이스 자체*의 구조적 AI 접근성을 진단·개선하려는 건가요, 아니면 *기능 구현/상류 검수/하네스 진단/전달 파이프라인*인가요?"

## 진행 방식

| Phase | 단계 | 핵심 산출물 |
|-------|------|-------------|
| 0 | 진단 (Assess) | 2축(Q/A) 진단 + L1~L5 등급(증거) + A축 격차 + 목표 + 백로그 — **진단 승인 게이트** |
| 1 | 구조·가드레일 (Guardrails) | 빌드 가드레일 설계(의존 방향 물리 강제 + 피드백 3차원 + 빌드/문서 분담) |
| 2 | 독립 실행 (Standalone) | 도메인 슬라이스 standalone 설계 + 독립 검증 커버리지 |
| 3 | 수용 증명·재측정 (Acceptance) | 수용 증명 인프라 설계 + 등급 재측정 Verdict(증거·한계·reward-hacking 점검) |
