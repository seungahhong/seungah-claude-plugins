# AI-Readable Codebase — 원리 · 2축/L1~L5 · 빌드 가드레일 · standalone · 수용 증명 · anti-pattern

> 이 문서는 ai-readable-codebase 하네스가 따르는 설계 원리의 근거다. 오케스트레이터(SKILL.md)와 각 에이전트가 참조한다.
> 1차 출처는 flex.team "AI가 읽을 수 있는 코드베이스" 5부작이며, 출처·인용·신뢰도(vote)·CAVEAT는
> [ai-readable-codebase-research.md](./ai-readable-codebase-research.md)에 정리되어 있다.

## 1. 무엇이 'AI 읽힘성(AI accessibility)'인가

핵심 명제: **코드 품질(Q축)과 AI 접근성(A축)은 다른 차원이다.** "코드 품질과 AI 접근성은 같은 것이 아니다"(flex 5/5).
사람에게 깨끗한 코드라도 패턴이 들쭉날쭉하고 빌드 피드백이 약하며 모듈 경계가 모호하면 에이전트에게는 적대적일 수 있다.

그리고 **"구조가 프롬프트보다 먼저다(structure before prompt)"**(flex 1/5). 팀이 뒤처지는 진짜 이유는 프롬프트 실력이 아니라
*에이전트가 기여할 수 있는 코드베이스 구조를 갖추지 못해서*다. 이 하네스는 그 구조적 격차를 진단·개선한다.

> 학술 보강(연구 dossier): LLM은 *의미 보존* 구문 변형만으로도 코드 이해가 최대 70% 흔들리고(arXiv:2505.10443, 3-0),
> 가장 깊은 구조적 추론을 요구하는 변형(loop unrolling)에서 가장 크게 무너진다(同, Qwen2.5-Coder −44.3pt). 즉 *구조·일관성*은
> AI가 코드를 얼마나 안정적으로 이해하는지에 실제로 영향을 준다. (단, "제어구조 변형에 가장 민감"이라는 더 강한 주장은 0-3으로 반박됨 — dossier 참조.)

## 2. 두 축 (Q-axis / A-axis)

- **Q축(코드 품질)** — 테스트 커버리지·안정성, 순환/인지 복잡도, 중복, 문서(API·주석·README), 의존성 최신성·보안, 아키텍처 준수(SOLID·관심사 분리). 기존 도구(SonarQube·CodeClimate 등)가 보는 차원.
- **A축(AI 접근성) 7개 지표**:
  1. **패턴 일관성** — 같은 문제를 같은 방식으로. 비일관 패턴은 에이전트가 임의 변형을 양산하게 한다.
  2. **빌드 피드백 품질** — 나쁜 코드에 즉각·명확한 에러. 기호적 피드백(컴파일러 출력)이 모호한 메시지보다 우월.
  3. **모듈 경계 예측성** — "새 코드를 어디에 둘지" 추론 가능. LLM은 외부 제약 없이 모듈 원칙을 내재화하지 못한다.
  4. **의존 방향 강제** — 금지 의존이 *빌드에서 물리적으로* 막힘(린트 규칙이 아니라).
  5. **독립 실행 가능성** — 부분을 격리 검증. 계산적 통제(테스트·린터·타입체커)가 추론적 통제보다 낫다.
  6. **에이전트 가이드** — CLAUDE.md가 빌드 규칙과 *동기화*돼 있는지.
  7. **자동 수용** — 변경의 자동 증명 가능성(건강한 데이터 생태계·접근 가능한 내부 시스템 필요).

> 핵심: **높은 Q축이 높은 A축을 보장하지 않는다.** flex 내부 평가에서 Q축이 같은(≈4.0) 두 서비스가 A축에서 1.4점 갈렸고,
> 차이는 오직 "의존 위반이 빌드 실패를 일으키는가(Convention Plugin) vs 통과하는가(ESLint)"였다(flex 5/5).

## 3. L1~L5 AI 접근성 등급 (증거 기반)

| 등급 | 이름 | 특징 |
|------|------|------|
| L1 (1.0–1.9) | AI-Hostile | 안정 빌드·모듈 구조·테스트 부재. 에이전트가 빌드 통과하는 깨진 코드 양산, 피드백 루프 없음, 에러 조용히 누적. |
| L2 (2.0–2.9) | AI-Tolerant | 빌드는 있으나 가드레일 약함. 의존 위반 허용. 단순 변경 성공, 모듈 횡단·신규 결정 실패. |
| L3 (3.0–3.6) | AI-Guided | 일관 빌드/CI·예측 가능 모듈·적정 테스트·정적분석. 가이드 따르기는 되나 경계 횡단은 사람. |
| L4 (3.7–4.5) | AI-Partnered | 빌드가 아키텍처 물리 강제·명확 모듈 경계·높은 패턴 일관성·standalone·E2E 수용 증명. 에이전트 자율 생성·검증·PR, 사람은 설계 결정. |
| L5 (4.6–5.0) | AI-Native | L4 + 에이전트 워크플로 빌드 네이티브 통합·자동 E2E+데모+PR·에이전트 간 위임. 현재는 지향점. |

**등급은 증거로 매긴다(자기보고 불신).** 가장 결정적 단일 측정: *"금지된 경계 위반이 빌드 실패를 일으키는가?"* — 막히면 L4+,
컴파일만 통과하고 아키텍처 위반은 통과하면 L2~L3. 위반 케이스를 직접 만들거나 빌드 설정을 file:line으로 인용해 판정한다.

## 4. 빌드 가드레일 — "빌드가 강제하고 문서가 설명한다"

자연어 가이드(CLAUDE.md)가 단독으로 실패하는 3대 이유(flex 1/5):
1. **해석 모호** — 경계가 회색지대(예: `@JsonProperty`가 "인프라"인가?).
2. **컨텍스트 경쟁** — 상세 규칙이 생성 시점에 코드와 어텐션을 다투다 후순위로 밀림.
3. **피드백 부재** — 위반이 사람 PR 리뷰 전까지 조용히 누적.

→ **역할 분담**: 빌드 가드레일은 "무엇이 *불가능*한지"(의존 방향·모듈 구조·스타일)를, CLAUDE.md는 "무엇이 *바람직*한지/왜"
(추상화 수준·패턴 의도·도메인 용어·비즈니스 규칙)를 맡는다. 빌드로 잡을 수 있는 것을 산문에 맡기지 않고, CLAUDE.md는 빌드 규칙과 동기화한다.

### 피드백 품질 3차원 (flex 2/5)
모든 빌드 피드백을 ① **위치 특정성**(파일·줄·열) ② **원인 명확성**(왜 실패했는지) ③ **수정방향 추론가능성**(메시지만으로 해법 유추)으로 평가·개선한다.
- 나쁨: `BUILD FAILED` / `expected "ISS-0001" but was: null`
- 좋음: `e: .../Issue.kt:3:8 Unresolved reference: springframework` / `Issue code generation failed: SequenceGenerator returned null for tenant=...`

### 피드백 계층 위계 & 앞단화
컴파일 에러(가장 빠름·특정적) → 의존성 해석 → 린트/정적분석(자동 수정) → 테스트 실패(가장 넓음·느림·해석적).
검증을 *앞단*으로 옮기고(컴파일 > 런타임, 타입 강제 > 컴파일 체크), **가장 중요한 아키텍처 규칙을 가장 빠른 계층(컴파일)**에서 잡는다. 테스트 실패도 *판정*이 아니라 *피드백*으로 설계한다.

> 학술 보강: 컴파일러-인-더-루프는 실제로 정확도를 올린다 — 구조화된 오류 안내 루프가 GPT-5의 Idris 해결을 22/56→54/56로 끌어올렸고
> *로컬 컴파일 에러* 피드백이 가장 큰 개선을 냈다(arXiv:2602.11481, 3-0/2-0). 그리고 self-repair의 병목은 생성이 아니라
> *피드백의 질*이다(arXiv:2306.09896, 3-0) — flex의 "피드백 3차원" 설계와 정확히 일치한다. 단, self-repair는 만능이 아니며
> 비용을 따지면 이득이 작거나 없을 수도 있고(同, 2-1) 이득은 첫 2라운드에 몰린다(arXiv:2604.10508, 3-0) — 무한 반복을 권하지 않는다.

## 5. Standalone — 도메인 슬라이스 독립 실행 (flex 3/5)

"전체를 올리지 않고 부분을 검증하는 법". Port/Adapter 의존성 역전으로 도메인 로직은 Port에만 의존하고, *운영 어댑터를 테스트 어댑터로 치환*한다(도메인 로직 불변).
- **어댑터 치환**: 예) OAuth2 Resource Server → 헤더 기반 인증, 엔드포인트별 권한 → permitAll. 복잡성은 Filter/경계에 격리하고 Service가 받는 추상(Requester)은 동일.
- **선택 조합·명시적 제외**: 필요한 도메인만, 도메인 간 의존 최소화, 운영 전용 모듈은 전이 의존에서 명시적 제외.
- **seed는 UseCase 경유**: Repository 직접 호출 금지. seed가 모든 도메인 검증·비즈니스 규칙을 거치게 한다(데이터 정합).
- **외부 의존 컨테이너화 + 단일 명령 기동** → 자기 완결적(self-contained) 환경.
- **AI 협업 3이점**: 빠른 피드백 루프·격리된 검증·수용 증거의 토대.

> 학술 보강: 동적 실행 특성(특히 *실행 트레이스 길이*)이 정적 속성보다 LLM 코드 추론에 더 큰 영향을 주며, 20스텝 내에 성능이 30%+ 급락한다
> (arXiv:2602.13962, 3-0). 짧고 격리된 실행 경로(작은 슬라이스)가 에이전트 추론에 유리하다는 간접 근거다.

## 6. 수용 증명(Acceptance) — 리뷰를 바꾼다 (flex 4/5)

리뷰에서 가장 비싼 질문은 "이거 진짜 되나?"다. 자동 E2E·데모로 **리뷰어가 PR을 열기 전에 동작 증명이 이미 존재**하게 하면, 리뷰가
"동작하나?"에서 "설계가 좋은가?"(UseCase 책임 범위·트랜잭션 경계·모듈 의존·확장성)로 이동한다. 핵심 철학: **"문서가 아니라 빌드가 지킨다."**
- 에이전트가 PR 생성 시 빌드/E2E 실패를 스스로 분석·교정하고, 사람 개입은 *수용이 증명된 뒤*. (예: `✅ E2E 8/8 passed | Demo: recorded`)
- **한계(반드시 명시)**: 자동 수용 증명은 *성능 저하·보안 취약(인가 우회)·동시성 데이터 정합·가독성/유지보수성*을 보장하지 못한다 — 사람 몫.

> 산업 맥락(BRONZE): AI 도입으로 코드 acceptance rate가 오르고 "리뷰 없이 머지되는 PR"이 증가한다는 telemetry가 있다
> (Faros AI Engineering Report 2026). 이는 *자동 수용 증명 + 설계 중심 리뷰*가 필요한 이유를 보강하나, 산업 리포트이므로 정량 수치는 CAVEAT와 함께만 본다.

## 7. anti-pattern (이 하네스가 피하는 것)

1. **Q축=A축 혼동** — 코드 품질이 높으니 AI도 잘 다룰 거라 가정. → 2축 분리 측정(§2).
2. **프롬프트/문서 과적합** — CLAUDE.md를 계속 늘려 구조 부재를 메우려 함. → 구조가 먼저, 빌드가 강제(§4).
3. **린트로 아키텍처 강제** — 통과 가능한 린트 규칙으로 의존 방향을 "지키게 함". → 빌드에서 물리적으로 불가능하게(§4).
4. **모호한 빌드 피드백** — `BUILD FAILED`. → 피드백 3차원(§4).
5. **느린 피드백 루프** — 전체 빌드/기동 후에야 검증. → 앞단화·모듈 컴파일·standalone(§4·§5).
6. **seed 지름길** — Repository 직접 seed로 비즈니스 검증 우회. → UseCase 경유(§5).
7. **수용 증명 과신** — "E2E 통과 = 좋은 코드". → 한계 명시(§6).
8. **등급 reward-hacking** — 자기보고로 등급 상승, green을 구조 변화로 착각. → 증거 기반 재측정(§3, SKILL Phase 3).
9. **무한 self-repair** — 피드백 루프를 무한 반복. → 이득은 첫 2라운드에 몰림(§4 학술 보강).

## 8. 인접 도메인 경계 (독립성 유지하며 변별)

이 하네스는 **애플리케이션 코드베이스 자체의 구조적 AI 접근성 진단·개선**에 특화한다. 다음은 *일반 개념*으로 범위 밖이며 의존하지 않는다.

- **한 기능의 실행 기반 구현·검증** — 서비스/API 설계·구현·빌드/테스트 검증은 백엔드 구현 도메인(이 하네스는 *한 기능*이 아니라 *코드베이스 구조*를 본다).
- **상류 산출물 핸드오프 게이트 검수** — 코드 착수 *전* 기획·디자인·계약·AC 검수는 별개 도메인.
- **하네스 자체 진단·개선** — CLAUDE.md/SKILL.md/agents("LLM 주위의 코드")의 결함 진단·고도화는 별개(표적이 *하네스*가 아니라 *제품 코드베이스*).
- **전달 파이프라인** — 커밋→프로덕션 CI/CD·릴리스·IaC는 별개(둘 다 '빌드'를 다루나, 이 하네스는 *빌드 계층 아키텍처 강제·구조*이지 전달이 아니다).
- **실행 가능 명세 작성** — spec=source of truth로 코드를 생성·검증하는 것은 별개.
- **컨텍스트 페이로드 조립** — 모델에 넣을 컨텍스트 조립·압축·관리는 별개.
- **완성 코드 리뷰·커밋/PR** — 이미 완성된 코드의 일반 리뷰·커밋은 별개.
