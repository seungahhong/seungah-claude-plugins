# AI-Readable Codebase — 코드베이스의 구조적 AI 접근성 진단·개선 오케스트레이터

코드베이스가 **AI 코딩 에이전트가 읽고 안전하게 기여할 수 있는 구조**인지를 진단하고 개선한다. 두 전제 위에 선다 —
**"구조가 프롬프트보다 먼저다(structure before prompt)"**, 그리고 **"코드 품질(Q축)과 AI 접근성(A축)은 다른 차원이다"**.
사람에게 깨끗한 코드라도 패턴 비일관·빌드 피드백 약함·모듈 경계 모호로 에이전트에게는 적대적일 수 있다.

## 핵심 전환 (왜 구조·빌드인가)

빌드 실패는 즉각적이고 모호하지 않은 신호다. 반면 자연어 가이드(CLAUDE.md)는 단독으로는 세 가지로 실패한다 —
해석 모호(회색지대), 컨텍스트 경쟁(상세 규칙이 생성 시 후순위로 밀림), 피드백 부재(위반이 PR 전까지 조용히 누적).
그래서 이 하네스는 **빌드가 강제하고(무엇이 불가능한지) 문서가 설명한다(무엇이 바람직한지/왜)**를 원칙으로, 가장 중요한
아키텍처 규칙을 *가장 빠른 피드백 계층(컴파일)*에서 잡도록 설계한다.

## 경계 (먼저 읽고 발동 여부를 판단하라)

이 하네스는 **'애플리케이션 코드베이스 자체의 구조적 AI 접근성을 진단·개선한다'**. 다음은 명시적으로 범위 밖이다.

- **한 기능의 실행 기반 구현·검증** — 서비스/API를 설계·구현하고 빌드·테스트로 검증하는 것은 *백엔드 구현 도메인*(backend-harness)이다.
  이 하네스는 *한 기능*이 아니라 *코드베이스 구조*를 진단·개선한다.
- **상류 산출물 핸드오프 게이트 검수** — 코드 착수 *전* 기획·디자인·API 계약·QA 인수조건을 검수하는 것은 review-harness다. 이 하네스는 *코드베이스*를 본다.
- **하네스 자체 진단·개선** — CLAUDE.md/SKILL.md/agents("LLM 주위의 코드")의 결함을 trace로 진단·고도화하는 것은 meta-harness다.
  표적이 *하네스*가 아니라 *제품 코드베이스*다(혼동 주의 — 둘 다 'CLAUDE.md'를 언급할 수 있으나 표적이 다르다).
- **전달 파이프라인** — 커밋→프로덕션 CI/CD·릴리스·IaC 파이프라인 설계는 cicd-harness다(둘 다 '빌드'를 다루나, 이 하네스는 *빌드 계층의 아키텍처 강제·구조*이지 *전달 파이프라인*이 아니다).
- **실행 가능 명세 작성** — 에이전트가 구현할 spec을 source of truth로 쓰는 것은 spec-driven-development다.
- **컨텍스트 페이로드 조립** — 모델에 넣을 컨텍스트를 조립·압축·관리하는 것은 context-engineering다.
- **완성 코드 리뷰·커밋/PR** — 이미 완성된 코드의 일반 리뷰, 커밋·PR은 frontend/git-harness다.

경계가 모호하면 한 질문으로 확인한다 — "*코드베이스 자체*의 구조적 AI 접근성을 진단·개선하려는 건가요, 아니면 *기능 구현/상류 검수/하네스 진단/전달 파이프라인*인가요?"

## 내재화 원칙 (모든 단계가 따른다)

- **A축 ≠ Q축(분리 측정)** — 코드 품질(테스트·복잡도·중복)과 AI 접근성(패턴 일관성·빌드 피드백·모듈 경계 예측성·의존 방향 강제·독립 실행·에이전트 가이드·자동 수용)을 *별개 축*으로 매긴다. 높은 Q축이 높은 A축을 보장하지 않는다.
- **구조가 프롬프트보다 먼저** — CLAUDE.md를 더 길게 쓰는 대신 에이전트가 *틀린 방향으로 갈 수 없게* 구조를 먼저 만든다.
- **빌드가 강제하고 문서가 설명한다** — 빌드는 "무엇이 불가능한지", CLAUDE.md는 "무엇이 바람직한지/왜". 빌드로 잡을 수 있는 것을 산문에 맡기지 않고, CLAUDE.md는 빌드 규칙과 *동기화*한다.
- **피드백을 앞단으로·가르치게 설계** — 검증을 앞단으로(컴파일 > 런타임, 타입 > 컴파일 체크), 피드백 메시지는 3차원(위치 특정성·원인 명확성·수정방향 추론가능성), 가장 중요한 규칙을 가장 빠른 계층(컴파일)에서. 테스트 실패도 *피드백*으로 설계한다.
- **독립 실행(standalone)** — 도메인 슬라이스를 전체 없이 검증 가능하게(port/adapter 치환·use-case 경유 seed). 계산적 통제(테스트·린터·타입체커가 실제로 도는 환경)가 추론적 통제보다 낫다.
- **수용 증명 우선 + 한계 명시** — 리뷰 전에 동작 증명이 존재하게 해 리뷰를 "동작하나→설계가 좋은가"로 옮기되, 수용 증명이 못 잡는 것(성능·보안·동시성·가독성)은 사람 몫으로 명시한다.
- **증거 기반 등급(reward-hacking 가드)** — 등급은 자기보고가 아니라 *증거*로 매긴다("경계 위반이 빌드 실패를 일으키는가"를 file:line으로). green ≠ 개선, 자기보고 불신.
- **generator/checker 분리** — 가드레일·standalone을 *설계*한 에이전트(Phase 1/2)와 등급을 *재측정*하는 검증자(Phase 3)를 분리한다.
- **스택 무관 일반화** — 근거 예시(Kotlin/Gradle/Hexagonal)는 예시일 뿐. 원리(빌드 강제·독립 실행·수용 증명)를 사용자 스택으로 환산한다. 특정 도구를 강요하지 않는다.
- **제안만(사람 집행)** — 진단·등급·설계 *제안*을 산출한다. 코드를 자동 수정하지 않는다(구조 변경 집행은 사람 승인 후 또는 구현 하네스로 라우팅).
- **과장 금지(정직성)** — 외부 정량 수치(CodeScene Code Health 등)는 vote/CAVEAT와 함께만 인용, "개선 N% 보장" 금지. 반박된 주장은 references dossier 투명성 섹션에만.

## 에이전트 팀

| Phase | 에이전트 | 역할 |
|-------|----------|------|
| 0 Assess | `accessibility-assessor` | 2축(Q/A) 진단 + L1~L5 등급(증거 기반) + A축 격차·목표 등급·우선순위 백로그 |
| 1 Guardrails | `guardrail-architect` | 빌드 가드레일 설계(의존 방향 물리 강제 + 피드백 3차원 + 빌드/CLAUDE.md 역할 분담) |
| 2 Standalone | `standalone-designer` | 도메인 슬라이스 독립 실행 설계(port/adapter 치환·use-case seed·커버리지) |
| 3 Acceptance & Re-grade | `acceptance-verifier` | 수용 증명 인프라 설계 + 등급 적대 재측정(reward-hacking 가드, 격차 해소 추적) |

# 인터랙티브 플로우

## Phase 0 — 진단 (Assess) · 진단 승인 게이트

`accessibility-assessor`를 호출해 코드베이스를 **2축(Q/A)으로 진단하고 L1~L5 등급을 증거 기반으로** 매긴다.

```
Agent(
  subagent_type="accessibility-assessor", model="opus",
  prompt="""
  [역할] 대상 코드베이스를 Q축(코드 품질)과 A축(AI 접근성)으로 분리 진단하고 L1~L5 AI 접근성 등급을 증거 기반으로 매긴다.
  [입력] 진단 대상: {대상 경로/서비스/모듈}, 빌드·언어·아키텍처: {있으면}
  [규칙] A축 7개 지표(패턴 일관성·빌드 피드백 품질·모듈 경계 예측성·의존 방향 강제·독립 실행·에이전트 가이드·자동 수용)로 평가한다.
         가장 결정적 측정은 "금지된 경계 위반이 빌드 실패를 일으키는가" — 위반 케이스나 빌드 설정을 file:line으로 인용해 판정한다(자기보고 불신).
         높은 Q축이 높은 A축을 보장하지 않음을 드러낸다. 증거 확보 불가 항목은 '검증 불가'로 분리(등급 미산입).
         현재/목표 등급과 A축 격차를 우선순위 백로그(각 항목에 담당 Phase 1/2/3 태깅)로 낸다. 코드를 고치지 않는다(진단만).
  [출력] AI 접근성 진단 리포트(Q축·A축 지표별·등급 판정·격차 백로그·가정).
  """
)
```

진단 리포트를 사용자에게 보여주고 **진단 승인 게이트**:

`[Phase 0] AI 접근성 진단 — 현재 L?, 목표 L?, A축 격차 N개. 다음: 빌드 가드레일 설계(Phase 1). 승인할까요?`

승인 전에는 개선 설계로 진행하지 않는다(잘못된 진단으로 개선 비용을 낭비하지 않기 위함). 승인된 진단을 `.claude/ai-readability/{대상-slug}/assessment.md`로 write한다(없을 때만 디렉토리 생성).

## Phase 1 — 구조·가드레일 (Guardrails)

`guardrail-architect`를 호출해 A축 격차 중 **빌드 계층에서 잡아야 할 규칙**을 골라 가드레일을 설계한다.

```
Agent(
  subagent_type="guardrail-architect", model="opus",
  prompt="""
  [역할] 핵심 아키텍처 규칙을 빌드/컴파일 계층으로 옮겨 물리적으로 강제하고, 빌드 피드백을 에이전트가 학습하게 설계한다.
  [입력] Phase 0 진단(A축 격차·백로그·현재/목표 등급): {Phase 0 산출물}, 빌드·언어·모듈 구조: {있으면}
  [규칙] '빌드가 강제하고 문서가 설명한다' — 빌드는 무엇이 불가능한지(의존 방향·모듈 구조·스타일), CLAUDE.md는 무엇이 바람직한지/왜.
         가장 중요한 가드레일은 '금지된 의존을 빌드에서 불가능하게' 만드는 것(린트 통과 가능 < 빌드 통과 불가).
         빌드 피드백을 3차원(위치 특정성·원인 명확성·수정방향 추론가능성)으로 개선하고, 검증을 앞단으로(컴파일>런타임, 타입>컴파일 체크) 옮긴다.
         규칙별로 빌드 강제/문서 설명/둘 다를 분담표로 명시하고, 빌드로 못 잡는 것(설계 판단·맥락)은 문서/리뷰로 분리한다.
         Convention Plugin/Gradle은 예시 — 원리를 사용자 스택으로 환산한다. 코드를 자동 수정하지 않고 설계·제안만 낸다.
  [출력] 빌드 가드레일 설계(빌드 강제 규칙·의존 방향 강제·피드백 개선·앞단화·역할 분담표·미해결).
  """
)
```

설계를 검토하고 `.claude/ai-readability/{대상-slug}/remediation-plan.md`에 Phase 1 섹션으로 write/append한다.

## Phase 2 — 독립 실행 (Standalone)

`standalone-designer`를 호출해 도메인 슬라이스를 **전체 시스템 없이 독립 실행**하게 설계한다.

```
Agent(
  subagent_type="standalone-designer", model="opus",
  prompt="""
  [역할] 도메인 슬라이스를 전체 시스템 없이 독립 실행·검증 가능한 self-contained 환경으로 설계한다.
  [입력] Phase 0 진단(독립 실행 격차) + Phase 1 가드레일 설계(모듈 경계·의존 방향): {Phase 0·1 산출물}
  [규칙] Port/Adapter 의존성 역전을 활용해 운영 어댑터를 테스트 어댑터로 치환한다(도메인 로직 불변, 복잡성은 Filter/경계에 격리).
         필요한 도메인만 선택 조합하고 운영 전용 모듈은 전이 의존에서 명시적으로 제외한다.
         seed 데이터는 Repository 직접 호출이 아니라 UseCase 경유로 생성한다(비즈니스 검증 경로를 타게 — 지름길 금지).
         외부 의존은 컨테이너화하고 단일 명령으로 기동되게 한다. 독립 검증 커버리지(독립 실행 가능 슬라이스 비율)를 낸다.
         Hexagonal/Spring/Gradle은 예시 — 원리를 사용자 스택으로 환산한다. 코드를 자동 수정하지 않고 설계·제안만 낸다.
  [출력] Standalone 실행 설계(모듈 조합·어댑터 치환·UseCase seed·외부 의존·기동·커버리지).
  """
)
```

설계를 검토하고 `remediation-plan.md`에 Phase 2 섹션으로 append한다.

## Phase 3 — 수용 증명·재측정 (Acceptance & Re-grade)

설계 에이전트와 **분리된** `acceptance-verifier`를 호출해 수용 증명 인프라를 설계하고 등급을 **적대적으로 재측정**한다.

```
Agent(
  subagent_type="acceptance-verifier", model="opus",
  prompt="""
  [역할] Phase 1·2 설계자와 분리된 독립 검증자로서 ① 수용 증명(자동 E2E·데모) 인프라를 설계하고 ② 등급을 증거 기반으로 적대 재측정한다.
  [입력] Phase 0 진단(목표 등급·격차) + Phase 1 가드레일 설계 + Phase 2 standalone 설계 (+ 개선이 일부 적용된 상태가 있으면): {Phase 0·1·2 산출물}
  [규칙] 수용 증명은 리뷰어가 PR을 열기 전 '이미 동작 증명이 존재'하게 해 리뷰를 '동작하나→설계가 좋은가'로 옮긴다(문서가 아니라 빌드가 지킨다).
         단, 수용 증명이 못 잡는 것(성능·보안 인가 우회·동시성 정합·가독성)은 사람 몫으로 반드시 명시한다(과신 금지).
         등급 재측정은 자기보고로 올리지 않는다 — '이제 경계 위반이 빌드 실패를 일으키는가'를 위반 케이스로 직접 확인한다. green ≠ 개선, reward-hacking 의심·역점검.
         Phase 0 백로그의 각 격차(G1, G2…)가 해소됐는지 증거와 함께 귀속하고, 미해소는 되돌릴 Phase로 라우팅한다.
         증거 확보 불가는 '미달'이 아니라 '검증 불가(BLOCKED)'로 분리한다(거짓 미달 방지). 코드를 자동 수정하지 않고 설계·Verdict만 낸다.
  [출력] 수용 증명 설계 + 등급 재측정 Verdict(증거·한계·격차 해소·BLOCKED·결론·라우팅).
  """
)
```

재측정 결과 분기:

- **목표 등급 도달 + 격차 증거 해소** → `acceptance-and-regrade.md`에 Verdict를 write하고 **완료 종료**.
- **격차 미해소 존재** → 어느 격차가 왜 미해소인지에 따라 라우팅한다 — 빌드 강제 미흡이면 `guardrail-architect`(Phase 1) 재호출, 격리 불가면 `standalone-designer`(Phase 2) 재호출, 진단 자체가 틀렸으면 `accessibility-assessor`(Phase 0)로 되돌린다.
- **검증불가(BLOCKED)** → 미달로 두지 말고 빌드·E2E 환경/위반 케이스 확보를 보정한다(거짓 미달 방지). 이 항목은 미달 카운트에 산입하지 않는다.
- **reward-hacking 의심(green인데 구조 미변경)** → 등급 상승을 거절하고 근거를 제시한 뒤 해당 Phase로 되돌린다.
