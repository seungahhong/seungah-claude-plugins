# review-harness

개발 착수 *전* **상류 산출물(기획·디자인·API 계약·QA 인수조건)** 을 핸드오프 시점에 '착수 게이트'로 검수하는 도메인 무관 멀티 에이전트 하네스입니다.

AI가 코드 작성을 자동화할수록 병목과 결함은 '작성'이 아니라 **상류 스펙 품질과 리뷰**로 이동합니다. happy-path만 그린 디자인, 음성 경로가 빠진 인수조건, 닫히지 않은 API 계약은 그대로 넘어가면 프론트엔드·백엔드가 추측으로 메우고 QA 막바지에 재작업으로 돌아옵니다. 이 하네스는 그 사각지대를 **코드 착수 전에** 잡습니다.

## 설치

이 저장소를 Claude Code 플러그인 마켓플레이스로 추가한 뒤 `review-harness` 플러그인을 활성화하면, `handoff-review` 오케스트레이터 커맨드와 4개 게이트 스킬을 사용할 수 있습니다.

## 스킬 / 커맨드

| 스킬·커맨드 | 역할 |
| --- | --- |
| `/handoff-review` (커맨드) | 오케스트레이터(진입점). 넘어온 산출물에 해당하는 게이트를 선택받아 **병렬 실행**하고, '착수 준비도(Readiness)' 통합 리포트와 각 팀에 되돌릴 질문을 산출 |
| `/dor-review` | 기획 산출물(PRD/유저스토리/인수조건/티켓)을 착수 준비도(Definition of Ready)로 검수 — DoR 게이트 · INVEST · GWT 완결성 · 모호성 린트 · 의존성 참조 |
| `/design-handoff-review` | 디자인 핸드오프(Figma/스펙)의 사각지대를 검수 — 에러/로딩/빈/요소 상태 누락 · 토큰 바인딩 · 컴포넌트↔코드 매핑 · 상태별 oracle |
| `/contract-review` | API 계약(OpenAPI/스키마)을 FE 착수 전 검수 — 엔드포인트 완결성 · breaking-change diff · 소비자(CDC) 커버리지 · 코드↔spec drift |
| `/test-coverage-review` | QA 상류 산출물(인수조건↔테스트)의 커버리지를 검수 — 테스트가능성 · AC↔테스트 매핑 · 커버리지 채점 · 누락 음성/엣지 시나리오 발굴 |

## 언제 쓰나 / 언제 다른 도구를 쓰나

**이 하네스를 쓰세요**
- 기획·디자인·계약·인수조건이 **개발 착수에 충분히 준비됐는지** 핸드오프 시점에 게이트로 확인하고 싶을 때
- 산출물의 **모호성·누락·사각지대·breaking change**를 코드 착수 전에 잡고 각 팀에 되돌릴 질문을 만들고 싶을 때
- FE가 BE 계약을 기다리지 않고 **mock으로 병렬 착수**해도 안전한 계약인지 판정하고 싶을 때

**다른 도구를 쓰세요**
- 완성된 **코드** 리뷰(품질·보안·성능·정합성) → `frontend-harness`의 `/review`
- PRD·사용자 스토리를 **처음부터 작성** → `product-spec-harness`의 `/product-spec`
- 커밋 메시지·PR 리뷰 → `git-harness`
- 하네스 자체를 생성·진단·개선 → `harness-generator` / `meta-harness`
- 검증 가능한 목표를 자율 반복으로 달성 → `loop-engineering`

이 하네스는 **상류 산출물을 검수**할 뿐, 코드를 작성·수정하거나 산출물 자체를 합의·수정하지 않습니다(읽기 위주). 이 경계는 plugin.json description과 `evals/trigger-eval.json`에 명시되어 있습니다.

## 진행 방식

1. **게이트 선택** — `/handoff-review`가 저장소의 상류 산출물을 자동 탐지해 보조하고, 검수할 게이트(dor / design / contract / test-coverage)를 사용자가 선택합니다.
2. **병렬 검수** — 선택된 게이트를 동시에 실행합니다(각 게이트는 독립 평가).
3. **착수 준비도 통합 리포트** — 게이트별 판정(PASS/보완/BLOCKED)과 각 팀(기획/디자인/BE/QA)에 되돌릴 질문, 종합 착수 판정(YES/보완 후/NO)을 제시합니다. 하나라도 Blocker가 있으면 착수 보류로 시작합니다.

개별 게이트만 빠르게 돌리려면 `/dor-review`처럼 스킬을 직접 호출하면 됩니다.

## 정직성(Honesty Guardrail)

모든 게이트 스킬은 동일한 정직성 규칙을 공유합니다.

- 정량 수치는 **검증된 근거만** 등급(GOLD/SILVER/BRONZE)·출처와 함께 인용합니다.
- **'개선 N%'를 약속하지 않습니다.** 효과의 크기는 조직이 직접 baseline을 측정해 추세로 판단합니다(baseline-before-target).
- 출처 추적에 실패했거나 반증된 수치(예: IBM '결함비용 1→100배', NIST 표의 'Example Only', Code Connect 'AI 코드 85~90%')는 사실로 인용하지 않고 '신화'로만 표기합니다.
- '참고/근거'는 **2025년 이후 논문**을 우선하고, 인용한 수치는 실재·연도·venue를 확인합니다. preprint·벤더 자기보고·소표본은 한계를 함께 적습니다.
- 자동 탐지(LLM)는 사람의 검토를 **대체하지 않고 보조**합니다 — LLM은 '이상적 템플릿'에서 벗어났다는 이유로 멀쩡한 산출물을 과대탐지하는 경향이 보고됩니다.

## evals

`evals/trigger-eval.json`은 이 하네스가 발동해야 하는 경우(should-trigger: 4개 게이트 검수)와 발동하면 안 되는 경우(should-not-trigger: 코드 리뷰·PRD 작성·커밋·하네스 진단·자율 반복 등)를 정의해 트리거 정확도와 자매 플러그인과의 경계를 점검합니다.
