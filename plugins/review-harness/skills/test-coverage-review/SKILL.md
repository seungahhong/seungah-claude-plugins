# Test Coverage Review — 인수조건↔테스트 커버리지 게이트

QA 산출물의 **커버리지**를 핸드오프 시점에 게이트로 리뷰한다. 코드가 정상이어도 인수조건이 테스트가능하게 정의되지 않으면 QA 막바지에야 누락이 드러나 재작업이 발생한다. 이 스킬은 인수조건이 착수 전에 측정가능한 형태로 정의됐는지 점검하고, 실행가능 스펙으로 변환하며, 빠진 음성·엣지 시나리오를 발굴한다.

**완성된 코드의 모듈 간 경계면**을 검증하는 작업이 코드 이후 단계라면, 이 스킬은 코드 이전의 **상류 산출물(인수조건·테스트 시나리오)**을 검증한다. 둘은 대상이 다르며 상호 보완한다(경계면 검증은 이 스킬의 범위 밖).

## 핵심 원칙

**1. 품질은 끝에서 검사하는 게 아니라 처음부터 내장한다.**
QA 산출물은 QA 단계 막바지에 던지는 인수 시트가 아니라, 기획 착수 시점부터 살아 있는 **재사용 가능한 테스트 인프라**다. 인수조건을 실행가능 스펙(Gherkin)으로 적어 두면 같은 문서가 명세이자 테스트 케이스이자 회귀 자산이 된다.

**2. 막바지 테스트만으로는 높은 결함제거효율에 도달하지 못한다 — 인수조건을 착수 전으로 당긴다.**
대부분의 테스트 단계는 단독 결함제거효율(DRE)이 50% 미만에 머무는 경향이 있어, 막바지 테스트만으로는 높은 결함제거효율을 만들 수 없다. 95%를 넘는 누적 DRE에 도달하려면 테스트 단독이 아니라 사전 인스펙션(정형 설계 인스펙션 평균 87%·정형 코드 인스펙션 85%)과 정적 분석(평균 85%, 범위 65~95%)을 테스트와 함께 운용해야 한다 — Capers Jones, "Software Defect Removal Efficiency" (2011) [GOLD, medium·dated·독점 데이터]. 결함을 줄이려면 인수조건 정의를 착수 이전(Shift-Left)으로 당겨 누락·모호함을 코딩 전에 제거한다.

**3. 누락을 메우는 테스트 생성은 실행 피드백 폐루프가 있어야 신뢰할 수 있다(생성은 이 게이트의 범위 밖).**
이 스킬은 커버리지를 *검수*할 뿐 테스트를 생성하지 않는다. 다만 Phase 4에서 발굴한 누락을 실제 테스트로 메울 때, raw LLM 생성 테스트는 환각으로 컴파일·런타임 오류와 낮은 커버리지를 낳으므로, 생성→컴파일→실행→수리 폐루프(경험적 수리 템플릿 + 커버리지 피드백)가 있어야 신뢰도가 오른다 — TestART, arXiv:2408.03095 (ACM TOSEM 2025) [SILVER, 2024 데이터·효과크기 단정 금지]. 생성·수리 루프 자체는 이 게이트의 범위 밖(실행 폐루프를 갖춘 별도 테스트 생성 단계)이며, 이 게이트는 그 입력(무엇을 어떤 경로로 생성해야 하는지)을 제공하는 상류 역할을 한다.

## Honesty Guardrail

이 스킬이 출력에 정량 수치를 인용할 때는 반드시 (1) 출처 등급, (2) 출처명·링크, (3) 가능하면 반대증거를 함께 표기한다.

- 등급 정의: GOLD = 피어리뷰·정부·DORA 등 대규모 연구 / SILVER = 방법론이 공개된 업계 리포트(벤더 자기보고 포함) / BRONZE = 벤더 블로그·일화.
- '개선 N%'를 약속하지 않는다. 검증된 소수 수치만 인용하고, 나머지는 조직이 직접 baseline을 측정하게 한다(baseline-before-target).
- 다음 수치는 출처 추적 실패 또는 반증되었으므로 **사실로 인용 금지**한다. 설명·주의용으로만, '신화'로 명시한다.

| 금지 수치 | 사유 |
| --- | --- |
| IBM Systems Science Institute '결함비용 1배→100배' | 1981 사내 교육 코스노트, 뒷받침 데이터 없음 |
| 결함수정비용 1/5/10/15/30배 | NIST 보고서가 표 제목에 'Example Only' 명기 |
| Crosstalk '결함비용의 64%가 요구·설계' | 원출처 추적 실패 |
| Flow Efficiency '보통 15~25%' | 검증 연구 0건, 통설 |
| DORA 'Elite가 Low 대비 182배/127배/2293배' | 블로그 합성물, DORA가 폐기한 프레이밍 |
| 결함탈출률 절대등급 'Good <5% / 우수 <2%' | 업계 표준 부재 — 자체 baseline으로 시작 [BRONZE] |

- **LLM-as-judge 채점은 캘리브레이션 없이는 신뢰 근거가 없다.** substring·단순 키워드 일치로 산출물 정오를 판정하면 인간 주석과의 일치도가 우연 수준(κ=0.049)에 그치고, 3개 LLM judge 앙상블이라야 보통 수준(κ=0.432, 보수적 편향)에 도달한다 — AgentProp-Bench, arXiv:2604.16706 (2026) [SILVER, 도구사용 에이전트 평가·100건 human-validated subset]. 따라서 Phase 3의 LLM-as-judge 점수를 인용·게이팅에 쓸 때는 (a) 최소 50개 인간 라벨과의 일치도, (b) 편향 방향(보수/관대), (c) raw judge 출력과 함께 캘리브레이션 추정치를 **함께 보고**한다. 이 셋이 없는 자동 점수는 합격 신호가 아니라 보조 신호로만 쓴다.

## 실행 절차

### Phase 0: 대상 선정

1. **리뷰 대상 산출물 수집**: 인수조건이 적힌 위치를 찾는다.

```bash
# 인수조건/실행가능 스펙 산출물 탐색
grep -rln 'Given\|When\|Then\|인수\s*조건\|Acceptance\|시나리오\|Scenario' \
  --include="*.md" --include="*.feature" --include="*.story" . 2>/dev/null | grep -v node_modules | head -40
```

```bash
# 코드 옆에 존재하는 테스트 흔적 탐색
git ls-files | grep -E '\.(feature|story)$|\.(test|spec)\.(t|j)sx?$|\.cy\.(t|j)sx?$|e2e' | head -40
```

2. **리뷰 모드 선택**: 다음 중 하나를 사용자에게 확인한다.

```
[Test Coverage Review] 리뷰 모드를 선택해주세요.

1. 변경 기반 — 이번 PR/브랜치가 추가·수정한 스토리와 인수조건만 (기본)
2. 기능 단위 — 특정 스토리/기능 하나의 인수조건↔테스트 전체
3. 도메인 전체 — 특정 도메인(studio/funding/광고 등) 산출물 전체
```

3. **착수 모드 확인**: 산출물이 아직 코드 착수 전 단계인지(사전 게이트), 코드와 함께 제출된 단계인지(핸드오프 게이트) 구분한다. 사전 게이트면 Phase 1~2를, 핸드오프 게이트면 Phase 1~4를 모두 적용한다.

### Phase 1: 인수조건 테스트가능성 점검

각 스토리의 인수조건이 **기계적으로 검증 가능한 형태**인지 점검한다. 모호한 서술("정상 동작한다", "빠르게 응답한다")은 테스트로 변환할 수 없으므로 FAIL로 분류한다.

스토리별로 다음을 확인한다.

| 점검 항목 | 기준 |
| --- | --- |
| 트리거(When) 명시 | 어떤 행위·이벤트가 동작을 일으키는지 구체적으로 적혔는가 |
| 사전조건(Given) 명시 | 시작 상태·전제 데이터가 명확한가 |
| 검증가능 결과(Then) 명시 | 관찰 가능한 결과·값·상태가 명시됐는가(주관적 형용사 금지) |
| 음성 경로 포함 | 권한 없음·검증 실패·중복 요청 등 거부 시나리오가 있는가 |
| 엣지·빈·에러 경로 포함 | 빈 입력·경계값·네트워크 실패·타임아웃이 다뤄졌는가 |
| 단일 책임 | 하나의 인수조건이 하나의 검증 대상만 다루는가 |

테스트가능하지 않은 인수조건은 **그 자리에서 Given-When-Then으로 변환 제안**한다.

```gherkin
# 변환 전(모호): "정산 신청이 정상 처리된다"
# 변환 후(실행가능 스펙)
Scenario: 잔액이 충분한 메이커의 정산 신청
  Given 정산 가능 잔액이 100,000원인 메이커가 로그인되어 있고
  When 80,000원 정산을 신청하면
  Then 신청 상태가 "정산 요청"으로 바뀌고
  And 정산 가능 잔액이 20,000원으로 줄어든다
```

### Phase 2: AC ↔ 테스트 매핑

각 인수조건에 대응하는 테스트가 실제로 존재하는지 양쪽을 대조한다. 인수조건과 테스트를 따로 읽으면 매핑 누락을 잡을 수 없으므로 **인수조건 목록과 테스트 파일을 같이 읽는다.**

```bash
# 코드 옆 테스트의 describe/it/Scenario 제목 수집
grep -rn "describe(\|it(\|test(\|Scenario:\|Scenario Outline:" \
  --include="*.feature" --include="*.test.ts" --include="*.test.tsx" \
  --include="*.spec.ts" --include="*.cy.ts" . 2>/dev/null | grep -v node_modules | head -50
```

대조 기준은 다음과 같다.

- 인수조건 하나당 최소 하나의 대응 테스트(단위·통합·E2E 중 적절한 레벨)가 존재하는가.
- Phase 1에서 음성·엣지로 분류한 경로에 대응 테스트가 있는가(정상 경로만 덮고 끝나지 않았는가).
- 테스트는 존재하나 대응 인수조건이 없는 경우(스펙 없는 테스트), 인수조건은 있으나 테스트가 없는 경우(검증 없는 스펙)를 각각 표시한다.
- 현재 루트 경로의 프로젝트 코드를 기준으로 참고한다: `*.feature`·E2E 흔적이 확인되지 않으면, 매핑 대상 테스트가 0건이라는 사실 자체를 갭으로 보고한다(없는 것을 있다고 추정하지 않는다).

### Phase 3: 커버리지 채점

시나리오 완결성·AC 정합·엣지/음성 가중 루브릭으로 0~100점을 매긴다. LLM-as-judge 방식으로, 각 점수에 근거 문장을 함께 남긴다. 루브릭 기반 LLM-as-judge에서 GPT-4o Mini 구성이 1차 시도 평가 완료율(ECR@1) 96.6%, 1천 건당 $1.01을 달성한 사례가 있다 — Huang et al., "LLM-as-a-Judge for Scalable Test Coverage Evaluation", arXiv:2512.01232 (2025) [SILVER, preprint·100스크립트 소규모]. 단 96.6%는 '채점을 정확히 했다'가 아니라 '평가가 1차 시도에 파싱·완료에 성공했다'는 운영 신뢰성 지표이며, 채점 정확도는 별도(MAAE 6.07)다. 두 수치는 LLM-as-judge 일반값이 아니라 GPT-4o Mini 한정값이고, 모델별 ECR@1은 85.4~100%·비용은 175배(최대 $78.96/1천) 차이가 난다 — 채점은 보조 신호로 쓰고 사람 검토를 대체하지 않는다.

| 항목 | 배점 | 평가 기준 |
| --- | --- | --- |
| 시나리오 완결성 | 30 | Given-When-Then 3요소가 모두 채워지고 관찰 가능한 결과를 가지는 비율 |
| AC 정합 | 25 | 인수조건↔테스트가 1:1 이상 대응되는 비율 |
| 음성 경로 커버리지 | 20 | 거부·실패·권한 시나리오가 덮인 비율(가중) |
| 엣지·빈·에러 커버리지 | 20 | 경계값·빈 입력·네트워크 실패가 덮인 비율(가중) |
| 단일 책임·가독성 | 5 | 하나의 시나리오가 하나만 검증하고 재사용 가능한지 |

점수 구간은 절대 합격선이 아니라 **상대 우선순위 신호**로만 사용한다. 합격 임계값은 조직이 자체 baseline을 측정해 정한다.

#### (선택) 3범주 루브릭 확장 + 객관:주관 50:50

위 5항목 루브릭은 *시나리오 명세 품질*에 초점을 둔다. 대상 산출물에 **실제 테스트 코드**가 함께 있으면(핸드오프 게이트), 산업 현장용 지속평가 프레임워크의 3범주 루브릭으로 확장해 더 균형 있게 본다 — 단발 학술 벤치마크는 모델 진화로 금세 낡으므로 도구 신뢰성을 *지속적으로 재평가*해야 한다는 전제다(Tracking the Moving Target, arXiv:2504.18985 (2025-04) [SILVER, 단일기업 종단 사례]).

| 범주 | 측정 | 비중 |
| --- | --- | --- |
| 코드 품질(정적) | 테스트 코드의 컴파일·정적분석·유지보수성 | 객관 |
| 화이트박스(실행) | 런타임 커버리지·철저성 | 객관 |
| 블랙박스(설계) | 테스트 방법론·설계의 전문가 평가 | 주관 |

- **객관(코드품질+화이트박스) 50% : 주관(블랙박스 전문가) 50%** 동일 가중으로, 자동 측정 가능 지표만으로 합격을 판정하지 않는다.
- 자동 측정값이 높아도 **전문가/사람 검토는 여전히 필수**다(같은 출처). 위 LLM-as-judge 캘리브레이션 가드(Honesty Guardrail)를 함께 적용한다.
- 이 확장은 조직이 켜고 끌 수 있는 옵션이며, 비중·임계값은 자체 baseline으로 정한다(baseline-before-target).

### Phase 4: 누락 시나리오 발굴

요구사항에서 명시되지 않았으나 반드시 필요한 시나리오를 제안한다. LLM이 생성한 Gherkin 인수 시나리오가 사람이 간과한 케이스를 드러내 유용했다는 산업 사례가 있다(AutoUAT가 만든 시나리오를 사용자가 95% 유용하다 평가) — Ferreira et al., "Acceptance Test Generation with LLMs: An Industrial Case Study", AST 2025 (arXiv:2504.07244) [SILVER, 단일 기업·자기보고]. 같은 도메인 공개 이슈트래커를 NLP+LLM으로 분석해 누락된 비자명 인수조건을 제안한 연구에서 생성 인수조건의 80~82%가 관련 요구를 추가했다 — Schwedt & Ströder, "From Bugs to Benefits … CrUISE-AC", arXiv:2501.15181 (2025) [SILVER, preprint·게재 미확정·5인 전문가/2개 도메인·충돌·중복 미검증]. 단 LLM 생성 시나리오는 누락·환각·의도 혼재가 관찰되므로 그대로 채택하지 말고 사람이 검토해 편입한다 — Hassani et al., "From Law to Gherkin", arXiv:2508.20744 (2025) [SILVER].

다음 축을 따라 빠진 시나리오를 찾는다.

- **음성 경로**: 권한 없는 사용자, 인증 만료, 입력 검증 실패, 중복·동시 요청.
- **에러 경로**: 외부 API 5xx, 타임아웃, 부분 실패, 재시도 후에도 실패.
- **권한·역할**: 역할별 가시성 차이, 본인/타인 리소스 접근 분리.
- **경계·빈 값**: 0건, 1건, 최대치, 페이지 경계, null/undefined.
- **상태 전이**: 이미 처리된 건의 재처리, 취소 후 재신청 등 비정상 전이.

발굴한 시나리오는 Phase 1과 동일한 Gherkin 형식으로 변환 가능한 형태로 제시해 곧장 인수조건에 편입할 수 있게 한다.

### Phase 5: 결과 보고

아래 템플릿으로 보고한다.

```markdown
# Test Coverage Review 결과

## 리뷰 범위
- 모드: 변경 기반 / 사전 게이트
- 대상 스토리: N개
- 대상 산출물: (파일 경로 목록)

## 1. 인수조건 테스트가능성

| # | 스토리 | When 명시 | Then 검증가능 | 음성/엣지 포함 | 결과 |
| --- | --- | --- | --- | --- | --- |
| 1 | 정산 신청 | ✅ | ✅ | ⚠️ 음성 누락 | FAIL |
| 2 | 광고 집행 | ✅ | ❌ | ❌ | FAIL |

## 2. AC ↔ 테스트 매핑

| # | 인수조건 | 대응 테스트 | 레벨 | 결과 |
| --- | --- | --- | --- | --- |
| 1 | 잔액 충분 시 정산 | settlement.feature:12 | E2E | ✅ |
| 2 | 잔액 부족 시 거부 | (없음) | - | ❌ |
| 3 | (인수조건 없음) | legacy.spec.ts:30 | 단위 | ⚠️ 스펙 없는 테스트 |

## 3. 커버리지 점수

| 항목 | 배점 | 획득 | 근거 |
| --- | --- | --- | --- |
| 시나리오 완결성 | 30 | 18 | Then이 비어 있는 시나리오 4건 |
| AC 정합 | 25 | 15 | 미매핑 인수조건 3건 |
| 음성 경로 | 20 | 6 | 거부 시나리오 대부분 누락 |
| 엣지·빈·에러 | 20 | 8 | 타임아웃·빈 입력 미정의 |
| 단일 책임·가독성 | 5 | 4 | 양호 |
| 합계 | 100 | 51 | (상대 신호, 절대 합격선 아님) |

## 4. 누락 시나리오 (편입 제안)

1. [음성] 잔액 부족 시 정산 신청 거부 + 에러 메시지 노출
2. [에러] 정산 API 타임아웃 시 재시도 안내
3. [권한] 타 메이커 정산 내역 접근 차단

## 5. 결함탈출률 추세 (자체 baseline)

- 결함탈출률·결함제거효율(DRE)에는 합의된 절대 등급이 없다. '<5%면 우수' 같은 절대선은 사용하지 않는다 [BRONZE].
- 조직이 직접 baseline을 측정한다: (요구·설계 단계에서 발견한 결함 수) / (전 단계 누적 발견 결함 수). 이 비율을 스프린트마다 기록해 추세로만 본다(baseline-before-target).
- 참고: 미국 평균 DRE는 약 85%, best-in-class는 99%를 넘는다 — Capers Jones [SILVER, c.2011·독점 데이터·dated]. 절대 목표가 아니라 방향 참고용으로만 인용한다.

## 종합

| 카테고리 | PASS | FAIL | 경고 |
| --- | --- | --- | --- |
| 테스트가능성 | N | N | N |
| AC 매핑 | N | N | N |
| 누락 시나리오 | - | - | N |

게이트 판정: 통과 / 보류(아래 Critical 해소 후 재리뷰)

### Critical (게이트 차단)
1. ...

### Warning (수정 권장)
```

## 체크리스트

- [ ] 모든 인수조건이 측정 가능한 Given-When-Then 형태로 적혀 있는가
- [ ] 각 인수조건에 관찰 가능한 결과(Then)가 명시됐는가(주관적 형용사 없음)
- [ ] 음성·엣지·빈·에러 경로가 정상 경로와 함께 정의됐는가
- [ ] 모든 인수조건에 대응하는 테스트가 적절한 레벨로 존재하는가
- [ ] 스펙 없는 테스트, 검증 없는 스펙을 각각 표시했는가
- [ ] 커버리지 점수에 각 항목별 근거 문장을 남겼는가
- [ ] 누락 시나리오를 Gherkin으로 편입 가능하게 제시했는가
- [ ] 인용한 정량 수치마다 출처 등급·출처명을 표기했는가
- [ ] 금지 신화(IBM 1→100배, 결함탈출률 절대등급 등)를 사실로 쓰지 않았는가
- [ ] 절대 합격선 대신 조직 자체 baseline 측정을 안내했는가
- [ ] LLM-as-judge 점수를 게이팅에 쓸 때 인간 라벨 일치도·편향·캘리브레이션을 함께 보고했는가(미보고 시 보조 신호로만)
- [ ] (테스트 코드 동반 시) 3범주 루브릭·객관:주관 50:50을 적용하고 전문가 검토 필수를 명시했는가
- [ ] 누락 메우기용 테스트 생성은 실행 폐루프를 갖춘 별도 테스트 생성 단계로 인계하고 이 게이트는 검수에 한정했는가

## 사용자 소통

- Phase 0에서 리뷰 모드와 착수 모드(사전 게이트/핸드오프 게이트)를 반드시 확인한다.
- Phase 1에서 테스트가능하지 않은 인수조건은 그 자리에서 Gherkin 변환안을 제시한다.
- Critical(게이트 차단) 항목 발견 시 즉시 사용자에게 알린다.
- 매핑 대상 테스트가 0건이면 없는 것을 있다고 추정하지 않고 갭으로 보고한다.

## 참고 / 근거

수치는 효과크기가 아니라 정성근거로만 인용한다(Honesty Guardrail). 표본·자기보고·preprint 한계를 함께 적는다.

**2025+ 근거 (LLM 시대의 인수테스트·커버리지)**

- LLM(GPT-4 Turbo) 인수테스트 생성: AutoUAT가 만든 Gherkin 인수 시나리오를 사용자가 95% 유용하다 평가(이전 간과 케이스도 드러냄), Test Flow가 생성한 Cypress 인수 테스트 케이스의 92%가 유용(60% 그대로 사용·8% 사소 수정·24% 재생성·8% 폐기) — Ferreira, Viegas, Faria, Lima, "Acceptance Test Generation with LLMs: An Industrial Case Study", AST 2025 (arXiv:2504.07244) [SILVER, 단일 기업·자기보고].
- 특화 LLM(AToMIC)이 JIRA 티켓·코드 변경에서 Gherkin·Page Object·실행가능 UI 테스트를 기능당 5분 미만에 생성(BMW 앱 170+ 화면, 13개 실이슈): Gherkin 93.3% 문법 정확, PageObject 78.8% 무수정 실행, UI 테스트 100% 정상 실행 — Fonseca, Lima, Faria, "Streamlining Acceptance Test Generation for Mobile Applications Through LLMs: An Industrial Case Study", ASE 2025 (arXiv:2510.18861) [SILVER]. ※ '문법·실행 가능성'은 '의미적 커버리지 완결성'이 아니다.
- 사람 준실험에서 LLM 생성 Given-When-Then 명세의 상위2등급 비율 relevance 95%·clarity 100%·completeness 94.2%였으나, 누락·환각·의도 혼재가 관찰되어 체계적 사람 검토가 필요 — Hassani, Sabetzadeh, Amyot, "From Law to Gherkin", arXiv:2508.20744 (2025) [SILVER, 표본 10인/60명세].
- 동일 프롬프트에도 LLM(ChatGPT/Gemini/Grok/Copilot) 간 유의차가 크고(인수조건 커버리지 평균 0.41~0.77, ChatGPT 0.7670 최고) '사람 평가를 생략할 만한 완전 자동화와는 아직 거리가 멀다' — "Automated Test Generation Using LLM Based on BDD: A Comparative Study", WEBIST 2025 (SCITEPRESS) [SILVER, 34 스토리/94 인수조건].
- 루브릭 기반 LLM-as-Judge에서 GPT-4o Mini가 1차 완료율(ECR@1) 96.6%·$1.01/1천 달성. 단 ECR@1은 운영 신뢰성이고 채점 정확도는 별도(MAAE 6.07)이며, 모델별 ECR@1 85.4~100%·비용 175배 편차 — Huang et al., "LLM-as-a-Judge for Scalable Test Coverage Evaluation", arXiv:2512.01232 (2025) [SILVER, preprint].
- 누락 인수조건 자동 발굴(CrUISE-AC): NLP+LLM 앙상블이 제안한 인수조건의 80~82%가 관련 요구를 추가(5인 전문가·2개 도메인). 충돌·중복은 미검증(저자 한계 명시) — Schwedt & Ströder, "From Bugs to Benefits …", arXiv:2501.15181 (2025) [SILVER, preprint·게재 미확정·5인 전문가/2개 도메인·충돌·중복 미검증].
- LLM-as-judge 신뢰성: substring 판정은 인간 일치도 κ=0.049(우연 수준), 3-LLM 앙상블 κ=0.432(보수 편향). LLM judge 사용 시 ≥50 인간 라벨 일치도·편향 방향·캘리브레이션 추정치를 함께 보고할 것을 권고 — "AgentProp-Bench", arXiv:2604.16706 (2026) [SILVER, 2,000-task/2,300-trace·100건 human-validated]. ※ 본 스킬의 LLM-as-judge 채점을 합격 게이트가 아닌 보조 신호로 두는 근거.
- 테스트 생성기 지속평가 3범주 루브릭(코드품질/화이트박스/블랙박스, 객관:주관 50:50, 전문가 검토 필수): 단발 벤치마크는 모델 진화로 낡으므로 지속 재평가가 필요 — "Tracking the Moving Target: A Framework for Continuous Evaluation of LLM Test Generation in Industry", arXiv:2504.18985 (2025) [SILVER, 단일기업 2024-04~2025-01 종단].
- 생성 테스트 신뢰도(실행 폐루프): raw LLM 단위테스트는 환각으로 컴파일/런타임 오류·낮은 커버리지 → 경험적 수리 템플릿(컴파일 50%/런타임 75% 수리) + 생성→컴파일→실행→수리 공진화 루프로 baseline(ChatGPT/EvoSuite/A3Test) 대비 pass +18%/coverage +20% — "TestART", arXiv:2408.03095 (ACM TOSEM 2025) [SILVER, **2024-08 데이터·최신 모델엔 수치 불일치 가능·효과크기 단정 금지**]. ※ 생성·수리는 실행 폐루프를 갖춘 별도 테스트 생성 단계(이 게이트의 범위 밖)가 담당, 이 게이트는 검수·입력 제공에 한정.

**고전 기반 (정성·방향 참고, 효과크기 아님·dated)**
