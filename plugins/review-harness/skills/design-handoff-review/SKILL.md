# 디자인 핸드오프 검수

디자인 핸드오프 산출물을 **개발 착수 전 게이트**에서 검수한다. 코드가 아니라 상류 산출물(디자인 스펙·상태 정의·토큰·컴포넌트 매핑)을 대상으로, 빠진 상태와 사각지대를 찾아 디자인팀에 되돌린다. AI가 코드를 대부분 작성하는 환경에서 병목은 '작성'이 아니라 '스펙 품질'과 '리뷰'로 이동했다. 이 스킬은 그 상류를 점검한다.

## 핵심 원칙

**핸드오프는 일회성 전달이 아니라 협업이다.**
핸드오프는 화면 이미지를 던지는 행위가 아니라 what(스펙)과 why(의도·제약)를 함께 넘기는 협업이다. 의도가 빠지면 프론트엔드는 보이는 픽셀만 보고 구현하며, 디자이너가 가정했던 제약은 코드에 반영되지 않는다.

**'이상적 화면'만 그리면 사각지대를 개발자가 추측한다.**
정상 데이터가 채워진 happy-path 한 장만 넘어오면, 에러·로딩·빈 상태·요소 상태는 프론트엔드가 임의로 추측해 구현한다. 추측은 재작업과 QA 막바지 결함으로 돌아온다. 사각지대를 비워둔 채 넘기지 않는 것이 검수의 목표다.

**자동 design-to-code(D2C)가 사각지대를 메워주지 않는다 — 스펙 품질이 병목이다.**
AI가 디자인→코드를 자동 생성하는 환경에서도 자동 D2C는 신뢰하기 어렵다. 동료심사 벤치마크에서 모델들은 시각 요소 회상과 레이아웃 생성에 가장 뒤처지고(AI 생성 페이지가 참조본과 '교환 가능'하다고 본 비율 49% — Design2Code, NAACL 2025 [GOLD]), 인터랙티브 요소 생성은 4가지 구조적 한계·10종 실패에 취약하며(Interaction2Code, ASE 2025 [GOLD]), 최신 모델조차 컴파일되고 시각적으로 정합한 코드를 일관되게 생성하지 못하고 세밀 편집이 부족하다(1D-Bench, 2026 [SILVER]). 단일 모놀리식 MLLM은 시각인지·레이아웃계획·코드합성을 한 모델에 통합하려다 복잡 UI에서 자주 실패하며(ScreenCoder, arXiv:2507.22827 (2025-07) [SILVER]), 생성 품질은 프레임워크 의존적이고(HTML/CSS 최고, Angular 최악) raw 출력은 거의 컴포넌트 기반이 아니다(React 컴포넌트화율 0.24%) — 게다가 MLLM의 UI 결함 자가탐지 정확도는 약 0.27로 자가검수에 의존할 수 없다(DesignBench, arXiv:2506.06251 (2025-06) [SILVER]). 즉 정적 happy-path 외 상태·인터랙션·레이아웃은 자동 생성이 메워주지 않으므로, 핸드오프 스펙이 사각지대를 닫아 두는 것이 더 중요해진다. 코드 작성이 자동화될수록 병목은 '작성'이 아니라 스펙 품질과 리뷰로 이동한다(코드 리뷰 병목: LAURA, ASE 2025 [GOLD, 인접 근거]).

> 검수↔생성 핸드오프: 이 게이트가 사각지대·토큰·매핑을 *검수*해 통과시키면, 그 산출(누락 상태·미매핑 컴포넌트 목록)은 다운스트림의 디자인→코드 생성(D2C) 단계 입력이 된다. 이 게이트는 생성 단계를 선행 조건으로 요구하지 않으며, 둘은 독립 단계다. 관계·근거는 [references/downstream-generation.md](references/downstream-generation.md) 참조.

## Honesty Guardrail — 수치 정직성

이 스킬이 출력에 정량 수치를 인용할 때는 반드시 (1) 출처 등급 `[GOLD/SILVER/BRONZE]`, (2) 출처명/링크, (3) 가능하면 반대증거를 함께 표기한다.

- 등급 정의: GOLD=피어리뷰·정부·DORA 등 대규모 연구. SILVER=방법론이 공개된 업계 리포트(벤더 자기보고 포함). BRONZE=벤더 블로그·일화.
- '개선 N%'를 약속하지 않는다. 검증된 소수 수치만 인용하고, 나머지는 조직이 직접 baseline을 측정하게 한다(baseline-before-target). 핸드오프 품질은 '사각지대 커버리지'와 '핸드오프 후 발생한 결함 수'를 조직이 직접 측정해 판단한다.

**사실로 인용 금지(신화로만 언급):**

- Code Connect 'AI 코드 생성 85~90% 정확' — 1차 출처 추적 실패(unverifiable). 반증 맥락으로는 'Figma 2025 AI Report'(표본 2,500명, 2025-04-24)의 'AI 출력을 신뢰/의존할 수 있다' 응답 32%를 쓸 수 있으나, 이 32%는 **'AI 출력 전반' 신뢰이지 'AI 생성 코드' 한정 수치가 아니며**, 출처도 'State of the Designer 2025'(핸드오프 91/92%)가 **아닌 별도 리포트**다 — 두 리포트를 섞어 인용하지 말 것.

## 실행 절차

### Phase 0: 입력 확보 및 검수 모드 선택

1. **Figma MCP 도구 가용성 확인.** 다음 도구를 `ToolSearch`로 로드해 사용 가능하면 사용한다. 로드 쿼리 예: `select:mcp__claude_ai_Figma__get_design_context,mcp__claude_ai_Figma__get_metadata,mcp__claude_ai_Figma__get_variable_defs,mcp__claude_ai_Figma__get_code_connect_map,mcp__claude_ai_Figma__get_screenshot`.

   | 도구 | 용도 |
   | --- | --- |
   | `get_metadata` | 프레임/노드 계층 추출 — 검수할 화면 목록 파악 |
   | `get_screenshot` | 화면 시각 확인 — happy-path 여부, 빈/에러 화면 존재 여부 |
   | `get_design_context` | 프레임의 레이아웃·텍스트·요소 상세 |
   | `get_variable_defs` | 토큰/변수 정의 추출 — 토큰화 커버리지 점검 |
   | `get_code_connect_map` | 프레임 ↔ 코드 컴포넌트 매핑 감사 |

2. **Figma 접근 불가 시 대체 입력.** MCP·토큰·URL이 없으면, 사용자가 붙인 디자인 스펙 문서·스크린샷 이미지·마크다운을 입력으로 받아 같은 항목을 점검한다. 입력이 전혀 없으면 검수를 진행하지 않고 입력을 요청한다.

3. **검수 모드 선택.**

```text
[Design Handoff Review] 검수 모드를 선택해주세요.

1. 단일 화면 — 특정 프레임/플로우 1개를 사각지대 관점으로 정밀 검수 (기본)
2. 기능 단위 — 한 기능의 전체 화면 세트를 검수 (목록↔상세↔폼 등)
3. 핸드오프 게이트 — 개발 착수 직전, 넘어온 산출물 전체를 게이트로 검수
```

### Phase 1: 사각지대 커버리지 점검

각 화면에 대해 아래 상태가 디자인으로 정의되어 있는지 확인한다. 정의가 없으면 '프론트엔드가 추측할 영역'으로 플래그한다.

- **검증 에러 상태**: 폼 입력 검증 실패 시 필드별 에러 표기, 에러 메시지 카피, 에러 시 포커스/스크롤 동작.
- **에러 페이지**: 404(없음), 403(권한 없음), 500/503(서버 오류), 네트워크 단절. 각 상황의 화면과 복구 동선(재시도·돌아가기).
- **로딩 상태**: 초기 로딩 스켈레톤, 부분 로딩(무한 스크롤·페이지네이션), 버튼/액션 진행 중 상태.
- **빈 상태(empty state)**: 데이터 0건, 검색 결과 0건, 권한은 있으나 콘텐츠 없음. 빈 상태 일러스트·안내 카피·다음 행동 유도(CTA).
- **파괴적 액션 알림**: 삭제·취소·되돌릴 수 없는 액션의 확인 다이얼로그, 경고 카피, 실행 후 피드백(토스트 등).
- **요소 상태(interaction states)**: 버튼·인풋·링크 등의 default / hover / focus / disabled / pressed(active) / loading. 특히 focus와 disabled는 누락이 잦다.
- **반응형 브레이크포인트**: 최소한 모바일과 데스크탑 두 폭의 레이아웃. 데스크탑 위주 화면이어도 모바일 폭 확인이 필요하다.
- **접근성 단서**: 탭 타깃 크기, 이미지 alt 텍스트 의도, 키보드 포커스 순서, 색 대비(텍스트/배경). 구현 단계의 접근성 점검은 별도 접근성 점검 워크플로로 이어간다.
- **실제 카피**: lorem ipsum·placeholder가 아닌 실제 문구. 글자 수가 가변일 때 최소/최대 길이의 오버플로 처리.
- **의도(why)**: 화면 옆에 "왜 이렇게 설계했는가"(제약·우선순위·엣지 의사결정)가 적혀 있는지. 의도가 없으면 디자인팀에 질문으로 되돌린다.

화면별로 위 항목의 정의 여부를 표로 집계해 커버리지%(정의된 항목 / 점검 대상 항목)를 계산한다.

### Phase 2: 토큰·변수 바인딩 점검

`get_variable_defs`(또는 스펙의 색·간격 명세)를 기준으로, 디자인 값이 토큰/변수에 묶여 있는지 점검한다.

- raw hex 값(`#1A1A1A` 등)이 색 토큰에 묶이지 않고 직접 박혀 있는 곳을 플래그한다.
- 임의 spacing(4의 배수가 아니거나 스케일 밖의 값)이 간격 토큰에 묶이지 않은 곳을 플래그한다.
- 타이포(폰트 크기·행간·굵기)가 텍스트 스타일에 묶이지 않은 곳을 플래그한다.
- **토큰화 커버리지%** = 토큰/변수에 바인딩된 속성 수 / 전체 색·간격·타이포 속성 수. 절대 목표치를 약속하지 말고, 조직이 baseline을 잡고 추적하도록 수치만 보고한다.

토큰 미바인딩이 많으면 코드에서 디자인 시스템 변수가 아닌 매직 넘버가 양산될 위험이 크다는 점을 보고한다.

### Phase 3: 컴포넌트 ↔ 코드 매핑 감사

`get_code_connect_map`으로 각 프레임의 주요 컴포넌트가 실제 코드 컴포넌트에 매핑되는지 확인한다.

- 매핑된 컴포넌트: 코드에 대응물이 있어 재사용 가능 → 정상.
- 미매핑 컴포넌트: 코드 대응물이 없거나 연결되지 않음 → 커스텀 난립 위험으로 플래그. 신규 컴포넌트인지(라이브러리 추가 필요), 기존 컴포넌트를 매핑만 안 한 것인지 구분해 보고한다.
- Code Connect를 쓰지 않는 환경이면, 디자인의 컴포넌트 이름과 코드 컴포넌트 이름의 대응 여부를 수동으로 점검한다.

매핑 결과를 감사 표로 정리한다. AI 코드 생성 정확도에 대한 벤더 수치는 인용하지 않는다(Honesty Guardrail 참조).

### Phase 4: 디자인 시점 인수조건 점검

각 상태별 '기대 화면'이 존재하는지를 테스트 oracle 관점으로 점검한다. 상태별 기대 화면이 있으면 인수조건(Given-When-Then)의 검증 기준이 된다.

- Phase 1에서 정의된 각 상태에 대해 "이 조건일 때 화면은 이렇게 보여야 한다"는 oracle이 명시되어 있는가.
- 다운스트림의 상태 전이↔코드 교차검증·인수조건→테스트 단계로 이어지는 입력이 되도록, 빠진 상태별 oracle을 목록화한다.
- 인수조건을 기획 단계에서 정의(Shift-Left)하고, 통합 기획/디자인 SoT가 부재하면 그 사실 자체를 게이트 실패로 보고한다.

### Phase 5: 결과 보고

아래 템플릿으로 출력한다.

```markdown
# 디자인 핸드오프 검수 결과

## 검수 범위
- 모드: 핸드오프 게이트
- 입력: Figma <fileKey>/<nodeId> (또는 스펙 문서/이미지)
- 검수 화면: N개

## 화면별 사각지대 커버리지

| 화면 | 검증에러 | 에러페이지 | 로딩 | 빈 상태 | 파괴적 알림 | 요소 상태 | 반응형 | 접근성 | 실제 카피 | 의도 | 커버리지% |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 펀딩 목록 | ✅ | ❌ | ✅ | ❌ | — | ⚠️ | ✅ | ⚠️ | ✅ | ❌ | 55% |
| 상세 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | 100% |

> 표기: ✅ 정의됨 / ❌ 누락 / ⚠️ 부분 정의 / — 해당 없음

## 토큰화 커버리지

| 분류 | 바인딩됨 | 미바인딩(raw) | 커버리지% |
| --- | --- | --- | --- |
| 색 | N | N | N% |
| 간격 | N | N | N% |
| 타이포 | N | N | N% |

## 컴포넌트 ↔ 코드 매핑 감사

| 프레임/컴포넌트 | 코드 매핑 | 결과 | 상세 |
| --- | --- | --- | --- |
| PrimaryButton | Button | ✅ | 매핑됨 |
| RewardCard | (없음) | ❌ | 미매핑 — 신규/커스텀 위험 |

## 누락 상태 리스트 (심각도)

### Blocker (착수 차단)
1. 펀딩 목록 — 에러 페이지 미정의(404/503). FE가 추측 구현 → 재작업 위험.

### Major (착수 전 보완 권장)
1. 펀딩 목록 — 빈 상태(검색 0건) 화면·CTA 미정의.

### Minor (참고)
1. 펀딩 목록 — 버튼 disabled 상태 시각 미정의.

## 디자인팀에 되돌릴 질문
1. 목록에서 데이터 0건일 때 보여줄 빈 상태 화면과 CTA가 있나요?
2. 권한 없는 사용자가 진입하면 403 화면으로 보낼지, 진입 자체를 막을지 의도가 무엇인가요?
3. RewardCard는 신규 컴포넌트인가요, 기존 컴포넌트 매핑 누락인가요?

## 게이트 판정
- PASS / FAIL: FAIL (Blocker 1건)
- 근거: 사각지대 커버리지 평균 N%, 미바인딩 토큰 N건, 미매핑 컴포넌트 N건
- 다음 단계: 디자인팀 질문 회신 후 재검수. 보완되면 구현·상태 교차검증·테스트 단계로 인계(이 게이트의 범위 밖).
```

## 체크리스트

### 사각지대 커버리지
- [ ] 각 화면에 검증 에러 상태가 정의되어 있다
- [ ] 404/403/500·503/네트워크 단절 에러 화면과 복구 동선이 있다
- [ ] 초기·부분·액션 로딩 상태가 정의되어 있다
- [ ] 데이터 0건·검색 0건 빈 상태와 CTA가 있다
- [ ] 삭제 등 파괴적 액션의 확인 다이얼로그와 실행 후 피드백이 있다
- [ ] 요소 상태(default/hover/focus/disabled/pressed/loading)가 모두 정의되어 있다
- [ ] 모바일·데스크탑 최소 두 폭의 반응형 레이아웃이 있다
- [ ] 탭 타깃·alt·포커스 순서·색 대비 단서가 있다
- [ ] lorem이 아닌 실제 카피와 최소/최대 길이 오버플로 처리가 있다
- [ ] 화면마다 설계 의도(why)가 적혀 있다

### 토큰·매핑
- [ ] raw hex·임의 spacing·미지정 타이포가 토큰/변수에 바인딩되어 있다
- [ ] 토큰화 커버리지%를 산출해 baseline으로 보고했다
- [ ] 주요 컴포넌트가 코드 컴포넌트에 매핑(Code Connect)되어 있다
- [ ] 미매핑 컴포넌트를 신규/매핑누락으로 구분했다

### 인수조건·정직성
- [ ] 상태별 기대 화면(oracle)이 존재해 인수조건의 기준이 된다
- [ ] 빠진 oracle을 다운스트림 상태 교차검증·테스트 단계의 입력으로 목록화했다
- [ ] 인용한 모든 수치에 등급·출처를 표기했다
- [ ] 금지된 신화 수치(Code Connect 85~90%, Zeplin 66/63% 등)를 사실로 쓰지 않았다
- [ ] '개선 N%'를 약속하지 않고 조직이 baseline을 측정하도록 안내했다

**2025+ 근거 (자동 D2C·핸드오프 품질)**

- 484개 실세계 웹페이지 D2C 벤치마크에서 모델은 시각 요소 회상·레이아웃 생성에 가장 뒤처지고, AI 생성 페이지가 참조본과 '교환 가능'하다고 본 비율은 49% — "Design2Code: Benchmarking Multimodal Code Generation for Automated Front-End Engineering", NAACL 2025 (arXiv:2403.03163) [GOLD, 평가 모델 구세대]. <https://aclanthology.org/2025.naacl-long.199/>
- 127개 웹페이지·374개 상호작용 벤치마크에서 MLLM의 인터랙티브 코드 생성에 4가지 한계(상호작용 생성 부실·10종 실패·시각적으로 미묘한 상호작용 저성능·단일 모달리티 한계) — "Interaction2Code", ASE 2025 [GOLD]. <https://conf.researchr.org/details/ase-2025/ase-2025-papers/8/Interaction2Code-Benchmarking-MLLM-based-Interactive-Webpage-Code-Generation-from-In>
- 최신 LLM조차 컴파일되고 시각적으로 정합한 UI 코드를 일관되게 생성하지 못하고 세밀 편집이 부족하며, 시각 피드백 기반 반복 편집이 렌더 성공률을 끌어올린다(단일/다중 라운드 구분 필요) — "1D-Bench: A Benchmark for Iterative UI Code Generation with Visual Feedback", arXiv:2602.18548 (2026) [SILVER, preprint·e커머스·204 평가셋].
- D2C 품질은 프레임워크 의존적(vanilla HTML/CSS 최고 CLIP>0.72·컴파일 100%; Angular 최악 컴파일 0.6~0.7·CLIP 0.45~0.55)이고, raw 출력의 컴포넌트화율이 낮으며(React 0.24%/Vue 5%/Angular 19%), 편집/수리는 code-only가 image보다 우수, MLLM의 UI 결함 자가탐지 정확도는 0.27 — "DesignBench: A Comprehensive Benchmark for MLLM-based Front-end Code Generation", arXiv:2506.06251 (2025-06) [SILVER, 900 샘플·모델 세대 의존]. ※ 절대수치는 효과크기로 쓰지 말고 '프레임워크 비대칭·컴포넌트화 필요·자가검수 불가'의 정성 근거로만.
- 단일 모놀리식 MLLM은 복잡 UI에서 실패하며, grounding/planning/generation 3단계 분해 + 책무별 전문 에이전트가 end-to-end보다 robustness·fidelity·interpretability에서 우월 — "ScreenCoder: Advancing Visual-to-Code Generation for Front-End Automation via Modular Multimodal Agents", arXiv:2507.22827 (2025-07) [SILVER]. ※ 다운스트림 D2C 생성의 3단계 분해 근거([references/downstream-generation.md](references/downstream-generation.md)).
