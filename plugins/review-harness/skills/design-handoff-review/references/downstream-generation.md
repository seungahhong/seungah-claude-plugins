# 다운스트림 코드 생성(D2C)과 검수 핸드오프

`design-handoff-review` 게이트의 **출력이 어떻게 후속 단계로 흘러가는지**를 설명하는 참고 문서다. 이 게이트는 코드를 만들지 않는다 — 다만 게이트가 검수해 통과시킨 디자인(과 누락/미매핑 목록)이 다운스트림의 **디자인→코드 생성(D2C)** 단계로 넘어간다. 이 문서는 그 관계와 근거를 *특정 플러그인·스킬에 의존을 선언하지 않고* 일반 개념으로 정리한다.

## 검수 → 생성 핸드오프 (개념)

- **이 게이트(검수)**: 개발 착수 *전* 핸드오프 산출물의 사각지대(에러/로딩/빈/요소 상태 누락·토큰 미바인딩·컴포넌트 미매핑)를 점검해 디자인팀에 되돌린다. 코드를 만들지 않는다.
- **다운스트림 생성(D2C)**: 사각지대를 닫은 디자인을 실제 프론트엔드 코드로 *생성*하는 별도 단계. 일반적으로 grounding(디자인 컨텍스트 정합·요소 식별) → planning(레이아웃·컴포넌트 매핑) → generation(코드 생성)으로 분해해 진행한다.
- **연결**: 이 게이트의 산출(누락 상태·미매핑 컴포넌트 목록)이 그 생성의 *이상적 입력*이 된다. 검수가 누락 상태를 잡아 두었기에, 다운스트림 생성은 happy-path 한 장이 아니라 상태가 정의된 디자인을 코드로 옮길 수 있다.

> 이 게이트는 다운스트림 생성 단계(어떤 플러그인·스킬로 구현되든)를 **선행 조건으로 요구하지 않는다.** 생성 단계 역시 이 게이트를 전제하지 않고 단독으로 동작할 수 있다 — 둘은 입력/출력으로 *연결될 수 있는* 독립 단계다.

## 왜 검수가 더 중요해지는가 — 자동 D2C가 사각지대를 메워주지 않는다

코드 작성이 자동화될수록 병목은 '작성'이 아니라 **스펙 품질과 리뷰**로 이동한다. 자동 D2C는 정적 happy-path 외 상태·인터랙션·레이아웃을 신뢰성 있게 메워주지 못하므로, 핸드오프 스펙이 사각지대를 닫아 두는 것이 더 중요해진다.

## 근거 (등급·연도·caveat — Honesty Guardrail 유지)

수치는 효과크기가 아니라 정성·방향성 근거로만 인용한다. 절대수치는 모델 세대 의존이므로 방향성으로만 쓴다.

- 단일 모놀리식 MLLM은 복잡 UI에서 시각인지·레이아웃계획·코드합성 통합에 자주 실패하며, grounding/planning/generation 3단계 분해 + 책무별 전문 에이전트가 end-to-end보다 robustness·fidelity·interpretability에서 우월 — "ScreenCoder: Advancing Visual-to-Code Generation for Front-End Automation via Modular Multimodal Agents", arXiv:2507.22827 (2025-07) [SILVER, preprint]. <https://arxiv.org/abs/2507.22827>
- 자동 D2C 품질은 프레임워크 의존적이고(HTML/CSS 비교적 안정·Angular 최고 난도), raw 출력은 거의 컴포넌트 기반이 아니며, 편집/수리는 code-only가 image 동반보다 우수(멀티모달 추가가 때로 악화), MLLM의 UI 결함 자가탐지 정확도는 낮다 — "DesignBench", arXiv:2506.06251 (2025-06) [SILVER, preprint·절대수치는 모델 세대 의존 → 방향성으로만]. <https://arxiv.org/abs/2506.06251>
- 자동 D2C에서 AI 생성 페이지가 참조본과 '교환 가능'하다고 본 비율 49% — "Design2Code", NAACL 2025 [GOLD, 평가 모델 구세대·절대수치 일반화 금지]; 인터랙티브 요소 생성의 4가지 구조적 한계·10종 실패 — "Interaction2Code", ASE 2025 [GOLD]; 최신 모델조차 컴파일·시각정합 일관 생성 못 하고 세밀 편집 부족 — "1D-Bench", 2026 [SILVER, preprint].
- 코드 리뷰가 시간소모·지식집약·숙련 리뷰어 부족으로 개발 병목이 됐다(병목이 작성→리뷰/검증으로 이동) — "LAURA", ASE 2025 (arXiv:2512.01356) [GOLD, 인접 근거: '코드 리뷰 병목'까지만, '스펙 핸드오프 병목'으로 확대 금지].
