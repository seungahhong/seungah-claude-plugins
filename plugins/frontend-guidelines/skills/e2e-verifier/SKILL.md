# E2E 브라우저 검증

개발이 완료된 기능이 실제 브라우저에서 정상 동작하는지 검증한다.
PRD의 인수 조건(acceptance criteria)을 기준으로 검증 시나리오를 구성하고, 사용자가 선택한 브라우저 자동화 도구로 시나리오를 실행한다.

## 입력

- **PRD 인수 조건** — 검증 시나리오의 기준이 되는 acceptance criteria
- **변경 파일 목록** — `git diff`로 확인한 수정된 파일 목록 (검증 범위 파악용)
- **개발 서버 URL** — 사용자에게 확인 (예: `https://www.example.kr`)

## 실행 절차

### Phase 0: 브라우저 자동화 도구 선택

사용자에게 사용할 브라우저 자동화 도구를 선택하도록 안내한다:

```
E2E 브라우저 검증에 사용할 도구를 선택해주세요:

1. Chrome MCP — Chrome DevTools Protocol 기반. 이미 열려 있는 Chrome 브라우저를 제어합니다.
2. Playwright MCP — Playwright 기반. 헤드리스/헤드풀 브라우저를 자동으로 실행합니다.
3. Agent-Browser — AI 에이전트 특화 브라우저. 자연어 기반 브라우저 제어를 지원합니다.

번호로 선택해주세요.
```

선택된 도구에 따라 해당 reference 파일의 패턴을 따른다:

| 선택 | 참조 파일 | 특징 |
|------|----------|------|
| Chrome MCP | `references/chrome-mcp.md` | Chrome 디버깅 모드 필요, 가장 낮은 수준의 제어 가능 |
| Playwright MCP | `references/playwright-mcp.md` | 자체 브라우저 실행, 스냅샷 기반 접근성 트리 활용 |
| Agent-Browser | `references/agent-browser.md` | AI 에이전트 친화적, 자연어 기반 인터랙션 |

### Phase 0-1: 사전 조건 확인

선택된 도구의 reference 파일에 정의된 **사전 조건 확인 절차**를 따른다. 각 도구마다 필요한 MCP 설정, 브라우저 실행 방식, 연동 확인 방법이 다르므로 반드시 해당 reference를 읽고 따른다.

**공통 확인 사항:**
- `.mcp.json`에 선택된 도구의 MCP 서버 설정이 있는지 확인
- 설정이 없으면 reference 파일에 정의된 설정 안내를 사용자에게 전달
- 개발 서버 URL 확인:

```
개발 서버가 실행 중인 URL을 알려주세요. (예: https://www.example.kr)
E2E 검증을 건너뛰려면 "건너뛰기"라고 입력해주세요.
```

사용자가 "건너뛰기"를 선택하면 검증을 건너뛰고 결과에 SKIP으로 기록한다.

### Phase 1: 검증 시나리오 생성

PRD 인수 조건에서 브라우저로 검증 가능한 항목을 추출하여 시나리오로 변환한다.

**시나리오 변환 규칙:**

| 인수 조건 유형 | 시나리오 동작 | 검증 방법 |
|---------------|-------------|----------|
| "~가 표시된다" | 페이지 이동 → 해당 요소 확인 | 스크린샷 + DOM 요소 존재 확인 |
| "~를 클릭하면 ~한다" | 요소 클릭 → 결과 확인 | 클릭 → 대기 → 스크린샷 |
| "~를 입력하면 ~한다" | 입력 필드에 값 입력 → 결과 확인 | 입력 → 대기 → 스크린샷 |
| "~목록이 N개 표시된다" | 페이지 이동 → 목록 카운트 | DOM 요소 카운트 |
| "~로 이동한다" | 동작 수행 → URL 또는 화면 변화 확인 | 스크린샷으로 이동 결과 확인 |
| "에러 메시지가 표시된다" | 잘못된 입력 → 에러 메시지 확인 | 입력 → 스크린샷으로 에러 메시지 확인 |

**브라우저로 검증 불가능한 항목** (API 응답, DB 상태, 서버 로직 등)은 시나리오에서 제외하고 "자동 검증 불가 — 수동 확인 필요" 항목으로 분류한다.

변환된 시나리오를 사용자에게 보여주고 확인을 요청한다:

```markdown
## E2E 검증 시나리오

PRD 인수 조건에서 N개의 브라우저 검증 시나리오를 생성했습니다.
사용 도구: {선택된 도구명}

| # | 인수 조건 | 시나리오 동작 | 검증 방법 |
|---|----------|-------------|----------|
| 1 | ... | ... | ... |

### 자동 검증 불가 항목 (수동 확인 필요)
- [ ] ...

이 시나리오로 진행할까요? 수정이 필요하면 알려주세요.
```

### Phase 2: 브라우저 검증 실행

선택된 도구의 reference 파일에 정의된 **실행 패턴**을 따라 각 시나리오를 실행한다.

**도구 공통 핵심 원칙:**
- **URL 직접 이동은 최초 1회만** — 이후 모든 페이지 이동은 화면 UI를 통해 수행한다
- **URL을 추측하거나 직접 구성하지 않는다** — 실제 사용자의 브라우저 사용 방식을 따른다
- **각 시나리오 전후 스크린샷을 캡처한다** — before/after 증거를 남긴다

**각 시나리오 실행 흐름** (도구별 구체적 도구 호출은 reference 참조):

```
각 시나리오에 대해:
  1. 현재 상태 캡처 (before)
  2. 시나리오 동작 수행 (이동 / 클릭 / 입력 / 스크롤)
  3. 결과 대기
  4. 결과 상태 캡처 (after)
  5. 검증 수행 (요소 존재 / 텍스트 내용 / 요소 카운트 / 화면 상태)
  6. 결과 즉시 출력
```

**각 시나리오 완료 직후 결과를 즉시 출력한다:**

```
[시나리오 #1] 광고 성과 위젯이 대시보드에 표시된다
도구: {선택된 도구명}
결과: ✅ PASS
검증: 대시보드 페이지에서 광고 성과 위젯 요소가 존재함을 확인
```

```
[시나리오 #2] 기간 선택 드롭다운 클릭 시 옵션 목록이 표시된다
도구: {선택된 도구명}
결과: ❌ FAIL
검증: 드롭다운 클릭 후 옵션 목록 요소가 발견되지 않음
사유: 드롭다운 메뉴가 열리지 않음 — 스크린샷 참조
```

### Phase 3: 검증 결과 정리

모든 시나리오 실행 후 결과를 종합한다.

```markdown
## E2E 검증 결과

### 전체 요약
- 사용 도구: {선택된 도구명}
- 총 시나리오: N개
- 성공: N개 (N%)
- 실패: N개
- 건너뜀: N개

### 시나리오별 결과
| # | 시나리오 | 결과 | 비고 |
|---|---------|------|------|
| 1 | ... | ✅ PASS | ... |
| 2 | ... | ❌ FAIL | ... |

### 실패 항목 상세
(실패한 시나리오별 스크린샷과 실패 사유)

### 자동 검증 불가 항목 (수동 확인 필요)
- [ ] ...
```

**전체 PASS인 경우:**

```
E2E 검증을 통과했습니다. 모든 시나리오가 정상 동작합니다.
```

**FAIL 항목이 있는 경우:**

```
E2E 검증에서 N건의 실패가 발생했습니다.

어떻게 진행할까요?
1. 코드 수정 후 재검증 — 실패 원인을 분석하고 코드를 수정한 뒤 다시 검증
2. 무시하고 계속 — 실패 항목을 경고로 기록하고 다음 단계 진행
3. 중단 — 워크플로우를 중단하고 현재까지 결과로 리포트 생성
```

## 주의사항

- **URL 직접 이동은 최초 진입 시에만 사용한다.** 이후 모든 이동은 화면 UI를 통해 수행한다.
- **URL을 추측하거나 직접 구성하지 않는다.** 실제 사용자의 브라우저 사용 방식을 따른다.
- **PRD 인수 조건에 없는 시나리오를 임의로 추가하지 않는다.** 검증 범위는 인수 조건에 한정한다.
- **브라우저에서 검증 불가능한 항목은 명시적으로 분류한다.** "수동 확인 필요"로 표기하여 사용자가 인지하게 한다.
- **모든 메시지는 한국어 존댓말로 출력한다.**
- **각 Phase 완료 후 반드시 사용자에게 결과를 보여주고 승인/거절을 확인받은 뒤 다음 Phase로 진행한다.** 사용자 승인 없이 자동으로 다음 단계를 진행하지 않는다.
- **각 시나리오 결과는 완료 즉시 출력한다.** 전체 완료까지 기다리지 않는다.
- **도구별 상세 실행 패턴은 반드시 해당 reference 파일을 읽고 따른다.** SKILL.md의 흐름만으로 실행하지 않는다.
