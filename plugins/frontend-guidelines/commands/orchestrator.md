# Orchestrator — 프론트엔드 개발 전체 워크플로우 오케스트레이터

사용자의 요구사항을 받아 **분석 → PRD 작성 → 가이드라인 기반 개발 → E2E 검증 → 품질 검사**까지 전체 흐름을 커맨드로 순차 실행하고, 최종 결과를 통합하여 전달하는 커맨드.

## 왜 오케스트레이터가 필요한가

프론트엔드 개발은 요구사항 분석, 설계, 구현, 검증이 긴밀하게 연결된다. 각 단계를 독립된 커맨드로 분리하면 단계별 전문성을 살리면서도 체계적인 작업이 가능하다. 오케스트레이터는 이 커맨드들 사이에서 산출물을 전달하고 흐름을 제어하는 지휘자 역할을 한다.

## 실행 절차

### Phase 0: 요구사항 접수

사용자에게 개발하려는 기능이나 프로젝트에 대해 설명을 요청한다. 이미 대화 중 요구사항이 제시되었다면 이 단계를 건너뛴다.

```
프론트엔드 개발 전체 워크플로우를 시작합니다.

분석 → PRD 작성 → 가이드라인 기반 개발 순으로 진행됩니다.
각 단계는 독립 커맨드가 수행하고, 저는 결과를 통합하여 전달합니다.

개발하려는 기능이나 프로젝트에 대해 설명해주세요.
```

사용자의 요구사항 원문을 그대로 보존한다. 이후 각 커맨드에 전달할 때 원문을 포함해야 맥락이 유지된다.

### Phase 1: Research 커맨드 실행 — 요구사항 분석

`./commands/workflow/research.md`를 읽고 그 프로세스를 따라 요구사항 분석을 수행한다.

research 커맨드는 5단계(초기 분석 → grill-me 인터뷰 → 리서치 → 명세 작성 → 확인)로 구성된다. **각 단계를 완료할 때마다 결과를 사용자에게 보여주고, 승인을 받은 뒤 다음 단계로 진행한다.** 전체 질문이나 분석 결과를 한꺼번에 제시하지 않는다.

**단계별 진행 규칙:**

1. **초기 분석** — 요구사항을 분석한 결과에서 **명시된 요구사항, 암묵적 요구사항, 기술적 제약**만 사용자에게 보여주고, 수정·보완 의견을 확인한 뒤 다음으로 넘어간다. **모호한 영역 목록은 절대 보여주지 않는다** — 모호한 영역은 2단계 인터뷰에서 1개씩 질문으로 해소한다.
2. **grill-me 인터뷰** — 서브에이전트를 spawn하여 사용자에게 질문한다. 질문은 반드시 **한 번에 1개만** 제시하고, 사용자의 답변을 받은 뒤 다음 질문을 결정한다. 모호한 영역이 6개여도 첫 번째 질문 1개만 먼저 묻고, 답변을 받은 후 다음 질문을 결정한다. 질문 목록이나 모호한 영역 전체를 한번에 나열하는 것도 금지한다. 모든 질문이 끝나면 확정된 요구사항 목록을 보여주고 승인을 받는다.
3. **리서치** — 코드베이스 탐색 및 기술 조사 결과를 사용자에게 보여주고, 추가 조사가 필요한 영역이 없는지 확인한다.
4. **명세 작성** — 요구사항 명세 초안을 사용자에게 보여주고, 수정 요청이 있으면 반영한다. 필요시 2단계(인터뷰)로 돌아갈 수 있다.
5. **최종 확인** — 확정된 명세를 요약하여 보여주고, Phase 2 진행 여부를 확인한다.

**수집할 산출물:**

- 요구사항 분석 결과 (기능/비기능 요구사항 표, 기술적 제약, 결정 로그)

Phase 1 완료 시:

```
[Research 완료] 요구사항 분석이 완료되었습니다.

- 기능 요구사항: N건
- 비기능 요구사항: N건
- 기술적 제약: N건

다음 단계(PRD 작성)로 진행할까요?
```

### Phase 2: PRD 커맨드 실행 — 개발 요구사항 정의서 작성

`./commands/workflow/prd.md`를 읽고 그 프로세스를 따라 PRD를 작성한다. Phase 1에서 수집한 요구사항 명세 전문을 입력으로 사용한다.

PRD 커맨드는 내부적으로 planner, architecture, critic 각각을 서브에이전트로 spawn하여 루프를 돌린다.

**수집할 산출물:**

- 확정된 PRD (구현 계획, 기술 아키텍처, 리스크 대응, 인수 조건 포함)

커맨드 완료 후, PRD 핵심 내용을 사용자에게 요약하여 보여준다:

```
[PRD 완료] 개발 요구사항 정의서가 확정되었습니다.

- 구현 단계: N개 Phase
- 수정 대상 파일: N개
- 리스크: 상 N건 / 중 N건 / 하 N건
- PRD 저장 위치: .claude/plans/{기능명}-prd.md

다음 단계(가이드라인 기반 개발)로 진행합니다.
```

### Phase 3: Frontend Guidelines 커맨드 실행 — 가이드라인 기반 개발

**[필수] 스킬 선택 — 2단계 문의:**

Frontend Guidelines 실행 전, 사용자에게 **2단계로 나누어** 적용할 스킬을 확인한다. 각 단계에서 사용자의 응답을 받은 후에만 다음으로 진행한다.

**Step 1: 기본 제공 스킬 선택 (멀티 체크)**

```
[Phase 3 준비 — 1/2] 가이드라인 기반 개발을 시작하기 전에 적용할 스킬을 선택해주세요.

기본 제공 스킬 (4개):
  □ 1. a11y — 웹접근성 (WAI-ARIA, 키보드 내비게이션, 스크린리더)
  □ 2. semantic-html — 시맨틱 HTML (랜드마크, 제목 계층, 의미적 마크업)
  □ 3. seo-geo — SEO/GEO 최적화 (메타태그, 구조화 데이터, AI 검색 대응)
  □ 4. tdd — 테스트 주도 개발 (Red-Green-Refactor 루프)

적용할 스킬 번호를 선택해주세요.
- 복수 선택 가능 (예: 1, 3)
- 전체 선택: "전체" 또는 "1, 2, 3, 4"
```

사용자가 번호를 선택하면 해당 스킬만 적용 대상으로 확정한다. "전체", "all", "모두", "1, 2, 3, 4" 등 전체 선택 의사를 표현하면 4개 모두 적용한다. 반드시 1개 이상 선택해야 하며, 선택하지 않으면 다시 안내한다.

**Step 2: 추가 스킬 확인**

Step 1에서 사용자의 응답을 받은 **이후에** 추가 스킬 여부를 묻는다. Step 1과 동시에 묻지 않는다.

```
[Phase 3 준비 — 2/2] 선택하신 스킬: {선택된 스킬 목록}

위 스킬 외에 추가로 적용하고 싶은 스킬이 있으신가요?
- 있다면 스킬명을 알려주세요.
- 없다면 "없음"이라고 답변해주세요.
```

사용자의 응답을 확인한 후, 추가 스킬이 있으면 Step 1에서 선택한 스킬과 함께 포함하여 frontend-guidelines 커맨드에 전달한다. 사용자가 "없음"이라고 답변하면 Step 1에서 선택한 스킬만으로 진행한다. 사용자가 "없음", "없어", "없습니다", "괜찮습니다" 등 유사한 표현으로 답변해도 추가 스킬 없이 진행한다.

---

`./commands/workflow/frontend-guidelines.md`를 읽고 그 프로세스를 따라 가이드라인 기반 개발을 진행한다. Phase 1의 요구사항 명세 전문과 Phase 2의 확정된 PRD 전문을 입력으로 사용한다.

frontend-guidelines 커맨드는 내부적으로 선택된 스킬(Step 1에서 선택한 기본 스킬 + Step 2에서 추가한 스킬) 각각을 서브에이전트로 spawn하여 병렬 실행한 뒤 통합한다.

**수집할 산출물:**

- 적용된 가이드라인 목록 및 각 스킬별 점검 결과
- 코드 수정 내역
- 최종 점검 결과표

커맨드 완료 후, 산출물을 사용자에게 요약하여 보여주고 다음 단계 진행을 안내한다:

```
[Frontend Guidelines 완료] 가이드라인 기반 개발이 완료되었습니다.

- 적용된 스킬: N개
- 수정된 파일: N개
- 점검 결과: 통과 N건 / 경고 N건 / 실패 N건

다음 단계(E2E 브라우저 검증)로 진행합니다.
```

### Phase 4: E2E 브라우저 검증 — 사용자 승인 후 서브에이전트로 실행

**사용자 승인 요청:**

E2E 검증을 시작하기 전에 사용자에게 실행 여부를 먼저 확인한다:

```
[Phase 4] E2E 브라우저 검증 단계입니다.
개발된 기능이 실제 브라우저에서 정상 동작하는지 검증할 수 있습니다.

E2E 브라우저 검증을 진행할까요?
1. 진행 — 브라우저 자동화 도구로 PRD 인수 조건을 검증합니다
2. 건너뛰기 — 검증을 건너뛰고 다음 단계로 진행합니다
```

사용자가 **"건너뛰기"**를 선택하면 Phase 4를 SKIP으로 기록하고 Phase 5로 진행한다.

사용자가 **"진행"**을 선택하면 아래 프로세스를 실행한다.

---

`./commands/workflow/verifier.md`를 읽고 그 프로세스를 따라 E2E 검증을 수행한다.

verifier 커맨드는 내부적으로 e2e-verifier 스킬을 로드하여 PRD 인수 조건에서 검증 시나리오를 생성하고, 사용자가 선택한 브라우저 자동화 도구(Chrome MCP / Playwright MCP / Agent-Browser)로 실제 브라우저에서 시나리오를 실행한다.

**서브에이전트로 실행:**

Agent 도구를 사용하여 **E2E 검증 서브에이전트**를 spawn한다.

**서브에이전트에게 전달할 프롬프트 구성:**

```
1. Phase 2에서 확정된 PRD의 인수 조건 전문
2. Phase 3에서 변경된 파일 목록 (git diff --name-only)
3. 커맨드 경로: ./commands/workflow/verifier.md — 이 커맨드의 프로세스를 따를 것
4. 산출물 요청: 시나리오별 PASS/FAIL 결과, 실패 항목 상세, 성공률
```

**수집할 산출물:**

- 검증 시나리오 목록 및 각 시나리오별 PASS/FAIL 결과
- 실패 항목 상세 (스크린샷, 실패 사유)
- 자동 검증 불가 항목 목록 (수동 확인 필요)
- 전체 성공률

**실패 처리:**

E2E 검증에서 실패 항목이 있는 경우, 사용자에게 다음 선택지를 제시한다:

```
E2E 검증에서 N건의 실패가 발생했습니다.

1. 코드 수정 후 재검증 — Phase 3로 돌아가 실패 컨텍스트와 함께 코드 수정 후 Phase 4 재실행
2. 무시하고 계속 — 실패 항목을 경고로 기록하고 Phase 5(타입/빌드 검사)로 진행
3. 중단 — 현재까지 결과로 리포트 생성
```

사용자가 "코드 수정 후 재검증"을 선택하면 실패 시나리오의 상세 정보(실패 사유, 스크린샷)를 컨텍스트로 Phase 3에 전달하여 해당 부분만 수정한 뒤 Phase 4를 다시 실행한다.

커맨드 완료 후:

```
[E2E 검증 완료] 브라우저 검증이 완료되었습니다.

- 총 시나리오: N개
- 성공: N개 (N%)
- 실패: N개
- 건너뜀: N개

다음 단계(타입 검사)로 진행합니다.
```

### Phase 5: 타입 검사 및 빌드 검사

TypeScript 타입 검사와 빌드 검사를 Bash 도구로 직접 수행한다. 이 단계에서는 **타입 검사와 빌드만** 확인한다.

```
[Phase 5 시작] 타입 검사 및 빌드 검사를 실행합니다.
```

**Step 1: 프로젝트 설정 감지**

`package.json`과 `tsconfig.json`을 확인하여 실행 가능한 검사를 결정한다:

- `tsconfig.json` 존재 → TypeScript 타입 검사 가능
- `package.json`의 `scripts.build` 존재 → 빌드 검사 가능

**Step 2: TypeScript 타입 검사**

```bash
npx tsc --noEmit
```

또는 `package.json`에 `typecheck` 스크립트가 있으면 `npm run typecheck`를 사용한다. `tsconfig.json`이 없으면 건너뛴다.

**Step 3: 빌드 검사**

```bash
npm run build
```

`package.json`에 `build` 스크립트가 없으면 건너뛴다.

**수집할 산출물:**

- TypeScript 타입 검사 결과 (PASS/FAIL, 에러 수)
- 빌드 검사 결과 (PASS/FAIL)

**실패 처리:**

검사에서 실패 항목이 있는 경우, 사용자에게 다음 선택지를 제시한다:

```
타입/빌드 검사에서 문제가 발견되었습니다.

- TypeScript: PASS/FAIL (N errors)
- Build: PASS/FAIL

1. 수정 후 재검사 — 에러 내용을 분석하고 코드를 수정한 뒤 다시 검사
2. 무시하고 계속 — 실패 항목을 경고로 기록하고 Phase 6(리포트)로 진행
3. 중단 — 현재까지 결과로 리포트 생성
```

커맨드 완료 후:

```
[Phase 5 완료] 타입 검사 및 빌드 검사가 완료되었습니다.

- TypeScript: PASS/FAIL
- Build: PASS/FAIL

다음 단계(최종 통합 리포트)로 진행합니다.
```

## 단계 간 산출물 전달 규칙

각 커맨드는 순차적으로 실행되며, 이전 단계의 산출물을 다음 커맨드 실행 시 입력으로 **전문 포함**하여 전달해야 한다. 요약이나 발췌가 아닌 전문을 전달하는 이유는, 이전 맥락이 유지되어야 정확한 작업이 가능하기 때문이다.

- Phase 2 (PRD 커맨드) → Phase 1의 요구사항 명세 전문을 포함
- Phase 3 (Frontend Guidelines 커맨드) → Phase 1의 명세 + Phase 2의 PRD 전문을 포함
- Phase 4 (E2E 검증) → Phase 2의 PRD 인수 조건 + Phase 3의 변경 파일 목록을 포함
- Phase 5 (Quality Gate) → 별도 입력 불필요 (프로젝트 전체 검사)

산출물이 지나치게 긴 경우(토큰 제한 우려), 핵심 섹션을 선별하되 "기능 요구사항 표", "구현 계획", "수정 대상 파일", "인수 조건" 등 구현과 검증에 직접 필요한 정보는 반드시 포함한다.

## 사용자 소통

- **각 Phase 완료 후 반드시 사용자에게 결과를 보여주고 승인/거절을 확인받은 뒤 다음 Phase로 진행한다. 사용자 승인 없이 자동으로 다음 단계를 진행하지 않는다.**
- 각 Phase 시작 시: "N단계(커맨드명)를 시작합니다" 안내
- 각 Phase 완료 시: 핵심 산출물 요약 + "다음 단계로 진행할까요?" 확인 요청
- 커맨드 실행 중 사용자 입력이 필요한 경우: 해당 질문을 사용자에게 전달하고 응답을 반영한다
  - Phase 1: grill-me 인터뷰 질문
  - Phase 3: 가이드라인 스킬 선택
  - Phase 4: 개발 서버 URL 확인, 검증 시나리오 확인, 실패 시 처리 방법 선택
  - Phase 5: 실패 시 처리 방법 선택
- 문제 발생 시: 어떤 단계에서 무엇이 문제인지 명확히 전달하고, 재시도 여부를 사용자에게 확인한다

## 조기 종료

사용자가 특정 단계까지만 진행하길 원할 수 있다. 예를 들어 "분석과 PRD까지만 해줘"라고 하면 Phase 2까지만 실행하고 통합 리포트를 생성한다. 통합 리포트에는 실행된 단계만 포함한다.

Phase 4(E2E 검증)와 Phase 5(Quality Gate)는 각각 건너뛰기가 가능하다:

- Phase 4: 사용자가 "건너뛰기"를 선택하면 SKIP으로 기록하고 Phase 5로 진행
- Phase 5: 사용자가 건너뛰기를 원하면 SKIP으로 기록하고 Phase 6(리포트)로 진행
- "개발까지만 해줘"라고 하면 Phase 3까지 실행 후 Phase 4, 5를 건너뛰고 리포트를 생성한다
