# PRD — 개발 요구사항 정의서 작성 커맨드

planner, architecture, critic 세 스킬을 각각 **서브에이전트**로 spawn하여 **계획 → 설계 → 비평** 루프로 순환하며, 실행 가능한 수준의 개발 요구사항 정의서를 작성한다.

## 왜 루프가 필요한가

한 번에 완벽한 PRD는 나오지 않는다. 계획을 세우면 설계 과정에서 누락이 드러나고, 설계를 비평하면 계획의 전제가 흔들리기도 한다. 이 세 관점을 반복적으로 순환하면서 빈틈을 메워야 실행 가능한 정의서가 된다.

## 루프 구조

```
┌─────────────────────────────────────────────┐
│                                             │
│   [1. 계획] ──→ [2. 설계] ──→ [3. 비평]     │
│       ▲                          │          │
│       │                          │          │
│       └──── 피드백 반영 ◀────────┘          │
│                                             │
│   비평 통과 시 → [4. PRD 확정]              │
│                                             │
└─────────────────────────────────────────────┘
```

## 실행 절차

각 Phase는 Agent 도구를 사용하여 **독립적인 서브에이전트**를 spawn하여 실행한다. 이 커맨드는 루프 흐름을 제어하고, 각 서브에이전트의 산출물을 수집·전달·통합하는 역할을 맡는다.

### Phase 1: 계획 서브에이전트 (Planner)

Agent 도구를 사용하여 **계획 서브에이전트**를 spawn한다.

**서브에이전트에게 전달할 프롬프트 구성:**
```
1. 사용자의 요구사항 원문
2. 스킬 경로: ./skills/planner/SKILL.md — 이 스킬의 프로세스를 따를 것
3. 코드베이스 컨텍스트 (현재 프로젝트 경로, 관련 파일 정보)
4. 산출물 요청: 초안 PRD (문제 정의, 범위, 제약 조건, 성공 기준, 구현 단계, 의존성 그래프)
```

**이 서브에이전트의 목표:**
- 사용자 인터뷰를 통해 문제 정의, 범위, 제약 조건, 성공 기준을 확정한다
- 코드베이스를 탐색하여 현재 상태를 파악한다
- 구현 단계를 분해하고 의존성 그래프를 작성한다

**산출물:** 초안 PRD (구현 계획 + 인수 조건)

서브에이전트 완료 후, 산출물을 수집하여 다음 Phase에 전달한다.

### Phase 2: 설계 서브에이전트 (Architecture)

Agent 도구를 사용하여 **설계 서브에이전트**를 spawn한다.

**서브에이전트에게 전달할 프롬프트 구성:**
```
1. Phase 1에서 수집한 초안 PRD 전문
2. 스킬 경로: ./skills/architecture/SKILL.md — 이 스킬의 프로세스를 따를 것
3. 코드베이스 컨텍스트
4. 산출물 요청: 기술 아키텍처 설계서 + 초안 PRD 갱신 사항
```

**이 서브에이전트의 목표:**
- 시스템 구조, API, 데이터 모델, 컴포넌트 구성을 구체화한다
- 기술 선택에 대한 비교 분석과 선택 근거를 문서화한다
- 동시성/안전성 전략을 정의한다

**산출물:** 기술 아키텍처 설계서

서브에이전트 완료 후, 설계 결과를 초안 PRD에 반영하여 갱신한다.

### Phase 3: 비평 서브에이전트 (Critic)

Agent 도구를 사용하여 **비평 서브에이전트**를 spawn한다.

**서브에이전트에게 전달할 프롬프트 구성:**
```
1. 갱신된 초안 PRD 전문
2. Phase 2의 기술 아키텍처 설계서 전문
3. 스킬 경로: ./skills/critic/SKILL.md — 이 스킬의 프로세스를 따를 것
4. 산출물 요청: 비평 보고서 (리스크 매트릭스 + 핵심 권고사항 3가지)
5. (2회차 이상) 이전 비평에서 지적된 사항과 반영 내용
```

**이 서브에이전트의 목표:**
- 7가지 관점(확장성, 안정성, 보안, 성능, 호환성, 운영, 누락 기능)에서 약점을 식별한다
- 각 리스크에 심각도와 발생 가능성을 부여한다
- 구체적인 해결 방안을 제시한다

**산출물:** 비평 보고서 (리스크 매트릭스 + 핵심 권고사항 3가지)

### 루프 판정: 계속 vs 확정

비평 서브에이전트의 보고서를 수집한 뒤, 다음 행동을 결정한다:

**루프 계속 조건** (하나라도 해당하면 Phase 1로 복귀):
- 심각도 **상** 리스크가 1건 이상 존재
- 계획의 전제나 범위 자체를 수정해야 하는 피드백이 있음
- 아키텍처의 핵심 구조 변경이 필요함

**루프 확정 조건** (모두 충족해야 함):
- 심각도 **상** 리스크가 0건
- 남은 리스크가 모두 설계 내에서 해결 방안이 반영되어 있음
- 사용자가 비평 결과에 동의

루프를 반복할 때는 비평에서 지적된 사항을 각 서브에이전트 프롬프트에 포함하여 변경된 부분만 재작업하도록 한다. 전체를 처음부터 다시 하지 않는다.

**최대 3회 반복.** 3회 루프 후에도 심각도 상 리스크가 남아 있으면, 해당 리스크를 "알려진 제한사항"으로 명시하고 사용자에게 판단을 맡긴다.

### Phase 4: PRD 확정

루프를 통과하면 최종 PRD를 작성한다.

**최종 PRD 구조:**

```markdown
# [프로젝트/기능명] 개발 요구사항 정의서

## 1. 개요
- 목적
- 범위 (포함/제외)
- 성공 기준

## 2. 요구사항
### 2.1 기능 요구사항
| ID | 요구사항 | 우선순위 | 인수 조건 |
|----|----------|----------|----------|

### 2.2 비기능 요구사항
| ID | 요구사항 | 기준 |
|----|----------|------|

## 3. 기술 아키텍처
(Phase 2 설계서 요약 — 상세는 별도 문서 참조)
- 시스템 구조 다이어그램
- 핵심 기술 선택 및 근거
- 데이터 모델

## 4. 구현 계획
### Phase N: [단계명]
- 작업 내용
- 수정 파일
- 산출물
- 의존성

## 5. 수정 대상 파일
| 파일 | 변경 유형 | 설명 |
|------|----------|------|

## 6. 리스크 및 대응
| 리스크 | 심각도 | 대응 방안 |
|--------|--------|----------|

## 7. 인수 조건
- [ ] 조건 1
- [ ] 조건 2

## 8. 결정 로그
| 항목 | 결정 | 근거 | 대안 |
|------|------|------|------|

## 부록
- 루프 이력 (각 이터레이션에서 변경된 사항 요약)
- 알려진 제한사항
```

확정된 PRD는 `.claude/plans/{기능명-케밥케이스}-prd.md` 경로에 저장한다.

## 루프 중 사용자 소통

- 각 Phase가 끝날 때마다 핵심 산출물을 사용자에게 요약하여 보여준다
- 비평 결과는 리스크 매트릭스와 핵심 권고사항 3가지를 중심으로 간결하게 전달한다
- 루프를 반복할 때는 "이전 비평에서 N건의 주요 리스크가 발견되어, 계획/설계를 수정 후 재검증합니다"라고 안내한다
- 사용자가 "이 정도면 충분하다"고 판단하면 루프를 조기 종료할 수 있다
