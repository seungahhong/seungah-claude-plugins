# 전환 프레임 — 정의를 돕는 렌즈 (Transformation Frames)

목적조직·AI native·일하는 방식을 사용자가 **자기 맥락에서 정의하도록 돕는 렌즈** 모음. 각 렌즈에는 트레이드오프와 흔한 안티패턴을 붙였다. 근거 등급은 [research/](./research/README.md)를 따른다.

> **불변식**: 이 렌즈는 **정의를 돕는 도구**이지 정답 틀이 아니다. AI가 "당신 목적조직은 X입니다"라고 단정하지 않는다 — 렌즈를 선택지로 제시해 사용자가 고르고 수정하게 한다. 어떤 렌즈도 은탄환으로 제시하지 않는다.

---

## A. 목적조직 (target/product operating model) 렌즈

'목적조직'은 팀마다 뜻이 다르다. 사용자가 무엇을 뜻하는지 아래 축으로 되묻는다(정의는 사용자 몫).

- **재편 중심축** — 무엇을 중심으로 팀을 나눌 것인가? (기능별 / 미션·목표별 / 제품·고객 여정별 / 도메인 경계별)
- **자율·오너십 정도** — 팀이 스스로 우선순위·해법을 정하는가, 위에서 내려오는가? "완전한 가치 슬라이스"를 팀이 오너십하는가?
- **정렬 메커니즘** — 자율을 주면 정렬은 무엇으로 하나(비전·전략·지표)?

**근거(MEDIUM, 실무 컨센서스)**: 성공한 제품 운영 모델은 팀이 *"a complete slice of value"* 를 오너십하고 조직도가 아니라 *"real domain boundaries"* 에 맞춰 설계된다. 실패는 대개 모델 자체가 아니라 **주변 시스템(아키텍처·경계·펀딩·팀 설계)이 안 맞아서** 온다.

**트레이드오프**: 자율↑ → 정렬·표준화·규모의 경제↓. 도메인 경계 재설계는 아키텍처·펀딩까지 건드려야 하는 큰 변화다.

**안티패턴**:
- **이름만 바꾸기** — 팀 이름을 "미션팀"으로 바꾸고 실제 오너십·경계는 그대로.
- **조직도 리팩터링** — 도메인 경계 없이 상자만 옮겨 조율 비용이 오히려 증가.
- **시스템 무시** — 아키텍처·펀딩·의사결정을 안 바꾸고 팀 구조만 바꿔 실패("model is rarely the constraint, the system is").
- **빅뱅 전면 재편** — 한 번에 전사 재편(잦은 재편·우선순위 불안정은 성과·웰빙 해침, DORA).

## B. AI native 렌즈

AI 도입은 데이터상 **개인 체감과 시스템 성과가 갈린다**(각도 3, STRONG). "AI native = 자동 생산성"이 아니다.

- **적용 지점** — 어떤 업무에 먼저? (코드 생성 / 리뷰 / 문서 / 기획 / 운영 / 고객대응) 왜 거기부터?
- **깊이** — 도구 도입인가, 일하는 방식 재설계인가, 조직/역할 재편인가?
- **통제 기반** — AI 산출물을 누가 어떻게 검증하나? 테스트·버전관리·피드백 루프·내부 플랫폼이 준비됐나?
- **성공 신호** — 무엇으로 성공을 볼 것인가(관찰형: 채택률·특정 업무 시간·품질 신호)?

**근거(STRONG·조건부, DORA)**: AI는 **조건을 증폭**한다 — 느슨한 결합+빠른 피드백 팀은 이득, 강한 결합+느린 프로세스 팀은 이득 거의 없음. 처리량·제품성과는 +지만 **delivery 안정성은 −**. 내부 플랫폼 품질이 AI 가치와 직접 상관. AI 코드 신뢰는 낮음(30~39%가 거의 불신).

**트레이드오프**: 속도 가속이 다운스트림 약점(부족한 테스트·강결합)을 노출. 도입·학습 비용, 초기 안정성 저하 가능.

**안티패턴**:
- **은탄환 기대** — "AI 넣으면 2배 빨라진다"(체감 ≠ 시스템 성과).
- **통제 없는 가속** — 테스트·버전관리·피드백 없이 AI만 도입해 안정성 악화.
- **전수 수기 검증** — 반대로 모든 AI 산출물을 수기 전수 검증해 이득 상쇄(충돌 신호).
- **"95% 실패" 같은 유행 수치 인용** — 반박된 마케팅 수치(research/ REFUTED). 방향성만 취한다.

## C. 스프린트/일하는 방식 렌즈

- **반복 주기** — 스프린트 도입/주기 변경/폐지? 계획·리뷰·회고를 어떻게?
- **산출물 관리** — 문서·결과물을 어디서 어떻게 만들고 관리·공유? (single source of truth?)
- **흐름 신호** — 개선하려는 관찰 지표(리드타임·WIP·배포 빈도)?

**근거(STRONG 게이팅 / MEDIUM 관행, DORA)**: 프로세스 개선은 *small batch sizes·robust testing* 같은 **기본기 없이는 delivery를 자동 개선하지 않는다**. 병목은 코드 생성이 아니라 프로덕션 투입. WIP 제한·flow metrics는 흐름을 드러내는 유용한 실무 도구(실증 인과는 약함).

**트레이드오프**: 더 짧은 주기·더 많은 정례 = 오버헤드. 지표를 목표로 삼으면 조작(Goodhart).

**안티패턴**:
- **의식만 도입** — 스탠드업·회고 형식만 들여오고 기본기(배치 크기·테스트)는 그대로.
- **지표 목표화** — 리드타임·속도를 목표로 걸어 숫자를 조작.
- **우선순위 불안정** — 잦은 방향 전환(성과·웰빙 해침, DORA).

## 렌즈 사용 절차 (vision-elicitor·gap-planner가 따른다)

1. 사용자가 방향(목적조직/AI native/…)을 말하면, 해당 렌즈의 **축을 선택지로** 제시해 정의를 되묻는다(임의 정의 금지).
2. 목표 상태를 **관측 가능하게** 잡고 **동인(왜)** 을 함께 캔다.
3. 모든 목표에 **트레이드오프 + 이 팀이 빠질 안티패턴**을 붙인다.
4. 은탄환 기대가 보이면 research/ 근거로 교정한다(체감 ≠ 성과, 조건부 이득).
5. 수치는 출처·등급과 함께 관찰값으로만. REFUTED 수치는 쓰지 않는다.
