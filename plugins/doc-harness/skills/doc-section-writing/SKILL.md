# 문서 섹션 작성

좋은 기술 문서는 변경을 처음 보는 동료가 한 번 읽고 이해하게 만든다. 각 섹션은 정해진 질문에 답해야 하며, diff를 나열하는 게 아니라 **의미를 설명**한다. 근거는 항상 `_workspace/context.md`/`meta.json`에서 끌어오고, 없는 사실은 지어내지 않는다.

## 문서 전체 목차 (고정 순서)

조립 시 항상 이 순서를 따른다. 검증도 이 목차를 기준으로 한다.

1. **요약** (조립자가 마지막에 생성)
2. **배경** — `section_background.md`
3. **목차** (조립자가 헤딩에서 자동 생성)
4. **기술스펙** — `section_techspec.md`
5. **변경사항** — `section_changes.md`
6. **테스트** — `section_tests.md`
7. **영향·리스크** — `section_impact.md`

각 작성자는 자기 `section_id` 하나만 만든다. 파일은 단일 H2 헤딩(`## 제목`)으로 시작한다.

## 섹션 카드

각 카드는 **답해야 할 질문 / 필수 요소 / 흔한 실패**를 정의한다.

### `background` — 배경

- **질문:** 왜 이 작업이 필요했는가? 어떤 문제·요구를 해결하나?
- **필수:** 문제 상황 또는 동기, 이 변경의 목표, (있으면) 관련 이슈/PR 링크.
- **근거:** 커밋 메시지·이슈 번호·PR 제목·코드 주석에서 동기를 추론한다.
- **실패:** "기능을 추가했다"처럼 변경 자체를 반복하는 것. 배경은 변경의 *이유*다.

### `techspec` — 기술스펙

- **질문:** 무엇을 어떻게 만들었는가? 구조와 인터페이스는?
- **필수:** 주요 모듈/컴포넌트와 역할, 데이터 흐름, **공개 인터페이스 변경**(함수 시그니처·API 라우트·환경변수·스키마·설정 키), 추가/변경된 의존성. 구조가 복잡하면 간단한 다이어그램(텍스트/mermaid)이나 표를 쓴다.
- **근거:** context.md의 "기술스펙 단서"를 우선 사용한다.
- **실패:** 파일을 그냥 나열하는 것. 모듈 간 *관계*와 *결정*을 설명한다.

### `changes` — 변경사항

- **질문:** 구체적으로 무엇이 바뀌었는가?
- **필수:** 의미 단위(기능/모듈)로 묶은 변경 목록 + 변경 성격 태그(feat/fix/refactor/...), 신규/수정/삭제 구분, **호환성 깨짐(breaking)** 명시. 큰 변경은 표로 정리한다.
- **근거:** context.md의 "변경사항 목록"을 정제한다.
- **실패:** `git diff` 라인 복붙. 사람이 읽을 요약으로 압축한다.

### `tests` — 테스트

- **질문:** 이 변경을 어떻게 검증했고, 어떻게 재현하나?
- **필수:** 추가/변경된 테스트와 커버하는 케이스, **테스트 실행 명령**, (있으면) 커버리지 수치/도구, 자동화로 못 잡는 수동 검증 시나리오.
- **근거:** context.md의 "테스트 현황".
- **실패:** "테스트 추가함"으로 끝내는 것. 무엇을·어떻게 검증하는지 적는다. 테스트가 없으면 "테스트 없음"을 명시하고 권장 검증 항목을 제안한다.

### `impact` — 영향·리스크

- **질문:** 배포하면 무엇이 달라지고 무엇을 주의해야 하나?
- **필수:** 동작 변화·마이그레이션 필요 여부·성능/보안 영향·롤백 방법·후속 작업. 영향이 없으면 "없음"으로 명시한다.
- **실패:** 형식적으로 "영향 없음"만 적는 것. breaking change·환경변수 추가 등은 반드시 짚는다.

## 공통 작성 규칙

- **명령형·간결체**로 쓴다. 군더더기 수식어를 뺀다.
- **근거 우선** — 컨텍스트에 없으면 "확인 필요"로 표기하고 추측하지 않는다.
- **중복 금지** — 다른 섹션이 책임지는 내용을 다시 쓰지 않는다.
- **코드/경로는 인라인 코드**로, 명령은 코드블록으로 표기한다.
- 근거가 부족해 채우지 못한 부분은 파일 끝에 `## 보강 필요` 메모로 남긴다.

## 재작성(피드백 반영)

`_workspace/grading.json`에서 자기 `section_id`의 `passed:false` 항목만 보강한다. 통과 항목은 건드리지 않는다. 미달 사유(`evidence`)가 "내용 부실"이면 근거를 더 끌어와 깊이를 더하고, "누락"이면 빠진 필수 요소를 추가한다.
