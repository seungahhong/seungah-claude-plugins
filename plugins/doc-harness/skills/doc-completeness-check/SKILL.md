# 문서 완성도 검증

검증은 생성-검증 루프의 품질 게이트다. 통과 기준이 느슨하면 빈 껍데기 문서가 통과하고, 너무 엄격하면 무한 루프에 빠진다. 그래서 **필수 목차 존재 → 섹션별 충실도 → 전체 정합성** 3단계로 채점하고, 각 판정에 문서 내 증거를 단다.

검증자는 `draft.md`만 본다. 작성 의도나 컨텍스트 원본을 받으면 같은 누락을 답습하므로, 오직 결과물의 완성도만 평가한다. 문서를 직접 고치지 않는다 — 미달 항목과 지시만 남긴다.

## 채점 절차

### 1단계 — 필수 목차 존재 (게이트)

다음 섹션이 모두 존재하고 비어 있지 않은지 확인한다. 하나라도 없으면 그 항목은 즉시 `passed:false`이고 전체 `verdict`는 `revise`다.

| section_id | 섹션 | 최소 통과 조건 |
| ---------- | ---- | ------------- |
| `summary` | 요약 | 3줄 이상, 무엇/왜/영향 중 2개 이상 포함 |
| `background` | 배경 | 변경의 *이유*가 1문장 이상 (변경 자체 반복은 미달) |
| `toc` | 목차 | 본문 헤딩과 일치하는 목차 + 앵커 |
| `techspec` | 기술스펙 | 모듈/구조 + 인터페이스·의존성 변경 중 1개 이상 명시 |
| `changes` | 변경사항 | 의미 단위 변경 목록 (diff 복붙은 미달) |
| `tests` | 테스트 | 테스트 내용 또는 "없음" 명시 + 실행/검증 방법 |
| `impact` | 영향·리스크 | 영향 서술 또는 "없음" 명시 |

### 2단계 — 섹션별 충실도

존재하는 섹션이 제 역할을 실제로 하는지 본다. 형식적 한 줄은 미달이다. 상세 루브릭과 점수화 방법은 `references/rubric.md` 참조.

### 3단계 — 전체 정합성

- 요약이 본문과 모순되지 않는가.
- 목차의 앵커가 실제 헤딩으로 연결되는가(dead link 없음).
- 섹션 간 사실(파일 수, 변경 성격)이 서로 어긋나지 않는가.
- 메타 머리말(브랜치/base/커밋 범위)이 채워져 있는가.

## 출력 — `_workspace/grading.json`

필드명을 변형하지 않는다(`text`/`passed`/`evidence` 정확히 사용).

```json
{
  "expectations": [
    { "section_id": "background", "text": "배경이 변경의 이유를 설명한다", "passed": true, "evidence": "'기존 토큰 만료 처리 부재로...' (배경 1문단)" },
    { "section_id": "tests", "text": "테스트 실행 방법이 있다", "passed": false, "evidence": "테스트 섹션에 실행 명령 없음" }
  ],
  "summary": { "passed": 9, "failed": 1, "total": 10, "pass_rate": 0.9 },
  "verdict": "revise"
}
```

## verdict 판정 규칙

- `verdict: "pass"` — 필수 목차(1단계) 전부 통과 **그리고** `pass_rate ≥ 0.9`.
- `verdict: "revise"` — 그 외. 미달 항목마다 `section_id`를 달아 작성자가 어디를 고칠지 알게 한다.

## 루프 종료 가이드

오케스트레이터는 최대 2회까지 재시도한다. 2회 후에도 `revise`면, 남은 미달 항목을 문서 말미 "알려진 한계" 또는 보고서에 명시하고 사용자에게 넘긴다. 무한 보강을 강요하지 않는다 — 근거가 코드에 없어 못 채우는 항목도 있기 때문이다.
