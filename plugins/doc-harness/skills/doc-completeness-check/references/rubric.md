# 상세 채점 루브릭

`doc-completeness-check` 2단계(섹션별 충실도)의 보충 자료. 각 섹션을 충실도 기준으로 채점하는 세부 항목과 자주 나오는 미달 사유를 모았다.

## 1. 충실도 채점 방식

각 섹션마다 1~3개의 `expectations` 항목을 만든다. 한 항목 = 한 가지 검증 가능한 기대치다. "좋은 문서다" 같은 주관적 항목은 금지하고, **존재/포함/일치** 같은 판정 가능한 형태로 쓴다.

각 항목은 `passed: true|false`와 `evidence`(문서 내 인용 또는 위치)를 갖는다. 증거 없이 통과시키지 않는다.

---

## 2. 섹션별 세부 기준

### 요약 (summary)
- [ ] 3줄 이상이며 변경의 핵심(무엇/왜/영향)을 압축한다.
- [ ] 본문에 없는 새 주장을 만들지 않는다.

### 배경 (background)
- [ ] 변경의 *이유/동기*를 설명한다(문제 상황, 요구, 기술 부채 등).
- [ ] 변경 내용 자체의 반복이 아니다.

### 기술스펙 (techspec)
- [ ] 주요 모듈/컴포넌트와 역할을 설명한다.
- [ ] 공개 인터페이스 변경(시그니처/라우트/환경변수/스키마/설정) 또는 의존성 변경을 명시한다.
- [ ] 단순 파일 나열이 아니라 구조·관계·결정을 설명한다.

### 변경사항 (changes)
- [ ] 의미 단위(기능/모듈)로 묶여 있다.
- [ ] 변경 성격 태그(feat/fix/refactor/...)와 신규/수정/삭제 구분이 있다.
- [ ] breaking change가 있으면 명시한다.
- [ ] `git diff` 원문 복붙이 아니다.

### 테스트 (tests)
- [ ] 추가/변경된 테스트와 커버 케이스를 설명하거나, 테스트가 없으면 "없음"을 명시한다.
- [ ] 테스트 실행 명령 또는 수동 검증 시나리오가 있다.

### 영향·리스크 (impact)
- [ ] 동작 변화/마이그레이션/성능/보안/롤백 중 해당 항목을 다루거나 "없음"을 명시한다.
- [ ] breaking change·환경변수 추가 등 놓치면 위험한 항목을 짚는다.

---

## 3. pass_rate 계산

```
pass_rate = passed / total   (소수 둘째 자리 반올림)
```

- `total` = 전체 `expectations` 수(1단계 게이트 항목 + 2단계 충실도 항목).
- 필수 목차(1단계) 항목은 가중치를 따로 두지 않되, 하나라도 미달이면 `pass_rate`와 무관하게 `verdict:"revise"`다(게이트 우선).

---

## 4. 흔한 미달 사유 → 개선 지시

`evidence`에 사유만 적지 말고, 작성자가 바로 고칠 수 있게 개선 방향을 함께 적는다.

| 미달 사유 | 개선 지시 예시 |
| -------- | ------------- |
| 배경이 변경 반복 | "왜 필요했는지(문제/요구)를 커밋 메시지·이슈에서 찾아 1문단 추가" |
| 기술스펙이 파일 나열 | "모듈 간 호출 관계와 공개 인터페이스 변경을 표로 정리" |
| 변경사항이 diff 복붙 | "기능 단위로 묶고 각 항목에 태그(feat/fix) 부여" |
| 테스트에 실행법 없음 | "package.json scripts 또는 README에서 테스트 실행 명령 확인 후 추가" |
| 영향이 형식적 '없음' | "breaking change·환경변수·마이그레이션 여부를 변경사항과 대조해 재확인" |
| 목차 dead link | "본문 헤딩 텍스트에서 앵커를 재생성" |
