# 다각도 문진 질문뱅크 (개발 · 회사 · 사업 3축)

grill-me 인터뷰 전략을 개발 방법론 선택에 특화한 질문뱅크다. **한 번에 한 질문**, 번호 선택지 제시, 직전 답이 다음 질문을 결정. 이 문서는 소진용 체크리스트가 아니라 **분기 지도**다 — 이미 안 것은 건너뛴다.

## 현행 진단 (Phase 0)

첫 질문은 **개괄**이다(아는 축부터 편하게). 그다음 축별로 빈 곳을 한 번에 하나씩 관측한다. **판단하지 말고 관측만** 한다.

- **개괄** — "지금 팀은 개발/회사/사업 측면에서 프로세스가 대략 어떻게 구성되어 있나요? 아는 만큼만요."
- **개발 현행** — 반복 주기가 있나요(스프린트 N주 / 연속 흐름 / 정해진 리듬 없음)? / 계획·리뷰·회고 같은 정기 미팅이 있나요? / 배포는 얼마나 자주, 얼마나 자동화돼 있나요? / 테스트·코드리뷰·CI가 있나요?
- **회사 현행** — 팀은 몇 명, 몇 개 팀인가요? / 원격/분산인가요? / 우선순위는 누가 정하나요? / 이 방식으로 일한 지 얼마나 됐나요?
- **사업 현행** — 무슨 사업/제품인가요? / 고객은 누구(내부/B2B/B2C)인가요? / 장애가 나면 대가가 큰 편인가요(규제·안전·돈)? / 릴리스 마감/계약 압력이 있나요?
- **페인** — 지금 가장 답답하거나 병목인 지점은 무엇인가요?

관측되지 않은 항목은 발명하지 말고 "확인 필요"로 남긴다.

## 개발 축

방법론 선택을 가장 크게 가르는 인자는 대개 **요구 불확실성**과 **피드백 주기**다. 여기부터 판다.

- **요구 불확실성** — 만들 것이 얼마나 확정돼 있나요?
  1. 대체로 고정(스펙이 명확, 잘 안 바뀜) 2. 반반(큰 틀은 있고 세부는 바뀜) 3. 자주 뒤집힘(탐색·PMF 단계)
  → *고정↔Waterfall/계획주도 쪽, 자주 바뀜↔반복·경험주도(Scrum/XP/Shape Up) 쪽*
- **피드백/반복 주기** — 만든 것을 사용자·이해관계자에게 얼마나 자주 검증받나요? (매주 / 매 스프린트 / 분기 / 릴리스 때만)
- **배포 빈도·자동화** — 프로덕션 배포 빈도와 자동화 수준은? (수동·드묾 / CI는 있음 / CD·트렁크 기반 잦은 배포)
  → *DevOps/CD 적합도. "자주 배포하고 싶다"면 자동화 역량이 전제인지 확인.*
- **품질/테스트 관행** — 자동화 테스트·코드리뷰·페어링·TDD가 있나요? (XP류 적합도)
- **기술 리스크·미지수** — 아키텍처·기술에 큰 미지수가 있나요? (스파이크·프로토타입 필요성)
- **작업 흐름 성격** — 계획 가능한 기능 개발인가요, 예측 불가한 인입(지원·운영·인시던트)인가요?
  → *인입·운영성↔Kanban(흐름·WIP), 계획형↔Scrum(스프린트)*

## 회사 축

- **팀 규모·팀 수** — 한 팀 몇 명? 여러 팀이 한 제품을 같이 만드나요?
  → *단일 소규모↔경량(Kanban/Scrum/Shape Up). 다수 팀 조율↔스케일링(LeSS/Nexus/SAFe) 검토(단, 규모 오버엔지니어링 경계).*
- **분산/집중** — 같은 시간대·같은 공간인가요, 분산·비동기인가요? (문서화·비동기 의식 필요성)
- **의사결정 구조** — 우선순위·범위를 누가 정하나요? PO/책임자가 상시 가용한가요? (Scrum PO 전제 확인)
- **조직 성숙도·이전 경험** — 애자일/프로세스를 해본 적 있나요? 실패한 적은? (성숙도 낮으면 의식 과부하 주의)
- **문화(자율 vs 통제)** — 팀이 스스로 방식을 정할 자율이 있나요, 위에서 지시받나요? (자기조직화 전제 확인)

## 사업 축

- **사업/성장 단계** — 스타트업(탐색)·스케일업·엔터프라이즈 중 어디인가요? (자금·런웨이 압력)
  → *탐색(pre-PMF)↔가벼운 반복·Kanban(Shape Up은 PMF 이후·부적합). 성숙·규제↔게이트·규율 강화.*
- **시장/요구 불확실성** — 시장이 검증됐나요, 아직 찾는 중인가요? (PMF 전/후)
- **criticality(장애의 대가)** — 장애가 나면 대가가 얼마나 큰가요? 1. 낮음(재시도 가능) 2. 중간(매출·평판) 3. 높음(안전·규제·금전 손실·인명)
  → *높을수록 규율·검증·문서(계획주도·V-Model·게이트) 비중↑ (Boehm-Turner criticality).*
- **규제·계약 의무** — 감사·인증·규제 릴리스 절차, 고정 범위/일정 계약이 있나요? (반복·잦은 배포와 충돌 가능)
- **릴리스 압력·케이던스** — 정해진 출시일/마감이 있나요? 얼마나 자주 내보내야 하나요?
- **고객 유형** — 내부/B2B(계약·SLA)/B2C(빠른 실험)인가요?

## 후속 질문 깊이 패턴

답이 나오면 한 단계 더 판다(grill-me 계승).

- **"왜" 파고들기** — "칸반을 원하신다고 했는데, 흐름 예측 불가 때문인가요, 스프린트 의식이 부담이어서인가요?"
- **반례 확인** — "지금은 5명이지만 6개월 뒤 15명이 돼도 그 방식이 성립할까요?"
- **숨은 가정 드러내기** — "잦은 배포를 원하시는데, 자동화 테스트·롤백이 갖춰져 있다고 전제해도 될까요?"
- **범위 확인** — "이 방법론 변경이 QA·운영 팀에도 영향을 주는데, 함께 고려할까요?"

## 충돌 감지 쌍

아래가 보이면 즉시 알리고 **우선순위를 묻는다**(둘 다 만족은 불가능할 수 있음을 명시).

| 충돌 | 왜 충돌인가 |
|------|-------------|
| 규제 감사 릴리스 + 하루 여러 배포 | 케이던스·게이트가 배포 빈도를 제약 |
| 요구 매주 변경 + 고정 스코프/일정 계약 | 경험주도 반복과 계획주도 계약의 상충 |
| 소규모 단일 팀 + 대규모 스케일링 프레임워크(SAFe) | 규모 오버엔지니어링(의식 과부하) |
| 높은 criticality + 최소 문서/게이트 | 장애 대가 대비 검증 부족 |
| 자기조직화 방법론(Scrum) + 위에서 지시하는 통제 문화 | 자율 전제와 통제 현실의 상충 |
| 잦은 배포 목표 + 수동 배포·테스트 부재 | CD의 자동화 전제 미충족 |
