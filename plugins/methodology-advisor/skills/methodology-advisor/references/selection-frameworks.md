# 맥락 적합(컨틴전시) 선택 프레임워크

방법론 제안을 **직관이 아니라 근거**로 하기 위한 사고 도구. 결정 인자(개발/회사/사업 3축)를 이 프레임워크에 위치시켜 후보군을 좁힌다. 핵심 메시지는 하나다 — **최선의 방법론은 없다. 맥락에 맞는 방법론이 있을 뿐이다(contingency).**

## 1. Cynefin (복잡성 감지)

Dave Snowden의 sense-making 프레임워크. 문제가 어느 **도메인**에 있는지에 따라 접근이 달라진다.

| 도메인 | 성격 | 접근 | 방법론 함의 |
|--------|------|------|-------------|
| Clear/Obvious | 인과가 자명, 정해진 답 | sense–categorize–respond | 정해진 절차·체크리스트(계획주도로 충분) |
| Complicated | 전문성으로 분석 가능(답이 있음) | sense–analyze–respond | 설계·전문가 주도(계획주도 + 리뷰) |
| **Complex** | 사후에야 인과가 보임(창발) | **probe–sense–respond** | **경험주도(반복·실험)** — 대부분의 신규 SW 개발 |
| Chaotic | 인과 불명, 즉각 행동 필요 | act–sense–respond | 안정화 우선(인시던트 대응) |
| Confusion/Disorder | 어느 도메인인지 모름 | 먼저 분류 | — |

**함의** — 신규 제품·기능 개발은 대개 **Complex**(무엇을 만들지조차 탐색). 그래서 짧은 반복으로 probe하는 애자일이 적합. 반면 잘 정의된·반복적인 작업은 Complicated/Clear라 계획주도가 낭비가 아니다. **한 조직 안에도 도메인이 섞여** 있으므로 하이브리드가 정당하다.

## 2. Stacey matrix (두 불확실성 축)

**요구 확실성(무엇을 만들지)** × **기술 확실성(어떻게 만들지)** 두 축으로 접근을 가른다.

- 둘 다 확실(합의 근접) → 단순/계획주도.
- 한 축이라도 불확실 → 복잡(complex) → 경험주도(반복·실험).
- 둘 다 매우 불확실 → 혼돈(chaos) → 탐색·안정화 우선.

> **주의(정직성)** — 애자일 커뮤니티가 대중화한 단순 "Stacey diagram"은 Ralph Stacey 본인이 후에 그 단순화된 규범적 사용과 거리를 뒀다. **거친 대화 도구**로만 쓰고 결정론적 지도처럼 과신하지 않는다.

## 3. Boehm & Turner — home ground · risk-based balancing

『Balancing Agility and Discipline』(2003). 애자일과 규율(계획주도)은 배타가 아니라 **스펙트럼**이며, 프로젝트를 5개 인자로 평가해 어느 쪽에 가까운지 판단한다.

**home ground 5인자** (레이더 차트로 시각화):

| 인자 | 애자일에 유리 | 계획주도에 유리 |
|------|---------------|-----------------|
| **Size(규모)** | 소규모 팀 | 대규모 팀 |
| **Criticality(장애 대가)** | 낮음(손실 감당 가능) | 높음(안전·다수 인명·큰 금전) |
| **Dynamism(변경 빈도)** | 요구가 자주 변함 | 요구가 안정적 |
| **Personnel(인력 숙련)** | 숙련·자율 인력 비중 높음 | 주니어 비중 높음(구조 필요) |
| **Culture(문화)** | 자유·혼돈 감내(권한 위임) | 질서·역할 명확 선호 |

5인자가 **한쪽으로 몰리면** 그 방법론이 home ground. **섞여 있으면**(흔함) → **하이브리드를 재단**한다.

**risk-based balancing(위험 기반 균형) — 5단계**(1차 IEEE Computer 2003 확인. *주의: 흔히 "4단계"로 잘못 인용되는데, 그것은 책의 예시 하위단계를 오독한 것이다 — 실제는 5단계이며 Step 3·5가 이 방법의 핵심 통찰이다. 근거: [research/README.md](./research/README.md) 교정 항목*):

1. **위험 분석** — 세 범주(환경적·애자일·계획주도)의 위험을 각각 평가.
2. **home ground 판정** — 프로젝트가 순수 애자일 또는 순수 계획주도 home ground에 해당하는지 평가.
3. **하이브리드 아키텍처(핵심)** — 명확한 home ground가 없으면(흔함), **애자일 강점이 통하는 부분엔 애자일, 나머지 기본엔 계획주도**로 아키텍처를 분할한다.
4. **통합 위험해소 전략** — 개별 위험해소책을 하나의 전략으로 통합.
5. **지속 모니터링·재조정(핵심)** — 진행하며 애자일/규율 균형을 계속 조정한다.

핵심은 "위험을 비교해 재단하고(Step 3), 진행하며 균형을 재조정한다(Step 5)"는 **동적 컨틴전시** 사고다 — 방법론 선택은 1회성 결정이 아니라 지속 재조정 대상이다.

## 4. 컨틴전시 축 → 방법론 매핑 (요약표)

결정 인자를 방법론 후보로 잇는 **휴리스틱**(결정론적 규칙이 아니다 — 교차로 판단하고 하이브리드를 우선 고려).

| 결정 인자 | 값 | 기우는 방향 |
|-----------|----|-------------|
| 요구 불확실성 | 높음 | 경험주도(Scrum/XP/Shape Up), 반복·피드백 |
| 요구 불확실성 | 낮음·고정 | 계획주도(Waterfall/V-Model) |
| 작업 흐름 | 예측 불가 인입(운영·지원) | Kanban(흐름·WIP·pull) |
| 작업 흐름 | 계획형 기능 개발 | Scrum(스프린트) / Shape Up(사이클) |
| criticality | 높음(안전·규제) | 계획주도·게이트·V-Model·강한 검증, CD는 게이트와 함께 |
| criticality | 낮음 | 경량 반복, 빠른 실험 |
| 팀 규모/수 | 소규모 단일 | 경량(Kanban/Scrum/Shape Up/XP) |
| 팀 규모/수 | 다수 팀·한 제품 | 스케일링(LeSS/Nexus/SAFe) — **단, 규모 오버엔지니어링 경계** |
| 배포 목표 | 자주·안전 배포 | DevOps/CD/DORA(자동화·테스트 전제 확인) |
| 품질/회귀 리스크 | 높음 | XP(TDD·페어·CI) |
| 조직 문화 | 자율·신뢰 | Shape Up/Scrum(자기조직화) |
| 조직 문화 | 통제·지시 | 경량 Kanban(최소 침습) — 자기조직화 전제 방법론은 마찰 |
| 인력 숙련·자율(Personnel) | 숙련·자율 비중 높음 | 애자일(자기조직화·XP 기술관행 감당) |
| 인력 숙련·자율(Personnel) | 주니어 비중 높음 | 구조·문서 강한 계획주도, 페어링/리뷰로 보완 |
| 거버넌스 요구 | 애자일 + 고정 예산/감사 | DSDM / 하이브리드(게이트 + 반복) |
| 프로세스 낭비·핸드오프 | 큼 | Lean(가치흐름) · Kanban |
| 성장 단계 | 스타트업·PMF 탐색 | Kanban 등 경량·짧은 반복, 실험 우선 (Shape Up은 PMF 이후·부적합) |
| 성장 단계 | 스케일업·엔터프라이즈 | 규율·스케일링·거버넌스 비중↑ |

**교차 규칙** — 단일 인자로 결정하지 않는다. 예: "요구 불확실성 높음"만 보면 Scrum이지만, "예측 불가 인입 + 통제 문화"가 겹치면 Kanban이 더 적합. 대개 **criticality × 요구 불확실성 × 팀 규모**가 1차 갈림.

## 5. 매칭 절차

1. **위치 지정** — 결정 인자를 Cynefin(도메인) · Stacey(두 축) · home ground(5인자)에 올린다.
2. **후보 추출** — 위 매핑표 + [methodology-catalog.md](./methodology-catalog.md)의 적합 조건과 대조해 2~3 후보.
3. **하이브리드 검토** — 5인자·도메인이 섞였으면(대부분) 순수 방법론보다 조합을 우선(예: 게이트 + CD, Scrumban).
4. **안티패턴 표시** — 각 후보에서 이 팀이 빠질 오적용을 카탈로그에서 가져와 붙인다.
5. **additive 우선** — 전면 전환보다 되돌리기 쉬운 최소 관행부터(WIP 한계·회고·CI 게이트 등).
6. **재평가 트리거** — 인자가 바뀌면(규모 2배·규제 진입·PMF 도달) 다시 위치 지정.

**정직성** — 이 프레임워크들은 *대화·판단 도구*이지 결정론적 계산기가 아니다. 근거로 인용하되 "프레임워크가 X라고 했으니 무조건 X"로 과신하지 않는다.
