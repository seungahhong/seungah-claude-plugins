# PR 생성 상세

Phase 3에서 사용하는 PR 생성의 상세 절차.

---

## 브랜치 확인 및 검증

### 현재 브랜치 확인

```bash
git branch --show-current
```

`main`, `master` 브랜치에서는 PR을 생성하지 않는다. 사용자에게 알리고 feature 브랜치로 전환을 안내한다.

### 타겟 브랜치 유효성

사용자가 입력한 타겟(base) 브랜치가 원격에 존재하는지 확인한다.

```bash
git ls-remote --heads origin <target-branch>
```

존재하지 않으면 사용자에게 다시 입력을 요청한다.

### push 상태 확인

```bash
git status
git log origin/<현재-브랜치>..HEAD --oneline 2>/dev/null
```

push되지 않은 커밋이 있으면 PR 생성 전 push가 필요함을 사용자에게 안내한다.

## 변경 사항 분석 (본문 작성 판단 프로세스 기반)

commit 스킬의 §1.4 **본문 작성 판단 프로세스**를 PR 수준으로 확장 적용한다.

diff를 시니어 개발자 관점에서 다음 순서로 검토한다:

1. **변경 범위 파악**: 수정된 파일 수, 변경 라인 수, 영향받는 모듈/레이어 확인
2. **의도 추론**: 단순 버그 픽스인지, 구조 변경인지, 새로운 요구사항 반영인지 판단
3. **숨겨진 영향 탐색**: 인터페이스 변경, 동작 차이, 하위 호환성, 성능 특성 변화 여부 확인
4. **테스트 변경 확인**: 테스트 파일 추가/수정이 있는지, 있다면 어떤 케이스를 커버하는지 확인

## PR 메시지 포맷

### 제목

```
이슈번호 type: 한국어 제목
```

- 브랜치명에서 `[A-Z]{2,}-\d+` 패턴으로 이슈 번호 추출
- Type은 commit 스킬과 동일 (feat/fix/docs/style/refactor/test/chore)
- 한국어 명령형으로 작성
- 70자 이내

### 본문

```markdown
## 요약
- 변경 내용의 핵심을 1~2문장으로 설명

## 배경 (선택)
- 왜 이 변경이 필요했는가
- diff만 봐서는 알 수 없는 비즈니스 맥락이나 기술 부채 배경

## 설계 (선택)
- 왜 이 방식인가. 대안이 있었다면 왜 이걸 골랐는지

## 영향
- 이전 동작과 달라진 점
- API 시그니처 변경, 기본값 변경, 삭제된 기능 등
- 영향이 없으면 "없음"으로 명시

## 테스트 시나리오
- 서비스 로직 관점에서 수동 검증이 필요한 항목
- 변경된 기능의 정상 동작, 엣지 케이스, 회귀 가능성 시나리오
```

**필수 섹션**: 요약, 영향, 테스트 시나리오
**선택 섹션**: 배경, 설계 — 내용이 있을 때만 포함. 억지로 채우지 않는다.

### 본문에 쓰지 않는 것

- 파일 단위 수정 사항 나열 (diff가 보여준다)
- 구현 단계 설명 (diff가 보여준다)
- diff에서 바로 읽히는 내용의 반복
- 출처 표기 문구 (예: `Co-Authored-By: ...`)

## PR 생성 명령

프리뷰를 사용자에게 보여주고 승인을 받은 뒤 실행한다.

```bash
# 1. push
git push -u origin <현재-브랜치>

# 2. PR 생성
gh pr create --base <target-branch> --title "<제목>" --body "$(cat <<'EOF'
<본문>
EOF
)"
```

생성된 PR의 URL을 사용자에게 출력한다.

## 예시

### 제목 예시

```
FRONTEND-1234 feat: PR 준비 워크플로우 스킬 추가
```

### 본문 예시

**기본 (필수 섹션만, 선택 섹션 불필요한 경우):**

```markdown
## 요약
- 리뷰 → 커밋 → PR 생성을 하나의 워크플로우로 통합하는 스킬 추가

## 영향
- 기존 /commit, /simplify, /review 스킬과 독립적으로 동작
- 새로운 스킬 추가이므로 기존 기능에 영향 없음

## 테스트 시나리오
- /review-to-pr 실행 시 gh CLI 미설치 상태에서 안내 메시지 출력 확인
- 리뷰 건너뛰기 후 커밋/PR 단계로 정상 진행 확인
- PR 생성 후 PR URL이 올바르게 출력되는지 확인
```

**모든 섹션 포함 (선택 섹션이 필요한 복잡한 변경):**

```markdown
## 요약
- 인증 미들웨어를 공통 모듈로 분리하여 라우트 핸들러의 중복 세션 검증 로직 제거

## 배경
- 각 라우트 핸들러에 중복되어 있던 세션 검증 로직을 한 곳에서 관리하기 위함
- 신규 라우트 추가 시 인증 로직 누락 사고가 2건 발생한 이력 있음

## 설계
- 미들웨어 패턴으로 추출. HOC 방식 대비 라우트 설정에서 의존성이 명확하게 드러남
- 인증이 필요 없는 공개 라우트는 `skipAuth` 옵션으로 명시적 제외

## 영향
- 기존 핸들러에서 직접 호출하던 `verifySession()`이 제거됨
- 토큰 갱신 정책 변경 시 수정 지점이 한 곳으로 집중

## 테스트 시나리오
- 미들웨어 적용된 라우트에서 유효한 세션으로 정상 접근 가능한지 확인
- 만료된 토큰으로 요청 시 401 응답 반환 확인
- 미들웨어 미적용 공개 라우트가 기존과 동일하게 동작하는지 확인
```
