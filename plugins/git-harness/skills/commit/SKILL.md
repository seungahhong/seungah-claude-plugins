# Commit Message Skill

좋은 커밋 히스토리는 `git log`, `git blame`, `git bisect`를 쓸모 있게 만든다.
이 스킬은 diff를 읽고 한국어 커밋 메시지를 작성한다.

---

## 0) 전제 조건

다음 명령을 순서대로 실행한다:

```bash
git status --porcelain
git branch --show-current
git diff --staged --stat
git diff --staged
git log --oneline -10
```

**대상 결정:**

- 스테이징된 변경이 있으면 → 스테이징된 변경만 대상
- 스테이징된 변경이 없으면 → 전체 변경 사항(`git diff`) 대상

**즉시 중단:**

- Diff에 secret 의심 문자열 (`API_KEY=`, `SECRET=`, `TOKEN=`, `password`, `Bearer `, `-----BEGIN` 등) → 사용자에게 알린다.
- 브랜치가 `main`/`master` → 사용자에게 알린다.

---

## 1) 메시지 형식

```
이슈번호 type: 제목
```

### 1.1) 이슈 번호

브랜치 이름에서 `[A-Z]{2,}-\d+` 패턴을 추출한다.

- `feature/FRONTEND-1234-login` → `FRONTEND-1234`
- 추출할 수 없으면 생략한다. 가짜 ID를 만들지 않는다.

### 1.2) Type

| type       | 용도              |
| ---------- | ----------------- |
| `feat`     | 기능 추가 및 수정 |
| `fix`      | 버그 수정         |
| `docs`     | 문서 작성         |
| `style`    | 코드 포맷팅       |
| `refactor` | 리팩토링          |
| `test`     | 테스트            |
| `chore`    | 환경 설정         |

### 1.3) 제목

- 한국어로 작성한다.
- 명령형으로 작성한다. (예: "추가", "수정", "제거", "변경")
- 변경 사항의 핵심을 간결하게 요약한다.

### 1.4) 본문

본문은 기본적으로 생략한다. 대부분의 커밋은 제목만으로 충분하다.

변경이 복잡하여 제목만으로 의도가 전달되지 않을 때만 본문을 추가한다.

#### 본문 작성 판단 프로세스

diff를 시니어 개발자 관점에서 다음 순서로 검토한다:

1. **변경 범위 파악**: 수정된 파일 수, 변경 라인 수, 영향받는 모듈/레이어 확인
2. **의도 추론**: 단순 버그 픽스인지, 구조 변경인지, 새로운 요구사항 반영인지 판단
3. **숨겨진 영향 탐색**: 인터페이스 변경, 동작 차이, 하위 호환성, 성능 특성 변화 여부 확인
4. **테스트 변경 확인**: 테스트 파일 추가/수정이 있는지, 있다면 어떤 케이스를 커버하는지 확인

검토 결과, 아래 순서대로 본문을 구성한다.
선택 섹션은 내용이 있을 때만 포함한다. 억지로 채우지 않는다.

| 순서 | 섹션 | 필수/선택 |
|------|------|-----------|
| 1 | # 요약 | 필수 |
| 2 | # 배경 | 선택 |
| 3 | # 설계 | 선택 |
| 4 | # 영향 | 필수 |
| 5 | # 테스트 시나리오 | 필수 |

**# 요약** (필수)
- 변경 내용의 핵심을 1~2문장으로 설명한다.

**# 배경** (선택)
- 왜 이 변경이 필요했는가. 문제 상황, 배경, 요구사항의 맥락.
- diff만 봐서는 알 수 없는 비즈니스 맥락이나 기술 부채 배경이 있을 때 작성한다.

**# 설계** (선택)
- 왜 이 방식인가. 대안이 있었다면 왜 이걸 골랐는지.
- 단순 구현이 아닌, 설계 선택이 있었을 때만 작성한다.

**# 영향** (필수)
- 이전 동작과 달라진 점. 마이그레이션, 성능, 호환성 등 눈에 안 띄는 부작용.
- API 시그니처 변경, 기본값 변경, 삭제된 기능 등 다른 개발자가 모르면 문제될 사항을 작성한다.
- 영향이 없는 변경이라면 "없음"으로 명시한다.

**# 테스트 시나리오** (필수)
- 코드 변경사항을 기반으로 서비스 로직 관점에서 수동 검증이 필요한 항목을 작성한다.
- 변경된 기능의 정상 동작, 엣지 케이스, 회귀 가능성이 있는 시나리오를 포함한다.
- 일반 목록(`-`)으로 작성한다.

각 섹션은 `# 섹션명` 헤딩 + 목록 형식으로 작성한다.

본문에 쓰지 않는 것:

- 파일 단위 수정 사항 나열 (diff가 보여준다)
- 구현 단계 설명 (diff가 보여준다)
- diff에서 바로 읽히는 내용의 반복

### 1.5) 금지 사항

- 출처 표기 문구 사용 금지 (예: `Co-Authored-By: ...`)

## 2) 예시

**기본 (본문 없음):**

```
FRONTEND-5678 feat: 로그인 페이지에 소셜 로그인 버튼 추가
```

**이슈 번호 없음:**

```
fix: 장바구니 수량 변경 시 총액 미반영 수정
```

**본문 포함:**

```
FRONTEND-2345 fix: 무한스크롤 중복 요청 수정

# 요약
- 스크롤 이벤트 디바운스 누락으로 동일 페이지가 중복 요청되던 문제 수정

# 영향
- API 호출 빈도가 감소하여 서버 부하 경감

# 테스트 시나리오
- 상품 목록 하단 스크롤 시 다음 페이지가 정상 로딩되는지 확인
- 빠르게 스크롤해도 동일 페이지가 중복 요청되지 않는지 네트워크 탭에서 확인
- 마지막 페이지 도달 시 추가 요청이 발생하지 않는지 확인
```

## 3) 실행 워크플로우

1. 전제 조건 확인 (0)
2. 브랜치에서 이슈 번호 추출 (1.1)
3. Type 결정 (1.2)
4. 제목 작성 (1.3)
5. 본문 필요 여부 판단 (1.4)
6. Validation (4)
7. 사용자에게 프리뷰 → 승인 후 커밋

본문 없는 경우:

```bash
git commit -m "FRONTEND-1234 feat: 결제 수단 선택 UI 추가"
```

본문 있는 경우:

```bash
cat > /tmp/commit-message.txt << 'EOF'
FRONTEND-2345 fix: 상품 목록 무한스크롤 중복 요청 수정

# 요약
- 스크롤 이벤트 디바운스 누락으로 동일 페이지가 중복 요청되던 문제 수정

# 영향
- 상품 목록 API 호출 빈도가 감소하여 서버 부하 경감

# 테스트 시나리오
- 상품 목록 하단 스크롤 시 다음 페이지가 정상 로딩되는지 확인
- 빠르게 스크롤해도 동일 페이지가 중복 요청되지 않는지 확인
EOF
git commit -F /tmp/commit-message.txt
```

---

## 4) Validation

커밋 직전에 확인:

- [ ] 한국어로 작성되었는가
- [ ] `이슈번호 type: 제목` 형식인가 (이슈번호 없으면 `type: 제목`)
- [ ] Type이 7가지 중 하나인가
- [ ] 제목이 명령형인가
- [ ] 본문에 필수 섹션(요약, 영향, 테스트 시나리오)이 포함되어 있는가
- [ ] 선택 섹션(배경, 설계)은 실제로 전달할 내용이 있을 때만 포함했는가
- [ ] 내용 없는 섹션이 억지로 채워지지 않았는가
- [ ] 본문이 있다면 `# 섹션명` 헤딩 + 목록 형식인가
- [ ] 출처 표기 문구가 없는가
- [ ] Secrets 미포함

---

## 5) 출력 포맷

1. 전체 커밋 메시지 프리뷰
2. 커밋 명령어
3. 사용자에게 다음 두 선택지를 제시한다 — 직접 `git commit`을 실행하지 않는다
   - **1) 승인** — 커밋을 실행한다
   - **2) 거절** — 사용자의 피드백을 반영하여 메시지를 수정한 뒤 다시 프리뷰한다
