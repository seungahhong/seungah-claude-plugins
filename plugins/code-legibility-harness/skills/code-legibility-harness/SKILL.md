  진단하고 승인하면 단계별로 적용", "주석이 코드랑 안 맞는 것 찾아 고쳐줘", "의미 없는 변수명·함수명 리네임 제안해줘",
  "컴포넌트/모듈명이 하는 일이랑 맞는지 봐줘", "stale 주석·주석처리된 코드 정리" 같은 요청을 할 때 사용한다.
  정직성: 등급(0~100)을 만들지 않는다(저장소 층 등급은 ai-readiness-cartography 소유). "테스트 통과 = 동작 보존"이라고

# code-legibility-harness

코드 **본문** 층위를 AI 에이전트가 읽기 좋게 만든다. 분석 → 제안 → **사용자 승인** → 단계별 적용.

**왜 이렇게 만드는가**: "AI가 읽기 좋은 코드"라고 하면 대부분 *주석을 더 달고 큰 함수를 쪼개는* 것을 떠올린다.
근거는 반대를 말한다([`references/research/`](references/research/)):

- **오도하는 주석 하나가 구조 변형 세 개를 합친 것과 맞먹는 해악**을 낸다(−13.47pp vs −14.04pp). CoT로도 방어되지 않는다.
- **자동 생성 주석은 최상위 모델도 ~20%가 부정확**하고, 특화 파인튜닝 모델조차 35~45%가 틀린다.
- **LLM 리팩터링의 19~35%가 기능적으로 비등가**이며, **그중 ≈21%는 기존 테스트를 전부 통과**한다.
- 구조 층의 강한 수치(92.7%·+12%·+32.8%)는 전부 **툴 측 그래프 인덱스의 효과**이지 본문 리팩터의 효과가 아니다.

그래서 이 하네스는 **삭제를 추가보다 먼저 하고, 리네임을 구조보다 먼저 하며, 구조는 기본적으로 손대지 않는다.**

## 진행 원칙 (전 구간 관통)

- **승인 게이트를 건너뛰지 않는다.** 근본 계약이다: 계획 승인(A) 없이 개별 적용 없음, 개별 승인(B) 없이 파일 쓰기 없음,
  최종 승인(C) 없이 "반영" 없음.
- **⚠️ 스코프 가드(게이트와 동급 불변식)**: Phase 0에서 사용자가 선택한 **개입 클래스 안에서만** 제안·적용한다.
  밖의 클래스는 "추가 안 함"으로 표기하고 확장 여부를 문의한다. 임의 확장 금지.
- **자기보고를 신뢰하지 않는다.** "고쳤습니다"는 증거가 아니다. `behavior-guard`가 센서를 **실행해 관측**한다.
- **generator ≠ checker.** 제안한 에이전트가 자기 제안을 검증하지 않는다.
- **등급을 만들지 않는다.** census(인벤토리)만 낸다. 0~100 등급은 자매 플러그인 `ai-readiness-cartography` 소관이다.
- **커밋하지 않는다.** 최종 승인 후 `git-harness` 핸드오프를 제안만 한다.

## 개입 클래스 (정본: [`references/intervention-catalog.md`](references/intervention-catalog.md))

| 클래스 | 개입 | 우선순위 | 동작 위험 | 기본값 |
|--------|------|---------|----------|--------|
| **C0** | 오도·stale 주석 **삭제/수정** | **P1** | 없음(주석은 실행되지 않음) | ON |
| **C1** | 계약·불변식 **주석 추가/수정** | P3 | 없음 (사실성 위험 ~20~45%) | ON |
| **C2** | 무의미·오도 식별자 **안전 리네임** | **P2** | 참조·컴파일 깨짐 | ON |
| **C3** | **구조 리팩터**(추출·이동·분할) | P4 | **높음** (19~35% 비등가) | **OFF · opt-in** |

> C0가 1순위인 이유는 효과가 가장 커서가 아니라 **효과가 확실하고 위험이 0**이기 때문이다(기대이익 기준).
> C3가 마지막인 이유는 효과가 없어서가 아니라 **효과가 추론이고 위험이 측정된 최대치**이기 때문이다.

## Phase 0 — 초기 문의 (Initial Inquiry)

목적: 대상과 **허용 스코프**를 확정한다. 두 질문 모두 스킵 가능하다.

### 1. 대상 경로 — 스킵 가능
"어느 경로를 진단할까요? (경로 입력 / 스킵 = 현재 저장소 루트)"
- 스킵 시 현재 작업 디렉토리를 루트로 삼는다. `git rev-parse --show-toplevel`로 저장소 루트를 확인한다.
- 하위 경로가 주어지면 그 경로만 스캔하되, 리네임 참조 검색은 **저장소 전체**를 대상으로 한다(참조는 밖에 있을 수 있다).

### 2. 개입 클래스 — 체크박스 다중선택 (**= 구속 스코프**)
```
[x] C0  오도·stale 주석 삭제/수정      (P1 · 위험 0 · 권장)
[x] C1  계약·불변식 주석 추가/수정      (P3 · 사실성 검증 필요)
[x] C2  무의미·오도 식별자 안전 리네임   (P2 · 컴파일/참조 센서 필요)
[ ] C3  구조 리팩터 (추출·이동·분할)     (P4 · 기본 OFF — 효과는 추론, 위험은 측정됨)
```
- C3를 켜려 하면 **경고를 먼저 보여준다**: "구조 리팩터가 에이전트 성공률을 올린다는 직접 측정 근거는 없습니다(추론).
  반면 LLM 리팩터링의 19~35%가 기능적으로 비등가이고, 그중 ≈21%는 기존 테스트를 통과합니다. 켜시겠습니까?"
- C3를 켜면 **테스트 스위트 존재를 먼저 확인**한다. 없으면 C3를 거부한다(등가성 확인 수단 부재).
- 자유형 답변은 위 스코프로 정규화한다.

이 선택이 **Phase 1의 스캔 축 · Phase 2 제안표 · Phase 3 허용 적용 범위**를 못박는다.
(이 단계는 게이트가 아니라 스코프 확정 — 첫 승인 게이트는 Phase 2의 게이트 A다.)

### 3. 안전 리네임 도구 탐지 (C2 선택 시)
저장소에서 다음을 탐지해 기록한다. **하나도 없으면 C2는 "제안만"으로 강등**하고 사용자에게 알린다.
| 언어 | 도구 |
|------|------|
| Python | `rope` · `pyright`/`pylsp` LSP rename |
| TS/JS | `ts-morph` · `tsserver` LSP rename · `jscodeshift` |
| Go | `gopls rename` |
| Rust | `rust-analyzer` |
| Java/Kotlin | LSP rename |

## Phase 1 — 결정론 census + 다각도 진단

목적: **결정론을 먼저, LLM은 보강.** 손진단부터 하지 않는다.

1. **census 스캔** (선택된 축만):
   ```bash
   python3 ${CLAUDE_PLUGIN_ROOT}/skills/code-legibility-harness/scripts/legibility_scan.py <root> \
     --axes <naming,comments,structure 중 선택된 것> \
     --out .claude/_legibility/census-before.json
   ```
   스캐너는 **stdlib만** 쓴다(Python 3.10+). 외부 패키지도, 다른 플러그인도 필요하지 않다.
   - 스캐너는 **등급을 내지 않는다.** 후보 목록 + `auto-high / auto-med / heuristic / report-only` 라벨을 낸다.
   - `skipped` 목록을 반드시 함께 읽는다. **측정 불가는 "결함 없음"이 아니다.**

2. **진단 에이전트 병렬 실행** — 선택된 클래스에 해당하는 것만 호출한다. 모두 `model: "opus"`.
   | 에이전트 | 조건 | 역할 |
   |---------|------|------|
   | [`comment-auditor`](../../agents/comment-auditor.md) | C0 또는 C1 선택 | 주석을 오도/stale/noise/유효로 4분류 |
   | [`naming-analyst`](../../agents/naming-analyst.md) | C2 선택 | 오도 > 무의미 > 모호 3축, 위험 등급·도구 판정 |
   | [`structure-cartographer`](../../agents/structure-cartographer.md) | **C3 선택 시에만** | 구조 후보(효과=추론 라벨 필수) |

   각 에이전트에 `census-before.json` 경로와 선택된 스코프를 전달한다. **에이전트는 파일을 수정하지 않는다.**

3. **AMBIGUOUS 승격**: `comment-auditor`가 "주석과 코드가 모순되는데 어느 쪽이 명세인지 모르겠다"고 표시한 항목은
   개입 후보가 아니라 **사용자에게 올리는 질문**이다. 주석이 아니라 **코드가 버그일 수 있다.**

## Phase 2 — 제안표 + 게이트 A (계획 승인)

목적: 전체 개입 계획을 **한 번에** 승인받는다. 승인 전에는 **어떤 파일도 수정하지 않는다.**

1. **랭킹**: [`references/intervention-catalog.md`](references/intervention-catalog.md)의 제안 랭킹 규칙을 적용한다
   (위험조정 기대이익; 동률이면 P 번호 우선).

2. **제안표**를 낸다:

   | # | 클래스 | 위치 | 현재 → 제안 | 근거(ID·tier) | 동작 위험 | 1급 센서 | 판정 상한 |
   |---|--------|------|------------|--------------|----------|---------|----------|
   | 1 | C0 | `pay.py:42` | 틀린 주석 삭제 | E-C1 `PRIMARY-2025+ 2025-04` | 없음 | 파싱 | VERIFIED |
   | 2 | C2 | `pay.py:88` | `d` → `settlement_date` | E-N1 `PRIMARY-2025+ 2025-10` | 참조 깨짐 | 타입체크+참조 | VERIFIED |
   | 3 | C1 | `pay.py:12` | 불변식 주석 추가 | E-C5 `PRIMARY-pre2025 2024-06 (배경)` | 없음(사실성 ~20%) | 코드 대조 | 사실성 별도 |

   - **스코프 밖 클래스는 "추가 안 함"으로 명시**하고 확장 여부를 문의한다.
   - C3 행이 있으면 **"예상 효과: 추론(직접 측정 근거 없음 — O-1)"** 을 반드시 표기하고, 판정 상한을 `UNVERIFIED`로 적는다.
   - `heuristic`·`report-only` 탐지(N4 심볼 충돌, 함수 크기)는 **개입 후보로 승격하지 않는다.** 진단 정보로만 보고한다.

3. **베이스라인 센서 실행**: `behavior-guard`(`model: "opus"`)를 호출해 개입 **전** 빌드/타입체크/테스트를 실행·기록한다.
   이미 빨간색이면 **그 사실을 먼저 보고하고 진행 여부를 묻는다**(개입 탓으로 오귀속 금지).

4. **게이트 A — 계획 승인**: "이 계획으로 진행할까요? (전체 승인 / 특정 행 수정·제외 / 스코프 재조정)"

## Phase 3 — 개별 순차 적용 + 게이트 B (개별 승인)

목적: 승인된 개입을 **하나씩** 적용한다. 한 개입마다 게이트를 통과해야 다음으로 간다.

각 개입에 대해:

1. **최소 단위로 쪼갠다** — 큰 함수를 통째로 재작성하지 않는다(위험이 복잡도에 비례: APPS 32.09% vs HumanEval 22.10%).
2. **diff를 먼저 보여준다** (적용 전).
3. **게이트 B — 개별 적용 승인**: "이 개입을 적용할까요? (적용 / 수정 후 적용 / 이것만 건너뜀 / 여기서 중단)"
   **승인 시에만** 파일을 쓴다(Write/Edit).
4. **클래스별 적용 규칙**:
   - **C0** — 주석 삭제. 단 주석이 계약·불변식을 담고 있고 코드가 그와 다르면 **삭제하지 말고 사용자에게 올린다**(코드가 버그일 수 있다).
   - **C1** — 계약·불변식만. 코드를 그대로 되뇌는 what-주석 생성 금지. 밀도 목표치 금지.
   - **C2** — **AST/LSP 도구로만 리네임한다. LLM이 문자열을 치환하지 않는다.** 도구가 없으면 적용하지 않고 제안만 남긴다.
     공개 API·직렬화 키·DB 컬럼·라우트·i18n 키에 노출된 심볼은 **컴파일러가 못 잡는 런타임 파괴**이므로 자동 적용 후보에서 제외하고 사람 판단을 구한다.
   - **C3** — 최소 단위 하나만. 다단계 연쇄 변경 금지(multi-step에서 성공률 53.5%→20.6%로 붕괴, 초기 설계 실수가 누적).
5. **`behavior-guard` 검증**(`model: "opus"`): 개입 후 동일 센서를 재실행해 관측한다.
   - 판정 어휘는 **`VERIFIED` / `FAILED` / `UNVERIFIED`** 셋뿐이다.
   - **C3는 최선의 경우에도 `UNVERIFIED`를 넘을 수 없다** — 기존 테스트로 등가성을 증명할 수 없기 때문이다(≈21% 누락).
     이 사실을 사용자에게 **그대로 보고한다.**
   - C2 후에는 **동적 참조 교차 확인**: `getattr`·문자열 키·템플릿·리플렉션을 grep으로 훑는다.
6. **`FAILED`면 즉시 롤백을 제안한다.** 커밋하지 않았으므로 워킹트리 되돌리기다.

**중단 지점**은 언제든 존중한다 — 사용자가 멈추면 지금까지 적용된 것만 남기고 Phase 4로 넘어간다.

## Phase 4 — 재확인·반영 + 게이트 C (최종 승인)

1. **census 재측정**:
   ```bash
   python3 ${CLAUDE_PLUGIN_ROOT}/skills/code-legibility-harness/scripts/legibility_scan.py <root> \
     --axes <Phase 1과 동일> --out .claude/_legibility/census-after.json
   ```
2. **델타 보고** — 반드시 다음 문장을 포함한다:
   > census 델타는 **개입이 반영되었는지의 결정론적 확인**이지 **에이전트 성공률의 대리 지표가 아니다.**
   공개 벤치마크(SWE-bench류) 점수로 개선을 주장하지 않는다(성공의 63%가 정답 *회수*였다).
3. **판정 요약**: 개입별 `VERIFIED / FAILED / UNVERIFIED` 집계. `UNVERIFIED`를 성공으로 포장하지 않는다.
4. **게이트 C — 최종 반영 재확인**: "이대로 반영(확정)할까요? (확정 / 특정 개입 되돌리기 / 추가 조정)"
   - **커밋/PR은 이 스킬의 범위가 아니다.** 재승인 후 원하면 `git-harness`로 핸드오프하겠다고 **제안만** 한다.
     `git-harness`는 **선택적 동반 플러그인**이다 — 설치돼 있지 않으면 그 사실을 알리고 커밋은 사용자에게 맡긴다.
     이 스킬은 어떤 경우에도 스스로 커밋하지 않으므로, 없어도 동작에 지장이 없다.

## 산출물 저장 (opt-in)

사용자가 원하면 `.claude/_legibility/`에 저장한다: `census-before.json` · `census-after.json` · `proposals.md` · `verdicts.md`.
묻지 않고 저장하지 않는다.

- [ ] 3개 게이트(계획 A → 개별 B → 최종 C)를 모두 통과했는가?
- [ ] 스코프 밖 클래스를 임의로 건드리지 않았는가? 밖은 "추가 안 함"으로 표기했는가?
- [ ] 등급(0~100)을 만들지 않았는가?
- [ ] "테스트 통과 = 동작 보존"이라고 말하지 않았는가?
- [ ] C3 제안에 "효과 = 추론(O-1)" 라벨과 `UNVERIFIED` 상한을 붙였는가?
- [ ] 리네임을 AST/LSP 도구로 했는가? (문자열 치환 금지) 도구가 없으면 제안만 남겼는가?
- [ ] `heuristic`·`report-only` 탐지를 개입 후보로 승격하지 않았는가?
- [ ] `skipped`(측정 불가) 목록을 보고했는가?
- [ ] 자기보고가 아니라 **실행 관측**으로 판정했는가?
- [ ] 커밋하지 않고 `git-harness` 핸드오프를 제안만 했는가?
