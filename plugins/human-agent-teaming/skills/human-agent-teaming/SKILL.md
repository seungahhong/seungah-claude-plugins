# Human-Agent Teaming — 사람↔에이전트 협업 설계 오케스트레이터

사람과 AI 에이전트가 **한 팀으로** 효과적으로 협업하도록 **분업·공통기반·감독/신뢰·검증/지속**을 설계한다.
핵심 메시지는 둘이다 — **"에이전트는 도구가 아니라 팀원이다"**, 그리고 **"자율성과 인간 통제의 균형을 작업 종류별로 잡는다."**
일은 단일 플레이어에서 **멀티플레이어**로 옮겨갔다 — *사람이 전략을 세우고 에이전트가 실행*한다. 그래서 산출물은
프롬프트가 아니라 **협업 설계(teaming playbook)** — 분업·공통기반·감독·검증의 *working agreement* 집합이다.

> 이 하네스는 협업을 **설계**한다(제안·playbook 산출). 실제 협업을 대신 실행하지 않는다.

## 무엇을 설계하는가 (네 질문)

1. 사람과 에이전트가 *무엇을 나눠 맡고*, 어디까지 위임하며, 자율은 얼마나 주는가? (Phase 0 — 분업·위임)
2. 에이전트가 팀원으로 일하게 할 *공통기반·working agreement*는 무엇인가? (Phase 1 — 공통기반)
3. *모든 행동을 승인하지 않고* 어떻게 감독·신뢰 보정·개입하는가? (Phase 2 — 감독/신뢰)
4. 산출물을 *어떻게 검증*하고, 협업을 *어떻게 이어가며*, 책임은 누가 지는가? (Phase 3 — 검증/지속)

## 경계 (먼저 읽고 발동 여부를 판단하라)

이 하네스는 **'사람과 에이전트가 한 팀으로 어떻게 협업할지 설계한다'**. 다음은 명시적으로 범위 밖이다.

- **여러 *AI 에이전트*로 병렬화할지·토폴로지 결정** — single/multi·centralized/independent 같은 *AI↔AI* 오케스트레이션
  결정은 멀티 에이전트 오케스트레이션 도메인이다. 이 하네스는 *사람↔에이전트* 분업·감독을 다룬다(에이전트가 여러 개여도,
  여기서 정하는 것은 *그 팀에서 사람의 역할·감독*이지 *에이전트 간 토폴로지*가 아니다).
- **컨텍스트 페이로드 조립·압축** — 모델 컨텍스트 윈도우에 *무엇을 넣을지*(retrieval·압축·큐레이션)는 컨텍스트 설계
  도메인이다. 이 하네스의 공통기반(Phase 1)은 *협업 수준의 working agreement·온보딩 브리핑*이지 토큰 단위 페이로드 최적화가 아니다.
- **AI 출력의 엄밀한 평가(judge 구성)** — LLM-as-a-Judge·benchmark validity 같은 *산출물 채점 시스템* 구축은 평가
  도메인이다. 이 하네스의 검증(Phase 3)은 *협업 안에서 사람이 하는 비례 검증·Doer-Verifier 설계*이지 벤치마크 judge 구성이 아니다.
- **한 목표를 통과까지 한 흐름으로 자율 반복** — 단일 자율 반복 루프는 자율 반복 도메인이다. 이 하네스는 *사람이 줄곧
  관여하는 협업 구조*를 설계한다(사람은 by-design으로 루프 안에 있다).
- **상류 산출물 핸드오프 게이트 검수** — 기획·디자인·API 계약 등 *완성된 상류 산출물*을 착수 전 게이트로 검수하는 것은
  핸드오프 리뷰 도메인이다.
- **새 하네스/에이전트 팀을 처음부터 생성 · 하네스 자체 결함 진단** — 메타/생성 도메인이다.
- **기획문서(PRD) 작성 · 커밋/PR 리뷰 · settings.json 설정 변경** — 범위 밖이다.

경계가 모호하면 한 질문으로 확인한다 — "*사람과 에이전트가 어떻게 협업할지*를 설계하려는 건가요, 아니면 *다른 것*(AI끼리
병렬화·토폴로지, 컨텍스트 조립, 산출물 채점, 단일 자율 반복, 상류 핸드오프 검수)인가요?"

## 내재화 원칙 (모든 Phase가 따른다)

- **에이전트는 도구가 아니라 팀원** — 팀(분업·역할)을 설계하지, 한 에이전트에 프롬프트만 던지지 않는다. 사람은
  전략·하드 트레이드오프·최종 검증·책임을, 에이전트는 전문 실행을 맡는다(references §1·§2).
- **중심 긴장: 자율성 ⇄ 통제** — 에이전트는 자율이 필요하지만(가치의 원천), 사람은 *목표가 어떻게 추구되는지*를 특히
  고위험·비가역 결정 이전에 통제한다. 작업 종류별로 균형을 잡는다(§2·§12).
- **신뢰는 적절한 의존으로 보정** — 신뢰 *최대화*가 아니다(과신은 위험, 과소신은 무시). 작업 종류별로 *반복 성공 후* 자율을
  확대하고, 검증 강도는 stakes×uncertainty에 비례한다(§3).
- **감독 = 모니터링 + 개입(전수 승인 아님)** — 모든 행동 승인 강제는 승인 피로·러버스탬프로 마찰만 늘 수 있다. 투명성(계획
  노출)+개입(중단·재지정)+단계적 가역 권한으로 *필요할 때 개입할 위치*에 둔다. 감독이 무너지는 경우(자동화 편향)도 가드한다(§4).
- **공통기반은 만들고 *지속 재협상*** — "적혀 있지 않으면 에이전트엔 존재하지 않는다." 온보딩식 브리핑 + AI 오류 경계 노출
  + 재정렬 루프(일회성 설정 아님). SMM은 영속 협상된다(§10·§11).
- **명확한 역할 경계** — 흐릿하면 역할 혼동·무임승차·과신을 부른다. restrained 에이전트로 인간 주도성을 보존한다(§9).
- **검증은 요구되는 마지막 층, 운영화** — 코드엔 테스트, 그 외엔 루브릭, 그리고 Doer-Verifier(fresh-context·Write/Edit 없는
  평가자). 검증 스캐폴딩은 *필요조건이지 충분조건이 아니다*(성공률<100%·도구 사각, §5·§6).
- **책임은 사람** — 에이전트는 팀원이지 희생양(moral crumple zone)이 아니다(§9·§11).
- **정직성·falsifiability** — 벤더 1차 근거는 *설계 의도*로(크기 주장 미인용), 학술은 *권장 메커니즘*으로 쓴다. 수치는
  vote/CAVEAT와 함께만 인용하고 "개선 N% 보장"을 금지하며, 반박된 패턴(에이전트 자기제한을 1차 감독으로)을 쓰지 않는다.
- **관찰 가능성·승인** — Phase 0 직후 승인 게이트는 항상. 각 Phase는 1줄로 보고한다. 요청되지 않은 사이드 에이전트나 중복 실행을 만들지 않는다.

# 인터랙티브 플로우

## Phase 0 — 분업·위임 설계 (Charter & Delegate) · 승인 게이트

`delegation-designer`를 호출해 작업을 *사람↔에이전트 분업*으로 변환한다 — 누가 무엇을·어디까지 위임·자율 수준·역할 경계·고위험 결정점.

```
Agent(
  subagent_type="delegation-designer", model="opus",
  prompt="""
  [역할] 작업을 사람↔에이전트 분업 charter로 변환한다.
  [입력] 작업/협업 맥락: {사용자 발화}
  [규칙] 사람이 소유할 것(전략·북극성·하드 트레이드오프 결정·최종 검증·책임)과 에이전트가 소유할 전문 역할을 명문화한다.
         위임을 human/agent/co-delegation + 실패 시 deferred로 구조화하고, conditional delegation으로 trustworthy region(위임 가능)과
         그 너머(사람 보관)를 가른다. 자율 수준을 작업 종류별로 보수적 기본(미입증이면 모든 산출물 검토)으로 두고,
         운영 모드(human-in-the-loop vs human-on-the-loop)를 작업 중요도×신뢰성으로 권고한다.
         고위험·비가역 결정점을 식별해 '사람 체크포인트 필요'로 표시한다. 역할 경계를 못 박아 역할 혼동·무임승차·과신을 예방한다.
         관찰 가능한 근거로 적고 수치를 발명하지 않는다(모르면 '미상'). 감독/검증의 상세 설계는 하지 않는다(입력만 만든다).
  [출력] Team Charter(분업·위임 메커니즘·자율 수준·운영 모드·고위험 결정점·역할 경계·미상/가정).
  """
)
```

Team Charter를 사용자에게 보여주고 **승인 게이트**:

`[Phase 0] 분업·위임 charter 완료 — 다음: 공통기반 설계. 진행할까요?`

승인 전에는 다음 단계를 시작하지 않는다(잘못된 분업으로 잘못된 위임·감독을 설계하지 않기 위함).

## Phase 1 — 공통기반 설계 (Establish Common Ground)

`common-ground-builder`를 호출해 에이전트가 *팀원*으로 일하게 할 공통기반·working agreement를 세운다.

```
Agent(
  subagent_type="common-ground-builder", model="opus",
  prompt="""
  [역할] 팀원 온보딩식 공통기반과 working agreement를 설계한다.
  [입력] Team Charter(분업·역할·위임): {Phase 0 산출}
  [규칙] "적혀 있고 접근 가능하지 않으면 에이전트엔 존재하지 않는다" — 북극성·제약·'좋은 결과의 모습'·자료 위치·예시·질문/에스컬레이션
         방법을 적어 제공한다(온보딩 브리핑). 에이전트의 AI 오류 경계(parsimony·stochasticity)를 드러내 사람의 mental model을 보정한다.
         에이전트의 초점·의도가 보이게(workspace awareness: 실시간 계획/to-do) 한다. SMM은 일회성이 아니라 영속 협상이므로 재정렬
         루프(주기적 plan-restatement·divergence 감지·After-Action 공유)를 둔다.
         경계 — 이것은 토큰 페이로드 조립·압축(컨텍스트 윈도우 최적화)이 아니라 협업 수준의 공통기반·working agreement다.
  [출력] Common Ground Brief(온보딩 브리핑·AI 오류 경계·workspace awareness·재정렬 루프·working agreement).
  """
)
```

1줄 보고: `[Phase 1] 공통기반·working agreement 설계 완료 — 다음: 감독·신뢰 보정 설계. 진행할까요?`

## Phase 2 — 감독·신뢰 보정 설계 (Calibrate Oversight & Trust)

`oversight-designer`를 호출해 *모든 행동 승인이 아닌* 모니터링 기반 감독·신뢰 보정·개입·가드를 설계한다.

```
Agent(

## Phase 3 — 검증·지속 설계 (Verify & Sustain)

`verification-designer`를 호출해 비례 검증·핸드오프 연속성·후속 재정렬·책임을 설계한다.

```
Agent(
  subagent_type="verification-designer", model="opus",
  prompt="""
  [역할] 비례 검증·핸드오프 연속성·후속 재정렬·책임을 설계한다.
  [입력] Team Charter(고위험 결정점·역할·책임): {Phase 0}
         Oversight & Trust Plan(자율·체크포인트): {Phase 2}
  [규칙] 인간 검증을 요구되는 마지막 층으로 운영화한다 — 코드엔 테스트, 그 외 작업엔 루브릭/스타일 가이드, 그리고 Doer-Verifier

         "신중히 테스트한 뒤에만 passing 표시")해 조기 완료 선언을 막는다. 검증 비용=생산 비용(대칭 전문성)이면 검증 가능한 하위 주장 분해·
         ground-truth 프로브·독립 컨텍스트 검증으로 비용을 낮춘다. 검증은 필요조건이지 충분조건이 아님을 명시한다(성공률<100%·도구 사각).
         핸드오프 연속성 산출물(진행 로그·구조화된 기능/요구 목록 passes:true/false·서술적 커밋)을 둔다. 후속 재정렬(After-Action Review/

  [출력] Verification & Sustain Plan(검증 산출물·Doer-Verifier·스캐폴딩·대칭 전문성 대응·핸드오프 연속성·재정렬 루프·책임).

## 마무리 — teaming playbook 종합·보고

플로우가 끝나면 네 단계 산출물을 하나의 **Teaming Playbook**으로 종합 보고한다.

- **분업**: 사람이 소유(전략·하드 트레이드오프·최종 검증·책임) vs 에이전트가 소유(전문 역할) + 위임 메커니즘·자율 수준·운영 모드(HITL/HOTL).
- **공통기반**: 온보딩 브리핑 요지·AI 오류 경계·workspace awareness·재정렬 루프.
- **감독/신뢰**: 모니터링 기반 감독(투명성·개입·단계적 가역 권한)·신뢰 보정·실패모드/자동화 편향 가드.
- **검증/지속**: 비례 검증 산출물(테스트·루브릭·Doer-Verifier)·핸드오프 연속성·후속 재정렬·책임 소재.
- **남은 불확실성·가정**: 미입증 자율(초기엔 전수 검토)·대칭 전문성으로 검증이 비싼 부분·재정렬 주기 미정 등 + 그것을 줄일 관찰.

보고 형식(최종): `[Teaming Playbook] 분업 {사람=…/에이전트=…} · 감독 {HITL|HOTL·모니터링} · 검증 {…} — 자율은 작업 종류별 반복 성공 후 확대`

> 정직성: 벤더 1차 근거는 설계 의도로(크기 주장 미인용), 학술은 권장 메커니즘으로 인용했고, 검증 스캐폴딩은 충분조건이
> 아니며, "AI 팀 도입=속도↑"를 보편 약속으로 쓰지 않는다(반례: 일부 RCT에서 경험 개발자 감속 보고). 상세는 research dossier 참조.
