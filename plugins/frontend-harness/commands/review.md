# Review — 통합 리뷰 및 배포 준비 커맨드

개발이 완료된 코드에 대해 **`/simplify`**, **`/review`**, **`security-audit`**, **`lighthouse-performance`**, **`qa-inspector`** 5가지 관점을 **병렬로 실행**하여 결과를 통합 보고하고, 수정이 필요한 경우 수정 → 재리뷰 루프를 돌려 배포 가능 상태에 도달시킨다.

> **커밋은 이 커맨드에서 다루지 않는다.** 커밋이 필요하면 별도 커밋 워크플로를 사용한다.

## 왜 통합 리뷰가 필요한가

다섯 가지 관점은 서로 보완 관계다. 함께 실행해야 사각지대가 사라진다.

| 관점 | 도구 | 잡는 것 |
| ---- | ---- | ------- |
| **간소화** | `/simplify` | 불필요한 복잡도, 중복, 가독성 저하 |
| **품질** | `/review` | 로직 버그, 컨벤션 위반, 누락된 케이스 |
| **보안** | `security-audit` | OWASP Top 10, 의존성 취약점, 시크릿 |
| **성능** | `lighthouse-performance` | Core Web Vitals(LCP, CLS, INP, TTFB) 회귀 |
| **정합성** | `qa-inspector` | API↔훅, 라우팅, 상태 전이 등 경계면 불일치 |

병렬 실행은 다섯 관점을 독립적으로 평가해 한쪽 결론이 다른 쪽 결과를 흐리지 않게 만든다.

## 실행 절차

### Phase 0: 사전 확인

1. **변경 사항 수집:**

```bash
git status --porcelain
git diff --stat HEAD
git diff --name-only HEAD
```

변경 파일이 없으면 사용자에게 알리고 종료한다.

2. **실행 항목 선택 (필수):**

5개 관점을 모두 나열하고 사용자가 실행할 항목을 선택하게 한다. 사용자가 답변하기 전에는 다음 단계로 진행하지 않는다.

```
[Review] 통합 리뷰를 시작합니다.

병렬로 실행할 항목을 선택해주세요:
  □ 1. /simplify              — 코드 간소화 (자동 적용)
  □ 2. /review                — PR 리뷰 (코드 품질, 버그)
  □ 3. security-audit         — OWASP Top 10, 의존성, 시크릿
  □ 4. lighthouse-performance — Core Web Vitals 측정
  □ 5. qa-inspector           — 모듈 간 경계면 정합성

선택 방법:
- 복수 선택 가능 (예: "1, 3, 5")
- 전체 선택: "전체" 또는 "1, 2, 3, 4, 5"
- 최소 1개 이상 선택해야 합니다
```

선택 해석 규칙:

- 번호 목록(`1, 3, 5`)을 받으면 해당 항목만 실행 대상으로 확정
- "전체", "all", "모두", "1~5", "1, 2, 3, 4, 5" 등 → 5개 모두
- 단일 번호(`3`)를 받으면 해당 항목 1개만 실행
- 0개 선택, 빈 응답, 범위 밖 번호 → 다시 안내
- 선택 결과를 사용자에게 한 번 더 확인 (예: "선택하신 항목: /simplify, security-audit, lighthouse-performance — 이대로 진행할까요?")

**선택 항목 수에 관계없이 항상 병렬 실행이다.** 선택이 2개든 5개든 모든 서브에이전트는 **하나의 어시스턴트 메시지 안에서 동시에 spawn**된다. 순차 실행 금지. 선택이 1개인 경우에만 단일 spawn으로 축소되고, 그 외에는 항상 병렬을 유지한다.

Phase 2의 통합 리포트, Phase 3의 수정 분기, Phase 4의 루프 종료 조건, Phase 5의 최종 리포트는 **선택된 항목만** 다룬다 — 미선택 항목은 표에서 "⏭️ (미선택)"으로 표기하거나 행을 생략한다.

3. **루프 한도 안내:**

```
선택된 항목으로 통합 리뷰를 진행합니다.

수정이 필요하면 적용 후 자동 재리뷰합니다.
최대 재리뷰 횟수: 3회 (이후엔 사용자 결정)
```

### Phase 1: 병렬 리뷰 실행

**Phase 0에서 선택된 항목**만 **단일 메시지에 병렬 Agent 호출**로 동시 spawn한다. 순차 실행 금지. 미선택 항목의 서브에이전트는 spawn하지 않는다. 모든 Agent 호출에는 `model: "opus"`를 명시한다.

#### 1-A. `/simplify` 서브에이전트

Agent 도구로 `code-simplifier` 서브에이전트를 spawn한다.

- 변경된 코드의 명확성·중복·가독성을 점검하고 **개선을 직접 적용**한다
- 기능 변경은 금지 — 동작 동일성 유지
- 산출물: 적용된 변경 diff, 적용하지 않은 제안 목록

#### 1-B. `/review` 서브에이전트

Agent 도구로 일반 리뷰 서브에이전트를 spawn한다. 프롬프트 안에서 빌트인 `/review` 스킬을 호출하도록 지시한다.

- 코드 품질, 컨벤션, 로직 버그, 누락된 케이스 점검
- 변경된 코드 전체를 대상으로 한다
- 산출물: Critical / Important / Suggestion 분류된 이슈 목록

#### 1-C. `security-audit` 서브에이전트

Agent 도구로 서브에이전트를 spawn하여 `../skills/security-audit/SKILL.md`의 절차를 따르게 한다.

- 변경 파일 기반 모드로 OWASP Top 10 점검 (A01, A03, A05, A07, A10)
- `npm audit`으로 의존성 취약점 스캔
- 보안 헤더 점검, 시크릿/민감정보 탐지
- 산출물: Critical / High / Medium 분류된 취약점 목록

#### 1-D. `lighthouse-performance` 서브에이전트

Agent 도구로 서브에이전트를 spawn하여 `../skills/lighthouse-performance/SKILL.md`의 절차를 따르게 한다.

- 사용자 확인된 URL(없으면 `package.json`의 `dev` 포트 추론)에서 실행
- Core Web Vitals(LCP, CLS, INP, TTFB, FCP) + 성능 점수
- Lighthouse CLI 실행 불가 시 코드 기반 정적 분석으로 대체
- 산출물: 지표별 PASS/WARNING/FAIL, 개선 기회(예상 절감) 목록

> **주의:** Lighthouse는 개발 서버나 빌드된 산출물에서 실행한다. 검사 URL을 사용자에게 한 번 확인한 뒤 다음 라운드부터는 동일 URL로 재사용한다.

#### 1-E. `qa-inspector` 서브에이전트

Agent 도구로 서브에이전트를 spawn하여 `../skills/qa-inspector/SKILL.md`의 절차를 따르게 한다.

- 변경 파일 기반 모드로 경계면 불일치 점검
- API 응답↔훅 타입, 라우팅, 상태 전이, 데이터 흐름 교차 비교
- 산출물: Critical / Warning 분류된 경계면 불일치 목록

**병렬 실행 규칙:**

- **선택된 항목 수에 관계없이** 모든 Agent 호출을 **하나의 어시스턴트 메시지 안에서** 동시에 발행한다 (선택 1개일 때만 단일 spawn)
- 미선택 항목은 spawn하지 않는다 — 결과 표에서도 제외하거나 "⏭️ (미선택)"으로 표기
- 결과를 모두 받기 전에는 다음 Phase로 넘어가지 않는다
- 어느 하나가 실패해도 나머지 결과는 보존하고 사용자에게 실패 사유를 보고한다

### Phase 3: 수정 여부 분기

#### Case A: 수정 필요 항목 없음

```
[Review 완료] 라운드 N에서 수정 필요 항목이 없습니다.

- /simplify:              N건 자동 적용
- /review:                0건 (Critical/Important)
- security-audit:         0건 (Critical/High)
- lighthouse-performance: PASS (Score N)
- qa-inspector:           0건 (Critical)

배포 가능 판정: YES

[참고] 커밋이 필요하면 별도 커밋 워크플로를 실행하세요.
```

→ Phase 5(최종 리포트)로 진행.

#### Case B: 수정 필요 항목 있음

사용자에게 수정 진행 여부를 확인한다.

```
[Review] 수정이 필요한 항목이 N건 있습니다.

[수정 항목 요약 — 위 종합 표 출력]

진행 방식을 선택해주세요:
1. 수정 후 재리뷰 — 위 항목을 모두 수정하고 라운드 N+1 실행 (기본)
2. 일부만 수정 — 항목 번호를 지정 (예: "1, 3, 4")
3. 수정 없이 종료 — 현재 결과로 최종 리포트 생성 (배포 가능 판정: NO)
```

**수정 적용 규칙:**

- Critical / High / lighthouse FAIL / qa-inspector Critical은 기본 적용 대상
- Important / Medium / Warning 이하는 사용자 확인 후 적용
- 수정 후 변경 diff를 사용자에게 보여준다
- 수정이 다른 곳에 영향을 줄 수 있는 경우(공용 함수 시그니처 변경 등) 사용자 승인을 별도로 받는다

수정 완료 후 **Phase 1로 돌아가** 동일한 5개 서브에이전트를 다시 병렬 실행한다 (라운드 카운터 +1).

### Phase 4: 재리뷰 루프 종료 조건

다음 중 하나라도 만족하면 루프를 종료한다.

| 종료 조건                                     | 판정                              |
| -------------------------------------------- | --------------------------------- |
| 수정 필요 항목 0건                            | PASS                              |
| 라운드 횟수가 3회에 도달                      | 사용자 결정 (계속 / 무시하고 종료) |
| 사용자가 "수정 없이 종료" 선택                | FAIL (배포 가능 NO)               |
| 동일 이슈가 2라운드 연속 잡힘                 | 사용자 결정 (수정 패턴 재고 필요) |

라운드 횟수 3회 초과 시:

```
[Review] 재리뷰 라운드가 3회에 도달했습니다.

남은 이슈:
- /review:                Critical N건 / Important N건
- security-audit:         Critical N건 / High N건
- lighthouse-performance: FAIL 지표 N개
- qa-inspector:           Critical N건

진행 방식을 선택해주세요:
1. 한 번 더 시도 — 라운드 4 진행
2. 무시하고 종료 — 남은 이슈를 WARNING으로 기록하고 리포트 생성
3. 중단 — 현재까지 결과로 리포트 생성
```

## 사용자 소통

- Phase 0에서 루프 한도(기본 3회)를 안내하고, 사용자가 변경하고 싶으면 받아들인다
- Lighthouse 검사 URL을 라운드 1에서 한 번만 확인하고 이후 라운드에서는 재사용한다
- 각 라운드의 통합 결과를 모두 보여준 뒤 수정 진행 여부를 묻는다 — 자동으로 수정을 강행하지 않는다
- Critical/High 이슈는 수정 코드를 구체적으로 제안한다
- 의존성 업데이트 권고 시 breaking change 여부를 함께 안내한다
- 동일 이슈가 반복해서 잡히면 수정 패턴 자체에 문제가 있을 수 있음을 지적한다

## 후속 단계 (별도 워크플로)

커밋은 이 커맨드 범위 밖이다. 이 커맨드 종료 후 커밋이 필요하면 별도 커밋 워크플로를 사용한다.

## 병렬 실행 안티패턴

다음은 하지 말 것:

- ❌ /simplify 먼저 끝내고 다른 검사를 시작하는 순차 실행 — 선택된 항목 수와 무관하게 항상 병렬
- ❌ 선택된 2~4개 항목을 "적으니까 그냥 순차로" 돌리기 — 항목 수가 줄어도 병렬 규칙은 동일
- ❌ /simplify 결과를 다른 검사의 입력으로 넣기 — 선택된 항목들은 독립적으로 평가되어야 한다
- ❌ Phase 0에서 사용자 선택을 받지 않고 5개 모두 spawn — 반드시 선택을 먼저 확인
- ❌ 미선택 항목을 라운드 N에서 spawn하지 않다가 라운드 N+1에서 갑자기 추가 — 선택은 세션 동안 고정 (사용자가 명시적으로 변경 요청한 경우만 갱신)
- ❌ 한 라운드에서 /simplify의 자동 적용을 거부하고 다음 라운드에서 같은 제안을 다시 받는 무한 루프 — 거부 항목은 라운드 간에 기억한다
- ❌ 보안 Critical 또는 lighthouse FAIL을 무시하고 PASS 판정 — 배포 가능 기준을 임의로 완화하지 말 것
- ❌ 라운드마다 Lighthouse URL을 다시 묻기 — 라운드 1에서 확정한 URL을 사용
