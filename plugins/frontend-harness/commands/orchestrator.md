# Orchestrator — 프론트엔드 개발 전체 워크플로우 오케스트레이터

사용자의 요구사항을 받아 **분석 → 설계 → 개발 → 리뷰 → 검증**까지 전체 흐름을 커맨드로 순차 실행하고, 최종 결과를 통합하여 전달하는 커맨드.

> **커밋은 이 오케스트레이터에서 다루지 않는다.** Phase 4(Review)는 `/simplify` + `/review` + `security-audit` + `lighthouse-performance` + `qa-inspector` 5개 관점을 병렬 실행하고 수정·재리뷰 루프를 수행한다. Phase 5(Verify)는 E2E + 타입/빌드 검사로 동작과 빌드 가능성을 확인한다. 종료 후 커밋이 필요하면 별도로 `/commit` (git-harness)을 사용한다.

## Phase 순서 설계 의도

리뷰가 검증보다 먼저 오는 이유: 정적 분석으로 잡을 수 있는 코드 품질·보안·성능·정합성 이슈를 먼저 해소한 뒤, 비싼 E2E 브라우저 검증을 수행한다. 그렇지 않으면 코드 결함이 있는 상태로 E2E를 돌려 실패가 누적되거나, E2E 통과 후 리뷰에서 문제를 잡아 다시 처음으로 돌아가는 낭비가 발생한다.

## 왜 오케스트레이터가 필요한가

프론트엔드 개발은 요구사항 분석, 설계, 구현, 검증, 리뷰가 긴밀하게 연결된다. 각 단계를 독립된 커맨드로 분리하면 단계별 전문성을 살리면서도 체계적인 작업이 가능하다. 오케스트레이터는 이 커맨드들 사이에서 산출물을 전달하고 흐름을 제어하는 지휘자 역할을 한다.

## 실행 절차

### Phase 0: 요구사항 접수

사용자에게 개발하려는 기능이나 프로젝트에 대해 설명을 요청한다. 이미 대화 중 요구사항이 제시되었다면 이 단계를 건너뛴다.

```
프론트엔드 개발 전체 워크플로우를 시작합니다.

분석 → 설계 → 개발 → 리뷰 → 검증 순으로 진행됩니다.
각 단계는 독립 커맨드가 수행하고, 저는 결과를 통합하여 전달합니다.

개발하려는 기능이나 프로젝트에 대해 설명해주세요.
```

사용자의 요구사항 원문을 그대로 보존한다. 이후 각 커맨드에 전달할 때 원문을 포함해야 맥락이 유지된다.

### Phase 1: Research 커맨드 실행 — 요구사항 분석

`./commands/research.md`를 읽고 그 프로세스를 따라 요구사항 분석을 수행한다.

research 커맨드는 5단계(초기 분석 → grill-me 인터뷰 → 리서치 → 명세 작성 → 확인)로 구성된다. **각 단계를 완료할 때마다 결과를 사용자에게 보여주고, 승인을 받은 뒤 다음 단계로 진행한다.** 전체 질문이나 분석 결과를 한꺼번에 제시하지 않는다.

**단계별 진행 규칙:**

1. **초기 분석** — 요구사항을 분석한 결과에서 **명시된 요구사항, 암묵적 요구사항, 기술적 제약**만 사용자에게 보여주고, 수정·보완 의견을 확인한 뒤 다음으로 넘어간다. **모호한 영역 목록은 절대 보여주지 않는다** — 모호한 영역은 2단계 인터뷰에서 1개씩 질문으로 해소한다.
2. **grill-me 인터뷰** — 서브에이전트를 spawn하여 사용자에게 질문한다. 질문은 반드시 **한 번에 1개만** 제시하고, 사용자의 답변을 받은 뒤 다음 질문을 결정한다. 질문 목록이나 모호한 영역 전체를 한번에 나열하는 것도 금지한다.
3. **리서치** — 코드베이스 탐색 및 기술 조사 결과를 사용자에게 보여주고, 추가 조사가 필요한 영역이 없는지 확인한다.
4. **명세 작성** — 요구사항 명세 초안을 사용자에게 보여주고, 수정 요청이 있으면 반영한다.
5. **최종 확인** — 확정된 명세를 요약하여 보여주고, Phase 2 진행 여부를 확인한다.

**수집할 산출물:**
- 요구사항 분석 결과 (기능/비기능 요구사항 표, 기술적 제약, 결정 로그)

Phase 1 완료 시:

```
[Research 완료] 요구사항 분석이 완료되었습니다.

- 기능 요구사항: N건
- 비기능 요구사항: N건
- 기술적 제약: N건

다음 단계(PRD 작성)로 진행할까요?
```

### Phase 2: PRD 커맨드 실행 — 개발 요구사항 정의서 작성

`./commands/prd.md`를 읽고 그 프로세스를 따라 PRD를 작성한다. Phase 1에서 수집한 요구사항 명세 전문을 입력으로 사용한다.

PRD 커맨드는 내부적으로 planner, architecture, critic 각각을 서브에이전트로 spawn하여 루프를 돌린다.

**수집할 산출물:**
- 확정된 PRD (구현 계획, 기술 아키텍처, 리스크 대응, 인수 조건 포함)

커맨드 완료 후, PRD 핵심 내용을 사용자에게 요약하여 보여준다:

```
[PRD 완료] 개발 요구사항 정의서가 확정되었습니다.

- 구현 단계: N개 Phase
- 수정 대상 파일: N개
- 리스크: 상 N건 / 중 N건 / 하 N건
- PRD 저장 위치: .claude/plans/{기능명}-prd.md

다음 단계(가이드라인 기반 개발)로 진행할까요?
```

### Phase 3: Develop — 가이드라인 기반 개발

**[필수] 스킬 선택 — 2단계 문의:**

Frontend Guidelines 실행 전, 사용자에게 **2단계로 나누어** 적용할 스킬을 확인한다. 각 단계에서 사용자의 응답을 받은 후에만 다음으로 진행한다.

**Step 1: 기본 제공 스킬 선택 (멀티 체크)**

```
[Phase 3 준비 — 1/2] 가이드라인 기반 개발을 시작하기 전에 적용할 스킬을 선택해주세요.

기본 제공 스킬 (4개):
  □ 1. a11y — 웹접근성 (WAI-ARIA, 키보드 내비게이션, 스크린리더)
  □ 2. semantic-html — 시맨틱 HTML (랜드마크, 제목 계층, 의미적 마크업)
  □ 3. seo-geo — SEO/GEO 최적화 (메타태그, 구조화 데이터, AI 검색 대응)
  □ 4. tdd — 테스트 주도 개발 (Red-Green-Refactor 루프)

적용할 스킬 번호를 선택해주세요.
- 복수 선택 가능 (예: 1, 3)
- 전체 선택: "전체" 또는 "1, 2, 3, 4"
```

사용자가 번호를 선택하면 해당 스킬만 적용 대상으로 확정한다. "전체", "all", "모두", "1, 2, 3, 4" 등 전체 선택 의사를 표현하면 4개 모두 적용한다. 반드시 1개 이상 선택해야 하며, 선택하지 않으면 다시 안내한다.

**Step 2: 추가 스킬 확인**

Step 1에서 사용자의 응답을 받은 **이후에** 추가 스킬 여부를 묻는다. Step 1과 동시에 묻지 않는다.

```
[Phase 3 준비 — 2/2] 선택하신 스킬: {선택된 스킬 목록}

위 스킬 외에 추가로 적용하고 싶은 스킬이 있으신가요?
- 있다면 스킬명을 알려주세요.
- 없다면 "없음"이라고 답변해주세요.
```

사용자의 응답을 확인한 후, 추가 스킬이 있으면 Step 1에서 선택한 스킬과 함께 포함하여 frontend-guidelines 커맨드에 전달한다. "없음", "없어", "없습니다", "괜찮습니다" 등 유사한 표현으로 답변하면 Step 1에서 선택한 스킬만으로 진행한다.

---

`./commands/frontend-guidelines.md`를 읽고 그 프로세스를 따라 가이드라인 기반 개발을 진행한다. Phase 1의 요구사항 명세 전문과 Phase 2의 확정된 PRD 전문을 입력으로 사용한다.

frontend-guidelines 커맨드는 내부적으로 선택된 스킬 각각을 서브에이전트로 spawn하여 병렬 실행한 뒤 통합한다.

**수집할 산출물:**
- 적용된 가이드라인 목록 및 각 스킬별 점검 결과
- 코드 수정 내역
- 최종 점검 결과표

커맨드 완료 후:

```
[Develop 완료] 가이드라인 기반 개발이 완료되었습니다.

- 적용된 스킬: N개
- 수정된 파일: N개
- 점검 결과: 통과 N건 / 경고 N건 / 실패 N건

다음 단계(Verify 검증)로 진행할까요?
```

### Phase 4: Review — 통합 리뷰 (5개 관점 병렬 + 재리뷰 루프)

개발이 완료된 코드에 대해 **`/simplify`**, **`/review`**, **`security-audit`**, **`lighthouse-performance`**, **`qa-inspector`** 5가지를 **병렬로 실행**하고, 수정 필요 항목이 있으면 수정 후 재리뷰 루프를 돌린다.

> **커밋은 이 Phase에서 다루지 않는다.** 종료 후 별도 플러그인의 `/commit` (git-harness)을 사용한다.

**Step 1 — 진행 여부 확인:**

```
[Phase 4] Review 단계입니다.
배포 가능 여부를 판정하기 위해 통합 리뷰를 수행합니다.

수정 필요 항목이 있으면 수정 후 재리뷰합니다 (최대 3회).

1. 진행 — 다음 단계에서 실행 항목을 선택합니다
2. 건너뛰기 — 리뷰를 건너뛰고 Verify로 진행
```

**Step 2 — 실행 항목 선택 (진행 선택 시):**

사용자에게 5개 관점을 모두 보여주고 실행할 항목을 선택받는다. 사용자가 답변하기 전에는 다음으로 진행하지 않는다.

```
[Phase 4] 병렬로 실행할 항목을 선택해주세요:
  □ 1. /simplify              — 코드 간소화 (자동 적용)
  □ 2. /review                — PR 리뷰 (코드 품질, 버그)
  □ 3. security-audit         — OWASP Top 10, 의존성, 시크릿
  □ 4. lighthouse-performance — Core Web Vitals 측정
  □ 5. qa-inspector           — 모듈 간 경계면 정합성

선택 방법:
- 복수 선택 가능 (예: "1, 3, 5")
- 전체 선택: "전체" 또는 "1, 2, 3, 4, 5"
- 최소 1개 이상 선택해야 합니다
```

선택 해석은 review.md의 규칙을 따른다 (전체/번호 목록/단일 번호 등). 선택이 끝나면 사용자에게 한 번 더 확인한 뒤 4-1로 진행한다.

#### 4-1. 통합 리뷰 실행

`./commands/review.md`를 읽고 그 프로세스를 따른다. **Step 2에서 선택된 항목만** 실행 대상이며, **선택된 항목 수에 관계없이 항상 병렬**(단일 메시지에서 동시 spawn)로 실행한다.

**병렬 실행 — 선택된 항목만 spawn:**

| # | 서브에이전트              | 역할                                          | 사용 도구                                    |
| - | ------------------------ | --------------------------------------------- | -------------------------------------------- |
| 1 | /simplify                | 코드 간소화 + 자동 적용                        | `code-simplifier` 서브에이전트                |
| 2 | /review                  | PR 리뷰 (Critical/Important/Suggestion 분류)  | 빌트인 `/review` 스킬                         |
| 3 | security-audit           | OWASP Top 10, 의존성, 보안 헤더, 시크릿        | `./skills/security-audit/SKILL.md`            |
| 4 | lighthouse-performance   | Core Web Vitals(LCP, CLS, INP, TTFB, FCP) 측정 | `./skills/lighthouse-performance/SKILL.md`    |
| 5 | qa-inspector             | API↔훅, 라우팅, 상태 전이, 데이터 흐름 교차 검증 | `./skills/qa-inspector/SKILL.md`              |

미선택 항목은 spawn하지 않으며, 통합 리포트의 해당 섹션은 "⏭️ (미선택)"으로 표기하거나 생략한다.

**Critical/High/FAIL 항목 발견 시 즉시 사용자에게 알리고 수정을 권한다.**

#### 4-2. 수정 → 재리뷰 루프

다섯 결과를 통합하여 수정 필요 항목을 정리한 뒤:

- 수정 항목 0건 → 종료
- 수정 항목 있음 → 사용자 승인 후 적용 → review의 Phase 1로 돌아가 재리뷰 (라운드 +1)
- 라운드 3회 도달 → 사용자에게 계속/종료 결정 위임

**Review 완료 시:**

```
[Review 완료] 통합 리뷰가 완료되었습니다.

- 라운드 수: N회
- /simplify:              누적 N건 자동 적용
- /review:                Critical N / Important N (잔여)
- security-audit:         Critical N / High N (잔여)
- lighthouse-performance: Score N / FAIL 지표 N
- qa-inspector:           Critical N / Warning N (잔여)
- 배포 가능 판정: YES / NO

다음 단계(Verify 검증)로 진행할까요?

[참고] 커밋은 git-harness 플러그인의 `/commit`을 별도로 실행하세요.
```

### Phase 5: Verify — 동작 및 빌드 검증

리뷰를 통과한 코드에 대해 **E2E 브라우저 검증**과 **타입/빌드 검사**를 수행하여 실행 시점의 동작과 배포 가능한 산출물 생성 여부를 확인한다.

**사용자에게 검증 항목을 확인한다:**

```
[Phase 5] Verify 검증 단계입니다.
리뷰를 통과한 코드의 동작과 빌드 가능성을 검증합니다.

검증 항목을 선택해주세요:
  □ 1. E2E 브라우저 검증 — PRD 인수 조건 기반 실제 브라우저 테스트
  □ 2. 타입/빌드 검사 — tsc --noEmit, npm run build

- 복수 선택 가능 (예: 1, 2)
- 전체 선택: "전체"
- 건너뛰기: "건너뛰기"
```

`./commands/verify.md`를 읽고 그 프로세스를 따른다.

#### 5-1. E2E 브라우저 검증 (선택 시)

`./commands/verifier.md`를 읽고 프로세스를 따른다.

- Chrome MCP / Playwright MCP / Agent-Browser 중 도구 선택
- PRD 인수 조건에서 검증 시나리오 생성
- 실제 브라우저에서 시나리오 실행

**서브에이전트로 실행 시 전달 입력:**
1. Phase 2에서 확정된 PRD의 인수 조건 전문
2. Phase 4에서 확정된 변경 파일 목록 (`git diff --name-only`)
3. 산출물 요청: 시나리오별 PASS/FAIL 결과, 실패 항목 상세, 성공률

#### 5-2. 타입/빌드 검사 (선택 시)

Bash 도구로 직접 수행한다.

```bash
# TypeScript 타입 검사 (npm run typecheck 우선)
npm run typecheck || npx tsc --noEmit

# 빌드 검사
npm run build
```

`tsconfig.json`이 없으면 타입 검사를 건너뛰고, `package.json`에 `build` 스크립트가 없으면 빌드 검사를 건너뛴다.

**실패 처리:**

```
Verify 검증에서 N건의 실패가 발생했습니다.

1. 수정 후 재검증 — 코드 수정 후 Phase 5 재실행
2. Phase 4(Review)로 돌아가기 — 코드 품질·정합성 이슈 가능성, 재리뷰 수행
3. 무시하고 계속 — 경고로 기록하고 Phase 6(리포트)로 진행
4. 중단 — 현재까지 결과로 리포트 생성
```

**Verify 완료 시:**

```
[Verify 완료] 검증이 완료되었습니다.

- E2E:      ✅/⚠️/⏭️ (N시나리오 중 N통과)
- 타입/빌드: ✅/⚠️/⏭️

최종 통합 리포트를 생성합니다.
```

## 단계 간 산출물 전달 규칙

각 커맨드는 순차적으로 실행되며, 이전 단계의 산출물을 다음 커맨드 실행 시 입력으로 **전문 포함**하여 전달해야 한다.

- Phase 2 (PRD) → Phase 1의 요구사항 명세 전문을 포함
- Phase 3 (Develop) → Phase 1의 명세 + Phase 2의 PRD 전문을 포함
- Phase 4 (Review) → Phase 3의 변경 파일 목록 + Phase 2의 PRD 핵심을 포함 (lighthouse·qa-inspector가 인수 조건과 데이터 모델을 참조)
- Phase 5 (Verify) → Phase 2의 PRD 인수 조건 + Phase 4 종료 시점의 변경 파일 목록(자동 수정 반영 후)을 포함

산출물이 지나치게 긴 경우(토큰 제한 우려), 핵심 섹션을 선별하되 "기능 요구사항 표", "구현 계획", "수정 대상 파일", "인수 조건" 등 구현과 검증에 직접 필요한 정보는 반드시 포함한다.

## 사용자 소통

- **각 Phase 완료 후 반드시 사용자에게 결과를 보여주고 승인/거절을 확인받은 뒤 다음 Phase로 진행한다. 사용자 승인 없이 자동으로 다음 단계를 진행하지 않는다.**
- 각 Phase 시작 시: "N단계(커맨드명)를 시작합니다" 안내
- 각 Phase 완료 시: 핵심 산출물 요약 + "다음 단계로 진행할까요?" 확인 요청
- 커맨드 실행 중 사용자 입력이 필요한 경우: 해당 질문을 사용자에게 전달하고 응답을 반영
- 문제 발생 시: 어떤 단계에서 무엇이 문제인지 명확히 전달하고, 재시도 여부를 사용자에게 확인

## 조기 종료

사용자가 특정 단계까지만 진행하길 원할 수 있다.

- "분석과 PRD까지만 해줘" → Phase 2까지만 실행 후 리포트
- "개발까지만 해줘" → Phase 3까지 실행 후 리포트
- "리뷰까지만 해줘" → Phase 4까지 실행 후 리포트
- "검증까지만 해줘" → Phase 5까지 실행 후 리포트 (= 전체 실행)

Phase 4(Review)와 Phase 5(Verify)의 각 하위 항목은 개별 건너뛰기가 가능하다.
