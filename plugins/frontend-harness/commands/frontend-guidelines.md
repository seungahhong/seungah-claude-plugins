# Frontend Guidelines — 프론트엔드 개발 가이드라인 커맨드

프론트엔드 개발 시 적용할 품질 가이드라인 스킬을 사용자와 함께 선택하고, 선택된 스킬을 각각 **서브에이전트**로 spawn하여 개발을 진행하는 커맨드.

## 사용 가능한 스킬

| # | 스킬 | 경로 | 설명 |
|---|------|------|------|
| 1 | 웹접근성 (a11y) | `./skills/a11y/SKILL.md` | WAI-ARIA 기반 웹접근성 점검 및 개선 |
| 2 | 시맨틱 HTML | `./skills/semantic-html/SKILL.md` | 시맨틱 태그 사용 점검 및 개선 |
| 3 | SEO/GEO | `./skills/seo-geo-optimizer/SKILL.md` | 검색엔진 + AI 검색 최적화 |
| 4 | TDD | `./skills/tdd/SKILL.md` | Red-Green-Refactor 루프 기반 테스트 주도 개발 |

## 실행 절차

### Phase 1: 스킬 선택

개발을 시작하기 전에, 사용 가능한 스킬 목록을 사용자에게 보여주고 실행 범위를 확인한다.

다음과 같이 안내한다:

```
프론트엔드 개발에 적용할 수 있는 가이드라인 스킬 목록입니다:

1. 웹접근성 (a11y) — WAI-ARIA 기반 접근성 점검/개선
2. 시맨틱 HTML — 시맨틱 태그 사용 점검/개선
3. SEO/GEO — 검색엔진 + AI 검색 최적화
4. TDD — Red-Green-Refactor 테스트 주도 개발

어떻게 진행할까요?
- **전체 실행**: 4개 스킬 모두 적용
- **일부 실행**: 필요한 스킬만 선택하여 적용
```

**전체 실행**을 선택하면 4개 스킬 모두를 적용 대상으로 설정한다.

**일부 실행**을 선택하면 각 스킬에 대해 하나씩 포함 여부를 확인한다:

```
각 스킬의 적용 여부를 알려주세요:

1. 웹접근성 (a11y) — 적용할까요? (Y/N)
2. 시맨틱 HTML — 적용할까요? (Y/N)
3. SEO/GEO — 적용할까요? (Y/N)
4. TDD — 적용할까요? (Y/N)

번호와 함께 답변해주세요. (예: "1, 3만 적용" 또는 "2번 빼고 전부")
```

사용자가 자유 형식으로 답변해도 의도를 파악하여 진행한다.

#### [필수] 추가 스킬 문의 — 반드시 사용자 응답을 받아야 다음 Phase로 진행

스킬 선택이 완료되면, **반드시(MUST)** 추가로 적용할 스킬이 있는지 사용자에게 확인한다. 이 질문은 생략하거나 건너뛸 수 없으며, **사용자의 명시적 응답을 받기 전까지 Phase 2로 진행하지 않는다.**

```
위 4개 외에 추가로 적용할 스킬이 있나요?

있다면 슬래시 커맨드로 입력해주세요.
예) /frontend-design — 디자인 기반 UI 개발

없으면 "없음"이라고 답변해주세요.
```

**진행 조건:**
- 사용자가 추가 스킬을 입력하면 → 해당 스킬을 적용 대상 목록에 포함하여 함께 실행한다.
- 사용자가 "없음"이라고 답변하면 → 다음 Phase로 진행한다.
- 사용자가 응답하지 않았거나 모호한 답변을 한 경우 → 다시 한번 명확히 확인한다.

> **주의:** 이 단계를 건너뛰고 Phase 2를 시작하면 안 된다. 사용자가 원하는 추가 스킬이 누락될 수 있기 때문이다.

### Phase 2: 서브에이전트 병렬 실행

선택된 각 스킬을 Agent 도구를 사용하여 **독립적인 서브에이전트**로 spawn하여 병렬로 실행한다. 각 서브에이전트는 자신의 스킬만 담당하며, 독립된 컨텍스트에서 작업한다.

**서브에이전트 spawn 규칙:**

선택된 스킬마다 Agent 도구를 사용하여 서브에이전트를 생성한다. 스킬 간 의존성이 없으므로 **모든 서브에이전트를 동시에 spawn**한다.

각 서브에이전트에게 전달할 프롬프트 구성:

```
1. 대상 코드/파일의 현재 상태 (경로, 컨텍스트)
2. 사용자의 개발 요구사항
3. 담당 스킬의 SKILL.md 경로와 해당 스킬의 지침을 따를 것
4. 산출물: 해당 스킬 관점의 분석 결과, 수정 제안, 체크리스트 점검 결과
```

| 스킬 | 서브에이전트 지시 | 스킬 경로 |
|------|------------------|-----------|
| 웹접근성 | WAI-ARIA 기반 접근성 점검 및 개선안 도출 | `./skills/a11y/SKILL.md` |
| 시맨틱 HTML | 시맨틱 태그 사용 점검 및 개선안 도출 | `./skills/semantic-html/SKILL.md` |
| SEO/GEO | 검색엔진 + AI 검색 최적화 점검 및 개선안 도출 | `./skills/seo-geo-optimizer/SKILL.md` |
| TDD | Red-Green-Refactor 기반 테스트 작성 및 개발 | `./skills/tdd/SKILL.md` |

각 서브에이전트는 스킬의 `references/` 디렉토리 참조 문서가 필요하면 자체적으로 읽어서 활용한다.

### Phase 3: 결과 통합 및 개발 진행

모든 서브에이전트의 실행이 완료되면, 각 결과를 수집하여 **하나의 통합된 개발 계획**으로 병합한다.

**통합 시 적용 우선순위:**

```
TDD 결과 (선택 시)
 └→ 테스트 정의가 구현의 기반이 되므로 최우선 반영

시맨틱 HTML 결과 (선택 시)
 └→ HTML 구조가 접근성과 SEO의 기반이 되므로 두 번째 반영

웹접근성 결과 (선택 시)
 └→ 시맨틱 구조 위에 ARIA, 키보드, 포커스 관리를 레이어링

SEO/GEO 결과 (선택 시)
 └→ 메타태그, 구조화 데이터, 콘텐츠 최적화를 최종 레이어로 적용
```

**통합 시 유의사항:**
- 스킬 간 겹치는 영역(예: 시맨틱 HTML의 랜드마크 구조와 접근성의 ARIA 역할)은 중복을 제거하고 통합하여 적용한다
- 서브에이전트 간 결과가 충돌하는 경우, 접근성 > 시맨틱 구조 > SEO 순으로 우선한다
- 통합된 결과를 바탕으로 실제 코드 수정을 진행한다
- 코드 리뷰 시 각 서브에이전트의 체크리스트를 기준으로 검토한다

### Phase 4: 최종 점검

개발 완료 후, 선택된 스킬별 체크리스트로 최종 점검을 수행한다.

```markdown
## 최종 점검 결과

### 적용된 가이드라인
| 스킬 | 상태 | 비고 |
|------|------|------|
| 웹접근성 | ✅/⚠️/❌ | ... |
| 시맨틱 HTML | ✅/⚠️/❌ | ... |
| SEO/GEO | ✅/⚠️/❌ | ... |
| TDD | ✅/⚠️/❌ | ... |

### 미충족 항목
- [ ] ...

### 권고사항
- ...
```
