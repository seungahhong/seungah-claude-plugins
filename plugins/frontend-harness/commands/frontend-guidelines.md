# Frontend Guidelines — 프론트엔드 개발 가이드라인 커맨드

프론트엔드 개발 시 적용할 품질 가이드라인 스킬을 사용자와 함께 선택하고, 선택된 스킬을 **서브에이전트**로 spawn하여 개발을 진행하는 커맨드. **TDD가 선택되면 TDD를 먼저 실행(선행)** 해 인수조건 기반 구현·테스트를 완성한 뒤, 나머지 스킬을 병렬로 실행한다.

## 입력 — 인수조건 계약 (PRD가 있으면 반드시 수령)

PRD(§2.3 사용자 스토리 + 인수조건, §7 인수조건 체크리스트)가 선행 단계에서 만들어졌다면, 그 **인수조건(AC-n.m) 목록을 입력으로 받는다**. 이 AC가 (1) TDD의 RED 테스트 목록 근거, (2) 최종 점검의 기능단위 판정 기준, (3) 이후 Verify·수렴 루프의 종료 기준이 된다.

**PRD가 없으면(단독 실행 등) Phase 1.5 미니 AC 도출을 수행한다** — 이 커맨드가 사용자와 함께 요구사항에서 관찰형 인수조건(Given/When/Then)을 도출하고, 각 AC에 **AC-ID와 검증 레인(E2E/TDD/수동)** 을 부여한 뒤 **사용자 승인**을 받고 Phase 2로 진입한다(스킬 선택 Phase 1 직후, TDD 서브에이전트 spawn 전). 요구·설계가 복잡하면 먼저 `/prd`를 권한다.

> **수렴 루프 재호출 모드** — 오케스트레이터 Phase 5(Verify)에서 E2E 자동 충족률이 100%가 아니어서 이 커맨드로 회귀한 경우, **미충족 E2E AC 목록만 입력으로 받는다**.
> - **(1) 최소 변경** — 전체 재개발이 아니라 해당 AC를 충족시키는 최소 변경(기능 추가/수정)에 집중하고, TDD가 선택되어 있으면 미충족 AC를 RED 테스트로 먼저 세운 뒤 GREEN으로 채운다.
> - **(2) 스킬 재선택 금지** — 최초 라운드에서 확정한 스킬·TDD 선택을 그대로 재사용하고 **Phase 1(스킬 선택·추가 스킬 문의)을 건너뛴다**(라운드마다 스킬을 다시 묻지 않는다).
> - **(3) 품질 레이어 재적용** — 변경된 파일에 한해 2-B 병렬 품질 스킬(웹접근성·시맨틱 HTML·SEO)을 다시 적용해 새 마크업에도 품질 레이어가 빠지지 않게 한다(이후 Review 라운드가 이를 다시 점검).

## 사용 가능한 스킬

| # | 스킬 | 경로 | 설명 |
|---|------|------|------|
| 1 | 웹접근성 (a11y) | `./skills/a11y/SKILL.md` | WAI-ARIA 기반 웹접근성 점검 및 개선 |
| 2 | 시맨틱 HTML | `./skills/semantic-html/SKILL.md` | 시맨틱 태그 사용 점검 및 개선 |
| 3 | SEO/GEO | `./skills/seo-geo-optimizer/SKILL.md` | 검색엔진 + AI 검색 최적화 |
| 4 | TDD | `./skills/tdd/SKILL.md` | Red-Green-Refactor 루프 기반 테스트 주도 개발 |

## 실행 절차

### Phase 1: 스킬 선택

> **호출측이 스킬 목록을 지정한 경우(오케스트레이터 Phase 3·수렴 루프 재호출)** — 이 Phase(스킬 선택·추가 스킬 문의 포함)를 **건너뛰고** 지정된 스킬 목록·TDD 포함 여부로 곧장 Phase 2에 진입한다(사용자에게 스킬을 중복으로 묻지 않는다). 아래 절차는 **이 커맨드를 단독 실행**할 때만 수행한다.

개발을 시작하기 전에, 사용 가능한 스킬 목록을 사용자에게 보여주고 실행 범위를 확인한다.

다음과 같이 안내한다:

```
프론트엔드 개발에 적용할 수 있는 가이드라인 스킬 목록입니다:

1. 웹접근성 (a11y) — WAI-ARIA 기반 접근성 점검/개선
2. 시맨틱 HTML — 시맨틱 태그 사용 점검/개선
3. SEO/GEO — 검색엔진 + AI 검색 최적화
4. TDD — Red-Green-Refactor 테스트 주도 개발

어떻게 진행할까요?
- **전체 실행**: 4개 스킬 모두 적용
- **일부 실행**: 필요한 스킬만 선택하여 적용
```

**전체 실행**을 선택하면 4개 스킬 모두를 적용 대상으로 설정한다.

**일부 실행**을 선택하면 각 스킬에 대해 하나씩 포함 여부를 확인한다:

```
각 스킬의 적용 여부를 알려주세요:

1. 웹접근성 (a11y) — 적용할까요? (Y/N)
2. 시맨틱 HTML — 적용할까요? (Y/N)
3. SEO/GEO — 적용할까요? (Y/N)
4. TDD — 적용할까요? (Y/N)

번호와 함께 답변해주세요. (예: "1, 3만 적용" 또는 "2번 빼고 전부")
```

사용자가 자유 형식으로 답변해도 의도를 파악하여 진행한다.

#### [필수] 추가 스킬 문의 — 반드시 사용자 응답을 받아야 다음 Phase로 진행

스킬 선택이 완료되면, **반드시(MUST)** 추가로 적용할 스킬이 있는지 사용자에게 확인한다. 이 질문은 생략하거나 건너뛸 수 없으며, **사용자의 명시적 응답을 받기 전까지 Phase 2로 진행하지 않는다.** (단, 이 필수 게이트는 **단독 실행 시**에만 적용된다 — 위 "호출측이 스킬 목록을 지정한 경우"에는 Phase 1 전체를 건너뛰므로 이 문의도 생략한다.)

```
위 4개 외에 추가로 적용할 스킬이 있나요?

있다면 슬래시 커맨드로 입력해주세요.
예) /frontend-design — 디자인 기반 UI 개발

없으면 "없음"이라고 답변해주세요.
```

**진행 조건:**
- 사용자가 추가 스킬을 입력하면 → 해당 스킬을 적용 대상 목록에 포함하여 함께 실행한다.
- 사용자가 "없음"이라고 답변하면 → 다음 Phase로 진행한다.
- 사용자가 응답하지 않았거나 모호한 답변을 한 경우 → 다시 한번 명확히 확인한다.

> **주의:** 이 단계를 건너뛰고 Phase 2를 시작하면 안 된다. 사용자가 원하는 추가 스킬이 누락될 수 있기 때문이다.

### Phase 2: 서브에이전트 실행 — TDD 선행 → 나머지 병렬

선택된 스킬을 Agent 도구로 **독립적인 서브에이전트**(모든 Agent 호출에 `model: "opus"` 명시)로 실행한다. **실행 순서는 TDD 선택 여부로 갈린다.**

```
TDD가 선택에 포함됨?
 ├─ 예 → 2-A. TDD 선행(단독·순차 완료) → 2-B. 나머지 선택 스킬 병렬
 └─ 아니오 → 2-B만: 선택된 모든 스킬을 한 번에 병렬 spawn
```

**왜 TDD를 먼저 실행하는가** — TDD의 Red-Green-Refactor는 인수조건을 테스트로 고정한 뒤 **실제 구현(GREEN)** 을 만들어낸다. 반면 웹접근성·시맨틱 HTML·SEO/GEO는 그 구현된 마크업 위에 **점검·개선을 레이어링**하는 스킬이다. 구현이 존재하기 전에 이들을 병렬로 돌리면 아직 없거나 흔들리는 코드를 검토하게 된다. 그래서 TDD로 인수조건을 충족하는 안정된 구현을 먼저 세우고, 그 위에 나머지 품질 레이어를 병렬로 얹는다.

#### 2-A. TDD 선행 (TDD 선택 시에만)

TDD 서브에이전트 **하나만** 먼저 spawn하고 **완료될 때까지 기다린다**(이 단계는 병렬 아님·순차). 완료 산출물이 이후 2-B 병렬 스킬들의 입력(대상 코드)이 된다.

TDD 서브에이전트 프롬프트 구성:

```
1. 대상 코드/파일의 현재 상태 (경로, 컨텍스트)
2. 사용자의 개발 요구사항
3. 인수조건 목록(AC-n.m, Given/When/Then) — 이것이 RED 테스트 목록의 근거다. AC 1건 → 테스트 ≥1개.
   (수렴 루프 재호출이면 미충족 AC만 대상)
4. 스킬 경로: ./skills/tdd/SKILL.md — 수직 슬라이싱(RED→GREEN을 AC 단위로)으로 진행할 것
5. 산출물: AC별 테스트 파일 + 통과하는 최소 구현(GREEN) + 리팩토링 결과, AC↔테스트 매핑표
```

TDD가 완료되면 **AC별 테스트 통과 여부**를 요약해 사용자에게 보여준 뒤 2-B로 진행한다.

#### 2-B. 나머지 선택 스킬 병렬 실행

TDD를 제외한 선택 스킬(웹접근성·시맨틱 HTML·SEO/GEO + 추가 스킬)을 **하나의 어시스턴트 메시지에서 동시에 spawn**한다(스킬 간 의존성 없음). TDD가 선택되지 않았다면 선택된 모든 스킬이 여기서 병렬 실행된다. 대상 코드는 2-A의 TDD 산출물(있으면) 또는 현재 코드다.

> **2-B 대상이 없으면(예: TDD 단독 선택) 2-B를 생략하고 곧장 Phase 3로 진행한다.** 공집합에 spawn을 시도하지 않는다.

각 서브에이전트에게 전달할 프롬프트 구성:

```
1. 대상 코드/파일의 현재 상태 (2-A TDD 산출물 경로 포함, 있으면)
2. 사용자의 개발 요구사항 + 관련 인수조건(AC)
3. 담당 스킬의 SKILL.md 경로와 해당 스킬의 지침을 따를 것
4. 산출물: 해당 스킬 관점의 분석 결과, 수정 제안, 체크리스트 점검 결과 (TDD 테스트를 깨지 않는 개선)
```

| 스킬 | 서브에이전트 지시 | 스킬 경로 | 실행 |
|------|------------------|-----------|------|
| TDD | 인수조건 RED→GREEN→Refactor 기반 구현·테스트 | `./skills/tdd/SKILL.md` | **선행(2-A, 순차)** |
| 웹접근성 | WAI-ARIA 기반 접근성 점검 및 개선안 도출 | `./skills/a11y/SKILL.md` | 병렬(2-B) |
| 시맨틱 HTML | 시맨틱 태그 사용 점검 및 개선안 도출 | `./skills/semantic-html/SKILL.md` | 병렬(2-B) |
| SEO/GEO | 검색엔진 + AI 검색 최적화 점검 및 개선안 도출 | `./skills/seo-geo-optimizer/SKILL.md` | 병렬(2-B) |

각 서브에이전트는 스킬의 `references/` 디렉토리 참조 문서가 필요하면 자체적으로 읽어서 활용한다.

> **주의:** TDD가 선택되었는데 2-A를 건너뛰고 TDD를 다른 스킬과 함께 병렬 spawn하지 않는다. 2-B의 병렬 개선이 TDD 테스트를 깨면, 해당 스킬의 개선을 조정해 테스트가 다시 통과하도록 맞춘다(테스트가 계약이다).

### Phase 3: 결과 통합 및 개발 진행

모든 서브에이전트의 실행이 완료되면, 각 결과를 수집하여 **하나의 통합된 개발 계획**으로 병합한다.

**통합 시 적용 우선순위:**

```
TDD 결과 (선택 시 · 2-A에서 이미 실행됨)
 └→ 인수조건을 고정한 테스트 + GREEN 구현이 나머지 개선의 기반(계약). 뒤 레이어는 이 테스트를 깨지 않는다.

시맨틱 HTML 결과 (선택 시 · 2-B 병렬)
 └→ HTML 구조가 접근성과 SEO의 기반이 되므로 두 번째 반영

웹접근성 결과 (선택 시 · 2-B 병렬)
 └→ 시맨틱 구조 위에 ARIA, 키보드, 포커스 관리를 레이어링

SEO/GEO 결과 (선택 시 · 2-B 병렬)
 └→ 메타태그, 구조화 데이터, 콘텐츠 최적화를 최종 레이어로 적용
```

**통합 시 유의사항:**
- 스킬 간 겹치는 영역(예: 시맨틱 HTML의 랜드마크 구조와 접근성의 ARIA 역할)은 중복을 제거하고 통합하여 적용한다
- 서브에이전트 간 결과가 충돌하는 경우, 접근성 > 시맨틱 구조 > SEO 순으로 우선한다
- 통합된 결과를 바탕으로 실제 코드 수정을 진행한다
- 코드 리뷰 시 각 서브에이전트의 체크리스트를 기준으로 검토한다

### Phase 4: 최종 점검 (기능단위 인수조건 평가 + 스킬별 체크리스트)

개발 완료 후 두 축으로 점검한다 — (1) **기능(FR/US) 단위로 인수조건(AC) 충족 여부**, (2) 선택된 스킬별 체크리스트. (1)의 개발 관점 충족률은 **선행 지표**이며, 수렴 루프의 **최종 종료 기준은 Verify의 E2E 자동 충족률**이다(TDD가 GREEN이어도 E2E는 실패할 수 있으므로 둘을 동일시하지 않는다).

```markdown
## 최종 점검 결과

### 기능단위 인수조건 평가 (개발 관점 · TDD 테스트 기준)
| FR / US | 인수조건(AC) | TDD 테스트 | 충족 |
|---------|--------------|-----------|------|
| FR-1 / US-1 | AC-1.1 | ✅ 통과 | ✅ |
| FR-1 / US-1 | AC-1.2 `[수동]` | (수동 확인) | ⏳ 사용자 확인 대상 |
| FR-2 / US-2 | AC-2.1 | ❌ 실패 | ❌ |
```
