# Verify — 동작 및 빌드 검증 커맨드

개발 완료된 코드에 대해 **E2E 브라우저 검증**과 **타입/빌드 검사**를 수행한다.

> 성능(Lighthouse), 통합 정합성(QA), 보안 감사, 코드 품질 리뷰는 이 커맨드의 범위가 아니다. 모두 `/review`에서 병렬로 처리한다.

## 왜 Verify가 필요한가

`/review`가 정적 분석으로 코드의 "안전한가/품질은 좋은가"를 본다면, Verify는 **실행 시점의 동작과 빌드 가능성**을 확인한다.

- **E2E**: PRD 인수 조건이 실제 브라우저에서 통과하는가
- **타입/빌드**: 배포 가능한 산출물을 만들 수 있는가

## 검증 파이프라인

```
┌──────────────────────────────────────────┐
│                                          │
│  /review 통과 코드                       │
│       │                                  │
│       ▼                                  │
│  ┌──────────────────┐                    │
│  │ 1. E2E 브라우저  │  PRD 인수 조건     │
│  │    검증          │  실제 시나리오 실행 │
│  └──────┬───────────┘                    │
│         │                                │
│         ▼                                │
│  ┌──────────────────┐                    │
│  │ 2. 타입/빌드     │  tsc --noEmit      │
│  │    검사           │  npm run build     │
│  └──────┬───────────┘                    │
│         │                                │
│         ▼                                │
│  ┌──────────────────┐                    │
│  │ 3. 검증 결과      │  통합 리포트       │
│  │    통합 보고      │  PASS/FAIL 판정    │
│  └──────────────────┘                    │
│                                          │
└──────────────────────────────────────────┘
```

## 실행 절차

### Step 0: 검증 항목 선택

```
[Verify] 검증 항목을 선택해주세요.

1. 전체 검증 — E2E + 타입/빌드 (기본)
2. E2E만 — PRD 인수 조건 기반 브라우저 검증
3. 타입/빌드만 — tsc --noEmit, npm run build
4. 건너뛰기
```

> **인수조건(AC) 목록 확보(E2E 검증 시)** — 오케스트레이터가 호출하면 Phase 2 PRD의 AC(§2.3·검증 레인 §2.4)를 전달받는다. **단독 실행이면** `.claude/plans/*-prd.md`를 찾아 사용하고, 여러 개면 사용자에게 대상을 확인한다. PRD·AC가 전혀 없으면 검증 기준이 없으므로 E2E를 SKIP하고 사용자에게 먼저 인수조건 확정(`/prd`)을 안내한다.

### Step 1: E2E 브라우저 검증 (인수조건 단위)

`./verifier.md`를 읽고 그 프로세스를 따른다. 내부적으로 `../skills/e2e-verifier/SKILL.md`를 로드한다. 이때 **`호출맥락=orchestrated`(상위가 메뉴 소유)를 전달**해 리프 스킬의 결정 메뉴를 억제한다 — 단독이든 오케스트레이션이든 결정 메뉴는 verify(§실패 처리) 또는 상위 오케스트레이터가 소유하므로 e2e-verifier는 메뉴를 띄우지 않고 충족률·미충족 AC만 보고한다.

- 도구 선택: Chrome MCP / Playwright MCP / Agent-Browser
- 입력: PRD 인수조건(AC-n.m) 목록 + 변경 파일 목록(`git diff --name-only`)
- **검증 단위는 E2E 레인 AC** — 브라우저로 관찰 가능한 AC(PRD §2.4의 검증 레인=E2E)만 시나리오로 만들어 PASS/FAIL을 판정한다.
- **TDD 레인 AC(단위 테스트로 검증되는 서버 로직 등)와 `[수동]` AC(E2E·TDD 모두 자동 검증 불가)는 E2E 자동 충족률 분모에서 제외**한다. TDD 레인은 Develop에서 검증되고, `[수동]`은 별도 수동 확인 체크리스트로 관리된다(둘 다 수렴 루프 대상 아님).
- 산출물: **AC별 PASS/FAIL**, 실패 항목 상세(스크린샷, 사유), **E2E 자동 충족률 = 통과 E2E AC / 전체 E2E 레인 AC**, 수동 확인 체크리스트(`[수동]` AC 목록)

**E2E 자동 충족률**이 이 커맨드의 핵심 지표다. 100% 미만이면(=E2E 레인 AC 중 미충족이 있으면) 수렴 루프(아래 "실패 처리")의 대상이 된다. **E2E 레인 AC가 0개면(예: 전부 TDD/`[수동]`) 충족률은 N/A(vacuously 통과)로 보고**하고 배포 판정은 TDD 레인·`[수동]` 확인으로 가른다.

**완료 시 사용자에게 결과를 보여주고 다음 단계 진행 여부를 확인한다.**

### Step 1-1: `[수동]` 인수조건 사용자 확인

E2E가 분리한 `[수동]` AC(브라우저로 관찰 불가능한 AC) 체크리스트를 사용자에게 제시하고 **항목별로 확인/보류를 수령**한다. 이 확인 상태가 최종 판정에서 PASS(전부 확인 완료)와 WARNING(미확인 잔여)을 가르는 게이트다.

```
[수동 확인 필요 인수조건] — 각 항목을 직접 확인하고 결과를 알려주세요.

- [ ] AC-1.2 — Given … When … Then … (확인 완료 / 보류)
- [ ] AC-3.1 — ...
```

- 사용자가 확인한 항목은 "수동 확인 완료"로 기록한다.
- 보류·미확인 항목이 남으면 최종 판정은 WARNING이 된다(PASS 아님).
- `[수동]` AC가 없으면 이 단계를 건너뛴다.
- **수렴 라운드 중 반복 방지** — 수렴 루프로 Verify가 여러 번 돌 때, `[수동]` 확인은 **E2E 자동 충족률 100%에 도달한 라운드에서 한 번** 수령한다. 앞 라운드에서 이미 확인한 항목은 상태를 캐시해 재요청하지 않는다(매 라운드 재질문 금지).
- **오케스트레이션 시에도 이 Step 1-1은 verify가 수행한다** — `[수동]` 확인/보류 수령은 "결정"이 아니라 "데이터 수집"이므로 §실패 처리의 메뉴 억제 대상이 아니다. verify가 확인/보류를 수령해 그 결과(확인 완료/미확인)만 상위(오케스트레이터)에 보고하고, 배포 게이트(오케스트레이터 §7)가 그 값을 소비한다(수집은 verify 소유, 배포 판정은 오케스트레이터 소유).

### Step 2: 타입/빌드 검사

Bash 도구로 직접 수행한다.

**Step 2-1. 프로젝트 설정 감지**

`package.json`과 `tsconfig.json`을 확인하여 실행 가능한 검사를 결정한다.

- `tsconfig.json` 존재 → TypeScript 타입 검사 가능
- `package.json`의 `scripts.typecheck` 존재 → `npm run typecheck` 우선 사용
- `package.json`의 `scripts.build` 존재 → 빌드 검사 가능

**Step 2-2. TypeScript 타입 검사**

```bash
# typecheck 스크립트가 있으면 우선
npm run typecheck

# 없으면 직접 실행
npx tsc --noEmit
```

`tsconfig.json`이 없으면 건너뛴다.

**Step 2-3. 빌드 검사**

```bash
npm run build
```

`build` 스크립트가 없으면 건너뛴다.

### Step 3: 검증 결과 통합

```markdown
# Verify 검증 결과

## 1. E2E 브라우저 검증 (인수조건 단위)

### 인수조건별 결과

| AC | 시나리오 | 결과   | 비고 |
| --- | -------- | ------ | ---- |
| AC-1.1 | ...   | ✅/❌ | ...  |
| AC-1.2 | ... `[수동]` | ⏭️ 수동 | E2E 충족률 분모 밖 · 사용자 수동 확인 |

- **E2E 자동 충족률: 통과 E2E AC K / 전체 E2E 레인 AC M = NN%** (M=0이면 N/A)
- TDD 자동 충족률(Develop에서 보고받음): 통과 TDD AC / 전체 TDD 레인 AC = NN%
- 수동 확인 체크리스트(`[수동]` AC): N개 (충족률 밖 · 사용자 수동 확인 대상)
- 미충족 E2E 인수조건: AC-... (수렴 루프 대상)

## 2. 타입/빌드 검사

| 검사 항목  | 상태     | 세부     |
| ---------- | -------- | -------- |
| TypeScript | ✅/❌/⏭️ | N errors |
| Build      | ✅/❌/⏭️ | ...      |

## 3. 종합 판정

| 검증     | 결과                          |
| -------- | ----------------------------- |
| E2E      | ✅ PASS / ⚠️ WARNING / ❌ FAIL |
| 타입/빌드 | ✅ PASS / ⚠️ WARNING / ❌ FAIL |
| **종합**  | **PASS / WARNING / FAIL**    |

## 4. 미해결 사항
- ...
```

**판정 기준:**

> **판정 대상 = 3 레인**: E2E 자동 충족률(이 커맨드가 측정) · TDD 자동 충족률(Develop에서 보고받음) · `[수동]` 확인. 어떤 레인의 AC가 0개면 그 레인은 N/A(vacuously 통과)로 본다.

- **PASS**: E2E 자동 충족률 **100%**(미충족 E2E AC 0건) + **TDD 자동 충족률 100%** + `[수동]` AC 전부 사용자 수동 확인 완료 + 타입/빌드 PASS
- **WARNING**: 다음 중 하나 — (a) E2E·TDD 레인은 전부 통과했으나 `[수동]` 수동 확인이 남음, (b) 미충족 E2E AC가 남았으나 **전부 사용자가 "알려진 미충족"으로 명시 승인**함, (c) 타입 경고
- **FAIL**: **사용자가 승인하지 않은** 미충족 E2E AC가 1건 이상, 또는 TDD 자동 충족률 100% 미만(단위 테스트 실패 → Develop 미완료), 또는 타입/빌드 FAIL

> WARNING(b)와 FAIL은 배타적이다 — 미충족 E2E AC가 **전부 승인**되면 WARNING(b), **하나라도 미승인**이면 FAIL. 승인 여부로만 가른다.

## 실패 처리 — 인수조건 수렴 루프 (Develop → Review → Verify)

**E2E 자동 충족률이 100% 미만**(=E2E 레인 AC 중 미충족 ≥1건)이면, 미충족 E2E AC를 없앨 때까지 **개발→리뷰→검증을 반복**한다. 이 루프의 목적은 E2E 레인 AC를 100% 충족시키는 것이다. TDD 레인 실패(단위 테스트 미통과)는 Develop에서, `[수동]` AC는 별도 수동 확인으로 처리한다(둘 다 루프 밖).

> **호출 맥락 판별(메뉴 소유권)** — **오케스트레이터 Phase 5로 호출된 경우** 아래 결정 메뉴를 **출력하지 않는다**. E2E 자동 충족률·미충족 E2E AC·`[수동]` 체크리스트만 보고하고, 수렴/회귀 결정은 **오케스트레이터 Phase 5-3 수렴 루프**에 위임한다(Develop 회귀는 오케스트레이터 소유이므로 verify가 주도하지 않는다). **이 커맨드를 단독 실행**할 때만 아래 메뉴를 제시한다(단독이므로 회귀는 사용자가 직접 커맨드를 순차 실행).

```
Verify: E2E 자동 충족률 NN% (미충족 E2E AC N건 · [수동] 미확인 M건)

미충족 E2E 인수조건:
- AC-2.1 — 사유: ...
- AC-3.2 — 사유: ...

어떻게 진행할까요? (단독 실행 — 아래는 사용자가 직접 실행할 순서 안내)
1. 개발로 회귀(권장) — 사용자가 `/frontend-guidelines`를 재실행해 미충족 E2E AC만 재구현 → `/review` → `/verify` 재검증(전 과정 자동화는 `/orchestrator`로 승격)
2. 리뷰로 회귀 — 코드 품질·정합성 이슈로 보이면 `/review` 먼저 재실행 후 재검증
3. 알려진 미충족으로 기록하고 계속 — 남은 미충족 E2E AC를 **"알려진 미충족(사용자 승인)"**으로 명시(충족률 100% 미달임을 리포트에 남김)하고 다음 단계로 진행
4. 중단 — 현재까지 결과로 리포트 생성
```

**수렴 루프 규칙:**

- **트리거는 "미충족 E2E AC ≥1건"이다** — E2E 레인 AC를 전부 통과했는데 `[수동]` AC만 남은 상태는 WARNING(수동 확인 대상)이지 루프 트리거가 아니다.
- **1번(개발로 회귀)이 기본 경로다.** 미충족 E2E AC 목록만 `frontend-guidelines`로 전달하면, 그 커맨드는 재호출 모드로 해당 AC만 재구현한다. **재구현 라우팅은 AC 검증 레인에 따른다** — 미충족 AC가 단위 테스트로 표현 가능하면 RED 테스트 선행(TDD), 순수 브라우저 통합 동작이면 구현 후 E2E 재검증(단위 RED 강요 안 함). 이어 `/review`로 회귀 검증 후 다시 이 Verify를 실행한다.
- **종료 조건**: E2E 자동 충족률 100% 도달(=미충족 E2E AC 0건) → 루프 종료. 남은 `[수동]` AC는 수동 확인으로 마무리(완료 시 PASS, 미확인 시 WARNING).
- **라운드 게이트**: 매 라운드 결과(충족률 변화·남은 AC)를 사용자에게 보여주고 계속 여부를 확인한다. 자동으로 무한 반복하지 않는다.
- **소프트 캡(기본 5회) + 정체 가드**: 라운드가 5회에 도달하거나, 동일 AC가 2라운드 연속 미충족이면(진전 없음) 사용자에게 계속/중단/원인 재검토를 위임한다. 진전 없는 반복은 인수조건 자체가 브라우저로 관찰 불가하거나(→`[수동]` 재분류) 구현 접근이 잘못됐을(→ PRD/설계 재검토) 신호다.
- **PRD에 없는 인수조건은 임의로 만들지 않는다** — 새 기능이 필요하다는 판단이 서면 PRD(§2.3)에 AC를 먼저 추가하고(추적 갱신) 루프에 편입한다.

## 사용자 소통

- Step 0에서 검증 항목을 반드시 확인한다
- E2E 실패 시 스크린샷과 실패 사유를 함께 보여준다
- 타입/빌드 에러는 첫 N건만 보여주고 전체 출력은 요청 시 제공한다
- 자동 검증 불가 항목은 수동 확인 가이드를 함께 제시한다
