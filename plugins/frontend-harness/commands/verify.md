# Verify — 동작 및 빌드 검증 커맨드

개발 완료된 코드에 대해 **E2E 브라우저 검증**과 **타입/빌드 검사**를 수행한다.

> 성능(Lighthouse), 통합 정합성(QA), 보안 감사, 코드 품질 리뷰는 이 커맨드의 범위가 아니다. 모두 `/review`에서 병렬로 처리한다.

## 왜 Verify가 필요한가

`/review`가 정적 분석으로 코드의 "안전한가/품질은 좋은가"를 본다면, Verify는 **실행 시점의 동작과 빌드 가능성**을 확인한다.

- **E2E**: PRD 인수 조건이 실제 브라우저에서 통과하는가
- **타입/빌드**: 배포 가능한 산출물을 만들 수 있는가

## 검증 파이프라인

```
┌──────────────────────────────────────────┐
│                                          │
│  /review 통과 코드                       │
│       │                                  │
│       ▼                                  │
│  ┌──────────────────┐                    │
│  │ 1. E2E 브라우저  │  PRD 인수 조건     │
│  │    검증          │  실제 시나리오 실행 │
│  └──────┬───────────┘                    │
│         │                                │
│         ▼                                │
│  ┌──────────────────┐                    │
│  │ 2. 타입/빌드     │  tsc --noEmit      │
│  │    검사           │  npm run build     │
│  └──────┬───────────┘                    │
│         │                                │
│         ▼                                │
│  ┌──────────────────┐                    │
│  │ 3. 검증 결과      │  통합 리포트       │
│  │    통합 보고      │  PASS/FAIL 판정    │
│  └──────────────────┘                    │
│                                          │
└──────────────────────────────────────────┘
```

## 실행 절차

### Step 0: 검증 항목 선택

```
[Verify] 검증 항목을 선택해주세요.

1. 전체 검증 — E2E + 타입/빌드 (기본)
2. E2E만 — PRD 인수 조건 기반 브라우저 검증
3. 타입/빌드만 — tsc --noEmit, npm run build
4. 건너뛰기
```

### Step 1: E2E 브라우저 검증

`./commands/verifier.md`를 읽고 그 프로세스를 따른다. 내부적으로 `./skills/e2e-verifier/SKILL.md`를 로드한다.

- 도구 선택: Chrome MCP / Playwright MCP / Agent-Browser
- 입력: PRD 인수 조건 + 변경 파일 목록(`git diff --name-only`)
- 산출물: 시나리오별 PASS/FAIL, 실패 항목 상세(스크린샷, 사유), 자동 검증 불가 항목, 성공률

**완료 시 사용자에게 결과를 보여주고 다음 단계 진행 여부를 확인한다.**

### Step 2: 타입/빌드 검사

Bash 도구로 직접 수행한다.

**Step 2-1. 프로젝트 설정 감지**

`package.json`과 `tsconfig.json`을 확인하여 실행 가능한 검사를 결정한다.

- `tsconfig.json` 존재 → TypeScript 타입 검사 가능
- `package.json`의 `scripts.typecheck` 존재 → `npm run typecheck` 우선 사용
- `package.json`의 `scripts.build` 존재 → 빌드 검사 가능

**Step 2-2. TypeScript 타입 검사**

```bash
# typecheck 스크립트가 있으면 우선
npm run typecheck

# 없으면 직접 실행
npx tsc --noEmit
```

`tsconfig.json`이 없으면 건너뛴다.

**Step 2-3. 빌드 검사**

```bash
npm run build
```

`build` 스크립트가 없으면 건너뛴다.

### Step 3: 검증 결과 통합

```markdown
# Verify 검증 결과

## 1. E2E 브라우저 검증

### 시나리오별 결과

| #   | 시나리오 | 결과   | 비고 |
| --- | -------- | ------ | ---- |
| 1   | ...      | ✅/❌ | ...  |

- 총 시나리오: N개 / 성공: N개 (N%) / 실패: N개
- 자동 검증 불가 항목: N개 (수동 확인 필요)

## 2. 타입/빌드 검사

| 검사 항목  | 상태     | 세부     |
| ---------- | -------- | -------- |
| TypeScript | ✅/❌/⏭️ | N errors |
| Build      | ✅/❌/⏭️ | ...      |

## 3. 종합 판정

| 검증     | 결과                          |
| -------- | ----------------------------- |
| E2E      | ✅ PASS / ⚠️ WARNING / ❌ FAIL |
| 타입/빌드 | ✅ PASS / ⚠️ WARNING / ❌ FAIL |
| **종합**  | **PASS / WARNING / FAIL**    |

## 4. 미해결 사항
- ...
```

**판정 기준:**

- **PASS**: E2E 100% + 타입/빌드 PASS
- **WARNING**: E2E 일부 실패(자동 검증 불가 항목 존재) 또는 타입 경고
- **FAIL**: E2E 다수 실패 또는 타입/빌드 FAIL

## 실패 처리

검증에서 FAIL 항목이 있으면 사용자에게 선택지를 제시한다.

```
Verify 검증에서 N건의 FAIL이 발생했습니다.

1. 수정 후 재검증 — 이슈를 수정하고 검증을 다시 실행
2. /review로 돌아가기 — 코드 품질·정합성 이슈일 가능성, /review를 다시 실행
3. 무시하고 계속 — WARNING으로 기록하고 다음 단계로 진행
4. 중단 — 현재까지 결과로 리포트 생성
```

## 사용자 소통

- Step 0에서 검증 항목을 반드시 확인한다
- E2E 실패 시 스크린샷과 실패 사유를 함께 보여준다
- 타입/빌드 에러는 첫 N건만 보여주고 전체 출력은 요청 시 제공한다
- 자동 검증 불가 항목은 수동 확인 가이드를 함께 제시한다
