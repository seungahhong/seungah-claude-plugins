# PRD — 개발 요구사항 정의서 작성 커맨드

planner, architecture, critic 세 스킬을 각각 **서브에이전트**로 spawn하여 **계획 → 설계 → 비평** 루프로 순환하며, 실행 가능한 수준의 개발 요구사항 정의서를 작성한다.

## 왜 루프가 필요한가

한 번에 완벽한 PRD는 나오지 않는다. 계획을 세우면 설계 과정에서 누락이 드러나고, 설계를 비평하면 계획의 전제가 흔들리기도 한다. 이 세 관점을 반복적으로 순환하면서 빈틈을 메워야 실행 가능한 정의서가 된다.

## 루프 구조

```
┌─────────────────────────────────────────────┐
│                                             │
│   [1. 계획] ──→ [2. 설계] ──→ [3. 비평]     │
│       ▲                          │          │
│       │                          │          │
│       └──── 피드백 반영 ◀────────┘          │
│                                             │
│   비평 통과 시 → [4. PRD 확정]              │
│                                             │
└─────────────────────────────────────────────┘
```

## 사용자 스토리·인수조건 — 개발·검증의 공통 계약 (내재화)

이 커맨드가 만드는 PRD는 **각 기능 요구사항을 사용자 스토리와 관찰형 인수조건(Given/When/Then)으로 전개**한다. 이 인수조건이 이후 모든 단계의 **공통 계약**이 되며, 그래서 PRD는 "읽고 끝나는 문서"가 아니라 개발·검증이 추측 없이 착수할 수 있는 실행 가능한 명세가 된다.

각 AC는 **검증 레인**으로 3분류된다(§2.4 매트릭스에 표기하고, 이 분류가 모든 충족률·게이트의 기준이다):

| 레인 | 정의 | 자동 판정 주체 | 충족률 |
|------|------|----------------|--------|
| **E2E** | 브라우저로 관찰 가능한 AC | Verify의 E2E | **E2E 자동 충족률** |
| **TDD** | 브라우저로는 못 봐도 단위 테스트로 검증 가능(서버 로직·순수 함수 등) | Develop의 TDD | **TDD 자동 충족률** |
| **수동(`[수동]`)** | E2E·TDD 어느 쪽으로도 자동 검증 불가(시각 뉘앙스·외부 시스템 상태·주관 판단) | 사람 | (없음 — 수동 확인) |

- **개발(frontend-guidelines)** — TDD가 E2E·TDD 레인 AC를 RED 테스트 목록의 근거로 삼고(AC 1건 → 테스트 ≥1개), 최종 점검을 **기능(FR/US) 단위**로 수행하며 **TDD 자동 충족률**(단위 테스트 통과율)을 산출한다.
- **검증(verify)** — E2E가 **E2E 레인 AC별로** 시나리오를 만들어 PASS/FAIL을 판정하고 **E2E 자동 충족률(통과 E2E AC / 전체 E2E 레인 AC)** 을 산출한다. TDD 레인·`[수동]`은 이 분수 밖이다.
- **수렴 루프** — E2E 자동 충족률이 100%가 아니면 미충족 E2E AC만을 대상으로 개발→리뷰→검증을 반복한다(오케스트레이터 Phase 5). TDD 레인 실패는 Develop(RED→GREEN)에서, `[수동]`은 사용자 확인으로 해소한다(둘 다 루프 밖).
- **배포 게이트** — **E2E 자동 충족률 100% AND TDD 자동 충족률 100% AND `[수동]` 전부 확인 완료.** 어떤 레인의 AC가 0개면 그 레인은 해당 없음(N/A=vacuously 통과)으로 처리한다(예: 전부 `[수동]`이면 E2E·TDD 레인은 N/A, `[수동]` 확인만으로 판정).

작성 규약(다른 플러그인에 의존하지 않고 이 커맨드에 **내재화** — 방법론은 표준 As-a / Given-When-Then / INVEST와 동일하되 프론트엔드 개발·검증용으로 흡수):

- **사용자 스토리** — `[역할]로서 [행동]하고 싶다, 왜냐하면 [가치]이기 때문이다.` 해결책(UI·구현)이 아니라 의도를 적는다.
- **인수조건** — `Given [전제] / When [행동] / Then [관찰 가능한 결과]`. "잘 된다 / 적절히 처리한다 / 직관적이다" 같은 판정불가 문장 금지 — 제3자가 화면·로그 관찰만으로 통과/실패에 동일하게 합의할 수 있어야 한다(브라우저로 관찰 가능한 결과를 우선한다).
- **식별자·추적** — 기능 요구사항 `FR-n`, 사용자 스토리 `US-n`(출처 FR 명시), 인수조건 `AC-n.m`(해당 US에 소속). 모든 FR이 ≥1개 US로, 모든 US가 ≥1개 AC로 커버되어야 한다(빠짐·고아 점검). 이 ID가 TDD·기능단위 점검·E2E·수렴 루프가 서로를 가리키는 **주소**다.

## 실행 절차

각 Phase는 Agent 도구를 사용하여 **독립적인 서브에이전트**를 spawn하여 실행한다(모든 Agent 호출에 `model: "opus"` 명시). 이 커맨드는 루프 흐름을 제어하고, 각 서브에이전트의 산출물을 수집·전달·통합하는 역할을 맡는다.

### Phase 1: 계획 서브에이전트 (Planner)

Agent 도구를 사용하여 **계획 서브에이전트**를 spawn한다.

**서브에이전트에게 전달할 프롬프트 구성:**
```
1. 사용자의 요구사항 원문
2. 스킬 경로: ./skills/planner/SKILL.md — 이 스킬의 프로세스를 따를 것
3. 코드베이스 컨텍스트 (현재 프로젝트 경로, 관련 파일 정보)
4. 산출물 요청: 초안 PRD (문제 정의, 범위, 제약 조건, 성공 기준, 구현 단계, 의존성 그래프,
   + 기능 요구사항(FR-n) → 사용자 스토리(US-n) → 관찰형 인수조건(AC-n.m, Given/When/Then))
5. 규약: 위 "사용자 스토리·인수조건 공통 계약"을 따른다. 각 FR을 1개 이상 US로,
   각 US를 1개 이상 AC로 전개하고 FR/US/AC에 ID를 부여한다. AC는 제3자가 관찰로 판정 가능해야 하며
   판정불가 문장("잘 된다/적절히")은 금지한다.
```

**이 서브에이전트의 목표:**
- 사용자 인터뷰를 통해 문제 정의, 범위, 제약 조건, 성공 기준을 확정한다
- 코드베이스를 탐색하여 현재 상태를 파악한다
- 구현 단계를 분해하고 의존성 그래프를 작성한다
- **각 기능 요구사항을 사용자 스토리와 관찰형 인수조건(Given/When/Then)으로 전개**하고 ID로 추적 가능하게 만든다

**산출물:** 초안 PRD (구현 계획 + 사용자 스토리 US-n + 인수조건 AC-n.m + FR↔US↔AC 추적)

서브에이전트 완료 후, 산출물을 수집하여 다음 Phase에 전달한다.

### Phase 2: 설계 서브에이전트 (Architecture)

Agent 도구를 사용하여 **설계 서브에이전트**를 spawn한다.

**서브에이전트에게 전달할 프롬프트 구성:**
```
1. Phase 1에서 수집한 초안 PRD 전문
2. 스킬 경로: ./skills/architecture/SKILL.md — 이 스킬의 프로세스를 따를 것
3. 코드베이스 컨텍스트
4. 산출물 요청: 기술 아키텍처 설계서 + 초안 PRD 갱신 사항
```

**이 서브에이전트의 목표:**
- 시스템 구조, API, 데이터 모델, 컴포넌트 구성을 구체화한다
- 기술 선택에 대한 비교 분석과 선택 근거를 문서화한다
- 동시성/안전성 전략을 정의한다

**산출물:** 기술 아키텍처 설계서

서브에이전트 완료 후, 설계 결과를 초안 PRD에 반영하여 갱신한다.

### Phase 3: 비평 서브에이전트 (Critic)

Agent 도구를 사용하여 **비평 서브에이전트**를 spawn한다.

**서브에이전트에게 전달할 프롬프트 구성:**
```
1. 갱신된 초안 PRD 전문
2. Phase 2의 기술 아키텍처 설계서 전문
3. 스킬 경로: ./skills/critic/SKILL.md — 이 스킬의 프로세스를 따를 것
4. 산출물 요청: 비평 보고서 (리스크 매트릭스 + 핵심 권고사항 3가지)
5. (2회차 이상) 이전 비평에서 지적된 사항과 반영 내용
```

**이 서브에이전트의 목표:**
- 7가지 관점(확장성, 안정성, 보안, 성능, 호환성, 운영, 누락 기능)에서 약점을 식별한다
- 각 리스크에 심각도와 발생 가능성을 부여한다
- 구체적인 해결 방안을 제시한다
- **인수조건 계약 검수** — (1) 모든 FR이 ≥1개 US로, 모든 US가 ≥1개 AC로 커버되는지(빠짐·고아 색출) (2) 각 AC가 Given/When/Then 관찰형이고 제3자 판정 가능한지(판정불가 문장 색출) (3) 인수조건이 TDD 테스트·E2E 시나리오로 곧장 전개될 만큼 구체적인지 점검한다. 이 셋 중 하나라도 실패하면 심각도 **상**으로 분류한다(개발·검증이 착수 불가).

**산출물:** 비평 보고서 (리스크 매트릭스 + 핵심 권고사항 3가지 + 인수조건 추적 매트릭스 미커버/판정불가 항목)

### 루프 판정: 계속 vs 확정

비평 서브에이전트의 보고서를 수집한 뒤, 다음 행동을 결정한다:

**루프 계속 조건** (하나라도 해당하면 Phase 1로 복귀):
- 심각도 **상** 리스크가 1건 이상 존재
- 계획의 전제나 범위 자체를 수정해야 하는 피드백이 있음
- 아키텍처의 핵심 구조 변경이 필요함

**루프 확정 조건** (모두 충족해야 함):
- 심각도 **상** 리스크가 0건
- 남은 리스크가 모두 설계 내에서 해결 방안이 반영되어 있음
- 사용자가 비평 결과에 동의

루프를 반복할 때는 비평에서 지적된 사항을 각 서브에이전트 프롬프트에 포함하여 변경된 부분만 재작업하도록 한다. 전체를 처음부터 다시 하지 않는다.

**최대 3회 반복.** 3회 루프 후에도 심각도 상 리스크가 남아 있으면, 해당 리스크를 "알려진 제한사항"으로 명시하고 사용자에게 판단을 맡긴다.

### Phase 4: PRD 확정

루프를 통과하면 최종 PRD를 작성한다.

**최종 PRD 구조:**

```markdown
# [프로젝트/기능명] 개발 요구사항 정의서

## 1. 개요
- 목적
- 범위 (포함/제외)
- 성공 기준

## 2. 요구사항
### 2.1 기능 요구사항
| ID | 요구사항 | 우선순위 | 대표 인수조건 ID |
|----|----------|----------|------------------|
| FR-1 | ... | 상/중/하 | AC-1.1, AC-1.2 |

### 2.2 비기능 요구사항
| ID | 요구사항 | 기준 |
|----|----------|------|

### 2.3 사용자 스토리 + 인수조건 (개발·검증 공통 계약)

각 기능 요구사항(FR)을 사용자 스토리(US)로 전개하고, 각 스토리에 관찰형 인수조건(AC)을 Given/When/Then으로 부여한다.

```
US-1 (출처: FR-1) — [역할]로서 [행동]하고 싶다, 왜냐하면 [가치]이기 때문이다.
  AC-1.1  Given [전제]  When [행동]  Then [관찰 가능한 결과]
  AC-1.2  Given [전제]  When [행동(예외/경계)]  Then [관찰 가능한 결과]
US-2 (출처: FR-2) — ...
  AC-2.1  Given ...  When ...  Then ...
```

- 인수조건은 판정불가 문장("잘 된다/적절히/직관적")을 금지하고, 브라우저·로그로 관찰 가능한 결과로 적는다.
- **자동 검증(E2E·TDD 모두)으로 판정할 수 없어 사람이 직접 확인해야 하는 항목**(예: 시각적 디자인 뉘앙스, 외부 시스템의 실제 상태, 주관적 판단)만 AC 옆에 `[수동]`으로 표시한다 — E2E·TDD 자동 충족률 모두에서 제외되고 수동 확인 체크리스트로 관리된다(수렴 루프 대상이 아니라 사용자 수동 확인 대상).
- **반례 주의**: 브라우저로는 관찰 못 해도 **TDD 단위 테스트로 검증 가능한 항목(서버 로직·순수 함수·DB 계층 등)은 `[수동]`이 아니다** — E2E 시나리오에서만 제외되고 **TDD 자동 충족률에는 산입**되는 일반 TDD 대상이다("브라우저 불가"를 "자동화 불가"로 오판해 수동으로 넘기지 않는다).

### 2.4 요구↔스토리↔인수조건 추적 매트릭스

각 AC를 한 행으로 두고 **검증 레인**(E2E / TDD / 수동 중 하나)을 명시한다.

| FR | US | AC | 검증 레인 |
|----|----|----|-----------|
| FR-1 | US-1 | AC-1.1 | E2E |
| FR-1 | US-1 | AC-1.2 | 수동(`[수동]`) |
| FR-2 | US-2 | AC-2.1 | E2E |
| FR-2 | US-2 | AC-2.2 | TDD |

- **빠짐 점검**: 모든 FR이 이 표의 US 열에, 모든 US의 AC가 AC 열에 등장하는지 확인한다.
- **고아 점검**: 어떤 FR에도 연결되지 않은 US, 어떤 US에도 속하지 않은 AC는 제거하거나 출처를 보강한다.
- **레인 점검**: 각 AC의 검증 레인이 정확한지 확인한다 — 브라우저 관찰형=E2E, 단위 테스트 가능=TDD, 둘 다 불가=수동. 레인이 충족률 분모와 배포 게이트를 가른다.

- 의존성

| 파일 | 변경 유형 | 설명 |
|------|----------|------|

## 6. 리스크 및 대응
```

## 다음 단계로의 계약 전달 (Develop·Verify가 소비)

이 PRD의 **§2.3 인수조건(AC)과 §7 체크리스트**는 이후 단계가 그대로 소비하는 계약이다.

- **Develop(frontend-guidelines)** — TDD 서브에이전트에게 AC 목록을 RED 테스트 목록의 근거로 전달한다(AC 1건 → 테스트 ≥1개). 최종 점검은 FR/US 단위로 AC 충족 여부를 표기한다.
- **Verify** — E2E가 E2E 레인 AC별로 시나리오를 만들어 PASS/FAIL과 **E2E 자동 충족률**(통과 E2E AC / 전체 E2E 레인 AC)을 산출한다. TDD 레인·`[수동]`은 분수 밖.
- **수렴 루프** — E2E 자동 충족률 < 100%면 미충족 E2E AC만 대상으로 Develop→Review→Verify를 반복한다(오케스트레이터 Phase 5). TDD 레인 실패는 Develop에서, `[수동]`은 사용자 확인으로 해소(둘 다 루프 밖).

## 루프 중 사용자 소통

- 각 Phase가 끝날 때마다 핵심 산출물을 사용자에게 요약하여 보여준다
- 비평 결과는 리스크 매트릭스와 핵심 권고사항 3가지를 중심으로 간결하게 전달한다
- 루프를 반복할 때는 "이전 비평에서 N건의 주요 리스크가 발견되어, 계획/설계를 수정 후 재검증합니다"라고 안내한다
- 사용자가 "이 정도면 충분하다"고 판단하면 루프를 조기 종료할 수 있다
