# Figma 시각 정합 검증 — 렌더 vs Figma 스냅샷 대조

E2E로 캡처한 실제 렌더 화면과 Figma에서 추출한 스냅샷(`.claude/design/{화면명}-figma.png`)을 대조하여 시각 정합성을 검증한다. verify 커맨드 **Step 1.5**에서 이 패턴을 따른다.

> 본 검증은 픽셀 완전일치가 아니라 **레이아웃 / 토큰 / 인텐트 3축 일치**를 100% 기준으로 본다. (픽셀 비요구)

## 전제: 동일 뷰포트 폭

3축 대조는 **렌더 캡처와 Figma 스냅샷의 뷰포트 폭이 같을 때만** 유효하다.

- Figma 스냅샷의 캡처 폭은 `.claude/design/{화면명}.json`의 `capturedViewportWidth` 값이다.
- E2E 렌더 캡처 전, 브라우저 뷰포트 폭을 `capturedViewportWidth`와 일치시킨다.
  - Chrome MCP: `emulate`의 `viewport` 폭을 `capturedViewportWidth`로 설정
  - Playwright MCP: `browser_resize`의 `width`를 `capturedViewportWidth`로 설정
  - Agent-Browser: 폭 조절 도구가 없으면 가장 근접한 표준 폭으로 캡처하고, 폭 불일치를 비고에 기록
- 폭을 맞출 수 없으면 3축 판정에 "뷰포트 폭 불일치(렌더 {W}px vs Figma {capturedViewportWidth}px)"를 명시하고, 반응형 차이로 인한 불일치는 FAIL로 간주하지 않는다.

## '100% 일치'의 현실 정의

| 축 | 100% 일치의 의미 | 일치로 보지 않는 예 |
|----|-----------------|--------------------|
| 레이아웃 | 정렬·순서·구조가 동일 (요소 배치 의도가 같음) | 안티앨리어싱·서브픽셀 렌더 차이, 폰트 힌팅 차이 |
| 디자인 토큰 | 색·간격·타이포 **값**이 토큰 범위 내 일치 | 동일 토큰의 1px 미만 렌더 반올림 |
| 인텐트 | 요소 누락 없음·텍스트 동일·상태(기본/호버/에러) 반영 | 더미 데이터 텍스트(실데이터로 대체된 경우) |

> 즉 '100%'는 픽셀 동일이 아니라 **3축 기준 항목이 모두 PASS**임을 뜻한다.

## 3축 체크리스트

대조 대상 화면마다 아래 3축을 점검하고, 각 항목을 ✅/❌로 판정한다.

### 축 1 — 레이아웃 (정렬·순서·구조)

- [ ] 주요 영역(헤더/콘텐츠/푸터 등)의 배치 순서가 Figma와 동일한가
- [ ] 요소 정렬(좌/우/중앙, 상/하)이 Figma 의도와 일치하는가
- [ ] 그리드·컬럼 구조, 요소 간 그룹핑이 동일한가
- [ ] 반응형 폭에서 의도된 레이아웃(랩/스택 전환)이 맞는가

### 축 2 — 디자인 토큰 (색·간격·타이포 값)

- [ ] 배경·텍스트·보더 **색상**이 `.claude/design/{화면명}.json`의 `tokens`와 일치하는가
- [ ] 패딩·마진·갭 등 **간격** 값이 토큰 범위 내인가
- [ ] 폰트 패밀리·크기·두께·행간 등 **타이포** 값이 일치하는가
- [ ] (KRW/KST/한글 컨텍스트) 통화·날짜·한글 폰트 렌더가 토큰 의도대로인가

### 축 3 — 인텐트 (요소 누락·텍스트·상태)

- [ ] Figma에 있는 요소가 렌더에서 **누락**되지 않았는가
- [ ] 렌더에 Figma에 없는 요소가 임의로 추가되지 않았는가
- [ ] 텍스트·라벨·아이콘 내용이 Figma 인텐트와 동일한가
- [ ] 상태(기본/호버/포커스/에러/빈 상태)가 Figma 명세대로 반영됐는가

## 대조 절차 (take_screenshot 재사용)

e2e-verifier의 캡처 도구 패턴을 그대로 재사용한다. 도구별 스크린샷 호출은 선택된 도구의 reference를 따른다.

| 선택 도구 | 캡처 호출 | 뷰포트 폭 맞춤 |
|----------|----------|---------------|
| Chrome MCP | `take_screenshot` | `emulate` viewport width |
| Playwright MCP | `browser_take_screenshot` | `browser_resize` width |
| Agent-Browser | `browser_screenshot` | (도구 없음 — 폭 불일치 기록) |

```
대조 대상 화면마다:
  1. 뷰포트 폭을 capturedViewportWidth로 맞춘다
  2. 렌더 화면을 캡처한다 (e2e-verifier의 take_screenshot 패턴 재사용)
  3. .claude/design/{화면명}-figma.png 를 나란히 놓고 3축 체크리스트로 대조
  4. 축별 ✅/❌ 와 불일치 사유를 기록
  5. 결과를 사용자에게 즉시 보여준다
```

## 비교 → 수정 → 재캡처 루프 (최대 3회 + 게이트)

3축 중 ❌ 항목이 있으면 수정 후 재캡처한다. **종료 조건:**

```
라운드 R (R = 1, 2, 3):
  - 3축 대조 → ❌ 항목 식별
  - ❌ 항목별 수정안을 사용자에게 보여주고 승인 게이트
    (승인 전 코드를 임의 수정하지 않는다)
  - 승인 시 수정 → 뷰포트 폭 맞춰 재캡처 → 재대조
  - 3축 전부 ✅ 이면 PASS로 종료
  - R == 3 이고도 ❌ 잔존 시: 잔여 불일치를 WARNING으로 기록하고 종료
    (무한 루프 금지. 잔여 항목은 사용자 판단에 위임)
```

- 각 라운드마다 사용자 승인 없이 다음 라운드(수정·재캡처)로 진행하지 않는다.
- 최대 3회까지만 자동 루프하며, 초과 시 사용자에게 판단을 위임한다.

## 판정 출력

```markdown
### Figma 시각 정합 — {화면명}

| 축 | 판정 | 비고 |
|----|------|------|
| 레이아웃 | ✅/❌ | ... |
| 디자인 토큰 | ✅/❌ | ... |
| 인텐트 | ✅/❌ | ... |

- 뷰포트 폭: 렌더 {W}px / Figma {capturedViewportWidth}px
- 라운드 수: N회
- 판정: ✅ 일치 / ⚠️ 부분 일치(잔여 N건) / ❌ 불일치
```

> Step 1.5 결과는 verify Step 3 통합 리포트의 "Figma 3축 판정·라운드 수"로 합산된다.
