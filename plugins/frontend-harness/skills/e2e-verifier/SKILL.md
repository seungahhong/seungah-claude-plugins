# E2E 브라우저 검증

개발이 완료된 기능이 실제 브라우저에서 정상 동작하는지 검증한다.
PRD의 인수 조건(acceptance criteria)을 기준으로 검증 시나리오를 구성하고, 사용자가 선택한 브라우저 자동화 도구로 시나리오를 실행한다.

## 입력

- **PRD 인수 조건** — 검증 시나리오의 기준이 되는 acceptance criteria
- **변경 파일 목록** — `git diff`로 확인한 수정된 파일 목록 (검증 범위 파악용)
- **개발 서버 URL** — 사용자에게 확인 (예: `https://www.example.kr`)

## 실행 절차

### Phase 0: 브라우저 자동화 도구 선택

사용자에게 사용할 브라우저 자동화 도구를 선택하도록 안내한다:

```
E2E 브라우저 검증에 사용할 도구를 선택해주세요:

1. Chrome MCP — Chrome DevTools Protocol 기반. 이미 열려 있는 Chrome 브라우저를 제어합니다.
2. Playwright MCP — Playwright 기반. 헤드리스/헤드풀 브라우저를 자동으로 실행합니다.
3. Agent-Browser — AI 에이전트 특화 브라우저. 자연어 기반 브라우저 제어를 지원합니다.

번호로 선택해주세요.
```

선택된 도구에 따라 해당 reference 파일의 패턴을 따른다:

| 선택 | 참조 파일 | 특징 |
|------|----------|------|
| Chrome MCP | `references/chrome-mcp.md` | Chrome 디버깅 모드 필요, 가장 낮은 수준의 제어 가능 |
| Playwright MCP | `references/playwright-mcp.md` | 자체 브라우저 실행, 스냅샷 기반 접근성 트리 활용 |
| Agent-Browser | `references/agent-browser.md` | AI 에이전트 친화적, 자연어 기반 인터랙션 |

### Phase 0-1: 사전 조건 확인

선택된 도구의 reference 파일에 정의된 **사전 조건 확인 절차**를 따른다. 각 도구마다 필요한 MCP 설정, 브라우저 실행 방식, 연동 확인 방법이 다르므로 반드시 해당 reference를 읽고 따른다.

**공통 확인 사항:**
- `.mcp.json`에 선택된 도구의 MCP 서버 설정이 있는지 확인
- 설정이 없으면 reference 파일에 정의된 설정 안내를 사용자에게 전달
- 개발 서버 URL 확인:

```
개발 서버가 실행 중인 URL을 알려주세요. (예: https://www.example.kr)
E2E 검증을 건너뛰려면 "건너뛰기"라고 입력해주세요.
```

사용자가 "건너뛰기"를 선택하면 검증을 건너뛰고 결과에 SKIP으로 기록한다.

### Phase 1: 검증 시나리오 생성

PRD 인수 조건에서 브라우저로 검증 가능한 항목을 추출하여 시나리오로 변환한다.

**시나리오 변환 규칙:**

| 인수 조건 유형 | 시나리오 동작 | 검증 방법 |
|---------------|-------------|----------|
| "~가 표시된다" | 페이지 이동 → 해당 요소 확인 | 스크린샷 + DOM 요소 존재 확인 |
| "~를 클릭하면 ~한다" | 요소 클릭 → 결과 확인 | 클릭 → 대기 → 스크린샷 |
| "~를 입력하면 ~한다" | 입력 필드에 값 입력 → 결과 확인 | 입력 → 대기 → 스크린샷 |
| "~목록이 N개 표시된다" | 페이지 이동 → 목록 카운트 | DOM 요소 카운트 |
| "~로 이동한다" | 동작 수행 → URL 또는 화면 변화 확인 | 스크린샷으로 이동 결과 확인 |
| "에러 메시지가 표시된다" | 잘못된 입력 → 에러 메시지 확인 | 입력 → 스크린샷으로 에러 메시지 확인 |

**브라우저로 검증 불가능한 항목** (API 응답, DB 상태, 서버 로직 등)은 E2E 시나리오에서 제외한다. 그중 **PRD가 `[수동]`으로 표시한 AC만** "수동 확인 필요"로 분류하고, TDD 단위 테스트로 검증되는 항목(서버 로직 등)은 개발 단계에서 이미 검증되므로 E2E 대상만 아닐 뿐 수동 확인 대상이 아니다.

변환된 시나리오를 사용자에게 보여주고 확인을 요청한다:

```markdown
## E2E 검증 시나리오

PRD 인수조건(AC)에서 N개의 브라우저 검증 시나리오를 생성했습니다.
사용 도구: {선택된 도구명}

| AC | 인수조건(Given/When/Then) | 시나리오 동작 | 검증 방법 |
|----|--------------------------|-------------|----------|
| AC-1.1 | ... | ... | ... |

- **인수조건 하나(AC)당 시나리오 하나 이상**을 만들어, 최종적으로 AC별 PASS/FAIL을 판정할 수 있게 한다.

### 자동 검증 불가 항목 (수동 확인 필요 · E2E 충족률 분모 밖 · 수동 확인 체크리스트)
- [ ] AC-... `[수동]` ...

이 시나리오로 진행할까요? 수정이 필요하면 알려주세요.
```

### Phase 2: 브라우저 검증 실행

선택된 도구의 reference 파일에 정의된 **실행 패턴**을 따라 각 시나리오를 실행한다.

**도구 공통 핵심 원칙:**
- **URL 직접 이동은 최초 1회만** — 이후 모든 페이지 이동은 화면 UI를 통해 수행한다
- **URL을 추측하거나 직접 구성하지 않는다** — 실제 사용자의 브라우저 사용 방식을 따른다
- **각 시나리오 전후 스크린샷을 캡처한다** — before/after 증거를 남긴다

**각 시나리오 실행 흐름** (도구별 구체적 도구 호출은 reference 참조):

```
각 시나리오에 대해:
  1. 현재 상태 캡처 (before)
  2. 시나리오 동작 수행 (이동 / 클릭 / 입력 / 스크롤)
  3. 결과 대기
  4. 결과 상태 캡처 (after)
  5. 검증 수행 (요소 존재 / 텍스트 내용 / 요소 카운트 / 화면 상태)
  6. 결과 즉시 출력
```

**각 시나리오 완료 직후 결과를 즉시 출력한다:**

```
[시나리오 #1] 광고 성과 위젯이 대시보드에 표시된다
도구: {선택된 도구명}
결과: ✅ PASS
검증: 대시보드 페이지에서 광고 성과 위젯 요소가 존재함을 확인
```

```
[시나리오 #2] 기간 선택 드롭다운 클릭 시 옵션 목록이 표시된다
도구: {선택된 도구명}
결과: ❌ FAIL
검증: 드롭다운 클릭 후 옵션 목록 요소가 발견되지 않음
사유: 드롭다운 메뉴가 열리지 않음 — 스크린샷 참조
```

### Phase 2-1: Figma 시각 정합 대조 (선택 — Figma 스냅샷이 있을 때)

`figma-extract`가 내려둔 Figma 스냅샷(`.claude/design/{화면명}-figma.png`)이 있으면, **E2E로 캡처한 실제 렌더 화면과 Figma에서 추출한 스냅샷을 대조**하여 시각 정합성을 검증한다. Figma 산출물이 없으면 이 단계를 건너뛴다.

- 대조는 픽셀 완전일치가 아니라 **레이아웃 / 디자인 토큰 / 인텐트 3축 일치**로 판정하며, 렌더 캡처와 Figma 스냅샷의 **뷰포트 폭(`capturedViewportWidth`)을 맞춘 뒤** 대조한다.
- 캡처는 Phase 2에서 사용한 도구의 스크린샷 패턴(`take_screenshot` 등)을 그대로 재사용한다.
- 상세 절차 — 3축 체크리스트, 동일 뷰포트 폭 맞춤, ❌ 항목 수정·재캡처 루프(최대 3회), 판정 출력 — 은 반드시 [references/figma-visual-diff.md](references/figma-visual-diff.md)를 읽고 따른다.
- 결과(축별 ✅/❌·라운드 수)는 Phase 3 검증 결과에 "Figma 시각 정합" 항목으로 합산한다.

### Phase 3: 검증 결과 정리

모든 시나리오 실행 후 결과를 종합한다.

```markdown
## E2E 검증 결과

### 전체 요약
- 사용 도구: {선택된 도구명}
- 자동검증 AC: M개 / `[수동]` AC: L개 / 시나리오: N개
- **E2E 자동 충족률: 통과 E2E AC K / 전체 E2E 레인 AC M = NN%** (이 스킬은 브라우저 전용이라 TDD 레인·`[수동]` AC는 이 분수 밖이다. TDD 레인은 Develop에서, `[수동]` 수동 확인 반영은 호출측(verify/orchestrator) 책임. E2E 레인 AC가 0개면 N/A)
- 미충족 E2E 인수조건: AC-... (호출측 수렴 루프 대상)
- 수동 확인 체크리스트(`[수동]`): AC-... (상위에서 사용자 수동 확인)

### 인수조건별 결과
| AC | 시나리오 | 결과 | 비고 |
|----|---------|------|------|
| AC-1.1 | ... | ✅ PASS | ... |
| AC-2.1 | ... | ❌ FAIL | ... |

### 실패 항목 상세
(실패한 시나리오별 스크린샷과 실패 사유)

### Figma 시각 정합 (Phase 2-1을 수행한 경우)
| 화면 | 레이아웃 | 디자인 토큰 | 인텐트 | 라운드 | 판정 |
|------|---------|------------|--------|--------|------|
| ... | ✅/❌ | ✅/❌ | ✅/❌ | N회 | ✅ 일치 / ⚠️ 부분 / ❌ 불일치 |

### 자동 검증 불가 항목 (수동 확인 필요)
- [ ] ...
```

> **호출 맥락에 따라 아래 처리가 갈린다 — 먼저 판별하라.** 판별 기준: 상위 커맨드(`verifier.md`·`verify.md`·`orchestrator`)가 이 스킬을 로드하며 `호출맥락=orchestrated`를 넘기면 오케스트레이션, 사용자가 이 스킬만 직접 지목했으면 단독이다. **플래그가 없으면 기본값은 orchestrated로 간주**(이 스킬은 `disable-model-invocation: true`라 통상 상위 커맨드를 통해서만 로드된다)하고 메뉴를 억제한다.
> - **오케스트레이션(orchestrated)으로 호출된 경우** — **아래 결정 메뉴를 출력하지 않는다.** E2E 자동 충족률·미충족 E2E AC·수동 확인 체크리스트만 보고하고, 다음 행동 결정은 전적으로 상위의 **인수조건 수렴 루프**(Develop→Review→Verify를 E2E 자동 충족률 100%까지 반복)에 위임한다(리프 스킬이 상위 루프와 중복·상충하는 메뉴를 띄우지 않는다).
> - **사용자가 이 스킬을 명시적으로 단독 실행한 경우에만** 아래 메시지를 제시한다.

**E2E 자동 충족률 100%인 경우:**

```
E2E 검증을 통과했습니다. E2E 레인 인수조건(AC)이 모두 충족되었습니다. (E2E 자동 충족률 100%)
남은 `[수동]` AC N건은 사용자 수동 확인이 필요합니다.
```

**미충족 E2E 인수조건이 있는 경우(E2E 자동 충족률 < 100%) · 단독 실행 시에만:**

```
E2E 검증: E2E 자동 충족률 NN% (미충족 E2E AC N건 · [수동] 미확인 M건)

미충족 E2E 인수조건: AC-...

어떻게 진행할까요? (단독 실행 — 사용자가 직접 실행할 순서 안내)
1. 개발로 회귀(권장) — 사용자가 /frontend-guidelines를 재실행해 미충족 E2E AC만 재구현 → /review → 재검증
2. 리뷰로 회귀 — 코드 품질·정합성 이슈로 보이면 /review 먼저 재실행 후 재검증
3. 알려진 미충족으로 기록하고 계속 — 미충족 E2E AC를 경고로 기록하고 다음 단계 진행
4. 중단 — 워크플로우를 중단하고 현재까지 결과로 리포트 생성
```

> 재검증 경로(1·2번)는 모두 **Review→Verify 순서**를 지킨다(리뷰 없이 재검증하는 경로를 두지 않는다). `[수동]` AC는 재구현 대상이 아니라 사용자 수동 확인 대상이다.

## 주의사항

- **URL 직접 이동은 최초 진입 시에만 사용한다.** 이후 모든 이동은 화면 UI를 통해 수행한다.
- **URL을 추측하거나 직접 구성하지 않는다.** 실제 사용자의 브라우저 사용 방식을 따른다.
- **PRD 인수 조건에 없는 시나리오를 임의로 추가하지 않는다.** 검증 범위는 인수 조건에 한정한다.
- **브라우저에서 검증 불가능한 항목은 명시적으로 분류한다.** "수동 확인 필요"로 표기하여 사용자가 인지하게 한다.
- **모든 메시지는 한국어 존댓말로 출력한다.**
- **각 Phase 완료 후 반드시 사용자에게 결과를 보여주고 승인/거절을 확인받은 뒤 다음 Phase로 진행한다.** 사용자 승인 없이 자동으로 다음 단계를 진행하지 않는다.
- **각 시나리오 결과는 완료 즉시 출력한다.** 전체 완료까지 기다리지 않는다.
- **도구별 상세 실행 패턴은 반드시 해당 reference 파일을 읽고 따른다.** SKILL.md의 흐름만으로 실행하지 않는다.
