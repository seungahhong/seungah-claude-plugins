# Harness Generator — 하네스 엔지니어링 자동 생성 메타 스킬

도메인 요구를 입력으로 받아 **에이전트 정의 + 스킬 + 오케스트레이터**로 구성된 하네스 한 묶음을 7단계로 설계·생성·검증한다. 특정 기술 스택(프론트엔드/백엔드/데이터)이나 특정 도구에 종속되지 않은 공통 메타 절차만 다룬다.

## 왜 하네스가 필요한가

단일 스킬이나 일회성 프롬프트로 처리하기 어려운 다단계 작업(분석 → 설계 → 실행 → 검증)이 있을 때, 산출물과 책임을 분리한 에이전트/스킬을 묶고 오케스트레이터가 흐름을 통제하면 다음 이점이 있다.

1. **재현성** — 같은 도메인 요청에 항상 동일한 절차로 응답한다.
2. **전문성 분리** — 각 에이전트가 한 가지 책임만 진다.
3. **점진적 진화** — 변경 이력이 남고 후속 실행이 안전해진다.

## 핵심 원칙

- **에이전트 팀이 기본** — 2명 이상 협업이 필요하면 `TeamCreate` 기반 팀을 우선 검토하고, 통신 오버헤드가 불필요할 때만 서브에이전트 모드를 선택한다.
- **에이전트 정의 파일 필수** — 빌트인 타입(`general-purpose`/`Explore`/`Plan`)을 쓰더라도 `agents/{name}.md`로 정의를 남긴다. Agent 호출 프롬프트에 역할을 직접 박지 않는다.
- **모든 Agent 호출은 `model: "opus"`** — 추론 품질이 하네스 품질을 좌우한다.
- **CLAUDE.md에는 포인터만** — 트리거 규칙과 변경 이력만 남기고, 에이전트/스킬 목록·디렉토리 구조는 적지 않는다.
- **스킬 본문은 < 500줄** — 초과분은 `references/`로 분리한다(Progressive Disclosure).
- **명령형/Why-First** — "~한다/~하라" 어조와 이유 설명을 함께 둔다. LLM은 이유를 알면 엣지 케이스에서 옳게 판단한다.

## 산출물 배치 규약

이 스킬은 다음 경로에 파일을 만든다(플러그인 안에서 동작 시):

```
plugins/{plugin}/
  skills/{domain}-orchestrator/SKILL.md   # 오케스트레이터 스킬
  skills/{domain}-orchestrator/references/ # 필요 시 보충
  skills/{agent-or-step}/SKILL.md         # 도메인별 단계 스킬들
  agents/{name}.md                        # (Claude Code 에이전트 정의)
  CLAUDE.md (또는 루트 CLAUDE.md)         # 하네스 포인터 + 변경 이력
```

플러그인 외부(루트 `.claude/`)에서 동작 시에는 `.claude/skills/`, `.claude/agents/`로 매핑한다. 디렉토리 형태는 환경에 맞춰 결정하되, **에이전트 정의는 별도 파일, 오케스트레이터는 단일 진입 스킬**이라는 구조는 항상 유지한다.

## 7단계 프로세스

### Phase 0: 현황 감사 (Audit)

설계 전에 이미 존재하는 하네스/에이전트/스킬과의 충돌을 차단한다.

**확인 항목:**
- `agents/`, `skills/`, `commands/` 디렉토리 존재 및 목록
- 루트 `CLAUDE.md`의 기존 하네스 포인터
- 동일 도메인 키워드를 트리거로 쓰는 기존 스킬

**분기:**
- 신규 도메인 → Phase 1로 진행
- 기존 하네스에 추가/확장 요청 → Phase 1을 축약하고 Phase 7-5(운영) 절차도 함께 적용
- 단순 수정/보완 요청 → Phase 7-5(운영)로 직접 진입

감사 결과를 사용자에게 1회 요약하고, 진행 방향에 대한 승인을 받는다.

### Phase 1: 도메인 분석 (Domain Analysis)

사용자 요청에서 다음을 추출한다. 모호하면 grill-me 식으로 **한 번에 한 질문씩** 명확화한다.

1. **도메인 범위** — 무엇을, 어디까지, 어떤 산출물 형태로
2. **작업 유형 분해** — 생성/검증/편집/분석 중 어느 조합인지
3. **기술/리소스 제약** — 사용 가능한 도구, 외부 API, 시간 제약
4. **사용자 숙련도** — 비전문 사용자라면 jargon(assertion, JSON schema 등) 사용 자제
5. **재실행 시나리오** — 이 하네스가 일회성인가, 반복 실행되는가

**산출:** 도메인 요약 카드(목적·작업 유형·제약·반복성).

### Phase 2: 아키텍처 설계 (Team Architecture)

#### 2-1 실행 모드 선택

| 모드 | 사용 시점 | 특징 |
| ---- | --------- | ---- |
| **에이전트 팀** (기본) | 2명 이상 실시간 협업 필요 | `TeamCreate` + `SendMessage` + `TaskCreate` 자체 조율 |
| **서브에이전트** | 결과 핸드오프만 충분 | `Agent` 직접 호출, `run_in_background`로 병렬화 |
| **하이브리드** | Phase별 특성 차이 큼 | Phase별 모드를 명시 |

**판단 흐름:** 팀 모드를 먼저 검토하고, 통신 비용 > 협업 이득일 때만 서브에이전트로 강등한다.

#### 2-2 아키텍처 패턴 선택

상세는 `references/agent-design.md` 참조.

| 패턴 | 구조 | 적합 |
| ---- | ---- | ---- |
| 파이프라인 | 순차 | 단계 의존이 강함 |
| 팬아웃/팬인 | 병렬 → 통합 | 다각 분석 후 결합(팀 필수) |
| 전문가 풀 | 라우팅 | 상황별 호출 |
| 생성-검증 | 쌍 반복 | 품질 게이트 |
| 감독자 | 동적 분배 | 가변 작업량 |
| 계층적 위임 | 재귀 분해 | 복잡 문제 |

#### 2-3 에이전트 분리 기준 4축

전문성 / 병렬성 / 컨텍스트 격리 / 재사용성. 한 에이전트가 두 축에서 모호하면 분리한다.

**산출:** 모드 선택 근거 + 패턴 + 에이전트 N명의 책임 카드.

### Phase 3: 에이전트 정의 작성 (Agent Definitions)

각 에이전트마다 `agents/{name}.md`를 만든다. 다음 섹션을 모두 포함한다.

```markdown
---
name: {agent-name}
description: {언제 호출되는가, pushy하게}
---

## Core Role
{한 문장 책임}

## Work Principles
- {원칙 1 — Why까지 한 줄 설명}
- ...

## Input / Output Protocol
- 입력: {형태와 출처}
- 출력: {형태와 저장 위치, 예: _workspace/{phase}_{agent}_{artifact}.md}

## Error Handling
- {예상 실패 모드와 대응}

## Collaboration
- {다른 에이전트와의 인계 방식}

## Team Communication Protocol  (팀 모드에서만)
- 송신 대상: {agent-name 목록 / 토픽}
- 수신 대상: {agent-name 목록 / 토픽}
- 작업 등록 범위: {TaskCreate에 올릴 작업의 종류}
```

**필수 규칙:**
- 빌트인 타입을 쓰는 에이전트도 정의 파일을 둔다(`subagent_type`은 Agent 호출 시 지정).
- 모든 Agent 호출에 `model: "opus"`를 명시한다.
- QA/검증 에이전트는 `general-purpose`(읽기 전용 `Explore` 금지), 모듈별로 점진 실행.

### Phase 4: 스킬 작성 (Skill Creation)

각 에이전트의 능력은 스킬로 분리한다. 한 에이전트가 여러 스킬을 가질 수 있고, 스킬은 에이전트 간 공유 가능하다.

**SKILL.md 작성 규칙은 `references/skill-guide.md`에 모았다. 핵심 요약:**

- **frontmatter 필수**: `name`, `description` (다른 키 추가 금지)
- **description은 트리거 메커니즘** — 무엇을/언제/유사 사례 구분 + 약간 pushy하게
- **본문 < 500줄** — 초과분은 `references/`로 분리
- **명령형 어조** — "~한다", "~하라"
- **Why-First** — 규칙보다 이유, 일반화된 원리로 기술
- **번들링 신호** — 같은 헬퍼/접근법이 3회 이상 재현되면 `scripts/`로 묶기
- **Progressive Disclosure** — 메타데이터(항상 로드) / 본문(트리거 시) / references(요청 시)

#### 출력 데이터 스키마 (필요 시)

평가/리포트가 필요한 도메인이라면 다음 스키마를 그대로 쓴다(필드명 변형 금지).

- `eval_metadata.json` — `eval_id`, `eval_name`, `prompt`, `assertions[]`
- `grading.json` — `expectations[]{text,passed,evidence}`, `summary{passed,failed,total,pass_rate}`
- `timing.json` — `total_tokens`, `duration_ms`, `total_duration_seconds`

서브에이전트 완료 알림에서만 `total_tokens`/`duration_ms`를 읽을 수 있으므로 즉시 저장한다.

### Phase 5: 오케스트레이션 (Orchestrator)

오케스트레이터는 모든 에이전트와 스킬을 묶는 단일 진입점 스킬이다. 모드별 생성 템플릿은 `references/orchestrator-generate.md` 참조.

**모든 오케스트레이터 공통 골격:**

```
Phase 0  — 컨텍스트 확인 (초기 / 후속 / 부분 재실행 자동 판별)
Phase 1  — 입력 분석 + _workspace/ 준비
Phase 2..N — 도메인 본 작업 (팀/서브/하이브리드 중 모드 명시)
Phase 마지막 — 통합 산출물 작성, _workspace/ 정리/보존 결정
```

**데이터 전달 전략 매트릭스:**

| 전략 | 도구 | 모드 | 적합 |
| ---- | ---- | ---- | ---- |
| 메시지 기반 | `SendMessage` | 팀 | 실시간 조정 |
| 작업 기반 | `TaskCreate`/`TaskUpdate` | 팀 | 진척/의존성 |
| 파일 기반 | Read/Write 합의된 경로 | 팀+서브 | 큰 데이터/감사 |
| 반환 기반 | `Agent` 반환 메시지 | 서브 | 단발 결과 수집 |

**파일 경로 규약:**
- 중간 산출물: `_workspace/{phase}_{agent}_{artifact}.{ext}`
- 최종 산출물: 사용자가 읽을 위치(예: 프로젝트 루트, `docs/`)
- `_workspace/`는 후속 실행을 위해 보존하고, 새 실행은 `_workspace_prev/`로 이동시킨 뒤 다시 만든다.

**에러 정책 (필수):**
- 1회 재시도, 그래도 실패하면 해당 결과만 누락 처리하고 진행
- 충돌 데이터는 삭제하지 말고 출처 태그로 보존

**팀 사이즈 가이드:**

| 작업량 | 권장 팀 사이즈 | 인당 작업 |
| ----- | ----------- | -------- |
| 5–10 | 2–3명 | 3–5 |
| 10–20 | 3–5명 | 4–6 |
| 20+ | 5–7명 | 4–5 |

**CLAUDE.md 포인터 등록 (필수):**

```markdown
## 하네스: {Domain}

**Goal:** {한 줄 목표}

**Trigger:** {도메인} 요청 시 `{orchestrator-skill}`를 사용한다. 단순 질의는 직접 답변.

**Change History:**
| Date | Change | Target | Reason |
| ---- | ------ | ------ | ------ |
| YYYY-MM-DD | Initial build | All | - |
```

에이전트/스킬 목록, 디렉토리 구조, 세부 규칙은 적지 않는다.

**후속 실행 지원 (필수):**
- 오케스트레이터 description에 후속 키워드 포함 — "재실행", "업데이트", "수정", "보완", "이전 결과 기반으로"
- Phase 1에 컨텍스트 체크: `_workspace/` 존재 여부 + 입력 변화로 초기/후속/부분 재실행 자동 판별
- 에이전트 정의에 재호출 가이드: 이전 산출물이 있으면 읽고 개선, 사용자 피드백이 있으면 선택적 수정

### Phase 6: 검증 (Validation)

#### 6-1 구조 검증
- 에이전트 정의 파일 위치/네이밍 정상
- 모든 SKILL.md frontmatter(`name`, `description`) 유효
- 에이전트-스킬 상호 참조 일관성
- `commands/`에 자동 생성하지 않음(스킬과 commands는 다른 메커니즘)

#### 6-2 모드별 점검
- **팀**: 통신 경로 그래프, 작업 의존성, 팀 사이즈 적정성
- **서브**: 입출력 연결, `run_in_background` 설정, 반환 수집 로직
- **하이브리드**: Phase별 모드 명시, 모드 경계에서 데이터 손실 없음

#### 6-3 스킬 실행 테스트
- 자연 프롬프트 2–3건씩 작성 → 실제 트리거 확인
- with/without 비교로 스킬 부가가치 측정
- 일반화 가능한 피드백은 본문에 반영, 좁은 수정은 피함

#### 6-4 트리거 검증
- **트리거 되어야 할 쿼리** 8–10개
- **트리거 되면 안 될 쿼리(near-miss)** 8–10개 — 의미가 인접한 경계 사례 우선

#### 6-5 Dry-run
- Phase 시퀀스 논리 점검
- Phase 간 입출력 매칭, dead link 식별
- 에러 시나리오의 폴백 가능성

#### 6-6 테스트 시나리오 문서화
오케스트레이터 본문에 `## 테스트 시나리오` 섹션을 두고 정상 흐름 1개 + 에러 흐름 1개 이상을 적는다.

### Phase 7: 진화 (Evolution)

#### 7-1 실행 후 피드백 요청
하네스 1회 실행 직후 개선 의견을 묻는다(필수 아님, 항상 제안).

#### 7-2 피드백 반영 매핑

| 피드백 유형 | 수정 위치 |
| ---------- | -------- |
| 결과 품질 | 에이전트의 스킬 |
| 에이전트 역할 | `agents/{name}.md` |
| 흐름 순서 | 오케스트레이터 |
| 팀 구성 | 오케스트레이터 + 에이전트 정의 |
| 트리거 누락 | 스킬 description |

#### 7-3 변경 이력 기록 (필수)
모든 수정은 CLAUDE.md의 Change History에 1줄로 남긴다. 날짜는 절대 날짜로 적는다.

#### 7-4 진화 트리거
- 사용자의 명시적 수정 요청
- 동일 피드백이 2회 이상 반복
- 동일 패턴의 에이전트 실패 반복
- 사용자가 매번 같은 우회책을 쓰고 있을 때

#### 7-5 운영/유지보수 워크플로우
1. **감사** — `agents/`/`skills/` ↔ 오케스트레이터/CLAUDE.md 드리프트 보고
2. **단일 변경** — 한 번에 하나만 수정
3. **CLAUDE.md 갱신** — 날짜·변경·대상·이유 기록
4. **검증** — Phase 6-1 구조 검증, 영향 범위가 크면 6-3/6-5도 실행

## 산출물 체크리스트

- [ ] `agents/` — 에이전트 정의 파일 (빌트인 타입도 정의 필수)
- [ ] `skills/` — SKILL.md(+ 필요 시 references/)
- [ ] 단일 오케스트레이터 스킬 (데이터 흐름 + 에러 정책 + 테스트 시나리오)
- [ ] 실행 모드 명시 (팀 / 서브 / 하이브리드, 하이브리드는 Phase별 모드 표기)
- [ ] 모든 Agent 호출에 `model: "opus"`
- [ ] 자동으로 `commands/`를 만들지 않음
- [ ] 기존 에이전트/스킬과 트리거 충돌 없음
- [ ] 스킬 description에 후속 작업 키워드 포함
- [ ] SKILL.md < 500줄
- [ ] 자연 프롬프트 2–3건으로 실제 트리거 확인
- [ ] 트리거 should/should-not 검증
- [ ] CLAUDE.md에 하네스 포인터(트리거 규칙 + 변경 이력)
- [ ] 오케스트레이터 Phase 1에 컨텍스트 체크(초기/후속/부분 자동 판별)

## 사용자 소통 규칙

- 각 Phase 종료 시 핵심 산출물을 사용자에게 보여주고 다음 Phase 진행을 명시적으로 확인한다.
- 인터뷰성 질문은 한 번에 1개만 던진다(목록 나열 금지).
- 도메인 jargon이 익숙하지 않은 사용자에게는 먼저 1줄 정의를 곁들인 뒤 진행한다.
- 어떤 Phase든 사용자가 "건너뛰기"를 요청하면 그대로 SKIP하고 결정 이력에 남긴다.
