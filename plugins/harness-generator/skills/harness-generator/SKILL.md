# Harness Generator — 하네스 엔지니어링 수동·인터랙티브 제너레이터 스킬

도메인 요구를 입력으로 받아 **에이전트 정의 + 스킬 + 오케스트레이터**로 구성된 하네스 한 묶음을 7단계로 설계·생성·검증한다. 특정 기술 스택(프론트엔드/백엔드/데이터)이나 특정 도구에 종속되지 않은 공통 메타 절차만 다룬다.

## 왜 하네스가 필요한가

단일 스킬이나 일회성 프롬프트로 처리하기 어려운 다단계 작업(분석 → 설계 → 실행 → 검증)이 있을 때, 산출물과 책임을 분리한 에이전트/스킬을 묶고 오케스트레이터가 흐름을 통제하면 다음 이점이 있다.

1. **재현성** — 같은 도메인 요청에 항상 동일한 절차로 응답한다.
2. **전문성 분리** — 각 에이전트가 한 가지 책임만 진다.
3. **점진적 진화** — 변경 이력이 남고 후속 실행이 안전해진다.

## 핵심 원칙

- **에이전트 팀이 기본** — 2명 이상 협업이 필요하면 `TeamCreate` 기반 팀을 우선 검토하고, 통신 오버헤드가 불필요할 때만 서브에이전트 모드를 선택한다.
- **에이전트 정의 파일 필수** — 빌트인 타입(`general-purpose`/`Explore`/`Plan`)을 쓰더라도 `agents/{name}.md`로 정의를 남긴다. Agent 호출 프롬프트에 역할을 직접 박지 않는다.
- **모든 Agent 호출은 `model: "opus"`** — 추론 품질이 하네스 품질을 좌우한다.
- **CLAUDE.md에는 포인터만** — 트리거 규칙과 변경 이력만 남기고, 에이전트/스킬 목록·디렉토리 구조는 적지 않는다.
- **스킬 본문은 < 500줄** — 초과분은 `references/`로 분리한다(Progressive Disclosure).
- **명령형/Why-First** — "~한다/~하라" 어조와 이유 설명을 함께 둔다. LLM은 이유를 알면 엣지 케이스에서 옳게 판단한다.

## 산출물 배치 규약

이 스킬은 다음 경로에 파일을 만든다(플러그인 안에서 동작 시):

```
plugins/{plugin}/
  skills/{domain}-orchestrator/SKILL.md   # 오케스트레이터 스킬
  skills/{domain}-orchestrator/references/ # 필요 시 보충
  skills/{agent-or-step}/SKILL.md         # 도메인별 단계 스킬들
  agents/{name}.md                        # (Claude Code 에이전트 정의)
  evals/trigger-eval.json                 # Phase 6-4 트리거 검증 산출물 (재검증 가능한 자산)
  CLAUDE.md (또는 루트 CLAUDE.md)         # 하네스 포인터 + 변경 이력
```

플러그인 외부(루트 `.claude/`)에서 동작 시에는 `.claude/skills/`, `.claude/agents/`로 매핑한다. 이때 eval 산출물은 하네스 루트의 `evals/`에 동일하게 둔다. 디렉토리 형태는 환경에 맞춰 결정하되, **에이전트 정의는 별도 파일, 오케스트레이터는 단일 진입 스킬**이라는 구조는 항상 유지한다.

## 7단계 프로세스

### Phase 0: 현황 감사 (Audit)

설계 전에 이미 존재하는 하네스/에이전트/스킬과의 충돌을 차단한다.

**확인 항목:**
- `agents/`, `skills/`, `commands/` 디렉토리 존재 및 목록
- 루트 `CLAUDE.md`의 기존 하네스 포인터
- 동일 도메인 키워드를 트리거로 쓰는 기존 스킬 — 마켓플레이스/작업 환경에 설치된 **모든 SKILL/command description을 grep**해 새 하네스의 트리거 키워드와 교집합을 표로 보고한다(예: '리뷰/PR/검증' 같은 흔한 키워드는 이미 설치된 다른 스킬과 겹치기 쉽다). 교집합이 있으면 그 충돌 스킬명을 Phase 6-4 near-miss 입력으로 넘긴다.

**분기:**
- 신규 도메인 → Phase 1로 진행
- 기존 하네스에 추가/확장 요청 → Phase 1을 축약하고 Phase 7-5(운영) 절차도 함께 적용
- 단순 수정/보완 요청 → Phase 7-5(운영)로 직접 진입

**확장/보완 분기 시 대상 결속(필수):** 확장 또는 보완으로 판별되면 작업 대상 하네스를 추측하지 말고, 루트/플러그인 `CLAUDE.md` 포인터(Goal/Trigger)와 그 하네스의 `agents/`·`skills/` 실파일로 **1:1 결속**한다. 동일 환경에 후보 하네스가 2개 이상이면 사용자에게 한 번에 1개 질문으로 어느 묶음인지 확정한 뒤 진행한다.

감사 결과를 사용자에게 1회 요약하고, 진행 방향에 대한 승인을 받는다. 보고는 1줄 형식을 쓴다. 예: `[Phase 0] 신규 도메인 (기존 하네스 없음, 트리거 충돌 없음) — 다음: Phase 1 도메인 분석. 진행할까요?` 확장 분기는 결속 대상을 명시한다. 예: `[Phase 0] 기존 리서치 하네스 확장 (대상: skills/research-orchestrator, 검증단계+에이전트 1 추가) — 다음: Phase 1 축약 + Phase 7-5 운영 병행. 진행할까요?`

### Phase 1: 도메인 분석 (Domain Analysis)

사용자 요청에서 다음을 추출한다. 모호하면 grill-me 식으로 **한 번에 한 질문씩** 명확화한다. 질문 순서는 하류 Phase가 의존하는 순서를 따른다 — **① 도메인 범위 → ② 산출물 형태 → ③ 반복성**이 불명확하면 이것부터 묻는다(범위 없이는 작업 분해가, 산출물 형태 없이는 수용 기준이, 반복성 없이는 후속 실행 설계가 성립하지 않는다).

1. **도메인 범위** — 무엇을, 어디까지, 어떤 산출물 형태로
2. **작업 유형 분해** — 생성/검증/편집/분석 중 어느 조합인지
3. **기술/리소스 제약** — 사용 가능한 도구, 외부 API, 시간 제약
4. **사용자 숙련도** — 비전문 사용자라면 jargon(assertion, JSON schema 등) 사용 자제
5. **재실행 시나리오** — 이 하네스가 일회성인가, 반복 실행되는가

**산출 ① 도메인 카드 (고정 필드 5개):**

```
purpose        — 무엇을 어디까지
task_types     — 생성|검증|편집|분석 조합
constraints    — 도구·API·시간 제약
recurrence     — 일회성|반복
user_expertise — 전문|비전문 (jargon 사용 여부 결정)
```

위 추출 항목 1~5와 필드가 1:1 대응한다 — 빈 필드는 곧 안 물어본 질문이므로, 비울 거면 비운 이유를 적는다.

**산출 ② 수용 기준 2~3건 (관찰형):** 완성된 하네스가 대표 요청을 처리했을 때 **관찰로 판정 가능한** 문장으로 적는다. Phase 6 검증이 이 문장들을 판정 기준으로 그대로 재사용한다 — 검증 시점에 기준을 즉석에서 지어내지 않기 위해서다. 각 기준은 두 유형으로 구분해 표기한다: **(설계정합형)** 산출물 구조로 관찰 가능한 것(예: "산출에 X 섹션이 포함된다")은 6-5 dry-run이 재사용하고, **(런타임형)** 실제 실행 결과라야 판정되는 것(예: "배포가 성공한다")은 6-5의 설계 정합 점검으로는 판정 불가이므로 6-3 실행 테스트로 라우팅한다. 확장 케이스(Phase 0 분기)에서 단계·에이전트를 추가하면, 추가분마다 관찰형 수용 기준을 최소 1건 증분 작성해 기존+증분을 합쳐 판정한다.

| 좋은 수용 기준 | 나쁜 수용 기준 |
| -------------- | -------------- |
| "산출에 X 섹션이 포함된다" | "좋은 결과를 낸다" |
| "Y 형식 파일이 생성된다" | "사용자가 만족한다" |
| "에러 흐름에서 누락 처리 후 진행한다" | "잘 동작한다" |

정답·자동채점기가 없는 개방형 도메인(UI 구현, 문서 작성 등)이면 관찰형으로 쓰되 그 한계를 도메인 카드에 함께 남긴다 — 없는 정답을 지어내지 않는다.

**진행 게이트:** 도메인이 너무 모호해 수용 기준을 한 건도 못 쓰면 지어내지 말고, 무엇이 막혔는지 밝힌 뒤 질문으로 돌아간다. 빈 명세로 Phase 2에 진행하지 않는다 — 수용 기준 없는 설계는 Phase 6에서 검증할 대상이 없다. 단, 사용자가 명시적으로 건너뛰기를 요청하면 사용자 소통 규칙("건너뛰기" 항목)이 우선하며, 수용 기준 미작성 사실을 결정 이력에 남기고 진행한다.

**Phase 0 축약 분기와의 관계:** Phase 0에서 기존 하네스 확장으로 분기해 Phase 1을 축약하는 경우, 기존 도메인 카드·수용 기준이 있으면 재사용·증분 갱신하고 새로 전부 작성하지 않는다. 재사용할 카드·기준의 회수 출처는 Phase 0에서 결속한 대상 하네스의 `CLAUDE.md`(또는 `_workspace/` 보존본)이며, 출처가 없으면 카드를 새로 작성하되 그 사실을 결정 이력에 남긴다.

### Phase 2: 아키텍처 설계 (Team Architecture)

#### 2-1 실행 모드 선택

| 모드 | 사용 시점 | 특징 |
| ---- | --------- | ---- |
| **에이전트 팀** (기본) | 2명 이상 실시간 협업 필요 | `TeamCreate` + `SendMessage` + `TaskCreate` 자체 조율 |
| **서브에이전트** | 결과 핸드오프만 충분 | `Agent` 직접 호출, `run_in_background`로 병렬화 |
| **하이브리드** | Phase별 특성 차이 큼 | Phase별 모드를 명시 |

**판단 흐름:** 팀 모드를 먼저 검토하고, 통신 비용 > 협업 이득일 때만 서브에이전트로 강등한다. '통신 비용'은 감으로 비우지 말고 구조에서 추정한다 — 비용 요인 표와 정량 앵커는 `references/agent-design.md` §1 '구조 기반 토큰비용 추정 요인' 참조.

#### 2-2 아키텍처 패턴 선택

상세는 `references/agent-design.md` 참조.

| 패턴 | 구조 | 적합 |
| ---- | ---- | ---- |
| 파이프라인 | 순차 | 단계 의존이 강함 |
| 팬아웃/팬인 | 병렬 → 통합 | 다각 분석 후 결합(팀 필수) |
| 전문가 풀 | 라우팅 | 상황별 호출 |
| 생성-검증 | 쌍 반복 | 품질 게이트 |
| 감독자 | 동적 분배 | 가변 작업량 |
| 계층적 위임 | 재귀 분해 | 복잡 문제 |

#### 2-3 에이전트 분리 기준 4축

전문성 / 병렬성 / 컨텍스트 격리 / 재사용성. 한 에이전트가 두 축에서 모호하면 분리하되, 각 축의 통합 신호를 역방향으로 함께 검사한다(상세 표는 `references/agent-design.md` §4) — 정당화되지 않는 에이전트는 늘리지 않는다.

**산출:** 모드 선택 근거 + 패턴 + 에이전트 N명의 책임 카드 + **구조 비용 메모 1줄**(에이전트 수·단계 수·예상 통신 방식 — 사용자가 이 trade-off를 보고 Phase 2 승인을 결정한다).

#### 2-4 품질 4차원 비후퇴 자기점검 (설계 시 정적 체크)

하네스 한 묶음을 설계할 때, 아래 4개 품질 차원을 *동시에* 의식하고 **한 차원을 끌어올리려고 다른 차원을 후퇴시키는 거래를 스스로 거절**한다. 이는 후보를 점수로 줄세우는 일이 아니라, 지금 설계 중인 단일 산출물이 한쪽으로 기울지 않았는지 보는 정적 렌즈다.

| 차원 | 좋은 방향 | 이 하네스에서 이미 보는 곳 |
| --- | --- | --- |
| 동작정합 | 대표 요청을 의도대로 처리 | Phase 1 관찰형 수용 기준 / Phase 6 검증 |
| 규칙본문비용 | 같은 정합을 더 적은 줄·더 적은 MUST/NEVER로 | 본문 < 500줄 / skill-guide §2 컨텍스트 절약 |
| 트리거정밀도 | should / should-not 경계가 또렷 | Phase 6-4 트리거 검증 |
| 일반화 | 한 발화가 아니라 결함 모드 자체를 닫음 | skill-guide §2 일반화 / 6-5 도메인 전용성 점검 |

**자기점검 질문 (설계 결정마다):**
- 수용 기준(동작정합)을 채우려고 본문 줄·MUST/NEVER를 불렸는가? 불렸다면 그만큼 정합 이득이 실제로 있는가? 이득 없는 본문 증가는 되돌린다.
- 한 입력 사례에만 들어맞는 좁은 규칙을 넣어 일반화를 깎지 않았는가? 좁은 우회책이면 원리 수준으로 일반화한다(skill-guide §2).
- description을 pushy하게 넓히다가 인접 발화까지 흡수해 트리거정밀도를 떨어뜨리지 않았는가? Phase 6-4로 재판정한다.

**왜:** 단일 점수(예: '정합만') 최적화는 '정합을 사려고 본문을 부풀리는' 국소 최적에 빠진다 — 더 많은 규칙이 더 나은 하네스가 아니다. 이유로 설득되는 더 적은 규칙이 우월하다. 한 차원 이득이 다른 차원 손실로 상쇄되면, 그 설계는 채택 전에 되돌려 더 작은 개입으로 다시 만든다. 이 점검은 단일 설계의 정적 자기점검이며, 후보 여러 개를 좌표로 채점해 고르는 일(자동 탐색·Pareto)은 이 스킬의 범위가 아니다.

### Phase 3: 에이전트 정의 작성 (Agent Definitions)

각 에이전트마다 `agents/{name}.md`를 만든다. 다음 섹션을 모두 포함한다.

```markdown
---
name: {agent-name}
description: {언제 호출되는가, pushy하게}
---

## Core Role
{한 문장 책임}

## Work Principles
- {원칙 1 — Why까지 한 줄 설명}
- ...

## Input / Output Protocol
- 입력: {형태와 출처}
- 출력: {형태와 저장 위치, 예: _workspace/{phase}_{agent}_{artifact}.md}

## Error Handling
- {예상 실패 모드와 대응}

## Collaboration
- {다른 에이전트와의 인계 방식}

## Team Communication Protocol  (팀 모드에서만)
- 송신 대상: {agent-name 목록 / 토픽}
- 수신 대상: {agent-name 목록 / 토픽}
- 작업 등록 범위: {TaskCreate에 올릴 작업의 종류}
```

**필수 규칙:**
- 빌트인 타입을 쓰는 에이전트도 정의 파일을 둔다(`subagent_type`은 Agent 호출 시 지정).
- 모든 Agent 호출에 `model: "opus"`를 명시한다.
- QA/검증 에이전트는 `general-purpose`(읽기 전용 `Explore` 금지), 모듈별로 점진 실행.

### Phase 4: 스킬 작성 (Skill Creation)

각 에이전트의 능력은 스킬로 분리한다. 한 에이전트가 여러 스킬을 가질 수 있고, 스킬은 에이전트 간 공유 가능하다.

**SKILL.md 작성 규칙은 `references/skill-guide.md`에 모았다. 핵심 요약:**

- **frontmatter 필수**: `name`, `description` (다른 키 추가 금지)
- **description은 트리거 메커니즘** — 무엇을/언제/유사 사례 구분 + 약간 pushy하게
- **본문 < 500줄** — 초과분은 `references/`로 분리
- **명령형 어조** — "~한다", "~하라"
- **Why-First** — 규칙보다 이유, 일반화된 원리로 기술
- **번들링 신호** — 같은 헬퍼/접근법이 3회 이상 재현되면 `scripts/`로 묶기
- **Progressive Disclosure** — 메타데이터(항상 로드) / 본문(트리거 시) / references(요청 시)

#### 출력 데이터 스키마 (필요 시)

평가/리포트가 필요한 도메인이라면 다음 스키마를 그대로 쓴다(필드명 변형 금지).

- `eval_metadata.json` — `eval_id`, `eval_name`, `prompt`, `assertions[]`
- `grading.json` — `expectations[]{text,passed,evidence}`, `summary{passed,failed,total,pass_rate}`
- `timing.json` — `total_tokens`, `duration_ms`, `total_duration_seconds`

서브에이전트 완료 알림에서만 `total_tokens`/`duration_ms`를 읽을 수 있으므로 즉시 저장한다.

### Phase 5: 오케스트레이션 (Orchestrator)

오케스트레이터는 모든 에이전트와 스킬을 묶는 단일 진입점 스킬이다. 모드별 생성 템플릿은 `references/orchestrator-generate.md` 참조.

**모든 오케스트레이터 공통 골격:**

```
Phase 0  — 컨텍스트 확인 (초기 / 후속 / 부분 재실행 자동 판별)
Phase 1  — 입력 분석 + _workspace/ 준비
Phase 2..N — 도메인 본 작업 (팀/서브/하이브리드 중 모드 명시)
Phase 마지막 — 통합 산출물 작성, _workspace/ 정리/보존 결정
```

**데이터 전달 전략 매트릭스:**

| 전략 | 도구 | 모드 | 적합 |
| ---- | ---- | ---- | ---- |
| 메시지 기반 | `SendMessage` | 팀 | 실시간 조정 |
| 작업 기반 | `TaskCreate`/`TaskUpdate` | 팀 | 진척/의존성 |
| 파일 기반 | Read/Write 합의된 경로 | 팀+서브 | 큰 데이터/감사 |
| 반환 기반 | `Agent` 반환 메시지 | 서브 | 단발 결과 수집 |

**파일 경로 규약:**
- 중간 산출물: `_workspace/{phase}_{agent}_{artifact}.{ext}`
- 최종 산출물: 사용자가 읽을 위치(예: 프로젝트 루트, `docs/`)
- `_workspace/`는 후속 실행을 위해 보존하고, 새 실행은 `_workspace_prev/`로 이동시킨 뒤 다시 만든다.

**에러 정책 (필수):**
- 1회 재시도, 그래도 실패하면 해당 결과만 누락 처리하고 진행
- 충돌 데이터는 삭제하지 말고 출처 태그로 보존

**팀 사이즈 가이드:**

| 작업량 | 권장 팀 사이즈 | 인당 작업 |
| ----- | ----------- | -------- |
| 5–10 | 2–3명 | 3–5 |
| 10–20 | 3–5명 | 4–6 |
| 20+ | 5–7명 | 4–5 |

**CLAUDE.md 포인터 등록 (필수):**

```markdown
## 하네스: {Domain}

**Goal:** {한 줄 목표}

**Trigger:** {도메인} 요청 시 `{orchestrator-skill}`를 사용한다. 단순 질의는 직접 답변.

**Change History:**
| Date | Change | Target | Reason |
| ---- | ------ | ------ | ------ |
| YYYY-MM-DD | Initial build | All | - |
```

에이전트/스킬 목록, 디렉토리 구조, 세부 규칙은 적지 않는다.

**후속 실행 지원 (필수):**
- 오케스트레이터 description에 후속 키워드 포함 — "재실행", "업데이트", "수정", "보완", "이전 결과 기반으로"
- Phase 1에 컨텍스트 체크: `_workspace/` 존재 여부 + 입력 변화로 초기/후속/부분 재실행 자동 판별
- 에이전트 정의에 재호출 가이드: 이전 산출물이 있으면 읽고 개선, 사용자 피드백이 있으면 선택적 수정

### Phase 6: 검증 (Validation)

#### 6-1 구조 검증
- 에이전트 정의 파일 위치/네이밍 정상
- 모든 SKILL.md frontmatter(`name`, `description`) 유효
- 에이전트-스킬 상호 참조 일관성
- `commands/`에 자동 생성하지 않음(스킬과 commands는 다른 메커니즘)

#### 6-2 모드별 점검
- **팀**: 통신 경로 그래프, 작업 의존성, 팀 사이즈 적정성
- **서브**: 입출력 연결, `run_in_background` 설정, 반환 수집 로직
- **하이브리드**: Phase별 모드 명시, 모드 경계에서 데이터 손실 없음

#### 6-3 스킬 실행 테스트
- 자연 프롬프트 2–3건씩 작성 → 실제 트리거 확인
- with/without 비교로 스킬 부가가치 측정
- 일반화 가능한 피드백은 본문에 반영, 좁은 수정은 피함

#### 6-4 트리거 검증 (산출물 필수)

- **트리거 되어야 할 쿼리** 8–10개
- **트리거 되면 안 될 쿼리(near-miss)** 8–10개 — 의미가 인접한 경계 사례 우선. 같은 환경에 이미 있는 스킬의 영역 발화를 반드시 포함한다(상호 오발동 가드). 이 중 **최소 2건은 Phase 0에서 grep으로 발견한 인접 스킬의 대표 발화**로 채운다 — 충돌 키워드가 검출됐는데 near-miss에 없으면 그 가드는 검증을 빠져나간다.

**절차 (사용자와 함께):**
1. 쿼리 초안을 표로 제시하고, 사용자가 실제로 쓸 법한 표현과 도메인 near-miss를 추가·수정하게 한다(질문은 한 번에 1개 — 소통 규칙 준수).
2. 각 쿼리를 생성된 description에 대조해 판정한다. 오분류가 나오면 description을 수정하고 전체를 재판정한다.
3. 확정본을 `{harness-root}/evals/trigger-eval.json`에 저장한다. 형식은 이 플러그인 자신의 `evals/trigger-eval.json`과 동일하게 쓴다:

       [
         { "query": "트리거 되어야 할 자연 발화", "should_trigger": true },
         { "query": "인접하지만 발동하면 안 되는 발화", "should_trigger": false }
       ]

**왜 파일로 남기는가:** description은 이후 보완(Phase 7)에서 가장 자주 수정되는 표적이다. 검증을 구두 확인으로 끝내면 다음 수정 때 같은 기준으로 재검증할 수 없다. 저장된 trigger-eval이 있어야 7-5 감사에서 회귀 검증이 되고, 검증이 1회성 행사가 아니라 재실행 가능한 자산이 된다.

#### 6-5 Dry-run (적대적 점검 + 리포트)

dry-run은 런타임 실행이 아니라 **설계 정합(design-conformance)** 점검이다. 중립 확인이 아니라 "이 하네스는 어디서 무너지는가"를 능동적으로 캔다 — 칭찬이 아니라 약점을 찾는 것이 점검자의 일이다.

**점검 항목:**
- Phase 시퀀스 논리 점검 — 대표 요청 1건을 Phase 골격에 논리적으로 흘려보고, 시퀀스가 그 요청을 실제로 푸는지 확인
- Phase 간 입출력 매칭, dead link(존재하지 않는 파일/스킬/에이전트 참조) 식별
- 에러 시나리오의 폴백 가능성
- **검증자 컨텍스트 오염** — 생성-검증 쌍에서 검증자가 생성자의 의도·사고 과정을 입력으로 받는 경로가 있는가 (`references/agent-design.md` §6 '컨텍스트 오염형 검증자' 참조)
- **비용 폭발 지점** — 상한 없는 반복 루프, 정당화되지 않은 팬아웃 폭
- **도메인 전용성(과적합) 점검** — 이 하네스 골격이 *이 도메인에서만* 성립하는 가정에 박제됐는지 캔다. 인접 도메인 probe 1건을 머릿속으로 흘려보고(예: 리서치 하네스라면 "같은 골격으로 코드리뷰 도메인을 처리하려면?"), 특정 키워드·산출 형식·도구 호출이 *하드코딩된 가정*으로 깨지는 지점을 찾는다. 깨지면 그 자체가 결함은 아니다 — 다만 "이 설계는 이 도메인 전용"임을 리포트에 명시해, 분리 4축의 *재사용성* 판단(`references/agent-design.md` §4)이 가설이 아니라 점검된 사실이 되게 한다. (도메인 무관 메타 도구가 도메인 종속 산출물을 내면 자기모순이다 — `references/skill-guide.md` §8의 '도메인 종속 예시 — 일반화하거나 references로 분리' 항목을 dry-run 단계로 확장한 것이다.) 단 인접 도메인 전이를 정량 점수로 채점해 후보를 고르는 일(전이성 채점)은 이 스킬의 범위가 아니다.

**절차와 마감:**
1. 정상 흐름 1개 + 에러 흐름 1개 시나리오로 수행하고, 결과는 아래 검증 리포트 템플릿을 따른다.
2. 판정 불가한 항목은 통과로 찍지 말고 "판정 불가"로 명시한다 — 확신을 가장하지 않는다. 약점을 못 찾았으면 "못 찾았다"고 쓰되, 위 점검 항목을 훑은 흔적을 리포트에 남긴다.
3. 결과 리포트를 사용자에게 제시하고, 발견된 약점마다 수정/수락을 함께 결정한다. 합의된 리포트는 `_workspace/6-5_validation_dryrun-report.md`(파일 경로 규약 준수)로 저장해 후속 실행과 Phase 7-5 드리프트 감사의 비교 기준선으로 쓴다.

**검증 리포트 템플릿:** 첫 줄에 "런타임 실행이 아닌 설계 정합 점검"임을 명시하고 아래 구성을 그대로 따른다.

    # {harness-name} 검증 리포트
    (첫 줄: 런타임 실행이 아니라 설계 정합 점검임을 명시)

    ## 시나리오 A — {대표 정상 입력} (정상)
    | Phase | 동작 | 입력 → 산출 | 점검 |
    (Phase별 1행씩 — Phase 간 데이터 흐름 정합을 1~2줄로 덧붙임)

    ## 시나리오 B — {에러 입력} (에러)
    (동일 표 — 발동한 폴백/에러 정책을 점검 열에 명시)

    ## dead link / 누락 점검
    - 오케스트레이터 → references / 단계 스킬: 실파일 해석 여부
    - 에이전트 정의 → 스킬 교차참조: 해석 여부
    - 에이전트 정의 ↔ 오케스트레이터 Phase 호출: 1:1 대응 여부

    ## 발견된 약점과 처리
    | 약점 | 심각도 | 처리(수정 / 사용자 수락) |

    **Verdict:** PASS / FAIL — {근거 1줄, 미해결 약점 포함}

**안티 패턴:**
- **칭찬형 점검** — 잘 된 점만 나열하고 실패 모드를 캐지 않음. 검증은 적대적이어야 한다.
- **정상 흐름만 점검** — 에러 시나리오 없는 dry-run은 절반의 검증이다.
- **구두 검증으로 종료** — 리포트를 파일로 남기지 않으면 Phase 7-5 드리프트 감사 때 비교 기준이 없다.
- **런타임형 기준을 설계정합으로 통과 처리** — "배포가 성공한다"처럼 실제 실행이라야 판정되는 (런타임형) 수용 기준을 6-5 설계 정합 점검에서 PASS로 찍지 않는다. 이런 기준은 판정 불가로 명시하고 6-3 실행 테스트로 라우팅한다(Phase 1 수용 기준 유형 구분 참조).
- **dry-run 입력 오염** — 설계 근거 메모·대화 이력을 입력에서 제외하고 생성된 파일만 읽는다(원리는 `references/agent-design.md` §6 참조).

#### 6-6 테스트 시나리오 문서화

오케스트레이터 본문에 `## 테스트 시나리오` 섹션을 두고 정상 흐름 1개 + 에러 흐름 1개 이상을 적는다. 각 시나리오는 `Phase | 동작 | 입력 → 산출 | 점검` 표로 기록한다 — 형식이 고정돼야 후속 실행·감사(Phase 7-5) 때 같은 기준으로 재점검할 수 있다. 6-5를 수행했다면 그 시나리오를 요약 전재하고(새로 짓지 않는다), 6-5가 SKIP된 경우에만 새로 작성한다.

### Phase 7: 진화 (Evolution)

#### 7-1 실행 후 피드백 요청
하네스 1회 실행 직후 개선 의견을 묻는다(필수 아님, 항상 제안).

#### 7-2 피드백 반영 매핑

| 피드백 유형 | 수정 위치 |
| ---------- | -------- |
| 결과 품질 | 에이전트의 스킬 |
| 에이전트 역할 | `agents/{name}.md` |
| 흐름 순서 | 오케스트레이터 |
| 팀 구성 | 오케스트레이터 + 에이전트 정의 |
| 트리거 누락 | 스킬 description |

#### 7-3 변경 이력 기록 (필수)
모든 수정은 CLAUDE.md의 Change History에 1줄로 남긴다. 날짜는 절대 날짜로 적는다.

#### 7-4 진화 트리거
- 사용자의 명시적 수정 요청
- 동일 피드백이 2회 이상 반복
- 동일 패턴의 에이전트 실패 반복
- 사용자가 매번 같은 우회책을 쓰고 있을 때

#### 7-5 운영/유지보수 워크플로우
1. **감사** — `agents/`/`skills/` ↔ 오케스트레이터/CLAUDE.md 드리프트 보고
2. **단일 변경** — 한 번에 하나만 수정
3. **CLAUDE.md 갱신** — 날짜·변경·대상·이유 기록
4. **검증** — Phase 6-1 구조 검증, description을 수정했으면 evals/trigger-eval.json 전체 재판정, 영향 범위가 크면 6-3/6-5도 실행

#### 7-6 비파괴 우선 (additive-first) — 보완 시 회귀 차단

자기가 만든 하네스를 7-5로 보완·수정할 때, **이미 동작하던 트리거 문구·Phase 흐름·산출 스키마를 고치기 전에 비파괴적 추가·완화를 먼저 시도한다.** 덮어쓰기는 이미 통과하던 동작을 조용히 깨뜨려(회귀) 후속 실행을 망가뜨리기 때문이다. 이 원칙은 7-5의 '감사 → 단일 변경 → 갱신 → 검증' 흐름에서 '단일 변경'의 *방향*을 정한다. 결함 유형별로:

- **트리거 결함** — 기존 should/should-not 문구를 바꾸지 말고 **누락 케이스를 추가**하거나 near-miss를 보강한다. (description 표면을 불가피하게 건드린 경우의 회귀 확인은 7-5 4단계의 `evals/trigger-eval.json` 전체 재판정으로 수행한다.)
- **흐름 결함** — Phase 순서를 갈아엎지 말고 **가드/폴백 분기를 덧댄다**.
- **스키마 결함** — 필드를 제거·개명하지 말고 **optional 필드 추가 또는 기본값 명시**로 완화한다.

비파괴 추가로 결함을 닫을 수 없다는 근거가 분명할 때만 덮어쓰기(destructive 수정)를 택하되, 그 사실과 깨질 수 있는 동작을 7-3 CLAUDE.md Change History에 함께 남긴다. 특히 7-4 진화 트리거상 '동일 피드백 2회 이상 반복'에 해당하거나 Change History에 같은 대상을 이미 한 번 고친 기록이 있으면 additive를 우선한다 — 같은 곳을 반복해 덮어쓰는 건 본문 비용만 늘리고 회귀 위험을 키운다.

> 이 원칙은 보완의 *방향*(덧대기 우선)을 정하며, 7-5의 '한 번에 하나만 수정'(변경 granularity)과는 직교한다 — 둘을 함께 지킨다.

### 인수 점검 (Acceptance Check) — 체크리스트의 evidence 판정

위 체크리스트(13항목)를 눈으로 훑고 끝내지 않는다. 각 항목을 evidence를 인용해 passed/failed로 판정하고, 결과 표를 사용자에게 제시해 하네스 인수 여부를 명시적으로 확인받는다.

- **evidence의 두 형태** — (a) 정적 구조 항목(예: `model: "opus"`, frontmatter, CLAUDE.md 포인터)은 **생성된 파일의 실제 위치(file:line 또는 섹션명)**를 인용한다. (b) 수행형 항목(자연 프롬프트 트리거 확인, should/should-not 검증, 트리거 충돌 부재)은 **검증 수행 기록(테스트 프롬프트와 관찰 결과 1줄 요약)**을 인용한다.
- **판정 형식은 Phase 4의 출력 데이터 스키마 중 `grading.json`을 그대로 쓴다** — 필드 정의는 `references/skill-guide.md` §7이 단일 출처다(필드명 변형 금지). 이 스킬이 생성물에 시키는 평가 형식을 자기 마감에도 쓴다.
- **evidence 없는 passed 금지** — 인용할 수 없으면 failed다. 자기 산출물에 관대해지는 칭찬형 자기평가를 막는 장치다.
- **failed 항목 처리** — (a) 즉시 수정 후 해당 항목만 재판정하거나, (b) 사용자가 의도적으로 수락하면 그 사실을 CLAUDE.md Change History에 1줄로 남긴다. 묵살 금지.
- 사용자가 원하면 판정 결과를 `_workspace/acceptance-check.json`(grading.json 형식)으로 저장한다. 후속 보완(Phase 7-5 감사) 시 직전 인수 상태와의 드리프트 기준선이 된다.

사용자 제시용 판정 표 예시(1행):

| 체크 항목 | 판정 | evidence |
|-----------|------|----------|
| 모든 Agent 호출에 `model: "opus"` | passed | `skills/research-orchestrator/SKILL.md` Phase 2 Agent() 호출 3건 전부 명시 |

여기서의 판정은 **단일 산출물의 자기 점검**이다 — 후보 여러 개의 비교, 품질×비용 좌표 채점(Pareto)은 이 스킬의 범위가 아니다(자동 탐색·채점 도메인). 후보 임계 채점(acceptance bar)과도 다른 개념이다.

## 사용자 소통 규칙

- **Phase 종료 보고 형식** — 각 Phase 종료 시 핵심 산출물을 보여준 뒤 `[Phase N] {핵심 결정 1줄} — 다음: {다음 Phase}. 진행할까요?` 1줄로 마감하고 명시적 확인을 받는다. 형식이 고정돼야 사용자가 승인 지점을 놓치지 않는다. 마지막 Phase 7 종료 시 "다음:" 슬롯은 "다음: 완료/피드백"으로 폴백한다.
- **파일 쓰기 게이트 (Phase 3·4·5 필수, Phase 6 산출물 포함)** — 파일을 생성/수정하는 Phase에서는 쓰기 *전에* 「생성할 파일 목록 + 각 파일의 description·핵심 섹션 구성」 미리보기를 제시하고 승인을 받은 뒤에만 쓴다. Phase 6이 만드는 검증 산출물(`evals/trigger-eval.json`, `_workspace/…_dryrun-report.md`, `_workspace/acceptance-check.json`)도 이 게이트 대상이다. 승인 없는 파일 생성 금지 — 사후 보고는 승인이 아니다. 동명 파일이 이미 있으면 덮어쓰기 전에 반드시 확인받는다.
- 인터뷰성 질문은 한 번에 1개만 던진다(목록 나열 금지).
- 도메인 jargon이 익숙하지 않은 사용자에게는 먼저 1줄 정의를 곁들인 뒤 진행한다.
- 어떤 Phase든 사용자가 "건너뛰기"를 요청하면 그대로 SKIP하고 결정 이력에 남긴다. 단 SKIP은 설계 논의의 생략이지 무승인 쓰기의 허용이 아니다 — 파일 쓰기 직전의 「파일 목록 1줄 확인」은 유지한다(미리보기 상세만 생략).
