## 1. 실행 모드 비교

| 항목 | 에이전트 팀 | 서브에이전트 |
| ---- | ----------------- | ------------------ |
| 통신 | 팀원 간 직접 (`SendMessage`) | 통신 불가, 결과만 반환 |
| 작업 관리 | 공유 작업 목록(`TaskCreate`) | 호출자가 수동 관리 |
| 비용 | 토큰 비용 큼 | 효율적 |
| 협업 품질 | 높음 (실시간 토론) | 제한적 |
| 적합 | 다각 분석/생성-검증 루프/감독자 | 독립 병렬/단발 작업 |

**선택 기준:** "에이전트 간 발견 공유와 실시간 토론이 결과 품질의 핵심인가?" → 그렇다면 팀, 아니면 서브에이전트.

복합/대형 워크플로우는 **기본적으로 팀**을 검토한 뒤, Phase별로 서브로 강등할 곳만 분리한다.

### 실행 모드 선택의 1차 근거 (Anthropic)

위 비교표의 trade-off 행(비용·협업 품질)과 "팀 기본" 원칙을 뒷받침하는 1차 근거. (표의 `SendMessage`/`TaskCreate` 등 도구 메커니즘은 Claude Code 고유이며 아래 출처의 직접 내용이 아니다.) Phase 2에서 사용자에게 모드 trade-off를 설명할 때 함께 제시한다. 외부 근거는 2025년 이후 발표 자료만 인용하며, 인용을 추가·교체할 때는 발표일을 확인해 링크 직후 (YYYY-MM) 형식으로 병기한다.

- **workflows vs agents 2분류** — [Effective Context Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)(2025-09)은 LLM 기반 workflows와 agents의 구분을 재확인하고, agent를 "LLMs autonomously using tools in a loop"(루프 안에서 자율적으로 도구를 쓰는 LLM)로 정의한다. 하네스 설계의 첫 질문은 "이 도메인은 고정 절차로 충분한가, 자율 루프(동적 판단)가 필요한가"다.
- **clean-context 서브에이전트 규약** — [Effective Context Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)(2025-09)에 따라 서브에이전트는 깨끗한 컨텍스트에서 작업하고 **1,000~2,000 토큰 요약만 반환**하게 설계한다. 서브에이전트 모드의 반환 메시지 규약으로 그대로 적용한다.
- **멀티에이전트 비용-품질 trade-off** — [Multi-agent Research System](https://www.anthropic.com/engineering/multi-agent-research-system)(2025-06): orchestrator-worker 구성이 breadth-first 탐색 태스크에서 단일 에이전트 대비 **+90.2%** 품질. 단 토큰비용은 일반 챗 대비 **약 15배**(단일 에이전트는 챗 대비 약 4배 — 단일 에이전트 대비로는 약 3.75배)다. 이득은 task-dependent다 — "팀 기본"은 품질 우선 가정이지 보편 법칙이 아니다. → 이 trade-off가 §1 선택 기준과 SKILL.md Phase 2-1 강등 규칙의 근거다.
- **반대 관점 (균형)** — Cognition [Don't Build Multi-Agents](https://cognition.ai/blog/dont-build-multi-agents)(2025-06)는 read-only 탐색 서브에이전트는 지지하되, 병렬 *의사결정* 에이전트 간 충돌을 경계한다. 병렬 단계에 의사결정 권한을 나눌 때는 팬인(통합자) 단계를 반드시 둔다.

### 구조 기반 토큰비용 추정 요인

"통신 비용 > 협업 이득"(SKILL.md Phase 2-1)을 판단하려면 비용을 어림이라도 추정해야 한다. 비용을 "모른다"로 비워두지 말고 설계 구조에서 추정해, 사용자에게 "이 구성이 왜 비싼가"를 설명한다. 위 비교표의 "비용: 토큰 비용 큼/효율적" 행을 정량화하는 절이다.

| 비용 요인 | 영향 |
| --- | --- |
| 에이전트 수 | spawn마다 컨텍스트 적재 비용 |
| 단계(Phase) 수 | 단계마다 호출 발생 |
| 팀 통신(`SendMessage`) | 메시지 건수·통신 경로 수에 비례 |
| 생성-검증 루프 | 반복 횟수 × (생성+검증) 비용 — 반복 상한 필수 |
| 다각 팬아웃 | 병렬 워커 수 × 비용 |

정량 앵커와 출처는 위 '실행 모드 선택의 1차 근거' 절의 Multi-agent Research System 항목 참조. 이득은 task-dependent이므로 협업이 품질의 핵심일 때만 정당화된다. 비용 요인이 두 개 이상 겹치는 설계(예: 팀 + 생성-검증 루프 + 팬아웃)는 비용이 곱으로 늘어나므로, 각 요인이 품질에 기여하는 이유를 Phase 2 산출(모드 선택 근거)에 명시한다. 팀 사이즈 가이드(SKILL.md Phase 5)와 함께 쓰면 "왜 이 인원인가"까지 설명할 수 있다.

## 2. 6가지 아키텍처 패턴

### 2-1 파이프라인 (Pipeline)
- **구조:** A → B → C 순차 흐름
- **목적:** 단계별 의존이 강하고 후공정이 전공정 산출 없이는 무의미할 때
- **모드:** 통신 필요성이 낮으면 서브, 단계 간 미묘한 조정이 필요하면 팀
- **예:** 요구사항 분석 → 설계 → 구현 계획

### 2-2 팬아웃/팬인 (Fan-out / Fan-in)
- **구조:** 다수 병렬 → 통합
- **목적:** 다각 관점에서 동시에 분석한 뒤 결합
- **모드:** **팀 필수** — 팀원 간 발견 공유가 결과 품질을 결정
- **예:** 보안 / 성능 / 테스트 리뷰어가 동시에 코드 점검 후 종합

### 2-3 전문가 풀 (Expert Pool)
- **구조:** 라우팅으로 상황별 전문가 호출
- **목적:** 입력 종류가 다양하고 매번 같은 전문가만 필요한 경우
- **모드:** 서브에이전트 적합
- **예:** 입력 파일 종류(SQL/JSON/YAML)에 따라 다른 분석가 호출

### 2-4 생성-검증 (Generate / Validate)
- **구조:** 생성자 ↔ 검증자 쌍의 반복 루프
- **목적:** 품질 게이트가 필요한 산출
- **모드:** 팀이 유리(피드백을 직접 주고받음). 반복 횟수 상한을 둔다(예: 2회).
- **예:** 글쓰기 + 비평, 코드 + 리뷰

### 2-5 감독자 (Supervisor)
- **구조:** 중앙 감독이 동적으로 작업을 분배
- **목적:** 가변 작업량/우선순위
- **모드:** 팀 (`TaskCreate` 공유 큐)
- **예:** 마이그레이션 작업을 파일별 복잡도에 따라 재분배

### 2-6 계층적 위임 (Hierarchical Delegation)
- **구조:** 상위 → 하위 → 더 하위로 재귀 분해
- **목적:** 복잡 문제를 단계적으로 잘게 나눠야 할 때
- **모드:** 팀 + 서브 혼합

### 2-7 업계 표준 매핑 — 표준 5패턴 ↔ 본 카탈로그 6패턴

본 카탈로그의 6패턴은 업계 표준 5패턴 워크플로우 분류(prompt chaining / routing / parallelization / evaluator-optimizer / orchestrator-workers)와 대부분 1:1로 매핑된다. Anthropic이 제안해 업계 표준으로 정착한 이 분류는 [Agentic RAG Survey, arXiv:2501.09136 §4](https://arxiv.org/abs/2501.09136)(2025-01, 인용한 §4는 v2 2025-02 추가)가 패턴별 적용 기준(When to Use)·세부 변형(parallelization의 sectioning/voting 포함)과 함께 동일하게 재정리한다. 사용자와 소통할 때 익숙한 쪽 이름을 쓰되, 매핑을 인지하고 필요하면 병기한다. 각 패턴의 적합 상황은 §2 해당 항목을 따른다.

| 표준 5패턴 | 본 카탈로그 6패턴 |
| --- | --- |
| prompt chaining | 파이프라인 |
| routing | 전문가 풀 |
| parallelization (sectioning/voting) | 팬아웃/팬인 |
| evaluator-optimizer | 생성-검증 |
| orchestrator-workers | 감독자 |
| *(없음)* | 계층적 위임 |

> 불일치: **계층적 위임**은 표준 5패턴에 없는 본 카탈로그 고유 패턴이다. 설계에 채택할 때는 "표준 외 고유 패턴"임을 사용자에게 표기하고, 감독자+전문가 풀 복합으로 대체 가능한지 한 번 더 검토한다.

## 3. 복합 패턴

실전에서는 단일 패턴이 드물고 다음 조합이 흔하다.

- **팬아웃 + 생성-검증** — 다각 분석 후 검증 게이트
- **파이프라인 + 팬아웃** — 순차 단계 안에서 일부 단계만 병렬
- **감독자 + 전문가 풀** — 감독자가 동적으로 풀에서 전문가 선택

복합 패턴은 기본적으로 **팀**으로 구성한다. 단, 특정 단계가 100% 독립이라면 그 단계만 서브로 분리해 비용을 줄인다(하이브리드).

## 4. 에이전트 분리 기준 4축

한 에이전트가 두 축 이상에서 모호하면 분리한다. 단, 분리 신호만 검사하면 에이전트 수가 늘기만 한다(에이전트 수는 곧 토큰비용이다) — 각 축의 **통합 신호를 역방향으로 함께 검사**하고, 통합 신호가 우세하면 한 에이전트로 합친다.

| 축 | 분리 신호 | 통합 신호 |
| -- | -------- | -------- |
| **전문성** | 요구되는 도메인/도구가 다름 (예: 코드 분석 vs 문서 작성) | 분리해도 매번 둘이 항상 같이 호출됨 |
| **병렬성** | 동시에 실행 가능하면 분리 → 처리 시간 단축 | 컨텍스트 공유가 필수라 사실상 직렬 |
| **컨텍스트 격리** | 한쪽이 다른 쪽의 컨텍스트를 보면 오염될 위험 (검토자가 작성 의도를 알면 객관성 상실) | 분리해도 핸드오프만 발생하고 격리 이득이 없음 |
| **재사용성** | 다른 하네스에서도 쓰일 가능성이 있으면 분리 | 재사용처가 가설일 뿐, 실제로 쓰는 다른 하네스가 없음 |

### 4-1 에이전트 능력 점검 — 분리(개수)와 직교하는 역량(내용) 축

§4의 4축이 '에이전트를 몇 개로 쪼갤지'를 정한다면, 이 절은 쪼갠 각 에이전트가 '어떤 역량을 켜야 하는지'를 정한다 — 둘은 직교한다. 책임 카드를 쓸 때 각 에이전트에 대해 아래 4가지 능력 중 *실제로 필요한 것만* 켠다. 켜지 않아도 되는 능력은 곧 토큰비용이다 — 만능 에이전트(§6)가 책임을 과잉으로 묶는 안티패턴이라면, 여기서 막는 건 *역량을 과잉으로 켜는* 안티패턴이다.

| 능력 | 켜는 신호 | 불필요하게 켜면 |
| --- | --- | --- |
| **계획(분해)** | 작업이 다단계이고 단계 간 의존이 강함 | 단순 단발 작업에 분해 단계를 넣어 호출 수만 늘림 |
| **추론(CoT/토론/반성)** | 산출 품질이 추론 깊이에 민감(애매한 판단·검증) | 정답이 자명한데 토론 루프를 돌려 비용 폭증 |
| **도구 사용** | 외부 상태·검색·API에 의존 | 내부 변환으로 충분한데 도구 호출 경로를 둠 |
| **메모리(회차/단계 간 상태)** | 재실행·후속 보완이 필요하거나 단계 간 누적 상태가 핵심 | 일회성 작업에 회차 상태를 유지해 복잡도만 키움 |

**Phase 2 산출에 반영:** 책임 카드마다 켠 능력을 1줄로 적고(예: "분석가 — 계획·추론, 도구·메모리 off"), 구조 비용 메모(SKILL.md Phase 2 산출)에서 "왜 이 능력 구성인가"를 함께 설명한다. 4축 분리가 '왜 이 인원인가'를 정당화한다면, 능력 점검은 '왜 이 인원이 이만큼만 무겁나'를 정당화한다.

> 주의: 이 점검은 단일 설계의 정적 체크다 — 능력 조합을 후보 여러 개로 탐색·채점하는 일(자동 탐색·Pareto)은 이 스킬의 범위가 아니다.

## 5. 패턴 선택 의사결정 트리

```
1. 작업이 순차 의존인가?
   └─ Yes → 파이프라인
   └─ No  → 2

2. 같은 입력에 다각 관점이 필요한가?
   └─ Yes → 팬아웃/팬인 (팀 모드)
   └─ No  → 3

3. 산출 품질에 별도 검증이 필요한가?
   └─ Yes → 생성-검증
   └─ No  → 4

4. 작업량이 예측 불가하고 동적 분배가 필요한가?
   └─ Yes → 감독자
   └─ No  → 5

5. 입력 종류가 다양하고 매번 한 전문가만 호출되는가?
   └─ Yes → 전문가 풀 (서브 가능)
   └─ No  → 단일 에이전트로 충분, 하네스 불필요할 수 있음
```

## 6. 안티 패턴

- **만능 에이전트** — 모든 책임을 한 정의에 몰아넣음. 컨텍스트가 서로 오염되고 재사용 불가.
- **에이전트 정의 없이 Agent 호출** — 프롬프트에 역할을 직접 박으면 재사용·진화·감사 불가.
- **비건 빌트인 호출** — `general-purpose`/`Explore`/`Plan`을 정의 파일 없이 부르고 끝. 다음 실행에서 동일 결과를 얻기 어렵다.
- **팀에서 리더만 통신** — 팀 모드인데 모든 메시지가 리더 경유면 사실상 서브 모드. 팀원 간 직접 통신을 활성화해야 협업 이점이 나온다.
- **컨텍스트 오염형 검증자** — 검증자가 생성자의 사고 과정을 그대로 받으면 같은 실수를 답습한다. 검증자에게는 산출물만 전달.
