## 1. 실행 모드 비교

| 항목 | 에이전트 팀 | 서브에이전트 |
| ---- | ----------------- | ------------------ |
| 통신 | 팀원 간 직접 (`SendMessage`) | 통신 불가, 결과만 반환 |
| 작업 관리 | 공유 작업 목록(`TaskCreate`) | 호출자가 수동 관리 |
| 비용 | 토큰 비용 큼 | 효율적 |
| 협업 품질 | 높음 (실시간 토론) | 제한적 |
| 적합 | 다각 분석/생성-검증 루프/감독자 | 독립 병렬/단발 작업 |

**선택 기준:** "에이전트 간 발견 공유와 실시간 토론이 결과 품질의 핵심인가?" → 그렇다면 팀, 아니면 서브에이전트.

복합/대형 워크플로우는 **기본적으로 팀**을 검토한 뒤, Phase별로 서브로 강등할 곳만 분리한다.

## 2. 6가지 아키텍처 패턴

### 2-1 파이프라인 (Pipeline)
- **구조:** A → B → C 순차 흐름
- **목적:** 단계별 의존이 강하고 후공정이 전공정 산출 없이는 무의미할 때
- **모드:** 통신 필요성이 낮으면 서브, 단계 간 미묘한 조정이 필요하면 팀
- **예:** 요구사항 분석 → 설계 → 구현 계획

### 2-2 팬아웃/팬인 (Fan-out / Fan-in)
- **구조:** 다수 병렬 → 통합
- **목적:** 다각 관점에서 동시에 분석한 뒤 결합
- **모드:** **팀 필수** — 팀원 간 발견 공유가 결과 품질을 결정
- **예:** 보안 / 성능 / 테스트 리뷰어가 동시에 코드 점검 후 종합

### 2-3 전문가 풀 (Expert Pool)
- **구조:** 라우팅으로 상황별 전문가 호출
- **목적:** 입력 종류가 다양하고 매번 같은 전문가만 필요한 경우
- **모드:** 서브에이전트 적합
- **예:** 입력 파일 종류(SQL/JSON/YAML)에 따라 다른 분석가 호출

### 2-4 생성-검증 (Generate / Validate)
- **구조:** 생성자 ↔ 검증자 쌍의 반복 루프
- **목적:** 품질 게이트가 필요한 산출
- **모드:** 팀이 유리(피드백을 직접 주고받음). 반복 횟수 상한을 둔다(예: 2회).
- **예:** 글쓰기 + 비평, 코드 + 리뷰

### 2-5 감독자 (Supervisor)
- **구조:** 중앙 감독이 동적으로 작업을 분배
- **목적:** 가변 작업량/우선순위
- **모드:** 팀 (`TaskCreate` 공유 큐)
- **예:** 마이그레이션 작업을 파일별 복잡도에 따라 재분배

### 2-6 계층적 위임 (Hierarchical Delegation)
- **구조:** 상위 → 하위 → 더 하위로 재귀 분해
- **목적:** 복잡 문제를 단계적으로 잘게 나눠야 할 때
- **모드:** 팀 + 서브 혼합

## 3. 복합 패턴

실전에서는 단일 패턴이 드물고 다음 조합이 흔하다.

- **팬아웃 + 생성-검증** — 다각 분석 후 검증 게이트
- **파이프라인 + 팬아웃** — 순차 단계 안에서 일부 단계만 병렬
- **감독자 + 전문가 풀** — 감독자가 동적으로 풀에서 전문가 선택

복합 패턴은 기본적으로 **팀**으로 구성한다. 단, 특정 단계가 100% 독립이라면 그 단계만 서브로 분리해 비용을 줄인다(하이브리드).

## 4. 에이전트 분리 기준 4축

한 에이전트가 두 축 이상에서 모호하면 분리한다.

| 축 | 분리 신호 |
| -- | -------- |
| **전문성** | 요구되는 도메인/도구가 다름 (예: 코드 분석 vs 문서 작성) |
| **병렬성** | 동시에 실행 가능하면 분리 → 처리 시간 단축 |
| **컨텍스트 격리** | 한쪽이 다른 쪽의 컨텍스트를 보면 오염될 위험 (검토자가 작성 의도를 알면 객관성 상실) |
| **재사용성** | 다른 하네스에서도 쓰일 가능성이 있으면 분리 |

**과도 분리 방지:** 분리해도 매번 둘이 동시에 호출된다면 한 에이전트로 통합한다.

## 5. 패턴 선택 의사결정 트리

```
1. 작업이 순차 의존인가?
   └─ Yes → 파이프라인
   └─ No  → 2

2. 같은 입력에 다각 관점이 필요한가?
   └─ Yes → 팬아웃/팬인 (팀 모드)
   └─ No  → 3

3. 산출 품질에 별도 검증이 필요한가?
   └─ Yes → 생성-검증
   └─ No  → 4

4. 작업량이 예측 불가하고 동적 분배가 필요한가?
   └─ Yes → 감독자
   └─ No  → 5

5. 입력 종류가 다양하고 매번 한 전문가만 호출되는가?
   └─ Yes → 전문가 풀 (서브 가능)
   └─ No  → 단일 에이전트로 충분, 하네스 불필요할 수 있음
```

## 6. 안티 패턴

- **만능 에이전트** — 모든 책임을 한 정의에 몰아넣음. 컨텍스트가 서로 오염되고 재사용 불가.
- **에이전트 정의 없이 Agent 호출** — 프롬프트에 역할을 직접 박으면 재사용·진화·감사 불가.
- **비건 빌트인 호출** — `general-purpose`/`Explore`/`Plan`을 정의 파일 없이 부르고 끝. 다음 실행에서 동일 결과를 얻기 어렵다.
- **팀에서 리더만 통신** — 팀 모드인데 모든 메시지가 리더 경유면 사실상 서브 모드. 팀원 간 직접 통신을 활성화해야 협업 이점이 나온다.
- **컨텍스트 오염형 검증자** — 검증자가 생성자의 사고 과정을 그대로 받으면 같은 실수를 답습한다. 검증자에게는 산출물만 전달.
