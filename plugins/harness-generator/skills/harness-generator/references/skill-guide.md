## 1. Description 작성 패턴

Description은 스킬의 유일한 트리거 메커니즘이다. Claude는 `available_skills` 목록에서 `name` + `description`만 보고 스킬 사용 여부를 결정한다.

### 트리거 메커니즘 이해

Claude는 자신의 기본 도구로 처리 가능한 단순 작업에는 스킬을 호출하지 않는 경향이 있다. 복잡·다단계·전문 작업일수록 트리거 확률이 높다. 단순 요청도 트리거되도록 하려면 description을 약간 pushy하게 작성한다.

### 작성 원칙

1. **무엇을 하는가** + **언제 트리거되는가**(구체적 상황)를 모두 기술
2. 유사하지만 트리거하면 안 되는 경계 사례 명시
3. 약간 "pushy" — Claude의 보수적 트리거 경향을 보상

### 좋은 예시

```yaml
description: "PDF 파일 읽기, 텍스트/테이블 추출, 병합, 분할, 회전, 워터마크,
  암호화/복호화, OCR 등 모든 PDF 작업을 수행. .pdf 파일을 언급하거나
  PDF 산출물을 요청하면 반드시 이 스킬을 사용. 단순 '읽기'가 아닌
  변환/편집/분석이 필요할 때 특히 유용."
```

```yaml
description: "엑셀/CSV/TSV 열 추가, 수식, 서식, 차트, 데이터 정제 등
  모든 스프레드시트 작업. 사용자가 스프레드시트 파일을 언급하면 — 
  '다운로드 폴더의 xlsx'처럼 캐주얼해도 — 이 스킬을 사용."
```

### 나쁜 예시

- `"데이터를 처리하는 스킬"` — 너무 모호
- `"PDF 관련 작업"` — 트리거 상황 미기술

### 후속 작업 키워드 (필수)

오케스트레이터/하네스 스킬의 description에는 다음 키워드를 자연스럽게 포함한다.

- "다시 실행", "재실행", "업데이트", "수정", "보완", "이전 결과 기반으로"

이 키워드들이 없으면 후속 요청이 같은 스킬로 라우팅되지 않을 수 있다.

## 2. 본문 작성 스타일

### Why-First

LLM은 이유를 이해하면 엣지 케이스에서도 옳게 판단한다.

**나쁜 예:**
```markdown
ALWAYS use library X. NEVER use library Y.
```

**좋은 예:**
```markdown
구조화 데이터 추출에는 라이브러리 X를 사용한다. Y는 평문 추출에 특화되어
행/열 구조를 보존하지 못하기 때문이다. X는 셀 경계를 인식해 구조를 유지한다.
```

### 일반화

피드백/테스트에서 문제가 보이면 좁은 수정 대신 원리 수준에서 일반화한다.

**오버피팅:**
```markdown
"Q4 매출" 열은 숫자로 변환하라.
```

**일반화:**
```markdown
열 이름에 "매출/금액/수량" 등 수치를 암시하는 키워드가 있으면 숫자 타입으로
변환한다. 변환 실패 시 원본을 유지한다.
```

### 명령형 어조

"~합니다" 대신 "~한다 / ~하라". 스킬은 지시서다.

### 컨텍스트 절약

모든 문장이 토큰 비용을 정당화하는지 자문한다.
- "Claude가 이미 알고 있는가?" → 삭제
- "이 설명이 없으면 실수하는가?" → 유지
- "예시 1개가 긴 설명보다 효과적인가?" → 예시로 대체

## 3. 출력 형식 정의 패턴

산출 형식이 중요한 스킬에서:

```markdown
## 보고서 구조
다음 템플릿을 그대로 따른다.

# [제목]
## 요약
## 핵심 발견
## 권장 사항
```

간결하게, 짧은 예시 1개를 함께 둔다.

## 4. 예시 작성 패턴

예시는 긴 설명보다 효과적이다.

```markdown
## 커밋 메시지 형식

**예시 1**
입력: JWT 기반 사용자 인증 추가
출력: feat(auth): JWT 기반 인증 구현

**예시 2**
입력: 비밀번호 표시 토글 버튼이 동작하지 않는 버그 수정
출력: fix(auth): 비밀번호 표시 토글 버튼 동작 수정
```

## 5. Progressive Disclosure 패턴

### 패턴 1: 도메인별 분리
```
data-skill/
├── SKILL.md (개요 + 도메인 선택 가이드)
└── references/
    ├── finance.md
    ├── sales.md
    └── product.md
```
사용자가 매출에 대해 물으면 finance.md만 로드.

### 패턴 2: 조건부 상세
```markdown
## 단순 편집
XML을 직접 수정한다.

**추적 변경이 필요하면**: [redlining.md](references/redlining.md) 참조
```

### 패턴 3: 대형 레퍼런스
300줄 이상의 reference는 상단에 목차를 둔다.

## 6. 스크립트 번들링 판단 기준

테스트 실행 트랜스크립트에서 다음 신호가 보이면 번들 대상.

| 신호 | 조치 |
| ---- | ---- |
| 테스트 3개 중 3개에서 동일 헬퍼를 즉석 작성 | `scripts/`에 번들링 |
| 매번 같은 의존성 설치 | 스킬에 의존성 명시 |
| 동일한 다단계 접근법 반복 | 본문에 표준 절차로 기술 |
| 같은 에러 후 같은 회피책 반복 | 본문에 알려진 문제와 해결법 |

번들 스크립트는 반드시 실행 테스트를 거친다.

## 7. 데이터 스키마 표준

평가/리포트가 필요한 도메인의 스킬은 다음 스키마를 그대로 사용한다(필드명 변형 금지).

### eval_metadata.json
```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "사용자 작업 프롬프트",
  "assertions": [
    "산출에 X가 포함된다",
    "Y 형식으로 파일이 생성된다"
  ]
}
```

### grading.json
```json
{
  "expectations": [
    { "text": "산출에 'A'가 포함됨", "passed": true, "evidence": "..." }
  ],
  "summary": { "passed": 2, "failed": 1, "total": 3, "pass_rate": 0.67 }
}
```
필드명: `text`, `passed`, `evidence` 정확히 사용 (`name`/`met`/`details` 금지).

### timing.json
```json
{ "total_tokens": 84852, "duration_ms": 23332, "total_duration_seconds": 23.3 }
```
서브에이전트 완료 알림 시점에만 접근 가능 → 즉시 저장한다.

## 8. 스킬에 포함하지 않을 것

- README.md / CHANGELOG.md / INSTALLATION_GUIDE.md 류
- 스킬 생성 과정의 메타 정보(테스트 결과, 반복 이력)
- 사용자 대상 설명서 — 스킬은 AI 에이전트를 위한 지시서
- Claude가 이미 아는 일반 지식
- 도메인 종속 예시(특정 프레임워크/언어 한정) — 일반화하거나 references로 분리
