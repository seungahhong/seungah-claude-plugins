# harness-generator

특정 도메인에 종속되지 않는 멀티 에이전트 하네스(에이전트 팀 + 스킬 + 오케스트레이터)를 **사람이 단계별로 직접 같이(수동·인터랙티브)** 설계·생성하는 Claude Code 메타 플러그인입니다. 매 Phase 산출물을 확인받으며 진행하므로, 새로운 도메인의 다단계 워크플로우(연구 → 설계 → 실행 → 검증)를 설계 결정까지 세밀하게 통제하며 만들고 싶을 때 사용합니다.

## 설치

```bash
claude plugin add seungahhong/seungah-claude-plugins
```

## 스킬

| Skill | Command | 설명 |
|-------|---------|------|
| Harness Generator | `/harness-generator` | 7단계 프로세스(Phase 0~7)로 도메인용 하네스를 한 묶음으로 설계·생성·검증·진화 |

## 언제 쓰나 — 그리고 언제 다른 도구를 쓰나

`harness-generator`는 **사람이 매 단계를 확인하며 1개 하네스를 정밀하게 만드는 수동 생성**에 특화되어 있습니다.

| 원하는 것 | 쓸 도구 |
|-----------|---------|
| 사람이 단계별로 **직접 같이** 하네스 만들기 (매 Phase 확인) | **`harness-generator`** (본 플러그인) |
| 후보 N개를 자동 탐색·Pareto 채점해 **최적 하네스를 뽑아** 만들기 | (자동 탐색·Pareto 도메인 — 본 플러그인 범위 밖) |
| **이미 있는** 하네스의 trace 기반 결함 진단·개선 | (trace 진단 도메인 — 본 플러그인 범위 밖) |
| **단일 스킬** 하나 작성·벤치마크 | `skill-creator` |

## 사용법

### 기본 호출

만들고 싶은 도메인/워크플로우를 함께 설명하며 호출합니다.

```
/harness-generator
```

예시 입력:

> "PR 리뷰부터 배포까지 자동화하는 하네스를 만들어줘. 리뷰 → 승인 → 배포 → 검증 단계로."

또는 자연어로: "하네스 만들어줘", "오케스트레이터 작성해줘", "에이전트 팀 구성해줘", "이 도메인용 스킬 묶음 생성해줘".

### 진행 방식 — 인터랙티브

- **매 Phase 종료 시** 핵심 산출물을 보여주고 다음 Phase 진행을 명시적으로 확인받습니다.
- 요구가 모호하면 **한 번에 한 질문씩** 명확화합니다(목록 나열 없음).
- 어떤 Phase든 "건너뛰기"를 요청하면 그대로 SKIP하고 결정 이력에 남깁니다.

### 생성 과정 — 7단계 프로세스(Phase 0~7)

| Phase | 단계 | 하는 일 |
|-------|------|---------|
| 0 | 현황 감사 | 신규 도메인 / 기존 하네스 확장 / 단순 보완을 판별해 분기 |
| 1 | 도메인 분석 | 입력·산출물·필요한 단계를 정의 |
| 2 | 아키텍처 설계 | 실행 모드(팀 / 서브에이전트 / 하이브리드) + 아키텍처 패턴(파이프라인 등) 선택, 에이전트 분리 기준 4축 적용 |
| 3 | 에이전트 정의 작성 | 역할·원칙·입출력·에러 처리·협업 프로토콜을 갖춘 `agents/{name}.md` 생성 |
| 4 | 스킬 작성 | 단계별 `skills/{step}/SKILL.md` + 필요 시 출력 데이터 스키마 |
| 5 | 오케스트레이션 | 흐름을 통제하는 진입점 오케스트레이터 작성 (`_workspace/` 컨텍스트 관리 + CLAUDE.md 포인터 등록) |
| 6 | 검증 | 구조 검증·모드별 점검·스킬 실행 테스트·트리거 should/should-not 검증·dry-run·테스트 시나리오 문서화 |
| 7 | 진화 | 실행 후 피드백 매핑·변경 이력 기록·운영/유지보수(드리프트 감사) |

### 재실행·보완

이미 만든 하네스를 다시 호출하면 Phase 0에서 기존 산출물을 감지해 **확장/부분 재실행/단순 보완**으로 분기합니다. "이 하네스에 검증 단계 추가해줘"처럼 요청하면 됩니다.

## 산출물

대상 위치에 아래 구조로 하네스 한 묶음이 생성됩니다.

```
{harness-root}/
  agents/{agent}.md            # 에이전트 정의 (모두 model: "opus")
  skills/{step}/SKILL.md       # 단계별 스킬
  (오케스트레이터 스킬)         # 전체 흐름 진입점
  CLAUDE.md                    # 하네스 포인터 + 변경 이력
```

## 트리거 평가 (evals)

`evals/trigger-eval.json`에 발동해야 할 발화(should_trigger)와 발동하면 안 될 발화가 들어 있습니다. should-not 케이스는 자동 탐색·Pareto 도메인·trace 진단 도메인·`skill-creator`(단일 스킬) 영역의 near-miss 발화를 중심으로 구성되어, cross-plugin 동시발동을 막는 **reciprocal 가드** 역할을 합니다(일반 비발동 케이스도 일부 포함). 스킬 description이나 트리거 표현을 바꾸면 이 파일을 함께 갱신하세요.
