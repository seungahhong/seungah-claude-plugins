# Improvement Playbook — 쉬운 것부터 안전하게 (Step 2)

개선 순서·메커니즘·안전장치의 정본. 원칙: **쉽고 근거 강한 것 먼저, 어렵고 근거 약한 구조는 뒤로·opt-in.** 사용자 지침("무거운 작업은 추후에, 쉬운 작업부터 단계별 승인")을 구조로 강제한다.

### Tier C — AI-맥락 신호 개선 (opt-in · report-only · 총점 무관)

프론트엔드/테스트 저장소에서만 의미 있는 **추가 개선 축**이다(사용자 요청: 시맨틱 태그·접근성·테스트 설명). **총점에 반영되지 않는 report-only 신호**라 A/B 기본 흐름과 별도로, 원할 때만 제안한다. 근거상 셋 다 개입 효과가 입증되지 않아 **삭제/교정·명백한 결함만** 다룬다.

## 섹션별 상세

### 1. A3 주석 — 삭제 우선 + '왜/맥락' 주석 opt-in 추가
**(a) 삭제(기본·위험 ~0)**: 코드와 모순되는(오도) 주석, 오래된(stale) 주석, 커멘트아웃된 죽은 코드, 의미 없는 noise 주석. 오도 주석 삭제만 위험이 실증적으로 0에 가깝다(코드 동작 불변).

**(b) 추가(opt-in·필수 아님·점수 무관)**: **코드만으로 문맥('왜')을 알기 어려운 곳**에 설명 주석을 넣는다. 스코어러가 report-only로 내는 **comment-gap 후보**를 seed로 쓴다:
- **침묵 예외 처리**(`except: pass`·빈 `catch{}`) — 왜 삼키는가(무시해도 되는 이유·복구 전략).
- **매직 넘버**(3자리+/소수) — 단위·의미·출처(예: `86400  # 초/일`, `0.0075  # 결제 수수료율`).
- **정규식** — 무엇을 매칭/치환하는지 + 입력 예시.
- **비자명 우회/워크어라운드** — 왜 이 방식인지(외부 버그·API 제약·성능 이유).

원칙(정직성):
- **what이 아니라 why**. 코드가 이미 말하는 것을 반복하는 주석(redundant what)은 넣지 않는다(이득 없음·drift 위험).
- **사람이 정확성을 검증한다.** 자동 생성 주석은 상당 비율 부정확하므로 사실로 단정해 넣지 않는다 — **오도 주석은 실측 해악**(LLM 이해를 측정 가능하게 저해). 정확성 게이트가 이 개입의 핵심.
- **이득의 정직한 프레이밍**: 올바른 주석의 *LLM 이해* 상승은 개입 근거가 약하다(통제 연구상 유의하지 않음). 그러나 (1) **사람** 이해를 돕고 (2) 코드로는 **파생 불가한 맥락**(의도·외부 제약·우회 이유)을 보충한다 — 이 두 가지가 추가의 근거다. "주석 추가로 AI 성공률 +N%"는 약속하지 않는다.
- **볼륨은 report-only**: 주석을 많이 넣는다고 점수가 오르지 않는다(가점 없음·Goodhart 방지).
- 검증된 계약(입력/출력/불변식) 주석도 이 경로로.
- 행위 무변경(주석은 코드 동작을 바꾸지 않음) → `behavior-guard`는 diff/정확성 리뷰만.

### 2·3. A1·A2 명명 — AST/LSP 위임
- **수동 문자열 치환 금지.** IDE/LSP의 rename symbol 또는 AST 변환에 위임한다(스코프 인식·오버라이드 추적).
- **런타임 파괴 주의**: 동적 접근(`getattr`·`obj[name]`)·직렬화 키(JSON/DB 컬럼)·리플렉션·공개 API·파일 기반 라우트·i18n 키는 리네임이 조용히 깨뜨린다 → `behavior-guard`에 격리 센서 요청.
- 명백히 오도하는 이름 교정만 개선 효과를 어느 정도 자신 있게 말할 수 있다 — LLM 대상 식별자 리네임 효과는 부호 불안정·과제 의존이고 *개입 연구가 없어* "성공률 +N%" 약속은 금지.

### 4. B3 중복 — 추출은 항상 옳지 않다
- Type-1/2 정확 중복만 대상. **3곳+·같은 이유로 함께 변하는 것**만 추출 후보.
- "잘못된 추상화보다 중복이 쌀 수 있다" — 우연한 중복(같아 보이지만 다른 이유로 변함)은 그대로 둔다.

### 5~8. 구조 리팩터 — opt-in·고위험
- **행위 센서로 baseline 대조.** 테스트가 있으면 재실행, 없으면 실제 실행·렌더.
- **"테스트 통과 ≠ 동작 보존."** green은 등가성 증명이 아니다(약한 오라클).
- 구조 리팩터 효과는 대체로 **추론**(직접 측정 근거 약함) → 점수 상승을 품질 인증으로 오독 금지.

### C-a·C-b·C-c. C1 시맨틱 마크업 & a11y — native 우선·자동 ARIA 금지
- **삭제/교정 우선**: native 요소로 대체 가능한 `role=`(`role="button"` on `<div>` 등)는 native `<button>/<nav>/<label>`로 바꾸는 게 정석('First rule of ARIA'). role로 native 시맨틱을 덮어쓰면 오히려 해로울 수 있다.
- **`<div/span onClick>` → `<button>` 승격**은 키보드·포커스·역할을 자동 획득하지만 **레이아웃/스타일·이벤트 전파를 바꿀 수 있다** → opt-in·행위 센서(실제 렌더·상호작용) 관측.
- **자동 ARIA/alt 생성 금지**: 기계·LLM 생성 ARIA/alt는 confident-but-wrong 사례가 많고(overlay 연구·WebAIM), 잘못된 ARIA는 없느니만 못하다. `alt`·`aria-label`은 **사람이 의미를 확인**해 추가한다.
- **ARIA/role 볼륨을 늘려 점수를 올리려는 시도 금지**(애초에 총점 무관·report-only). 이득은 모델 역량 조건부라 "N% 개선" 약속 금지.

### C-a·C-d. C2 테스트 설명 & 셀렉터 — 삭제 우선·자동 어서션 금지
- **오도/모호 제목 삭제·리네임**: `it('works')`·`test1` 같은 저정보 제목을 *대상·동작·기대결과*를 담은 이름으로. **오도 제목(본문과 모순)은 삭제/교정** — 통제 연구상 올바른 설명의 이해 이득은 미미하고 오도 설명만 측정 가능하게 해롭다(비대칭).
- **자동 어서션/오라클 생성 금지**: 테스트를 LLM 최적화 오라클로 쓰면 구현을 굳혀 버그를 초록으로 은폐한다(overfitting·green-locks-bug). LLM 생성 오라클은 *기대*가 아니라 *현재 구현*을 인코딩하는 경향.
- **testid/CSS/xpath → getByRole 이관**은 refactor-robust하지만 접근성 이름 계산에 의존해 깨질 수 있고 **getByRole가 green이어도 a11y가 깨질 수 있다**(axe 대체 아님) → opt-in·행위 센서.

## 안전·결정 권한 불변식 (요약)

1. **위험은 경고, 수용은 사용자.** opt-in·개별 승인된 고위험 변경을 회귀 위험을 이유로 조용히 보류·강등하지 않는다.
2. **자동 테스트 부재 = 행위 센서 승격 사유**(거부 아님). 모든 센서 불가 시에만 사용자에게 escalate.
3. **UNVERIFIED는 관측 한계 라벨**이지 거부 신호 아님 — red(실제 파괴) 센서만 롤백.
4. **한 번에 한 섹션·한 승인.** 묶어 적용 금지.
5. **커밋하지 않는다.** git 워크플로/사용자의 몫.
6. **인도 정합**: "리팩터" 요청을 린트 설정·툴링으로 대체 충족하지 않는다. 인도분이 요청 클래스와 다르면 정직히 보고.

## generator ≠ checker

개선안을 제안·적용하는 `staged-refiner`와 행위를 검증하는 `behavior-guard`는 **분리**된다. 만든 사람이 자기 것을 통과시키는 편향을 막는다.
