# design-principle-harness

> 저장소를 **두 큰 틀**로 다뤄 AI 코딩 에이전트가 **읽고 안전하게 기여할 수 있는** 코드베이스로 만드는 인터랙티브 멀티 에이전트 하네스 — **Track A 코드 품질**(명명·주석·설계 원칙·시맨틱/테스트)과 **Track B AI 접근성**(에이전트가 코드를 이해·안전하게 수정하기 쉬운가).

## 두 트랙 (Phase 0에서 선택)

**Phase 0**에서 원하는 트랙(A / B / 둘 다)을 고르고, 각 틀 안에서 **다시 단계별로 문의 후 선택한 방향으로만** 승인 개선한다(둘 다면 A→B 순).

- **Track A 코드 품질** — `score.py`로 채점 → 쉬운 것부터 섹션별 승인 개선 → 종합.
- **Track B AI 접근성** — `ai_access.py`로 6지표 측정 → 지표별 승인 개선 → 종합.

---

# Track A — 코드 품질

임의 저장소를 결정론 스코어러로 채점하고, 그 점수표가 매긴 **개선 순서(쉬운 것부터)**대로 섹션마다 사람 승인을 받아 하나씩 개선한 뒤, 재측정 델타와 함께 종합 결과를 낸다.

### Step A1 — 점수화 (두 층위)

`score.py`(stdlib only, Python 3.10+)가 100점을 두 층위로 나눠 채점한다:

- **Tier A 표면 가독성** /60 — 변수·함수·클래스/컴포넌트 **명명 명료성**, 명명 **규약 일관성**, **주석 건강도**.
- **Tier B 설계 원칙** /40 — **복잡도(KISS)**, **응집·결합(SRP)**, **중복(DRY)**, **순환 의존**, **과대추상(DIP·DI/IoC·YAGNI)**.
- **Tier C AI-맥락 신호** (report-only · **총점 100에 미포함**) — **C1 Semantic Markup & a11y**(시맨틱 HTML 태그·ARIA/WCAG), **C2 Test Legibility**(테스트 설명·getByRole 접근성 셀렉터). 프론트엔드/테스트 저장소에만 적용. 사용자 요청으로 측정·개선하되, 세 신호 모두 직접 개입 근거가 없어(→ 정직성) **점수로 합산하지 않고 개선 후보 census로만** 낸다.

각 섹션은 점수와 함께 **confidence(strong/medium/weak/report-only)**, 근거 대상(human/LLM/web-agent/both), 개선 난이도, caveat를 단다.

```bash
python3 skills/design-principle-harness/scripts/score.py <repo> --json scorecard.json
```

### Step A2 — 쉬운 것부터 섹션별 개선

점수표의 개선 순서(**주석 정리 → 명명 리네임 → 중복 제거 → (opt-in) 구조 리팩터**)대로, 섹션마다 **계획 승인 → 개별 적용 → 행위 검증**을 거친다. SOLID·복잡도·순환 같은 **무거운 구조 작업은 뒤로 미뤄 opt-in**으로만 건드린다.

**주석은 삭제만이 아니라 추가도 한다**: 오도·stale·죽은 코드 주석은 삭제(기본·위험 ~0)하고, **코드만으로 문맥('왜')을 알기 어려운 곳**(침묵 예외 처리·매직 넘버·정규식·비자명 우회 등 스코어러가 내는 *comment-gap 후보*)에는 **'왜/맥락' 설명 주석을 opt-in으로 추가**한다(필수 아님). what이 아니라 why(의도·외부 제약·우회 이유)를 담고, **사람이 정확성을 검증**한다(자동 생성 주석을 사실로 단정하지 않음 — 오도 주석은 실측 해악). 주석을 많이 단다고 점수가 오르지는 않는다(볼륨은 report-only).

### Step A3 — 종합 결과

`score.py`를 재실행해 before→after 델타를 내고, 무엇을 바꿨는지·무엇을 미뤘는지·잔여 위험을 종합 리포트로 정리한다.

## 왜 두 층위인가 (정직성)

이 하네스는 사용자 요청대로 설계 원칙을 **점수화하지만**, 점수를 **인증이나 최적화 목표가 아니라 개선 우선순위용 진단 지표**로 다룬다. 근거:

- **명명·주석은 사람 이해 근거가 견고**하고, 오도/불일치/stale한 이름·주석은 LLM에도 해롭다. → Tier A 가중을 높였다. 단 **LLM 대상 식별자 리네임의 효과는 부호가 불안정·과제 의존적**(요약 과제엔 크게, 알고리즘 코드엔 거의 0)이라 하나의 수치로 못 쓴다.
- **고전 구조 지표(cyclomatic·LCOM·Maintainability Index)는 약한 프록시** — 코드 길이를 통제하면 LLM 성능과 무상관이고 전부 사람 대상 검증(=AI 가독성 외삽)이다. → Tier B는 낮은 confidence·진단.
- **설계 원칙 준수를 점수·게이트로 만들면 Goodhart/reward-hacking** — 정적 스멜을 오라클로 쓰면 표면 변형으로 게임된다. → 총점을 목표로 최적화하지 않는다.
- **Tier C(시맨틱 마크업·a11y·테스트 설명)는 report-only·총점 미포함** — 시맨틱 HTML 이득은 **모델 역량 조건부**(강한 모델=full HTML 유리·약한 모델=a11y-tree 유리, WorkArena L1)이고 landmark/button-vs-div 격리 연구가 없다; **ARIA를 더 쓰면 오히려 오류와 상관**(WebAIM·상관≠인과)이라 native 우선·자동 ARIA 금지; **올바른 테스트 설명의 LLM 이해 이득은 미미하고 오도 설명만 해로우며**, 테스트를 LLM 오라클로 쓰면 버그를 굳힌다(overfitting). → 존재/볼륨 가점 없이 오도·모호·native-대체가능만 개선, 자동 ARIA/alt/어서션 생성 금지. [근거: references/research/semantic-a11y-test-dossier.md]

사용자의 "쉬운 것부터, 무거운 SOLID는 추후에"라는 순서가 이 근거와 정확히 맞물린다.

## 안전장치

- 리네임은 **AST/LSP 위임**(수동 문자열 치환 금지) — 동적 참조·직렬화·공개 API 파괴 방지.
- 오도·죽은 코드 주석은 **삭제 우선**(위험 ~0). 자동 주석 생성은 부정확할 수 있어 검증된 계약만 추가.
- 구조 리팩터는 **행위 센서 관측·opt-in** — **"테스트 통과 ≠ 동작 보존."**
- **위험은 경고, 수용은 사용자.** 승인된 고위험 변경을 조용히 보류하지 않는다.
- **측정 단계는 코드 수정 없음**, 개선 단계는 승인 게이트 뒤에만 수정하고 **커밋하지 않는다**.

# Track B — AI 접근성

**AI 코딩 에이전트가 코드베이스를 이해하고 안전하게 수정하기 쉬운가**를 6지표로 측정·개선한다. `ai_access.py`(stdlib only, 독립)가 결정론 assessment를 내고, 지표별로 문의 후 승인 개선한다.

### 6지표

| 지표 | 유형 | 무엇을 보나 |
|------|------|------------|
| **M1 의존성 방향 강제** | **SCORED** | dependency-cruiser·eslint-boundaries·import-linter·ArchUnit·Nx·TS refs 설정 존재 + CI에서 금지 import가 실패하는가 |
| **M2 독립 실행 가능성** | **SCORED** | 부분을 전체 없이 build/test 가능한가 + **독립 ground-truth oracle**(에이전트 저작 테스트와 구별) |
| **M3 빌드 피드백 품질** | report-only | 잘못된 코드에 빌드가 명확한 에러(strict 타입·typecheck·CI) |
| **M4 모듈 경계 예측 가능성** | report-only | 에이전트가 "어디에 코드를 놓을지" 추론할 관례적 구조 |
| **M5 패턴 일관성** | report-only | 같은 문제를 같은 방식으로(모듈 시스템 등; 명명은 Track A) |
| **M6 에이전트 가이드** | report-only | 빌드가 못 잡는 설계 의도가 CLAUDE.md/AGENTS.md에 문서화 |

### 왜 SCORED는 M1·M2뿐인가 (정직성)

- **build enforces, docs explain** — 기계 검증 가능 속성(강제된 의존 방향·실행 테스트)만 신뢰할 사실. 나머지 지표의 '에이전트 이득'은 **측정된 개입이 아니라 인간-SE 이득+첫 원리로부터의 추론**이다(저장소 구조를 통제 변경해 같은 에이전트 성공을 잰 연구 없음).
- **에이전트 가이드 존재 ≠ 성능** — context 파일이 성공을 안 높이고 LLM-생성본은 5/8에서 낮췄다(arXiv:2602.11988) → **자동 생성 금지**, 사람이 non-inferable 내용만.
- **tool-index ≠ code-structure** — LocAgent(92.7%)·RepoGraph(+32.8%) localization 이득은 툴 그래프 인덱스의 것이지 저장소 layout의 공로가 아니다.
- **유일하게 측정된 레버 = 독립 oracle** — 에이전트 저작 테스트의 격리-green이 실제 정답이 아닌 경우가 실패의 81~100%(arXiv:2606.26978) → '격리-green=correct' credit 금지.
- 개선(가드레일·가이드)은 승인 게이트 뒤에만·**커밋하지 않는다**. "가드레일 추가로 AI 성공률 +N%"는 말하지 않는다.

## 사용 예

```
[Track A] "이 저장소 설계 품질 점수 매기고 주석·이름부터 정리해줘"
[Track A] "SOLID·응집/결합·복잡도·중복 진단하고 쉬운 것부터 승인받아 고쳐줘, 구조는 나중에"
[Track B] "AI 에이전트가 이 코드베이스 이해·수정하기 쉬운지 측정하고 승인받아 개선해줘"
[Track B] "의존성 방향 가드레일·CLAUDE.md 가이드 점검하고 정비해줘"
```

## 경계

- 이 하네스는 **두 축**을 다룬다 — **Track A 코드 본문 설계 품질**(명명·주석·설계 원칙·시맨틱/테스트)과 **Track B AI 접근성**(의존 방향 강제·모듈 경계·빌드 피드백·독립 실행·에이전트 가이드 등 저장소 구조·강제 층위). 사용자가 선택한 트랙만 수행한다.
- 완성 코드 리뷰·PR·커밋, 상류 산출물(기획/디자인/API 계약/QA) 핸드오프 게이트, 하네스 자체 진단·생성, 한 기능 구현·실행검증은 범위 밖.
- **다른 마켓플레이스 플러그인을 참조·의존하지 않는다**(score.py·ai_access.py·루브릭·근거 dossier 모두 내재화·단독 설치로 동작).
