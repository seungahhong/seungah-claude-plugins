# QA Agent Harness — 리스크→오라클→실행→트리아지 오케스트레이터

소프트웨어 QA를 **에이전틱 라이프사이클**로 다룬다 — 커버리지를 채우는 게 아니라 *리스크 노출(발생확률 × 영향)을
커버리지 우선순위로 변환*하고, 각 시나리오에 *진짜 판정 오라클*을 명시하고, 테스트를 생성·실행하되 *실패를 맹목 재시도하지
않고* 결함/플래키/환경으로 먼저 분류한다. 전 과정은 *행동 전 제안·HITL 게이트·설명 가능한 로깅된 결정*으로 관통된다.

핵심 메시지: **"커버리지 극대화나 그린 신호 자체가 reward-hacking 가능한 대리지표다 — 오라클이 약하면 연기만 나고 경보는
울리지 않는다. 테스트의 가치는 실행 경로 수가 아니라 어서션(오라클)의 강도에 있다."**

> 이 하네스는 *QA 특화 파이프라인*(오라클·자가치유·플래키 클러스터링)이다. *검증 가능한 목표로의 도메인 무관 자율 반복
> 루프*(정지조건까지 반복·교차세션 학습)나 *AI 출력을 오프라인 judge로 채점·측정*하는 것이 아니다 — 그건 자율 반복·평가
> 도메인이다(아래 경계).

## 무엇을 하는가 (네 단계)

1. 요구사항·변경·리스크를 *리스크 노출 등급·커버리지 우선순위*로 정규화하는가? (Phase 0 — Risk-Driven Test Strategy)
2. 스토리/API를 *리스크 매핑 시나리오 + 명시적 오라클*을 갖춘 살아있는 라이브러리로 설계하는가? (Phase 1 — Scenario Design)
3. *오라클 우선* 테스트를 생성·실행하고 드리프트를 *점수화·로깅·게이트된 자가치유*로 다루는가? (Phase 2 — Generation + Self-Healing Execution)
4. 실패를 *결함/플래키/환경으로 분류*하고 공유 근본원인 클러스터링·변경 기반 재실행 우선순위화하는가? (Phase 3 — Intelligent Failure Triage)

## 경계 (먼저 읽고 발동 여부를 판단하라)

이 하네스는 **'리스크 기반 전략→오라클 명시 시나리오→오라클 우선 생성·게이트된 자가치유 실행→결함 vs 플래키 지능형
트리아지의 독립 end-to-end 에이전틱 QA 라이프사이클'** 에 특화한다. 다음은 명시적으로 범위 밖이다.

- **넓은 프론트엔드 개발 워크플로우 안의 FE 한정 QA 단계** — 접근성/Lighthouse/특정 페이지 감사처럼 *FE 개발 흐름 내부의
  한정 단계*는 FE 개발 도메인이다. 이 하네스는 *도메인 무관 QA 라이프사이클*로 자가치유·실패 트리아지·리스크 전략을 1급으로 다룬다.
- **구현 흐름 내부의 좁은 test-generator 스킬** — *특정 기능을 구현하며 유닛 테스트 하나 붙이는 것*은 백엔드 구현 도메인이다.
  이 하네스는 기능 구현이 아니라 *전략→설계→실행→트리아지의 독립 end-to-end QA 방법론*(FE+API 무관)이다.
- **AI 출력을 오프라인 LLM-as-a-Judge로 채점·측정** — 산출물을 *루브릭으로 채점하고 benchmark validity를 감사*하는 것은
  평가 도메인이다. 이 하네스는 *실제 소프트웨어 테스트 스위트를 런타임에 생성·실행·자가치유·트리아지*한다(측정이 아니라 실행·수리).
- **라이브 텔레메트리로 프로덕션 인시던트 RCA** — traces/logs/metrics로 *운영 장애 탐지·국소화·완화*는 운영 도메인이다.
  이 하네스는 *프리프로덕션 테스트 스위트 실패*를 결함 vs 플래키 vs 환경으로 트리아지한다.
- **코딩 착수 전 상류 인수조건↔테스트 커버리지 핸드오프 게이트 검수** — 코드 착수 *전* 상류 산출물을 게이트로 검수하는 것은
  핸드오프 리뷰 도메인이다. 이 하네스는 *코드가 존재한 후* 테스트를 실행·트리아지한다.
- **검증 가능한 목표로의 도메인 무관 자율 반복 루프** — 정지조건까지 반복하고 교훈을 메모리에 distill하는 것은 자율 반복
  도메인이다. 정지조건 개념은 재사용 가능하나 이 하네스는 *QA 특화*(오라클·자가치유·플래키 클러스터링) 파이프라인이다.
- **여러 AI 에이전트로 병렬화할지·토폴로지 결정** — single/multi·토폴로지 결정은 멀티 에이전트 오케스트레이션 도메인이고,
  **사람↔에이전트 분업·감독 설계**는 사람-에이전트 협업 도메인이다. 이 하네스는 *고정 4단계 QA 방법론*이며 QA 행동(치유·트리아지·파괴적 조작)에 특화된 점진적 자율성·HITL 게이트를 구체적으로 내장한다.
- **LLM 입출력 안전 가드레일(주입·PII·독성)** — 각 해당 도메인이다.

경계가 모호하면 한 질문으로 확인한다 — "*리스크→오라클→실행→트리아지의 에이전틱 QA 라이프사이클로 테스트를 만들고
실행·자가치유·분류*하려는 건가요, 아니면 *다른 것*(FE 한정 감사·기능 구현+테스트·산출물 채점·프로덕션 장애 RCA·상류
핸드오프 검수·도메인 무관 자율 반복·에이전트 병렬화·사람 분업·입출력 가드레일)인가요?"

## 내재화 원칙 (모든 Phase가 따른다)

- **리스크 노출이 노력을 배분한다** — 균일 커버리지가 아니라 발생확률×영향으로 테스트 깊이·순서를 정한다(references §1).
- **오라클 우선** — 테스트 가치는 어서션의 강도다. 커버리지는 결함 탐지력과 약하게만 상관하므로 스모크/동어반복을 금지하고 시나리오마다 기대 상태·불변식을 명시한다(references §2).
- **오라클 독립 검증** — 할루시네이션된 어서션은 모델 간 비일관적이라 다중 표본 합의(panel/consensus)로 거른다(references §3; CANDOR).
- **자가치유는 점수화·로깅·게이트** — role→data-testid→접근성 트리→가시 텍스트 우선순위로 후보를 비교하고, 치유가 실제 기능 변경/회귀를 초록으로 은폐하지 않게 로깅·검토한다(references §4).
- **재시도 전 트리아지** — 실패를 결함 vs 플래키 vs 환경으로 먼저 분류한다. 플래키는 공유 근본원인으로 클러스터링해 하나의 수정으로 다수를 해소한다(references §5).
- **변경 기반 우선순위화** — 회귀 전량 재실행이 아니라 코드 변경 영향+리스크로 테스트를 선택·정렬한다(references §6).
- **점진적 자율성 + HITL 게이트** — 행동 전 제안(suggest-before-act), 파괴적/고위험 행동은 사람 승인 게이트를 통과시킨다(references §7).
- **설명 가능한 로깅된 결정** — 전략 우선순위·치유 선택·트리아지 판정을 근거와 함께 기록해 감사·재현 가능하게 한다(references §8).
- **커버리지 연극·리워드 해킹 가드** — 그린 신호는 대리지표다. 약한 어서션·회귀 은폐 치유·그린까지 재시도·현재 구현 오버피팅 오라클을 위반으로 잡는다(references §9).
- **정직성·falsifiability** — S06 조직 도입·성숙도 5단계·변경관리 층은 제외, 회사별 수치·CANDOR 0.834→0.971은 특정 세팅 관찰값, "개선 N% 보장" 금지(references §10).
- **관찰 가능성·승인** — Phase 0 직후 승인 게이트는 항상. 각 Phase는 1줄로 보고하고, 요청되지 않은 사이드 에이전트나 중복 실행을 만들지 않는다(references §11).

# 인터랙티브 플로우

## Phase 0 — 리스크 기반 테스트 전략 (Risk-Driven Test Strategy) · 승인 게이트

`test-architect`를 호출해 요구사항·변경·리스크를 *리스크 노출 등급·커버리지 우선순위*로 정규화한다.

```
Agent(
  subagent_type="test-architect", model="opus",
  prompt="""
  [역할] 요구사항·최근 변경·비즈니스/기술 리스크를 리스크 노출 등급과 커버리지 우선순위를 담은 테스트 전략으로 변환한다.
  [입력] QA 대상 맥락: {사용자 발화} / (선택) 요구사항·스토리·최근 변경(diff/PR)·기존 결함 이력·시스템 경계.
  [규칙] 각 기능/영역의 리스크를 발생확률 × 영향으로 등급화하고(리스크 노출), 균일 커버리지가 아니라 리스크 노출도가
         테스트 깊이·순서를 배분하게 한다. 무엇을 어느 깊이(스모크/기능/경계/부하 등)로 테스트할지와 각 영역의 판정 기준
         (무엇이 성공/실패인가의 뼈대)을 적는다. 리스크 근거(변경 영향·과거 결함·외부 의존)를 설명 가능하게 로깅한다.
         테스트를 직접 생성·실행하지 않는다(전략만). 커버리지 수치를 목표로 삼지 않는다(대리지표 경고). 관찰 가능한 근거로만
         적고 리스크 등급을 발명하지 않는다(모르면 '미상'·가정 명시). 조직 도입·성숙도 모델 같은 비하네스 층은 다루지 않는다.
  [출력] Test Strategy(영역별 리스크 노출 등급·커버리지 우선순위·테스트 깊이·판정 기준·미상/가정).
  """
)
```

Test Strategy를 사용자에게 보여주고 **승인 게이트**:

`[Phase 0] 리스크 기반 테스트 전략 완료 — 다음: 시나리오 설계. 진행할까요?`

승인 전에는 다음 단계를 시작하지 않는다(잘못된 리스크 배분으로 잘못된 시나리오·테스트를 만들지 않기 위함).

## Phase 1 — 시나리오 설계·살아있는 라이브러리 (Scenario Design)

`scenario-designer`를 호출해 스토리/API를 *리스크 매핑 시나리오 + 명시적 오라클*로 설계한다.

```
Agent(
  subagent_type="scenario-designer", model="opus",
  prompt="""
  [역할] 스토리/API/도메인 규칙을 리스크에 매핑된 시나리오로 번역하되 각 시나리오에 명시적 판정 오라클을 부착한다.
  [입력] Test Strategy: {Phase 0 산출} / (선택) 사용자 스토리·API 계약·도메인 규칙·기대 동작.
  [규칙] Phase 0의 리스크 우선순위를 따라 시나리오를 설계하고, 각 시나리오에 '무엇이 성공/실패인가'의 명시적 오라클
         (기대 상태·불변식)을 붙인다 — 스모크/동어반복 오라클(예: 200 떴으니 통과, 요소 존재만 확인)을 금지한다.
         시나리오를 요구사항·리스크로 추적 가능하게 태깅하고(traceability), 제품과 함께 진화하는 살아있는 라이브러리로
         유지한다(정적 회귀팩 지양). 현재 구현 동작에 오버피팅한 오라클(명세가 아니라 지금 코드가 하는 대로)을 만들지 않는다.
         테스트 코드를 생성·실행하지 않는다(설계만). 오라클이 약하거나 정의 불가한 영역은 '오라클 격차'로 표시한다.
  [출력] Scenario Library(리스크 매핑 시나리오 목록·시나리오별 명시적 오라클·추적 태그·오라클 격차·미상/가정).
  """
)
```

1줄 보고: `[Phase 1] 시나리오 설계 완료(시나리오 {N}·오라클 {M}) — 다음: 오라클 우선 생성·자가치유 실행. 진행할까요?`

## Phase 2 — 오라클 우선 생성 + 자가치유 실행 (Generation + Self-Healing Execution)

`test-engineer`를 호출해 *오라클 우선* 테스트를 생성·실행하고 드리프트를 *점수화·로깅·게이트된 자가치유*로 다룬다.

```
Agent(
  subagent_type="test-engineer", model="opus",
  prompt="""
  [역할] Scenario Library를 실행 가능한 테스트로 생성·실행하고 로케이터/DOM 드리프트를 점수화·로깅·게이트된 방식으로 자가치유한다.
  [입력] Scenario Library: {Phase 1} / Test Strategy: {Phase 0} / (선택) 테스트 러너·앱 실행 환경·기존 스위트.
  [규칙] 시나리오의 명시적 오라클을 실행 가능한 어서션으로 옮긴다 — 스모크/동어반복 어서션 금지, 진짜 오라클(기대 상태·
         불변식)을 넣는다. 할루시네이션 우려 오라클은 다중 표본 합의로 교차 확인한다(정답에서 실패·버그를 통과시키는 어서션
         제거). 실행 중 로케이터가 깨지면 role→data-testid→접근성 트리→가시 텍스트 우선순위로 후보를 점수화 비교해 자가치유
         하되, 모든 치유를 로깅하고 신뢰도 낮은 치유·기능 변경 의심 신호는 사람 검토로 게이트한다 — 실제 기능 변경/회귀를
         초록으로 은폐하는 재바인딩을 금지한다(회귀 은폐 = 위반). 파괴적/고위험 행동(테스트 데이터 대량 삭제·외부 시스템
         변조·프로덕션 접촉)은 실행 전 사람 승인(HITL). 실패를 분류·재시도하지 않는다(Phase 3의 일). 테스트를 통과시키려
         어서션을 약화·삭제·하드코딩하지 않는다.
  [출력] Execution Report(생성 테스트·오라클·실행 결과 PASS/FAIL·자가치유 로그(점수·게이트 상태)·HITL 대기·미상/가정).
  """
)
```

1줄 보고: `[Phase 2] 생성·실행 완료(PASS {p}/FAIL {f}·자가치유 {h}·게이트 {g}) — 다음: 실패 트리아지. 진행할까요?`

> 파괴적/고위험 행동이나 신뢰도 낮은 자가치유에 사람 승인이 필요하면 *그 행동을 보류*하고 승인을 요청한 뒤 진행한다(회귀
> 은폐·비가역 손상 금지).

## Phase 3 — 실패 트리아지 & 근본원인 (Intelligent Failure Triage)

`failure-triager`를 호출해 실패를 *결함/플래키/환경으로 분류*하고 클러스터링·재실행 우선순위화한다.

```
Agent(
  subagent_type="failure-triager", model="opus",
  prompt="""
  [역할] Execution Report의 실패를 재시도 전에 결함 vs 플래키 vs 환경으로 분류하고, 공유 근본원인으로 클러스터링하며 변경 기반 재실행을 우선순위화한다.
  [입력] Execution Report: {Phase 2} / Scenario Library·Test Strategy: {Phase 1·0} / (선택) 변경 영향(diff)·실패 로그·재실행 이력.
  [규칙] 각 실패를 실제 결함(오라클이 진짜 위반을 드러냄) vs 플래키(비결정·타이밍·순서 의존) vs 환경(네트워킹·외부 의존·
         자원)으로 분류한다 — 근거(오라클·로그·재현성)와 함께. 그린까지 맹목 재시도·격리로 통과시키지 않는다(플래키 억제
         금지). 플래키는 공유 근본원인(네트워킹·외부 의존 등)으로 클러스터링해 하나의 수정으로 다수를 해소하도록 묶고,
         체계적(systemic) 플래키를 고립 사건으로 오판하지 않는다. 재실행/회귀 범위는 변경 영향+리스크로 우선순위화한다
         (전량 재실행 아님). 모든 트리아지 판정을 근거와 함께 설명 가능하게 로깅한다. 코드/테스트를 직접 고치지 않는다
         (분류·클러스터·우선순위·수정 제안만). 실제 결함을 플래키로 오분류해 삼키지 않는다.
  [출력] Triage Report(실패별 분류(결함/플래키/환경)+근거·공유 근본원인 클러스터·변경 기반 재실행 우선순위·수정 제안·미상).
  """
)
```

1줄 보고: `[Phase 3] 트리아지 완료(결함 {d}·플래키 {fl}·환경 {e}·클러스터 {c}) — 종합 보고로 마무리할까요?`

> **트리아지 사람 게이트**: 결함/플래키 경계가 모호하거나 클러스터 수정이 광범위하면 사람 판단을 요청한다. 실제 결함을
> 플래키로 오분류해 삼키는 것을 막는다.

## 마무리 — 종합 보고

플로우가 끝나면 네 단계 산출물을 하나로 종합 보고한다.

- **리스크 전략**: 영역별 리스크 노출 등급·커버리지 우선순위·판정 기준.
- **시나리오**: 리스크 매핑 시나리오·시나리오별 명시적 오라클·추적 태그·오라클 격차.
- **실행**: 생성 테스트·실행 PASS/FAIL·점수화·로깅된 자가치유(게이트 상태)·HITL 결과.
- **트리아지**: 결함/플래키/환경 분류·공유 근본원인 클러스터·변경 기반 재실행 우선순위.
- **남은 불확실성·가정**: 오라클 격차·UNVERIFIED 영역·사람 판단 필요 항목.
