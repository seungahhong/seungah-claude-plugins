> 생성·게이트된 자가치유 실행→결함 vs 플래키 트리아지)* 만 추출하고, 세션의 헤드라인 신규성인 *조직 도입·AI 성숙도 5단계
> 모델·변경관리 층*은 Claude Code로 따를 수 없어 **명시적으로 제외**했다. 각 설계 원칙은 1차/2차 자료로 접지한다.
> [qa-agent-harness-principles.md](./qa-agent-harness-principles.md)의 원칙과 상호 참조된다(§N은 이 dossier의 절 번호).
>
> **정직성 가드(머리말).** (1) 세션의 *조직 도입·성숙도 5단계·변경관리* 층은 **하네스화 불가**로 본 플러그인에서 제외한다 —
> 담긴 것은 *harnessable core*(에이전틱 QA 파이프라인)뿐이다. (2) 세션이 보고한 *회사별 수치*(1.54x QA 산출·~49% AI
> 대체·68% 태스크 시간 절감)는 **특정 조직 관찰값**이지 약속이 아니다 — 인용하되 "개선 N% 보장"으로 쓰지 않는다. (3)
> CANDOR의 오라클 정확도 0.834→0.971은 *특정 세팅 관찰값*이며 보편 법칙이 아니다. (4) 근거는 등급(HIGH/MEDIUM)을 붙여
> 구분하고 벤더 자료(Katalon)는 *운영모델 어휘*로만 쓰며 성과 주장을 인용하지 않는다. (5) "개선 N% 보장"을 쓰지 않는다.

## §1. 에이전틱 테스팅 — 멀티에이전트 QA 프레임 (1차)

- **제목**: "The Rise of Agentic Testing: Multi-Agent Systems for Robust Software Quality Assurance"
- **arXiv ID / 출처**: arXiv:2601.02454 · https://arxiv.org/pdf/2601.02454
- **신뢰도**: **HIGH**
- **핵심**: 소프트웨어 QA를 *단일 도구*가 아니라 **협업하는 다중 에이전트 라이프사이클**로 조직한다 — 전략·설계·생성·실행·
  자가치유·트리아지가 역할 분업된 에이전트로 흐른다. 본 하네스의 4단계(전략→시나리오→생성/자가치유 실행→트리아지)와
  *자가치유·실패 분류를 1급으로 다루는* 골격의 근거다.
- **인용(요지)**: 에이전트는 테스트를 *생성·실행·유지보수·자가치유*하며, 실패를 *분류·트리아지*하는 협업 시스템으로
  묶인다 — 커버리지를 채우는 단일 스크립트 실행이 아니라 *품질 라이프사이클*이다.
- **CAVEAT**: 프레임/서베이 성격이 강한 부분이 있어 *구조적 스캐폴드*로 쓴다. 특정 수치 우월성 주장으로 인용하지 않는다.

## §2. CANDOR — 할루시네이션에서 합의로, 정확한 오라클 생성 (1차)

- **제목**: "Hallucination to Consensus: Multi-Agent LLMs for End-to-End Test Generation with Accurate Oracles" (CANDOR)
- **arXiv ID / 출처**: arXiv:2506.02943 · https://arxiv.org/html/2506.02943v4
- **신뢰도**: **HIGH**
- **핵심**: LLM이 생성한 테스트 오라클은 *할루시네이션*(그럴듯하지만 의미적으로 틀림)에 취약하다 — CANDOR는 **다중 에이전트
  합의(panel/consensus)** 로 오라클을 교차 검증해 정확도를 끌어올린다. 본 하네스 §3(오라클 독립 검증)의 load-bearing 근거.
- **인용(요지)**: 단일 모델 오라클은 비일관적이라 *여러 에이전트의 합의*로 걸러야 정답 프로그램에서 실패하거나 버그를
  통과시키는 어서션을 제거할 수 있다. 보고된 오라클 정확도 개선은 **0.834→0.971**(특정 세팅).
- **CAVEAT**: **0.834→0.971은 특정 데이터셋·세팅 관찰값**이지 보편 법칙이 아니다 — "개선 N% 보장"으로 인용하지 않는다.

## §3. All Smoke, No Alarm — 오라클 신호 (1차·load-bearing)

- **제목**: "All Smoke, No Alarm: Oracle Signals in Agent-Authored Test Code"
- **arXiv ID / 출처**: arXiv:2606.18168 · https://arxiv.org/pdf/2606.18168
- **신뢰도**: **HIGH**
- **핵심**: 에이전트가 작성한 테스트는 *실행 경로는 많이 밟지만 어서션(오라클)이 약해* 결함을 못 잡는 경우가 많다 —
  **커버리지는 결함 탐지력과 약하게만 상관**한다. "연기만 나고 경보는 울리지 않는다(All Smoke, No Alarm)"가 본 하네스의
  핵심 메시지이자 §2(오라클 우선)·§9(커버리지 연극 가드)의 직접 근거다.
- **인용(요지)**: 높은 커버리지·그린 신호가 품질을 보장하지 않는다 — *오라클 신호의 강도*가 실제 결함 탐지를 좌우한다.
- **CAVEAT**: 경험 연구다 — 방향(오라클 강도가 핵심)은 견고하나 *효과 크기*는 세팅 의존이라 인용하지 않는다.

## §4. Systemic Flakiness — 공동발생 플래키 클러스터 (1차)

- **제목**: "Systemic Flakiness: An Empirical Analysis of Co-Occurring Flaky Test Failures"
- **arXiv ID / 출처**: arXiv:2504.16777 · https://arxiv.org/html/2504.16777
- **신뢰도**: **HIGH**
- **핵심**: 플래키 테스트 실패의 *상당수가 공동발생(co-occurring) 클러스터*를 이룬다 — 개별 사건이 아니라 **공유 근본원인**
  (네트워킹·외부 의존 등)에서 온다. 따라서 하나의 수정으로 다수 플래키를 해소할 수 있다. 본 하네스 §5(재시도 전 트리아지·
  클러스터링)의 근거 — 체계적(systemic) 플래키를 고립 사건으로 오판하지 말 것.
- **인용(요지)**: 플래키를 개별로 격리·재시도하면 systemic 근본원인을 놓친다 — *클러스터로 묶어 공유 원인을 고쳐야* 한다.
- **CAVEAT**: 경험 분석 — 클러스터 비율은 코퍼스 의존 관찰값이다. 방향(클러스터링이 유효)만 차용한다.

## §5. LLM 기반 테스트 오라클 생성의 이해 (2차·보강)

- **제목**: "Understanding LLM-Driven Test Oracle Generation"
- **arXiv ID / 출처**: arXiv:2601.05542 · https://arxiv.org/abs/2601.05542
- **신뢰도**: **MEDIUM**
- **핵심**: LLM이 생성한 오라클은 *현재 구현 동작에 오버피팅*하거나 명세 위반을 못 잡는 실패모드를 가진다 — 화이트박스
  통과·홀드아웃 블랙박스 실패. 본 하네스 §2·§9(테스트 오버피팅 anti-pattern)의 보강 근거.
- **인용(요지)**: 오라클은 *명세/기대*에 접지해야지 *지금 코드가 하는 대로* 굳히면 리워드 해킹형 오라클이 된다.
- **CAVEAT**: 보강 근거(MEDIUM). 실패모드 서술을 *가드로 운영화*하되 효과 수치를 인용하지 않는다.

## §6. 요구사항 기반 테스트 우선순위화 — 리스크 인자 (2차·산업연구)

- **제목**: "Requirements based test prioritization using risk factors: An industrial study" (Information and Software Technology)
- **출처**: https://www.sciencedirect.com/science/article/abs/pii/S095058491500155X
- **신뢰도**: **MEDIUM**
- **핵심**: 테스트 우선순위를 *요구사항의 리스크 인자(발생확률 × 영향)* 로 정하면 결함 조기 검출·노력 절감에 유리하다 —
  균일 커버리지가 아니라 **리스크 노출도가 노력을 배분**한다. 본 하네스 §1(리스크 노출)·§6(변경 기반 우선순위화)의 근거.
- **인용(요지)**: 리스크 기반 우선순위화가 *유출 증가 없이 노력을 절감*(산업 세팅 관찰값).
- **CAVEAT**: 산업 사례 연구(MEDIUM) — 절감 크기는 *특정 세팅 관찰값*이다. 방향(리스크가 우선순위를 정함)만 차용한다.

## §7. Agentic QA 운영모델 (2차·벤더, 어휘만)

- **제목**: "Agentic QA as a Quality Operating Model" (Katalon)
- **출처**: https://katalon.com/resources-center/blog/agentic-qa-as-a-quality-operating-model
- **신뢰도**: **MEDIUM** (벤더 자료 — *운영모델 어휘*로만 사용)
- **핵심**: 에이전틱 QA를 *행동 전 제안·점진적 자율성·HITL 승인 게이트·설명 가능한 결정*을 갖춘 운영모델로 서술한다. 본
  하네스 §7(점진적 자율성·suggest-before-act·HITL 게이트)·§8(설명 가능한 로깅)의 어휘 근거.
- **인용(요지)**: 자율성은 *신뢰가 쌓이며 단계적으로 확장*하되 파괴적/고위험 행동은 사람 승인을 통과시킨다.
- **CAVEAT**: **벤더 자료** — 개념·어휘(suggest-before-act·HITL)만 차용하고 *성과 주장·제품 수치는 인용하지 않는다*.

## 제외 범위 (하네스화 불가 — 투명성)

> Tech-Verse 2026 S06 세션의 *헤드라인 신규성*이지만 Claude Code 하네스로 따를 수 없어 **명시적으로 제외**한 것들이다.
> 본 플러그인은 아래를 담지 않는다.

- **조직 도입·AI 성숙도 5단계 모델** — 팀/조직이 에이전틱 QA를 채택하는 성숙도 단계(레벨 모델)는 *조직 변화*이지 에이전트
  파이프라인이 아니다 — 하네스화 불가로 제외.
- **변경관리(change-management) 층** — 사람·프로세스·인센티브 재설계는 하네스 범위 밖 — 제외.
- **회사별 성과 수치의 보편화** — 1.54x QA 산출·~49% AI 대체·68% 태스크 시간 절감은 *특정 조직 관찰값*이다. 세션 맥락으로
  기록하되 **"개선 N% 보장"으로 쓰지 않는다**.
- **CANDOR 정확도의 보편화** — 0.834→0.971은 *특정 세팅 관찰값*이지 "합의 쓰면 항상 이만큼 오른다"가 아니다(§2 CAVEAT).

## 방법론 — 세션 harnessable-core 추출 + 1차 자료 접지

- **출발점**: Tech-Verse 2026 S06 "10x Speed With QA Agent Platform" 세션 정독 → *하네스화 가능한 에이전틱 QA 파이프라인*과
  *하네스화 불가한 조직/성숙도/변경관리 층*을 분리(위 제외 범위).
- **접지**: 파이프라인의 각 원칙(리스크 우선순위화·오라클 우선·오라클 합의 검증·게이트된 자가치유·플래키 클러스터링·변경
  기반 우선순위화·점진적 자율성)을 *1차/2차 자료*로 접지 — HIGH(2601.02454·2506.02943·2606.18168·2504.16777) +
  MEDIUM(2601.05542·요구사항 리스크 산업연구·Katalon 운영모델).
- **정직성·정확 귀속 가드**: 1차(HIGH)와 보강(MEDIUM)·벤더 자료를 절마다 구분한다. 벤더 자료는 *어휘만*, 수치는 *특정
  세팅·조직 관찰값*으로 한정, "개선 N% 보장" 금지. 시점 민감성(소스 2015~2026-06)을 명기한다.
