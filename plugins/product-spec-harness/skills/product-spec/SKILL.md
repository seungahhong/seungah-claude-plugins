# Product Spec Harness — 제품 기획 인터랙티브 오케스트레이터

기획자(PM)의 요구·문제(또는 **기획안 초안**)를 입력으로 받아 **요구 정의 → 기획안 DoR 평가 → 기획문서(PRD) → 사용자 스토리 → 적대적 검증**의 5 Phase를 인터랙티브하게 진행한다. 특정 기술 스택이나 도구에 종속되지 않은, **개발 착수 전** 제품 기획 산출물에 특화한 공통 절차다.

두 가지 입력 모드를 지원한다.

- **(A) 기획안 입력 모드** — 사용자가 기획안(초안)을 인자/대화로 제공하면, 그 내용에서 문제 정의를 **추출**한다. 이어 **생성 *전*에 원본 기획안의 착수 준비도(DoR)를 먼저 평가**(Phase 1, 사용자 승인 시)해 보강점을 도출하고, 그 피드백을 반영해 PRD·사용자 스토리를 **반드시 생성**한다(초안을 받았다고 PRD·스토리 생성을 건너뛰지 않는다). 마지막에 산출물을 적대적 검증한다. **DoR 평가 결과 파일(`product-spec-review.md`)은 PRD·스토리 생성 후 마무리에서 사용자가 저장을 선택할 때만** 기록한다.
- **(B) 대화형 인터뷰 모드** — 기획안이 없으면 한 번에 한 질문으로 문제를 정의(Phase 0)한 뒤 PRD·스토리를 만든다. 평가할 입력 기획안이 없으므로 Phase 1(기획안 DoR 평가)은 건너뛰고, 필요 시 마무리에서 *생성 산출물*을 대상으로 DoR 평가·저장을 선택 적용한다.

## 경계 (먼저 읽고 발동 여부를 판단하라)

이 하네스는 **'기획자의 제품 기획(도메인 무관, 개발 전)'** 이다. 다음은 명시적으로 범위 밖이다.

- **프론트엔드 개발용 PRD·구현 요구사항·코드 작성** — 화면 구현·컴포넌트·기술 설계(설계 → 코드)로 들어가는 요청이면 이 하네스를 발동하지 말라. 이 하네스는 개발 전 *제품 기획문서*만 다룬다.
- **코드/하네스/커밋 작업** — 하네스 자체를 생성·진단·개선하는 작업, 코드 리뷰, 커밋 메시지 작성은 이 하네스가 다루지 않는다.
- **완성 산출물의 핸드오프 게이트 검수** — *이미 완성된* PRD/유저스토리/디자인/계약/인수조건을 (작성이 아니라) 착수 게이트로 검수만 하는 작업은 범위 밖이다(이 하네스는 *작성*과 입력 기획안 평가에 한정).

경계가 모호하면 한 질문으로 의도를 확인한다 — "개발 전 *제품 기획문서*를 만드는 건가요, 아니면 *구현/코드*로 들어가는 건가요?"

## 왜 단계별·인터랙티브인가

제품 기획은 한 번에 정답이 나오지 않는다. 잘못된 문제 정의 위에 쌓은 PRD·스토리는 전부 폐기된다. 그래서 이 하네스는 **매 Phase 종료 시 핵심 산출물을 보여주고 사용자 승인을 받은 뒤에만** 다음으로 넘어간다. 승인 게이트는 비용이 가장 싼 시점(아직 한 Phase 산출물만 있을 때)에 방향 오류를 잡는 장치다.

## 내재화 원칙 (모든 Phase가 따른다)

- **승인 게이트** — 매 Phase 종료 시 핵심 산출물 + 1줄 보고를 제시하고 사용자 승인을 받는다. 보고 형식:
  `[Phase N] {핵심결정} — 다음: {다음}. 진행할까요?`
- **파일 쓰기 전 미리보기 승인** — 파일을 쓰기 전 전체 내용을 미리보기로 보여주고 승인받은 뒤에만 기록한다. 승인 없이 파일을 만들지 않는다.
- **관찰형 수용기준** — 사용자 스토리 수용기준과 성공지표는 제3자가 관찰로 판정 가능해야 한다. Given/When/Then으로 적고, "좋다 / 만족스럽다 / 직관적이다" 같은 판정 불가 문장은 금지한다.
- **완전성·추적** — 모든 PRD 핵심 요구가 ≥1개 스토리로 커버되는지 요구↔스토리 매트릭스로 추적한다.
- **적대적 검증** — 검증 단계는 칭찬형 금지. 누락·모호·비일관을 능동적으로 캔다.
- **기획안 착수 준비도(DoR) 평가 내재화** — 모드 A에서 PRD·스토리 생성 **전(Phase 1)**, **진입 질문(평가할까요?)을 반드시 먼저 제시하고**(생략·자동 스킵 금지) 사용자가 동의(예/모호한 긍정)하면 원본 기획안을 DoR(착수 준비도) 관점에서 평가해 `# DoR Review 결과` + 보강 체크리스트를 채팅으로 공유하고, 그 보강점을 PRD·스토리 생성에 **반영**한다. 평가 결과의 **파일 저장(`product-spec-review.md`)은 PRD·스토리 생성 후 마무리에서 사용자가 저장을 선택할 때만** 수행한다(평가 시점엔 채팅 제시만, 즉시 저장 금지). 평가 방법론은 외부 플러그인을 참조하지 않고 [references/dor-review-rubric.md](./references/dor-review-rubric.md)에 **내재화**돼 있다.
- **additive-first** — 이미 합의된 PRD/스토리를 보완할 때, 기존 항목을 뒤엎기 전에 비파괴 추가·완화를 먼저 제안한다.
- **한 번에 한 질문** — 인터뷰하듯 질문은 한 번에 하나만. jargon(예: INVEST, Gherkin, DoR)은 처음 등장 시 1줄 정의를 곁들인다.

## 에이전트 팀

| Phase | 에이전트 | 책임 |
|-------|----------|------|
| 0 | `requirements-analyst` | 요구/문제/사용자 정의 (Discovery) — 기획안 있으면 카드 추출 |
| 1 | `dor-evaluator` | **원본 기획안** 착수 준비도(DoR) 평가 (모드 A에서 **진입 질문 제시 필수** · 평가 실행만 예/아니오 · 생성 전 선행, 채팅 제시·즉시 저장 안 함) |
| 2 | `prd-writer` | 기획문서(PRD) 작성 (Phase 1 보강점 반영) |
| 3 | `story-writer` | 사용자 스토리 + 수용기준 도출 (Phase 1 보강점 반영) |
| 4 | `spec-reviewer` | 생성 산출물(PRD·스토리) 적대적 검증 (채팅 표시만, 파일 저장 안 함) |

각 에이전트 정의는 `../../agents/{name}.md`에 있다. **모든 Agent 호출은 `model: "opus"`를 명시한다** — 추론 품질이 기획 산출물 품질을 좌우한다.

> Phase 1(`dor-evaluator`)와 Phase 4(`spec-reviewer`)의 차이 — 대상이 다르다. Phase 1은 **입력으로 받은 원본 기획안**을 생성 *전*에 DoR로 평가해 보강 체크리스트를 주고(피드백을 PRD·스토리에 반영), 그 결과는 채팅으로만 제시한다(파일 저장은 마무리에서 사용자가 선택할 때만). Phase 4는 **이번에 생성한 PRD·스토리**의 완전성·일관성·관찰성을 적대적으로 검증하며 결과는 **채팅으로만** 제시한다(파일 저장 안 함).

## 산출물 배치

별도 지정이 없으면 작업 폴더 `_workspace/`(또는 사용자가 지정한 위치)에 다음을 만든다. 모든 문서 언어는 **한국어**.

```
_workspace/
  PRD.md                   # Phase 2 산출
  user-stories.md          # Phase 3 산출
  product-spec-review.md   # Phase 1 기획안 DoR 평가 결과 — 마무리에서 사용자가 저장을 선택할 때만 기록
```

- **`product-spec-review.md`는 PRD·스토리 생성 후 마무리에서 사용자가 저장을 선택한 경우에만 저장한다** — Phase 1 평가 시점에는 채팅으로만 제시하고 파일을 만들지 않는다.
- **Phase 4 적대적 검증 리포트는 파일로 저장하지 않는다** — 채팅으로만 제시하고, 결함은 additive-first로 PRD.md·user-stories.md에 반영한다.
- Phase 0의 문제 정의 카드는 채팅에서 합의한 뒤 PRD.md의 도입부로 반영한다(별도 파일이 필요하면 `_workspace/problem-card.md`).

# 인터랙티브 플로우

시작 시 두 가지를 정한다 — (1) 산출물 위치를 한 번 확인하고(기본 `_workspace/`), (2) **입력 모드를 판별**한다.

## 입력 모드 판별 (시작 시 1회 · 판별 결과를 사용자에게 명시 선언)

- 사용자가 **기획안(초안)** 을 인자/대화로 제공했거나 파일 경로를 가리키면 → **기획안 입력 모드(A)**. 원본 기획안 전문을 보존하고(파일이면 경로 기록), Phase 0은 인터뷰가 아니라 **카드 추출**로 시작한다. 이어 Phase 1(기획안 DoR 평가, 생성 전 선행)을 거치고, 기획안을 받았다는 이유로 PRD·스토리 생성을 건너뛰지 않는다 — **PRD·스토리(Phase 2·3)는 반드시 수행**한다.
- 기획안이 없으면 → **대화형 인터뷰 모드(B)**. Phase 0을 한 번에 한 질문 인터뷰로 진행하고, 입력 기획안이 없으므로 Phase 1은 건너뛴다.

> **무엇이 '기획안'인가(모드 A 임계)** — 문제·목표·기능·대상·제약 중 **둘 이상을 서술한 사용자 제공 텍스트/파일**이면 분량·형식과 무관하게 기획안(초안)으로 간주한다(자유서술 문단·메모·붙여넣기 포함). 한 줄짜리 단순 요구만 있으면 모드 B다.
> **모호하면 A로 안전 디폴트** — 기획안인지 단순 요구인지 경계가 모호하면 **모드 A로 간주**하고 모드 B로 떨어뜨리지 않는다(평가는 어차피 Phase 1 진입 질문에서 사용자가 거절할 수 있으므로 A쪽이 안전하다). **모드를 못 정한 채 Phase 2(PRD)로 직행하지 않는다.**
> **모드 선언(사용자 가시)** — 판별 결과를 채팅에 1줄로 **명시**한다: `[입력 모드] A(기획안 입력) — 다음: 카드 추출 → 기획안 DoR 평가(Phase 1)` 또는 `[입력 모드] B(인터뷰) — 다음: 카드 인터뷰 → PRD(Phase 2); 입력 기획안 없어 Phase 1 DoR 평가는 건너뜀`. 이 모드 라벨이 이후 **모든 Phase 전이의 기준**이며, **Phase 1 진입 시 한 번 더 확인**한다.

이후 Phase 0부터 순차 진행하며, 각 Phase는 **반드시 승인 게이트를 통과한 뒤에만** 다음으로 넘어간다. 사용자가 보완을 요청하면 additive-first로 같은 Phase 안에서 보완한다.

## Phase 0 — 요구/문제 정의 (Discovery)

`requirements-analyst`를 호출해 문제를 정의한다. 산출은 다음 고정 필드를 채운 **문제 정의 카드**다.

- **모드 A(기획안 입력)**: 기획안 전문을 입력으로 넘겨 카드를 **추출**한다. 기획안에서 도출되는 필드는 채우고(근거 1줄), 추출 불가하거나 모호·판정불가한 필드만 **한 번에 한 질문**으로 보강한다. 처음부터 다시 인터뷰하지 않는다.
- **모드 B(인터뷰)**: **grill-me: 한 번에 한 질문**으로 인터뷰하며 카드를 채운다.

| 필드 | 내용 |
|------|------|
| `problem` | 풀려는 문제(현상·페인포인트) |
| `target_users` | 대상 사용자(세그먼트·맥락) |
| `goals` | 목표 — 비즈니스 목표 / 사용자 목표 구분 |
| `constraints` | 제약(범위·일정·정책·기술 전제 등) |
| `success_metrics` | 성공지표 — **관찰형**(제3자가 측정·관찰 가능) |

호출:

```
Agent(
  subagent_type="requirements-analyst", model="opus",
  prompt="""
  [역할] 문제 정의 카드를 채운다. 기획안이 있으면 추출 우선, 없으면 한 번에 한 질문 인터뷰.
  [입력] 입력 모드: {A 기획안 입력 / B 인터뷰}
         기획안(있으면 전문 또는 경로): {기획안}
         사용자 초기 요구: {사용자 발화}
  [규칙] 모드 A면 기획안에서 5필드를 추출하고 근거를 1줄씩 남긴다. 추출 불가·모호 필드만 한 번에 한 질문으로 보강(처음부터 재인터뷰 금지).
         모드 B면 한 번에 한 질문으로 채운다. 공통: jargon은 1줄 정의, success_metrics는 관찰형, 기획안에 없는 사실 발명 금지.
  [출력] 고정 필드(problem/target_users/goals{비즈니스·사용자}/constraints/success_metrics) 채운 문제 정의 카드.
          아직 비어 있거나 모호한 필드는 다음 질문 1개로 표시.
  """
)
```

카드가 모두 채워지면 사용자에게 보여주고 승인 게이트를 띄운다. **이때 '다음'은 위에서 선언한 모드 하나의 행선지만 적는다**(두 모드를 슬래시로 병기하지 않는다 — 병기는 모드 혼선·스킵의 원인이다).

- **모드 A** → `[Phase 0] 문제·대상·목표·성공지표 합의 — 다음: 기획안 DoR 평가(Phase 1). 진행할까요?`
- **모드 B** → `[Phase 0] 문제·대상·목표·성공지표 합의 — 다음: PRD 작성(Phase 2; 입력 기획안 없어 DoR 평가 건너뜀). 진행할까요?`

승인 전에는 다음 Phase로 넘어가지 않는다. **모드 A에서 Phase 0이 승인되면 다음 행동은 반드시 Phase 1 진입 질문 제시다 — Phase 2(PRD)로 직행하지 않는다.**

## Phase 1 — 기획안 평가 (DoR Review, 내재화 · 모드 A 진입 질문 **필수** · 생성 전 선행)

PRD·스토리를 만들기 **전에**, 입력으로 받은 **원본 기획안을 착수 준비도(DoR) 관점에서 평가**해 보강점을 도출하고 그 피드백을 이후 생성(PRD·스토리)에 반영한다. 사용자에게 **평가 진행 여부를 먼저 묻고, 승인할 때만** 진행한다(별도 승인 게이트).

> **'조건부'의 의미를 혼동하지 말 것** — 조건부는 **평가 실행 여부**(예/아니오)에만 걸린다. **진입 질문을 던지는 것 자체는 모드 A에서 필수이며 생략할 수 없다.** 모드 A에서 Phase 1을 건너뛰는 합법 경로는 단 둘뿐이다 — ① 모드 B(입력 기획안 없음), ② 모드 A에서 사용자가 진입 질문에 **명시적으로 "아니오"**라고 답함. **진입 질문을 띄우지 않은 채 Phase 2로 직행하는 것은 금지**(보고된 0→2 스킵 결함의 직접 원인).

> jargon 1줄 정의: **DoR(Definition of Ready)** = 산출물이 개발·QA가 추측 없이 착수할 수 있는 상태인지 판정하는 착수 게이트.

진입 질문(승인 게이트 · 모드 A에서 **반드시 제시**):

`[Phase 1] 원본 기획안을 DoR(착수 준비도)로 먼저 평가해서, 보강점을 PRD·스토리에 반영할까요? (예/아니오 — 모호하면 '예'로 진행)`

- **예 (또는 '진행해/알아서 해' 등 모호한 긍정)** → `dor-evaluator`를 호출해 평가한다. **모호한 응답은 거절이 아니라 '예'로 해석**한다(평가는 채팅 전용·비파괴라 기본값으로 안전). 결과는 **채팅으로만** 제시한다 — **이 시점에는 파일로 저장하지 않는다**(저장은 PRD·스토리 생성 후 마무리에서 별도로 묻는다).
- **아니오 (명시적 거절일 때만)** → 평가를 건너뛰고 바로 Phase 2(PRD)로 간다.
- **모드 B(입력 기획안 없음)** → 이 Phase는 건너뛴다(평가할 입력 기획안이 없음). 생성 산출물 DoR 평가가 필요하면 마무리에서 선택적으로 제안한다.

평가는 [references/dor-review-rubric.md](./references/dor-review-rubric.md)에 **내재화**된 루브릭(DoR 게이트 · INVEST 스코어카드 · Given-When-Then 완결성 · 모호성 린트 · 의존성/계약 참조)을 따른다. 외부 플러그인을 참조하지 않는다.

호출:

```
Agent(
  subagent_type="dor-evaluator", model="opus",
  prompt="""
  [역할] 원본 기획안의 착수 준비도(DoR)를 생성 전에 평가한다. 기획안을 직접 고치지 않고 보강 후보만 제시.
  [입력] 평가 대상(원본 기획안): {기획안 전문/경로}, 문제 정의 카드: {Phase 0 카드}
  [참조] references/dor-review-rubric.md (DoR 게이트·INVEST·GWT 완결성·모호성 린트·의존성, # DoR Review 결과 템플릿·체크리스트·Honesty Guardrail).
  [규칙] 각 FAIL에 위반 문장 인용. INVEST의 Testable=0 또는 Independent=0이면 착수 차단.
         음성 경로(엣지/에러/빈/권한) 누락마다 확인 필요 질문 생성. 모호성은 Before→After(수치 발명 금지).
         정량 근거는 Honesty Guardrail(등급·출처)에 따라서만 인용, '개선 N%' 약속 금지.
         결과는 채팅 제시용으로만 낸다 — 파일 저장은 오케스트레이터가 마무리에서 사용자 동의 시 수행한다.
  [출력] `# DoR Review 결과` 보고서 — 8개 섹션(대상/DoR 게이트/INVEST/GWT 완결성/모호성/의존성/종합 판정/보강 체크리스트) 전부 포함.
          특히 '# DoR Review 결과' 하위 내용과 '보강 체크리스트'(미체크=보강 대상)를 반드시 포함한다.
  """
)
```

평가 결과(`# DoR Review 결과` + 보강 체크리스트)를 **채팅으로 제시**한다. 보강 체크리스트의 미체크(`[ ]`) 항목은 이후 Phase 2·3에서 PRD·스토리로 **채워 넣을 대상**이다. 이 결과는 **이 시점에 파일로 저장하지 않는다**(저장 여부는 마무리에서 묻는다 — 산출물 배치 참조).

승인 게이트:

`[Phase 1] 기획안 DoR 평가({착수가능|보완 후|보류} / 보강 {n}건, 채팅 제시) — 다음: PRD 작성(보강점 반영). 진행할까요?`

## Phase 2 — 기획문서(PRD) 작성

승인된 문제 정의 카드를 입력으로 `prd-writer`를 호출한다. **Phase 1에서 DoR 평가를 했다면 그 보강 체크리스트를 함께 입력**해, 식별된 누락·모호를 PRD에서 additive하게 채운다. PRD는 [references/prd-template.md](./references/prd-template.md)의 표준 구조를 따른다.

구조: **배경·문제 / 목표·성공지표(관찰형) / 범위(In·Out 명시) / 핵심 요구사항(기능·비기능) / 가정·리스크 / 마일스톤**.

호출:

```
Agent(
  subagent_type="prd-writer", model="opus",
  prompt="""
  [역할] 문제 정의 카드 → PRD 초안. Phase 1 DoR 보강점이 있으면 반영.
  [입력] 문제 정의 카드: {Phase 0 카드}
         Phase 1 DoR 보강 체크리스트(있으면): {미체크 항목·확인 필요 질문}
  [참조] references/prd-template.md 표준 구조와 작성기준.
  [규칙] 목표·성공지표는 관찰형. 범위는 In/Out 모두 명시. 핵심 요구사항은 기능/비기능 구분하고 각 요구에 식별자(R1, R2…)를 부여한다(Phase 3 추적용). DoR 보강점은 해당 요구·가정·리스크로 흡수한다.
  [출력] PRD 초안(표준 구조). 모호하거나 카드/보강점에서 결정되지 않은 부분은 [확인필요] 마커로 1개씩 표시.
  """
)
```

PRD에 `[확인필요]`가 있으면 **한 번에 한 질문**으로 해소한 뒤 반영한다. PRD 초안이 완성되면 **파일 쓰기 전 전체 미리보기**를 보여주고 승인받은 뒤 `_workspace/PRD.md`에 기록한다.

승인 게이트:

`[Phase 2] PRD 합의(범위 In/Out·핵심 요구 R1…Rn) — 다음: 사용자 스토리 도출. 진행할까요?`

## Phase 3 — 사용자 스토리 도출

승인된 PRD의 핵심 요구사항(R1…Rn)을 입력으로 `story-writer`를 호출한다. **Phase 1 DoR 보강 체크리스트가 있으면** 음성 경로(엣지/에러/빈/권한) 등 식별된 누락을 스토리·수용기준으로 채운다. 작성 규약은 [references/user-story-guide.md](./references/user-story-guide.md)를 따른다.

각 스토리는:

- **형식** — "…로서 …하고 싶다, 왜냐하면 …" (As a / I want / so that)
- **수용기준** — Given/When/Then (관찰형, 제3자가 판정 가능)
- **INVEST 자가점검** — Independent / Negotiable / Valuable / Estimable / Small / Testable (각 jargon 1줄 정의 곁들임)
- **추적** — 각 스토리가 커버하는 요구 식별자(R#)를 명시

호출:

```
Agent(
  subagent_type="story-writer", model="opus",
  prompt="""
  [역할] PRD 핵심 요구 → 사용자 스토리 + 수용기준. Phase 1 DoR 보강점(음성 경로 등) 반영.
  [입력] PRD 핵심 요구사항(R1…Rn): {PRD 요구 목록}
         Phase 1 DoR 보강 체크리스트(있으면): {미체크 항목·누락 경로}
  [참조] references/user-story-guide.md (As a/I want/so that + Gherkin + INVEST).
  [규칙] 수용기준은 Given/When/Then 관찰형. 판정불가 문장 금지. 각 스토리에 커버 요구(R#) 표기. INVEST 자가점검 결과 포함. DoR이 지적한 엣지/에러/빈/권한 경로를 시나리오로 포함.
  [출력] 스토리 목록 + 요구↔스토리 커버 표(어떤 R#가 어떤 스토리로 커버되는지).
  """
)
```

스토리 초안이 완성되면 **파일 쓰기 전 미리보기**를 보여주고 승인받은 뒤 `_workspace/user-stories.md`에 기록한다.

승인 게이트:

`[Phase 3] 스토리 N개 + 수용기준(G/W/T)·요구 커버 표 — 다음: 적대적 검증. 진행할까요?`

## Phase 4 — 검증 (적대적)

`spec-reviewer`를 호출해 **칭찬 금지·결함 색출** 모드로 PRD와 스토리를 교차 검증한다. 점검 축:

1. **완전성** — 요구↔스토리 추적 매트릭스를 만들어 커버 안 된 요구(R#)를 찾는다.
2. **INVEST 통과** — 각 스토리가 INVEST를 만족하는지, 위반 항목을 지목한다.
3. **수용기준 관찰성** — Given/When/Then이 제3자 관찰로 판정 가능한지. 판정불가 문장을 색출한다.
4. **일관성** — 목표 ↔ 요구 ↔ 스토리 사슬의 불일치(목표에 없는 요구, 요구 없는 스토리 등).
5. **모호/판정불가 문장** — PRD·스토리 전반의 애매어·미정의 용어를 캔다.

호출:

```
Agent(
  subagent_type="spec-reviewer", model="opus",
  prompt="""
  [역할] 적대적 검증 — 칭찬 금지, 누락·모호·비일관을 능동적으로 캔다.
  [입력] PRD: {PRD.md}, 스토리: {user-stories.md}, 문제 정의 카드: {Phase 0 카드}
  [점검] 1) 완전성(요구↔스토리 추적 매트릭스로 미커버 요구 색출)
         2) INVEST 통과/위반
         3) 수용기준 관찰성(판정불가 문장 색출)
         4) 일관성(목표↔요구↔스토리)
         5) 모호/미정의 용어
  [출력] 검증 리포트 — 추적 매트릭스 + 발견 결함 목록(심각도·위치·근거·수정 제안). 결함 0건이면 그 근거를 명시.
  """
)
```

검증 리포트는 **채팅으로만 제시한다 — 별도 `.md` 파일로 저장하지 않는다.** (적대적 검증은 그 자체가 산출물이 아니라 PRD·스토리를 다듬기 위한 피드백이다.)

**결함이 있으면 additive-first로 보완한다** — 합의된 PRD/스토리를 뒤엎기 전에 비파괴 추가(누락 스토리 추가, 수용기준 보강)·완화를 먼저 제안하고, 사용자 승인 후 해당 Phase(2 또는 3)로 돌아가 반영한다(반영 결과는 `PRD.md`·`user-stories.md`에만 기록). 구조적 뒤집기는 사용자가 명시적으로 승인할 때만 한다.

승인 게이트:

`[Phase 4] 적대적 검증 완료(결함 {n}건 / 추적 커버율 {x}%) — 마무리할까요?`

# 마무리

## 1. 기획안 DoR 평가 결과 저장 (PRD·스토리 생성 후 · opt-in)

PRD·스토리가 만들어진 **뒤에만**, Phase 1에서 수행한 기획안 DoR 평가 결과를 파일로 남길지 묻는다. **사용자가 저장을 선택한 경우에만** 저장한다(요청·평가 시점에 자동 저장 금지).

`[저장 확인] Phase 1의 기획안 DoR 평가 결과(# DoR Review 결과 + 보강 체크리스트)를 product-spec-review.md로 저장할까요? (예/아니오)`

- **예** → 평가 결과 전체를 **파일 쓰기 전 미리보기**로 다시 보여주고, 승인 후 `_workspace/product-spec-review.md`에 기록한다.
- **아니오** → 저장하지 않는다(채팅 제시로만 남긴다).
- **Phase 1을 건너뛴 경우(모드 B 등 입력 기획안 없음·평가 미실시)** → 저장할 평가 결과가 없다. 필요하면 *생성된 PRD·스토리*를 대상으로 DoR 평가를 지금 수행할지 한 번 묻고(opt-in), 수행했다면 동일하게 저장 여부를 묻는다(보고서에 대상이 "생성 산출물"임을 명시).

## 2. 산출물 요약

만들어진 산출물의 위치를 요약해 보고한다 — `PRD.md` / `user-stories.md`, 그리고 저장을 선택했다면 `product-spec-review.md`(기획안 DoR 평가 + 보강 체크리스트). **적대적 검증(Phase 4) 리포트와, 저장을 선택하지 않은 DoR 평가 결과는 파일로 남기지 않는다** — 그 피드백은 PRD·스토리 보완으로 흡수된다. 추가 보완 요청은 항상 additive-first로, 해당 Phase에서 한 번에 한 질문으로 진행한다.
