# 사용자 스토리 작성 가이드

PRD의 요구사항(FR/NFR)을 사용자 관점의 **스토리**와 **관찰형 수용기준**으로 전개하는 가이드다. Phase 2(`story-writer`)가 이 가이드를 따르고, Phase 3(`spec-reviewer`)가 이 기준으로 검증한다. 모든 산출 문서는 한국어로 작성한다.

> jargon 1줄 정의: **사용자 스토리(User Story)** = 기능을 "누가·무엇을·왜"의 사용자 관점 한 문장으로 표현한 작업 단위.

---

## 1. 사용자 스토리 형식

```
[역할]로서 [목표/행동]하고 싶다, 왜냐하면 [이유/가치]이기 때문이다.
```

- **[역할]**: 구체적 사용자 유형. ("사용자" 같은 뭉뚱그림보다 "신규 가입자", "관리자"처럼 구체화.)
- **[목표/행동]**: 사용자가 하고 싶은 일. *해결책(UI·구현)이 아니라 의도*를 적는다.
- **[이유/가치]**: 왜 그것이 가치 있는지. 이유가 약하면 스토리의 필요성(Valuable)을 재검토한다.

**예시**
- 좋음: "신규 가입자로서 이메일만으로 빠르게 가입하고 싶다, 왜냐하면 결제 전 부담 없이 제품을 둘러보고 싶기 때문이다."
- 나쁨: "가입 폼에 이메일 input 박스를 추가한다." (역할·이유 없음 — 작업 지시이지 스토리가 아님.)

각 스토리에는 고유 ID(예: `US-1`)와, 출처가 된 PRD 요구사항 ID(예: `FR-1`)를 함께 기록한다. → [§5 추적 연결](#5-prd-요구와의-추적-연결) 참조.

---

## 2. 수용기준: Gherkin (Given / When / Then)

수용기준(Acceptance Criteria)은 **제3자가 관찰로 참/거짓을 판정**할 수 있어야 한다. 표준 형식은 Gherkin이다.

```
Given  [전제 상황 / 초기 상태]
When   [사용자의 행동 / 트리거]
Then   [관찰 가능한 결과 / 시스템 반응]
```

- **Given**: 시나리오의 출발 상태. ("로그인하지 않은 신규 방문자가 가입 화면에 있다.")
- **When**: 검증 대상 행동. ("유효한 이메일과 비밀번호를 입력하고 '가입'을 누른다.")
- **Then**: 관찰 가능한 결과. ("3초 이내에 온보딩 화면으로 이동하고, 환영 이메일이 발송된다.")
- 하나의 스토리는 보통 정상 경로 1개 + 예외/경계 경로 1~N개의 시나리오를 가진다.

> jargon 1줄 정의: **Gherkin** = Given/When/Then 키워드로 행동을 기술하는 수용기준 표준 문법.

**예시 (스토리 US-1, 출처 FR-1)**
```
시나리오: 유효한 이메일로 가입 성공
  Given 로그인하지 않은 방문자가 가입 화면에 있고
  When  미사용 이메일과 8자 이상 비밀번호를 입력하고 '가입'을 누르면
  Then  온보딩 화면으로 이동하고, 해당 이메일로 인증 메일이 1통 발송된다.

시나리오: 이미 가입된 이메일 (예외)
  Given 이미 가입된 이메일이 존재하고
  When  같은 이메일로 가입을 시도하면
  Then  '이미 가입된 이메일입니다' 메시지가 표시되고, 가입은 진행되지 않는다.
```

## 3. INVEST 6기준

좋은 사용자 스토리의 6가지 속성. 각 스토리는 작성 후 아래 점검질문으로 자가점검한다.

| 기준 | 1줄 설명 | 점검질문 |
|------|----------|----------|
| **I** — Independent (독립적) | 다른 스토리에 묶이지 않고 따로 다룰 수 있다 | 이 스토리를 단독으로 진행/검증할 수 있는가? 다른 스토리가 끝나야만 의미가 생기지 않는가? |
| **N** — Negotiable (협상가능) | 세부 구현이 아니라 의도를 담아 조정 여지가 있다 | UI·구현 방식을 못박지 않고 "무엇을·왜"만 담았는가? |
| **V** — Valuable (가치있음) | 사용자/비즈니스에 분명한 가치가 있다 | '왜냐하면' 절이 실제 가치를 설명하는가? 가치가 없으면 왜 만드는가? |
| **E** — Estimable (추정가능) | 규모를 가늠할 만큼 충분히 명확하다 | 모호함 때문에 크기를 추정 못 할 부분은 없는가? |
| **S** — Small (작음) | 한 반복 주기에 들어갈 만큼 작다 | 'and/또는'로 여러 기능이 뭉쳐 있지 않은가? 쪼갤 수 있는가? |
| **T** — Testable (검증가능) | 관찰형 수용기준으로 통과/실패를 판정할 수 있다 | Given/When/Then으로 제3자가 판정 가능한가? |

**S와 T가 가장 자주 깨진다**: 접속사로 여러 기능을 묶었으면(S 위반) 분리하고, 수용기준이 모호하면(T 위반) §4 표로 다시 쓴다.

## 4. 좋은 vs 나쁜 수용기준 (관찰 가능 vs 모호)

수용기준은 "좋다/만족/잘 된다" 같은 **판정불가 문장을 금지**한다. 제3자가 관찰만으로 동일한 결론에 도달해야 한다.

| 나쁜 예 (모호·판정불가) | 좋은 예 (관찰 가능) |
|------------------------|---------------------|
| "가입이 잘 된다" | "Then 가입 완료 후 3초 이내 온보딩 화면으로 이동한다" |
| "에러를 적절히 처리한다" | "Then '이미 가입된 이메일입니다' 메시지가 표시되고, 폼은 초기화되지 않는다" |
| "성능이 충분하다" | "Then 가입 응답이 p95 기준 500ms 이내로 반환된다" |
| "사용자가 만족한다" | "Then 가입 완료 화면에 사용자 이름이 표시된다" (관찰 가능한 사실로 대체) |
| "보안이 안전하다" | "Then 비밀번호는 응답·로그·화면 어디에도 평문으로 노출되지 않는다" |

**판정 체크**: Then을 읽고 "두 사람이 같은 화면/로그를 보면 통과/실패에 동일하게 합의하는가?"가 'No'면 다시 쓴다.

## 5. PRD 요구와의 추적 연결

모든 스토리는 **출처 PRD 요구사항(FR/NFR ID)을 명시**한다. 이것이 Phase 3의 **요구↔스토리 추적 매트릭스**의 근간이다.

- 한 PRD 요구사항은 1개 이상의 스토리로 커버된다(완전성: 커버되지 않은 요구가 있으면 누락).
- 한 스토리는 보통 하나의 핵심 요구를 출처로 가진다(여러 요구를 묶으면 INVEST의 S/I 위반 의심).
- 추적 표기 예:

| 스토리 ID | 스토리 요약 | 출처 요구(PRD) | 수용기준 수 |
|-----------|-------------|----------------|-------------|
| US-1 | 이메일로 가입 | FR-1 | 정상1 / 예외2 |
| US-2 | 가입 응답 속도 | NFR-1 | 1 |

- **빠짐 점검**: PRD의 모든 FR/NFR ID가 이 표의 '출처 요구' 열에 최소 1회 등장하는지 확인한다. 등장하지 않는 요구 = 스토리 누락(또는 PRD에서 Out으로 명시되었어야 함).
- **고아 점검**: 어떤 PRD 요구에도 연결되지 않은 스토리 = 출처 불명. PRD에 요구를 추가하거나 스토리를 제거한다.

## 작성 후 자가 점검

- [ ] 모든 스토리가 "[역할]로서 …하고 싶다, 왜냐하면 …" 형식인가?
- [ ] 모든 수용기준이 Given/When/Then이고 관찰 가능한가(§4 판정 체크 통과)?
- [ ] 각 스토리가 INVEST 6기준을 통과하는가(특히 S·T)?
- [ ] 모든 스토리에 출처 PRD 요구 ID가 연결되었는가?
- [ ] PRD의 모든 요구가 최소 1개 스토리로 커버되는가(빠짐 점검)?
