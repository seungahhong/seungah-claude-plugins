# PRD 표준 구조 (Product Requirements Document)

기획자(PM)가 개발 착수 *전* 단계에서 작성하는 제품 기획문서의 표준 구조와 각 섹션 작성 기준이다. Phase 1(`prd-writer`)이 이 템플릿을 따른다. 모든 산출 문서는 한국어로 작성한다.

> jargon 1줄 정의: **PRD(Product Requirements Document)** = 무엇을·왜 만드는지를 정의하는 제품 기획문서. "어떻게 구현하는지"(설계·코드)는 담지 않는다.

---

## 섹션 구성 (고정 6섹션)

PRD는 아래 6섹션을 순서대로 포함한다. 빈 섹션을 두지 말고, 해당 없음이면 "해당 없음 — (사유)"로 명시한다.

### 1. 배경·문제 (Background & Problem)

- **무엇을 쓰나**: 이 제품/기능이 왜 필요한지, 현재 어떤 문제가 있는지, 누구의 문제인지.
- **작성 기준**:
  - 문제를 "해결책 없이" 기술한다. ("로그인 화면이 없다"는 해결책 진술 — "신규 사용자가 자기 데이터를 다시 찾지 못한다"가 문제 진술.)
  - 문제의 대상 사용자(target_users)와 빈도/영향을 함께 적는다.
  - Phase 0 문제 정의 카드의 `problem` / `target_users`를 근거로 확장한다.

### 2. 목표·성공지표 (Goals & Success Metrics)

- **무엇을 쓰나**: 이 제품이 달성하려는 비즈니스 목표·사용자 목표와, 성공을 **관찰로 판정**할 수 있는 지표.
- **작성 기준**:
  - 목표는 비즈니스 목표와 사용자 목표를 분리한다.
  - 성공지표는 반드시 **관찰형**(제3자가 측정·관찰로 참/거짓 판정 가능)으로 쓴다. 아래 [관찰형 성공지표 기준](#관찰형-성공지표-좋은-예-vs-나쁜-예) 표를 따른다.
  - 각 지표는 기준선(현재값)·목표값·측정 방법·측정 시점을 포함한다.

### 3. 범위 (Scope: In / Out)

- **무엇을 쓰나**: 이번 산출물이 다루는 것(In)과 **명시적으로 다루지 않는 것(Out)**.
- **작성 기준**:
  - In-scope: 이번에 만들 핵심 항목을 불릿으로 나열.
  - **Out-of-scope를 반드시 명시한다** — 아래 [범위 Out 명시의 중요성](#범위-out-명시의-중요성) 참조.
  - 경계가 모호한 항목은 In/Out 중 하나로 강제 분류하고 사유를 적는다.

### 4. 핵심 요구사항 (Requirements: 기능·비기능)

- **무엇을 쓰나**: 제품이 충족해야 할 요구사항. **기능 요구사항(FR)**과 **비기능 요구사항(NFR)**을 분리한다.
- **작성 기준**:
  - 각 요구사항에 고유 ID 부여(예: `FR-1`, `NFR-1`). Phase 2 스토리 추적과 Phase 3 요구↔스토리 매트릭스가 이 ID를 참조한다.
  - 기능 요구사항(FR): 사용자가 무엇을 할 수 있어야 하는지. ("사용자는 이메일로 가입할 수 있다.")
  - 비기능 요구사항(NFR): 성능·보안·접근성·가용성 등 품질 속성. 가능하면 관찰형 임계값 포함. ("가입 API 응답은 p95 기준 500ms 이하.")
  - 각 요구사항은 단일 책임(한 문장 = 한 요구). 접속사로 묶인 복합 요구는 분리한다.

> jargon 1줄 정의: **FR(Functional Requirement)** = 시스템이 *무엇을 하는가*. **NFR(Non-Functional Requirement)** = 시스템이 *얼마나 잘 하는가*(속도·안정성 등).

### 5. 가정·리스크 (Assumptions & Risks)

- **무엇을 쓰나**: 참이라고 전제한 사항(가정)과, 틀릴 경우 산출물을 위협하는 요인(리스크).
- **작성 기준**:
  - 가정: 검증되지 않았지만 전제로 깐 사실. ("결제 PG사 API가 90일 내 제공된다고 가정.")
  - 리스크: 영향도·발생가능성·완화책(mitigation)을 함께. 가능하면 표로.
  - 가정이 깨지면 어떤 요구사항(FR/NFR ID)이 영향받는지 연결한다.

### 6. 마일스톤 (Milestones)

- **무엇을 쓰나**: 산출물을 단계로 나눈 일정·게이트. 개발 일정이 아니라 **기획 산출물의 단계**다.
- **작성 기준**:
  - 각 마일스톤에 목표·포함 요구사항(ID)·완료 판정 기준(관찰형)을 적는다.
  - 일정은 상대 표기(예: M1, M2) 또는 확정 시 절대 표기.

## 관찰형 성공지표: 좋은 예 vs 나쁜 예

성공지표는 **제3자가 관찰·측정만으로 참/거짓을 판정**할 수 있어야 한다. "좋아짐/만족" 같은 주관·판정불가 문장은 금지한다.

| 구분 | 나쁜 예 (판정불가·모호) | 좋은 예 (관찰형) |
|------|------------------------|------------------|
| 전환 | "가입이 쉬워진다" | "가입 시작→완료 **전환율을 42% → 60%**로 (출시 후 4주, GA 퍼널 기준)" |
| 사용성 | "사용성이 좋아진다" | "신규 사용자의 **첫 핵심행동 도달 시간(TTV)을 중앙값 8분 → 3분**으로" |
| 만족 | "사용자가 만족한다" | "출시 후 30일 **NPS를 +12 → +30**으로 (인앱 설문 응답 ≥200건 기준)" |
| 성능 | "빠르게 동작한다" | "주요 화면 **로딩 p95를 1.2초 이하**로 (RUM 측정)" |
| 리텐션 | "재방문이 늘어난다" | "신규 코호트 **D7 리텐션을 18% → 28%**로" |
| 오류 | "에러가 줄어든다" | "결제 단계 **실패율을 5.0% → 1.5% 이하**로 (서버 로그 기준)" |

**판정 체크**: 지표를 읽고 "서로 다른 두 사람이 같은 데이터를 보면 동일한 참/거짓 결론에 도달하는가?"가 'No'면 다시 쓴다.

> jargon 1줄 정의: **TTV(Time To Value)** = 사용자가 제품의 핵심 가치를 처음 경험하기까지 걸린 시간. **NPS(Net Promoter Score)** = 추천 의향 기반 만족 지표(-100~+100).

## 범위 Out 명시의 중요성

**Out-of-scope를 적지 않으면 모든 것이 In-scope로 오해**된다. Out 명시는 다음을 막는다.

- **스코프 크리프(scope creep)**: 합의 없이 범위가 슬금슬금 늘어나는 것. Out에 적으면 "그건 이번이 아니다"의 합의 근거가 된다.
- **잘못된 기대**: 이해관계자가 당연히 포함될 거라 가정한 항목을 Out에 박아 기대를 정렬한다.
- **검증 누락 오판**: Phase 3 검증에서 "이 요구가 왜 스토리가 없지?"를 "Out이라서 의도된 제외"로 구분할 수 있다.

작성 요령: In의 각 항목에 대해 "이것과 헷갈릴 인접 항목은?"을 묻고, 그 인접 항목을 Out에 명시한다. (예: In "이메일 가입" → Out "소셜 로그인(차기)", Out "비밀번호 없는 매직링크").

## 작성 후 자가 점검

- [ ] 6섹션이 모두 채워졌는가(해당 없음도 사유와 함께 명시)?
- [ ] 모든 성공지표가 관찰형인가(위 판정 체크 통과)?
- [ ] 범위에 Out이 명시되었는가?
- [ ] 모든 요구사항에 고유 ID가 있고 단일 책임인가?
- [ ] 가정/리스크가 영향받는 요구사항 ID와 연결되었는가?
