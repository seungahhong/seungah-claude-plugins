# 기획 완성도 점검(DoR) 루브릭 (Definition of Ready)

Phase 1(`dor-evaluator`)가 **원본 기획안**(인자로 받은 기획 초안)을 **기획 완성도 점검(DoR · Definition of Ready)** 관점에서 평가할 때 따르는 루브릭이다. PRD·스토리를 만들기 **전**에 진행하는 **생성 전 선행** 단계다(모드 A에선 진입 질문 제시가 필수이고, 평가 *실행*만 사용자 동의 조건). 평가 결과는 `# 기획 완성도 점검 결과(DoR Review)` 보고서 + 점검표로 산출해 **기획자에게 무엇을 보강해야 하는지** 공유하고, 그 보강점을 이후 PRD·스토리 생성에 반영한다. 평가 시점에는 **채팅으로만** 제시하며, 파일(`product-spec-review.md`) 저장은 PRD·스토리 생성 후 마무리에서 **사용자가 저장을 선택할 때만** `.claude/_docs/<기획서 슬러그>/`에 한다. 모든 산출 문서는 한국어로 작성한다.

> 쉬운 말 1줄 정의: **기획 완성도 점검(DoR, Definition of Ready)** = 한 산출물이 다운스트림(개발·QA)이 추측 없이 착수할 수 있는 상태인지 판정하는 착수 게이트. (완료 판정인 **완료 기준(DoD, Definition of Done)**과 구별.)

> **내재화 노트**: 이 루브릭은 핸드오프 게이트 방법론(기획 완성도 게이트(DoR 게이트) · 좋은 스토리 6가지 기준 점수표(INVEST 스코어카드) · 조건·행동·결과(Given-When-Then) 완결성 · 모호성 점검 · 의존성 참조)을 이 플러그인에 **내재화**한 것이다(외부 플러그인 참조·의존 없음 — 이 플러그인만 설치해도 독립 동작). 단, 이미 *완성된* 상류 산출물을 핸드오프 시점에 독립 게이트로 통합 검수하는 작업(여러 게이트 병렬·디자인/계약/QA 동시)은 이 하네스의 범위가 아니다. 여기서는 **PRD·스토리 생성 전에 입력 기획안의 완성도를 평가**해 보강점을 생성에 반영하는 선행 피드백 단계로 한정한다.

---

## 핵심 원칙 (평가 전 먼저 읽는다)

**원칙 1 — 모호한 요구사항은 곧 다운스트림 재작업이다.** 명확성은 착수 전에 강제한다. "빠르게", "쉽게", "적절히" 같은 표현은 구현자와 QA에게 서로 다른 해석을 남기고, 그 차이는 QA 막바지나 릴리스 후 재작업으로 청구된다. 불완전·은닉 요구사항은 요구공학에서 가장 많이 지목되는 문제이며(NaPiRE 설문 228명 중 109명=48%, Méndez Fernández et al. 2017 [GOLD]), 회피 가능한 재작업은 프로젝트 노력의 상당 부분을 차지한다(Boehm & Basili 2001 [GOLD]). 모호성을 발견하면 구체적 재작성안을 제시하고, **의도를 알 수 없으면 지어내지 말고** 기획에 되돌릴 질문으로 만든다.

**원칙 2 — 인수조건은 테스트 가능해야 하고 정상경로만으로는 부족하다.** 모든 인수조건은 조건·행동·결과(Given-When-Then)로 표현 가능해야 하며, 정상 흐름(happy path)뿐 아니라 경계(edge)·에러(error)·빈 상태(empty)·권한(authz) 흐름을 포함해야 한다. 예외 흐름(negative path)이 없는 인수조건은 미완성으로 본다.

**원칙 3 — 자동 탐지(LLM)는 사람 검토를 보조하지, 그 자체로 게이트를 통과시키지 않는다.** LLM·NLP는 유저스토리 품질 결함과 모호성을 탐지하는 데 도움이 되지만(in-context 예시로 모호 요구사항 분류 성능이 0-shot 대비 평균 20.2% 향상 — Bashir/Ferrari et al., ICSME 2025 [GOLD]), 한계가 분명하다. 최신 LLM은 '이상적 템플릿'에서 벗어났다는 이유로 전문가가 acceptable로 본 스토리를 과대탐지하고 환각·불안정성을 보이며(GPT-5; 룰 기반 AQUSA가 정밀도 0.61로 false positive가 더 적음 — Perkusich et al. 2025 [SILVER]), '무엇이 결함인가'는 프로젝트·도메인·이해관계자 해석에 따라 달라지는 맥락 의존 개념이다(Unterbusch & Vogelsang, ICSE-NIER 2026 [SILVER]). 따라서 평가 결과는 **확정 판정이 아니라 보강 후보**로 제시하고, 도메인·맥락 판정은 사람(기획자)이 한다.

## 평가 절차

### Step 0 — 대상·범위 확정

평가 대상을 명시한다. 우선순위는 **원본 기획안**이다.

- **원본 기획안 있음(기본)**: 인자/대화로 받은 기획 초안 전문을 대상으로 한다. 목적은 *입력 기획안*에서 무엇이 모호·누락됐는지 짚어 기획자에게 보완할 점 점검표(보강 체크리스트)를 주는 것.
- **원본 기획안 없음(인터뷰 모드)**: 평가할 입력 기획안이 없으므로, 사용자 동의 시 이번에 *생성한* PRD·사용자 스토리를 대상으로 같은 루브릭을 적용할 수 있다(선택). 이 경우 대상이 "생성 산출물"임을 보고서에 명시한다.

기획안에 없는 의도를 추론으로 채우지 않는다. 모르면 "확인 필요" 질문으로 돌린다.

### Step 1 — 기획 완성도 게이트(DoR 게이트) 점검

기획안이 착수 가능한 상태인지 항목별로 PASS(✅)/FAIL(❌)을 매긴다. 각 FAIL에는 **위반한 문장/구절을 인용**한다.

| 점검 항목 | 통과 기준 |
| --- | --- |
| 표준 포맷 | 사용자·목적이 명시됨(`…로서 …하고 싶다` 또는 동등하게 누가·무엇을·왜가 드러남) |
| 인수조건 측정 가능 | 인수조건이 조건·행동·결과(Given-When-Then)로 표현 가능하고 관찰 가능한 결과를 명시 |
| 비즈니스 규칙 | 계산·제약·우선순위 등 규칙이 문서화됨 |
| 예외·에러·권한 흐름 | 빈 상태, 실패 응답, 권한 없음 흐름이 기술됨 |
| 디자인 참조 | 화면이 있는 기능이면 디자인 링크/시안이 있고 대상 화면과 매칭됨 (없으면 "해당 없음" 사유) |
| 의존성 식별 | 참조하는 API 계약·타 팀·선행 작업이 명시됨 |
| 크기·추정 | 1스프린트 내로 분할 가능하고 추정이 있음 |
| QA 시나리오 | 테스트 시나리오가 식별됨(기획·개발·QA 3자 공동검토(Three Amigos)) |
| 공유된 이해 | 열린 질문(open question)이 0건 |

판정 규칙:

- 하나라도 FAIL이면 종합 판정은 **보완 후 착수** 또는 **착수 보류**로 시작한다.
- "디자인 참조" 또는 "의존성 식별" FAIL은 FE가 BE 계약/시안을 기다리며 멈추는 병목 직결 신호이므로 우선 표기한다.

### Step 2 — 좋은 스토리 6가지 기준 점수표(INVEST 스코어카드)

유저스토리(또는 기획안의 요구 단위) 품질을 좋은 스토리 6가지 기준(INVEST) 6축으로 채점한다. 각 축 **0~2점**(0=없음/위반, 1=부분, 2=충족), 총 12점 만점.

| 축 | 의미 | 0점 예 | 2점 예 |
| --- | --- | --- | --- |
| **I**ndependent (독립적) | 독립적으로 착수·릴리스 가능 | 다른 미정 스토리에 강결합 | 선행 의존이 명시·해소됨 |
| **N**egotiable (협상가능) | 구현 방식이 강제되지 않음 | 해법까지 못박음 | 무엇/왜 중심, 어떻게는 협상 가능 |
| **V**aluable (가치있음) | 사용자/비즈니스 가치가 분명 | 가치 진술 없음 | so that(왜냐하면) 절에 가치 명시 |
| **E**stimable (추정가능) | 추정 가능 | 정보 부족으로 추정 불가 | 규모 추정에 합의됨 |
| **S**mall (작음) | 스프린트 내 완료 가능 | 에픽 규모 | 분할된 작은 단위 |
| **T**estable (검증가능) | 테스트 가능한 인수조건 존재 | 인수조건 없음/검증 불가 | 조건·행동·결과(Given-When-Then)로 검증 가능 |

**차단 규칙(Blocking):**

- `Testable(검증가능) = 0` 또는 `Independent(독립적) = 0`이면 총점과 무관하게 **착수 차단**으로 판정한다. 테스트할 수 없는 스토리는 완료를 정의할 수 없고, 독립적이지 않은 스토리는 착수 자체가 다른 미정에 묶인다.
- 그 외 총점 가이드: 0~6 차단 / 7~9 보완 후 착수 / 10~12 착수 가능. 단 점수는 신호이며, 차단 규칙과 Step 1 FAIL이 우선한다.

### Step 3 — 인수조건 조건·행동·결과(Given-When-Then) 완결성

각 인수조건(또는 기획안이 암시하는 동작)을 조건·행동·결과(Given-When-Then)로 매핑하고, 흐름 커버리지를 점검한다.

1. **조건·행동·결과 매핑** — 인수조건을 `Given <전제> / When <행동> / Then <관찰 가능한 결과>`로 재구성한다. 재구성이 불가능하면 그 인수조건은 테스트 불가로 표기한다.
2. **흐름 커버리지** — 다음 흐름이 누락되지 않았는지 확인한다.

   | 흐름 | 점검 |
   | --- | --- |
   | 정상 흐름(happy) | 기대 입력 → 기대 결과 |
   | 경계(edge) | 경계값, 최대/최소, 동시성, 중복 제출 |
   | 에러(error) | API 실패, 검증 실패, 타임아웃 시 사용자 피드백 |
   | 빈 상태(empty) | 데이터 0건일 때 화면/문구 |
   | 권한(authz) | 비로그인·권한 없음·만료 상태의 처리 |

3. **예외 흐름 요구** — 정상 흐름만 있는 인수조건은 미완성으로 본다. 빠진 흐름마다 확인 필요 질문을 생성한다(예: 결제 API가 타임아웃되면 사용자에게 무엇을 보여주는가?).

### Step 4 — 모호성 점검(모호한 표현 다듬기)

문장 단위로 약한 표현을 탐지하고 구체적 재작성안을 제시한다. 의도를 알 수 없으면 재작성하지 말고 질문으로 돌린다.

- **약한 단어**: "빠르게", "쉽게", "적절히", "등등", "해야 함", "유연하게", "최적화", "사용자 친화적"
- **대명사 모호**: "그것", "해당", "이 값"이 무엇을 가리키는지 불명확
- **정량자 없는 주장**: "많은", "대부분", "충분히" 처럼 기준 수치가 없는 표현

검색 보조(기획안이 로컬 파일일 때):

```bash
grep -rniE "빠르게|쉽게|적절히|유연하게|사용자 친화|등등|최적화|충분히|대부분|해야 ?함" --include="*.md" --include="*.txt" <기획안경로>
```

각 발견 항목은 Before(모호) → After(명확) 형태로 재작성안을 적는다. 단, **수치를 임의로 지어내지 않는다.** 기준이 필요하면 "기획이 임계값을 지정해야 함"으로 표기한다.

### Step 5 — 의존성 / 계약 참조 점검

이 기획안이 착수되려면 필요한 외부 참조가 명시됐는지 본다. 미명시는 FE가 BE 계약을 기다리며 멈추는 병목으로 이어진다.

| 점검 | 통과 기준 |
| --- | --- |
| API 계약 참조 | 호출할 엔드포인트/스키마(OpenAPI 등)가 합의·링크됨. 미합의면 API 먼저 합의(Contract-First: 선합의 → mock 병렬) 권고 |
| 디자인 링크 | 대상 화면의 시안이 링크되고 상태(빈/에러)별 시안 존재 |
| 타 팀 의존성 | 선행/병행 작업과 담당, 완료 기준이 명시됨 |
| 용어 일관성 | 도메인 용어가 기획안 내부 및 저장소 용어집과 충돌하지 않음 |

### Step 6 — 결과 보고: `# 기획 완성도 점검 결과(DoR Review)`

아래 템플릿으로 통합 보고한다. 점수·PASS/FAIL을 표로 제시하고, 마지막에 **보완할 점 점검표(보강 체크리스트)**와 **기획에 되돌릴 질문 목록**을 둔다.

```markdown

- 평가 대상: 원본 기획안 / (인터뷰 모드 시) 생성 PRD·스토리
- 산출물: <파일 경로 또는 "대화로 제시된 기획안">

## 1. 기획 완성도 게이트(DoR 게이트)

| 점검 항목 | 결과 | 위반/근거 (인용) |
| --- | --- | --- |
| 표준 포맷 | ✅/❌ | "..." |
| 인수조건 측정 가능 | ✅/❌ | "..." |
| 비즈니스 규칙 | ✅/❌ | "..." |
| 예외·에러·권한 흐름 | ❌ | "정상 흐름만 기술, 실패 흐름 없음" |
| 디자인 참조 | ❌ | 시안 링크 없음 |
| 의존성 식별 | ❌ | 결제 API 계약 미참조 |
| 크기·추정 | ✅/❌ | "..." |
| QA 시나리오 | ✅/❌ | "..." |
| 공유된 이해 | ✅/❌ | "열린 질문 N건" |

## 2. 좋은 스토리 6가지 기준 점수표(INVEST 스코어카드)

| 축 | 점수(0~2) | 근거 |
| --- | --- | --- |
| Independent(독립적) | 1 | ... |

| Valuable(가치있음) | 2 | ... |

| Testable(검증가능) | 0 | 인수조건이 검증 불가 → 차단 |
| **총점** | **8 / 12** | Testable=0 → **착수 차단** |

| 인수조건 | 조건·행동·결과(Given-When-Then) | 정상 | 경계 | 에러 | 빈 화면 | 권한 |
| --- | --- | --- | --- | --- | --- | --- |
| AC1 | Given .. When .. Then .. | ✅ | ⚠️ | ❌ | ❌ | ❌ |

## 8. 보완할 점 점검표(보강 체크리스트) — 기획자 공유용

표기 규칙: **충족한 항목은 `[O]`로 한 줄 표기**하고, **아직 안 된 항목은 `[ ]`로 둔 뒤 바로 아래에 `↳ 이렇게 보강:` 한 줄(Before→After 예시)**을 붙인다. 기획자가 `[ ]` 항목과 그 보강 예시만 보면 무엇을·어떻게 채우면 되는지 바로 알 수 있게 한다. (아래는 형식 예시 — 실제 평가 결과의 충족/미충족에 맞춰 `[O]`/`[ ]`를 채운다.)

- [O] 표준 포맷(`…로서 …하고 싶다, 왜냐하면 …`)을 갖췄다
- [ ] 모든 인수조건이 조건·행동·결과(Given-When-Then)로 표현 가능하다
  ↳ 이렇게 보강: "가입이 잘 된다"(판정 불가) → "Given 미사용 이메일로 When 가입하면 Then 3초 내 온보딩 화면으로 이동한다"
- [ ] 정상 흐름 외 경계(edge)·에러(error)·빈 상태(empty)·권한(authz) 흐름이 기술됐다
  ↳ 이렇게 보강: 정상 가입만 적힘 → "이미 가입된 이메일", "형식 오류", "네트워크 실패" 시 각각 무엇을 보여줄지 1줄씩 추가
- [O] 비즈니스 규칙(계산·제약·우선순위)이 문서화됐다
- [ ] 디자인/시안 링크가 있고 상태별 시안이 있다(화면이 있는 기능)
  ↳ 이렇게 보강: 링크 없음 → 기본/빈 화면/에러 상태 시안 URL을 각 화면 옆에 첨부
- [ ] 참조 API 계약이 합의·링크됐다 (없으면 API 먼저 합의(Contract-First) 권고)
  ↳ 이렇게 보강: 미참조 → 호출 엔드포인트·요청/응답 예시를 합의해 링크, 미정이면 "API 먼저 합의 후 mock 병렬" 표기
- [ ] 타 팀/선행 의존성이 담당·완료 기준과 함께 식별됐다
  ↳ 이렇게 보강: "결제팀 협업 필요" → "결제팀 PG 연동 API(담당 ○○○, 완료 기준: 샌드박스 키 발급) — M2까지 필요"
- [O] 1스프린트 단위로 분할·추정됐다
- [ ] QA 테스트 시나리오가 식별됐다(기획·개발·QA 3자 공동검토(Three Amigos))
  ↳ 이렇게 보강: 미식별 → 핵심 인수조건별로 정상 1 + 예외 1 시나리오를 표로 정리
- [O] 열린 질문이 0건이다
- [ ] 좋은 스토리 6가지 기준(INVEST)의 Testable(검증가능)·Independent(독립적)가 모두 1점 이상이다
  ↳ 이렇게 보강: 인수조건이 검증 불가(Testable=0) → 각 인수조건을 조건·행동·결과(Given-When-Then)로 다시 써 관찰 가능하게
- [ ] 약한 단어·모호 대명사·정량자 없는 주장에 재작성안 또는 질문이 달렸다
  ↳ 이렇게 보강: "빠르게 로딩" → "초기 콘텐츠 표시 임계값 지정 필요(기획이 수치 결정)"
```

점검표는 보고서 본문에 그대로 포함해, 기획자가 **`[O]` = 충족 / `[ ]` + 보강 예시 = 보완 대상**으로 즉시 파악하고, 무엇을 어떻게 채우면 되는지 예시로 바로 알 수 있게 한다.

## Honesty Guardrail

이 평가가 출력에 정량 수치를 인용할 때는 반드시 (1) 출처 등급 [GOLD/SILVER/BRONZE], (2) 출처명/링크, (3) 가능하면 반대증거를 함께 표기한다.

- 등급 정의: **GOLD** = 피어리뷰·정부·DORA 등 대규모 연구 / **SILVER** = 방법론이 공개된 업계 리포트(벤더 자기보고 포함) / **BRONZE** = 벤더 블로그·일화.
- **개선 N%를 약속하지 않는다.** 검증된 소수 수치만 인용하고, 목표 수치는 조직이 직접 baseline을 측정해 정한다(baseline-before-target). 이 평가는 "기획 완성도 점검(DoR)을 도입하면 재작업 N% 감소" 같은 약속을 하지 않는다.
- 다음 수치는 출처 추적 실패 또는 반증되었으므로 **사실로 인용 금지**(신화로만 언급): IBM '결함비용 1→100배', Crosstalk '결함비용의 64%가 요구·설계'. '늦게 고치면 100배' 류의 정확한 지연 배수도 약속하지 않는다 — 171개 프로젝트 분석에서 일관된 지연효과가 확인되지 않았다(Menzies et al. 2017 [GOLD]).
- Standish CHAOS의 '불완전 요구사항이 실패요인 12.3%'는 수치 자체는 1994 원본에 실재하나 정의·방법론이 강하게 비판받았으므로(Eveleens & Verhoef, IEEE Software 2010 — "misleading, one-sided … meaningless") **단정 인용을 피하고 '논쟁적 통념'으로만** 표기한다. 본문 주장은 NaPiRE(2017)·Boehm & Basili(2001)로 떠받친다.
- LLM/NLP로 유저스토리 품질 결함을 탐지할 수 있다는 것은 근거가 있으나(룰 기반 AQUSA: 재현율 97.9% / 정밀도 84.8% — Lucassen et al. RE 2016 [GOLD]; 단 AQUSA는 NLP 룰 기반이며 LLM이 아니고, 정밀도는 데이터셋에 의존해 다른 산업 데이터셋에선 0.61로 보고됨 — Perkusich et al. 2025 [SILVER]), '>90% 정밀도'를 일반값으로 인용하지 않는다. 최신 LLM은 멀쩡한 산출물을 과대탐지하는 경향이 있으므로, 이 평가의 자동 탐지는 사람의 검토를 대체하지 않고 보조한다.

## 참고 / 근거

수치를 인용할 때는 효과크기가 아니라 정성근거로만 쓴다(Honesty Guardrail). 표본·자기보고·preprint 한계를 함께 적는다.

**2025+ 근거 (LLM 시대의 요구사항·유저스토리 품질)**

- LLM in-context learning(10-shot)이 모호 요구사항 분류 성능을 0-shot 대비 평균 20.2% 향상시키고 설명 품질이 전문가 평가 3.84/5에 이르나, 도메인 특화 용어 이해 부족이 한계로 지적됨(철도 산업 3개 데이터셋) — Bashir, Ferrari et al., "Requirements Ambiguity Detection and Explanation with LLMs: An Industrial Study", ICSME 2025 (Industry Track) [GOLD].
- 룰 기반 AQUSA와 GPT-5·GPT-5-mini·GPT-4 비교(산업 3개 데이터셋·유저스토리 182건): GPT-5-mini가 최고 재현율(0.81)·F1(0.62), AQUSA가 최고 정밀도(0.61)·현저히 적은 false positive. GPT-5는 환각·불안정성, GPT-4는 과소탐지 → 구조검증(룰)+경량 LLM 맥락보강의 'Dual-gate' 권고 — Perkusich et al., "Evaluating the Quality of User Stories: An Extended Comparative Study of Multiple LLMs and Rule-Based Tools", Springer, 2025 [SILVER, 표본 182·소중규모]. DOI 10.1007/978-3-032-22375-3_10
- '무엇이 결함인가'는 프로젝트·도메인·이해관계자 해석에 따라 달라지는 맥락 의존 개념이며, 사람이 검증한 소수 예시(20건)를 되먹이는 Human-LLM 협업이 표준 few-shot·fine-tuned BERT를 능가(Mercedes-Benz 요구사항 1,266건) — Unterbusch & Vogelsang et al., "Context-Adaptive Requirements Defect Prediction through Human-LLM Collaboration", arXiv:2601.01952, ICSE-NIER 2026 [SILVER, preprint·NIER·단일 기업].

**고전 기반 (정성·방향 참고, 효과크기 아님)**

- 불완전·은닉 요구사항이 요구공학(RE)에서 가장 많이 지목된 문제(응답자 228명 중 109명=48%) — Méndez Fernández et al., "Naming the Pain in Requirements Engineering", Empirical Software Engineering, vol.22(5) pp.2298–2338, 2017 [GOLD]. DOI 10.1007/s10664-016-9451-7
- 회피 가능한 재작업이 프로젝트 노력의 약 40~50%를 차지하며, 재작업의 약 80%가 결함의 약 20%에서 비롯됨 — Boehm & Basili, "Software Defect Reduction Top 10 List", IEEE Computer 34(1):135–137, 2001 [GOLD]. DOI 10.1109/2.962984
- 결함을 늦게 고칠수록 비싸진다는 통설의 정확한 배수는 일반화할 수 없음(171개 프로젝트에서 일관된 지연효과 미확인) — Menzies et al., "Are Delayed Issues Harder to Resolve?", Empirical Software Engineering, 2017 [GOLD]. DOI 10.1007/s10664-016-9469-x
- 룰 기반 NLP 도구 AQUSA가 유저스토리 품질결함을 탐지(QUS 프레임워크, 18개 회사 1,023개 스토리, 재현율 97.9%/정밀도 84.8%) — Lucassen et al., "Improving agile requirements: the Quality User Story framework and tool", Requirements Engineering 21(3):383–403, 2016 [GOLD]. ※ AQUSA는 LLM이 아니라 NLP 룰 기반이며, 정밀도는 데이터셋 의존(다른 데이터셋 0.61).
- 기획 완성도 점검(DoR)/좋은 스토리 6가지 기준(INVEST)/조건·행동·결과(Given-When-Then) 정의 — Bill Wake, "INVEST in Good Stories, and SMART Tasks"(2003); North, "Introducing BDD"(2006). [BRONZE, 정의·관행 출처]
