# product-spec-harness

기획자(PM)가 요구·문제로부터 **제품 기획문서(PRD)와 사용자 스토리**를 단계별로 작성·검증하도록 돕는 도메인 무관 인터랙티브 멀티 에이전트 하네스입니다. 개발 착수 *전* 단계의 제품 기획 산출물에 특화합니다.

매 Phase 산출물을 보여주고 사용자 승인을 받은 뒤에만 다음 단계로 넘어갑니다. 한 번에 한 가지씩 인터뷰하듯 질문하며, 전문 용어에는 1줄 정의를 곁들입니다.

## 설치

이 저장소를 Claude Code 플러그인 마켓플레이스로 추가한 뒤 `product-spec-harness` 플러그인을 활성화하면, `product-spec` 스킬이 자동으로 트리거되거나 직접 호출할 수 있습니다.

## 스킬

| 스킬 | 역할 |
|------|------|
| `product-spec` | 오케스트레이터(진입점). 문제 정의 → PRD 작성 → 사용자 스토리 → 적대적 검증의 4단계를 인터랙티브로 진행하며, 각 단계에서 전용 에이전트(requirements-analyst / prd-writer / story-writer / spec-reviewer)를 호출한다. |

## 언제 쓰나 / 언제 다른 도구를 쓰나

**이 하네스를 쓰세요**
- 새 제품·기능의 문제와 요구를 정리해 **기획문서(PRD)**를 만들고 싶을 때
- PRD에서 **사용자 스토리와 수용기준(Given/When/Then)**을 도출하고 싶을 때
- 기획 산출물의 누락·모호·비일관을 **적대적으로 검증**하고 싶을 때
- 개발 착수 *전*, 도메인 무관한 제품 기획 단계일 때

**다른 도구를 쓰세요**
- 프론트엔드 개발용 PRD·구현 요구사항·코드 작성(planner→architecture→critic→코드) → `frontend-harness`
- 코드 리뷰·커밋 메시지·PR 리뷰 → `git-harness`
- 하네스 자체를 생성·진단·개선 → `harness-generator` / `meta-harness`

이 하네스는 코드/하네스/커밋 작업을 수행하지 않습니다. 트리거 충돌을 막기 위해 이 경계는 plugin.json description과 `evals/trigger-eval.json`에 명시되어 있습니다.

## 진행 방식 (인터랙티브 4단계)

| Phase | 단계 | 핵심 산출물 |
|-------|------|-------------|
| 0 | 요구/문제 정의 (Discovery) | 문제 정의 카드 (problem / target_users / goals / constraints / success_metrics) |
| 1 | 기획문서(PRD) 작성 | 배경·문제 / 목표·성공지표 / 범위(In·Out) / 핵심 요구사항(기능·비기능) / 가정·리스크 / 마일스톤 |
| 2 | 사용자 스토리 도출 | "…로서 …하고 싶다, 왜냐하면 …" 스토리 + 수용기준(Given/When/Then) + INVEST 자가점검 |
| 3 | 검증 | 요구↔스토리 추적 매트릭스 / INVEST 통과 / 수용기준 관찰성 / 일관성 / 모호·판정불가 색출 |

각 Phase 종료 시 `[Phase N] {핵심결정} — 다음: {다음}. 진행할까요?` 형식으로 1줄 보고하고, 파일을 쓰기 전 미리보기로 승인을 받습니다.

## 산출물

- **제품 기획문서(PRD)** — 배경·목표·범위·요구사항·리스크·마일스톤을 담은 한국어 문서
- **사용자 스토리 세트** — 관찰형 수용기준(Given/When/Then)과 INVEST 점검을 포함
- **검증 리포트** — 완전성·일관성·관찰성·모호성 점검 결과 (보완 시 additive-first)

## evals

`evals/trigger-eval.json`은 이 하네스가 발동해야 하는 경우(should-trigger)와 발동하면 안 되는 경우(should-not-trigger, 예: 프론트엔드 개발 PRD·코드·커밋)를 정의해 트리거 정확도를 점검합니다.
