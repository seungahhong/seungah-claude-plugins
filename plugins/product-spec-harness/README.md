# product-spec-harness

기획자(PM)가 요구·문제(또는 **기획안 초안**)로부터 **제품 기획문서(PRD)와 사용자 스토리**를 단계별로 작성·검증하도록 돕는 도메인 무관 인터랙티브 멀티 에이전트 하네스입니다. 개발 착수 *전* 단계의 제품 기획 산출물에 특화합니다.

매 Phase 산출물을 보여주고 사용자 승인을 받은 뒤에만 다음 단계로 넘어갑니다. 한 번에 한 가지씩 인터뷰하듯 질문하며, 전문 용어에는 1줄 정의를 곁들입니다.

## 입력 모드

- **기획안 입력 모드** — 기획안(초안)을 인자/대화로 주면, 그 내용에서 문제 정의를 **추출**합니다. 이어 **생성 *전*에 원본 기획안의 착수 준비도(DoR)를 먼저 평가**(동의 시)해 `# DoR Review 결과` + 보강 체크리스트로 무엇을 고치면 좋을지 알려주고, 그 보강점을 반영해 PRD·사용자 스토리를 **반드시 생성**합니다(초안을 줬다고 생성을 건너뛰지 않습니다). 마지막에 산출물을 적대적 검증합니다. **DoR 평가 결과 파일(`product-spec-review.md`)은 PRD·스토리를 만든 뒤 저장할지 물어, 저장을 선택한 경우에만** 기록합니다.
- **대화형 인터뷰 모드** — 기획안이 없으면 한 번에 한 질문으로 문제를 정의한 뒤 PRD·스토리를 만듭니다(입력 기획안이 없어 DoR 평가는 건너뛰고, 필요 시 마무리에서 생성 산출물 대상으로 선택 적용).

## 설치

이 저장소를 Claude Code 플러그인 마켓플레이스로 추가한 뒤 `product-spec-harness` 플러그인을 활성화하면, `product-spec` 스킬이 자동으로 트리거되거나 직접 호출할 수 있습니다.

## 스킬

| 스킬 | 역할 |
|------|------|
| `product-spec` | 오케스트레이터(진입점). 문제 정의 → 기획안 DoR 평가(생성 전) → PRD 작성 → 사용자 스토리 → 적대적 검증의 5단계를 인터랙티브로 진행하며, 각 단계에서 전용 에이전트(requirements-analyst / dor-evaluator / prd-writer / story-writer / spec-reviewer)를 호출한다. |

## 언제 쓰나 / 언제 다른 도구를 쓰나

**이 하네스를 쓰세요**
- 새 제품·기능의 문제와 요구를 정리해 **기획문서(PRD)**를 만들고 싶을 때
- **기획안(초안)을 주고** 거기서 PRD·사용자 스토리를 뽑고 싶을 때
- PRD에서 **사용자 스토리와 수용기준(Given/When/Then)**을 도출하고 싶을 때
- 기획 산출물의 누락·모호·비일관을 **적대적으로 검증**하고 싶을 때
- 생성 전에 **원본 기획안의 착수 준비도(DoR)를 평가**해 무엇을 보강할지 체크리스트로 받고, 그 피드백을 PRD·스토리에 반영하고 싶을 때
- 개발 착수 *전*, 도메인 무관한 제품 기획 단계일 때

**이 하네스가 다루지 않는 것 (범위 밖)**
- 프론트엔드 화면 구현·컴포넌트·기술 설계용 PRD·구현 요구사항·코드 작성
- 코드 리뷰·커밋 메시지·PR 리뷰
- 하네스(에이전트팀·스킬·오케스트레이터) 자체를 생성·진단·개선하는 작업
- **이미 완성된** PRD/유저스토리/디자인/계약/인수조건을 (작성이 아니라) 핸드오프 시점에 독립 게이트로 검수만 하는 작업

이 하네스는 개발 착수 *전* 제품 기획문서·사용자 스토리 **작성**과 입력 기획안의 **DoR 평가**에 한정합니다(위 작업은 수행하지 않음). 트리거 충돌을 막기 위해 이 경계는 plugin.json description과 `evals/trigger-eval.json`에 명시되어 있습니다.

> **독립 플러그인**: DoR 평가는 외부 플러그인에 의존하지 않습니다. 방법론(DoR 게이트·INVEST 스코어카드·GWT 완결성·모호성 린트·의존성 참조 + Honesty Guardrail·근거)을 이 플러그인 안(`skills/product-spec/references/dor-review-rubric.md`)에 **내재화**해, 이 플러그인만 설치해도 단독으로 동작합니다. 여기서는 *작성 흐름의 생성 전 선행 평가*로 입력 기획안을 평가해 PRD·스토리에 반영합니다.

## 진행 방식 (인터랙티브 5단계)

| Phase | 단계 | 핵심 산출물 |
|-------|------|-------------|
| 0 | 요구/문제 정의 (Discovery) | 문제 정의 카드 (problem / target_users / goals / constraints / success_metrics) — 기획안 있으면 추출 |
| 1 | 기획안 DoR 평가 (생성 전·조건부) | 원본 기획안의 `# DoR Review 결과` (DoR 게이트 · INVEST 스코어카드 · GWT 완결성 · 모호성 린트 · 의존성) + **보강 체크리스트** — **채팅 제시**, 보강점을 이후 PRD·스토리에 반영 |
| 2 | 기획문서(PRD) 작성 | 배경·문제 / 목표·성공지표 / 범위(In·Out) / 핵심 요구사항(기능·비기능) / 가정·리스크 / 마일스톤 (Phase 1 보강점 반영) |
| 3 | 사용자 스토리 도출 | "…로서 …하고 싶다, 왜냐하면 …" 스토리 + 수용기준(Given/When/Then) + INVEST 자가점검 (Phase 1 보강점 반영) |
| 4 | 적대적 검증 | 요구↔스토리 추적 매트릭스 / INVEST 통과 / 수용기준 관찰성 / 일관성 / 모호·판정불가 색출 (**채팅 표시만, 파일 저장 안 함**) |
| 마무리 | 기획안 DoR 평가 결과 저장 (opt-in) | Phase 1 평가 결과 → `product-spec-review.md` (**PRD·스토리 생성 후 사용자가 저장을 선택할 때만**) |

각 Phase 종료 시 `[Phase N] {핵심결정} — 다음: {다음}. 진행할까요?` 형식으로 1줄 보고하고, 파일을 쓰기 전 미리보기로 승인을 받습니다. Phase 1(기획안 DoR 평가)은 사용자가 동의할 때만 진행하는 조건부 단계이며, 그 결과의 파일 저장도 PRD·스토리 생성 후 별도 동의가 있을 때만 합니다.

## 사용 예시 (기획안 입력 모드)

거친 기획 초안을 인자로 주고 PRD·사용자 스토리를 만드는 전형적인 흐름입니다. **DoR 평가(Phase 1)가 입력의 빈틈을 생성 *전*에 먼저 잡아, 그 보강점이 PRD·스토리에 반영되는 점**에 주목하세요. (아래는 진행 예시를 요약한 것입니다.)

**입력** — 초안을 붙여넣어 호출:

> "신규 사용자가 이메일로 가입하게 해줘. 가입이 **빠르고 쉬웠으면** 좋겠고, 가입 후 바로 서비스를 둘러볼 수 있게. 이걸로 PRD랑 사용자 스토리 만들어줘."

**Phase 0 — 요구/문제 정의 (초안에서 카드 추출)**

```
## 문제 정의 카드
- problem: 가입 절차가 번거로워 신규 사용자가 첫 경험 전에 이탈
- target_users: 결제 전 제품을 둘러보려는 신규 방문자
- goals: (비즈니스) 가입 전환율 향상 / (사용자) 부담 없는 빠른 가입
- constraints: (미정 — 확인 필요)
- success_metrics: (미정 — 관찰형으로 확인 필요)
❓ 성공을 어떻게 측정할까요? 예: "가입 시작→완료 전환율"
```
`[Phase 0] 문제·대상·목표·성공지표 합의 — 다음: 기획안 DoR 평가(모드 A) / PRD 작성(모드 B). 진행할까요?`

**Phase 1 — 기획안 DoR 평가 ("예" 선택 · 채팅 제시, 파일 저장 안 함)**

원본 초안을 DoR로 진단해 무엇이 빠졌는지 체크리스트로 공유합니다(이 보강점을 다음 단계에서 채웁니다).

```
# DoR Review 결과
## 1. DoR 게이트
| 항목 | 결과 | 근거(인용) |
| 인수조건 측정 가능 | ❌ | "빠르고 쉽게" → 제3자 판정 불가 |
| 엣지/에러/권한    | ❌ | 중복 이메일·형식 오류·네트워크 실패 경로 없음 |
## 8. 보강 체크리스트
- [x] 표준 포맷(누가·무엇을·왜)을 갖췄다
- [ ] 정상경로 외 엣지/에러/빈/권한 경로가 기술됐다
- [ ] 약한 단어("빠르게/쉽게")에 관찰형 기준이 달렸다
- [ ] 관찰형 성공지표가 있다
```
`[Phase 1] 기획안 DoR 평가(보완 후 / 보강 3건, 채팅 제시) — 다음: PRD 작성(보강점 반영). 진행할까요?`

**Phase 2 — PRD (Phase 1 보강점 반영)**

```
## 2. 목표·성공지표(관찰형)
- 가입 시작→완료 전환율 42% → 60% (출시 4주, 퍼널 기준)   ← "빠르게/쉽게" 대체
## 4. 핵심 요구사항
- FR-1 이메일+비밀번호로 가입할 수 있다
- FR-2 이미 가입된 이메일·형식 오류 시 안내한다            ← DoR 에러 경로 반영
- NFR-1 가입 API 응답 p95 ≤ 500ms
```
`[Phase 2] PRD 합의(범위 In/Out·핵심 요구 R1…Rn) — 다음: 사용자 스토리 도출. 진행할까요?`

**Phase 3 — 사용자 스토리 (음성 경로 포함)**

```
### US-1 (covers FR-1, FR-2)
신규 가입자로서 이메일로 가입하고 싶다, 왜냐하면 결제 전 부담 없이 둘러보고 싶기 때문이다.
- Given 미가입 방문자가 가입 화면에서 When 미사용 이메일+8자 비밀번호로 가입하면 Then 온보딩으로 이동한다
- Given 이미 가입된 이메일로 When 가입 시도하면 Then "이미 가입된 이메일" 안내가 뜨고 진행되지 않는다   ← DoR 반영
```
`[Phase 3] 스토리 3개 + 수용기준(G/W/T)·요구 커버 표 — 다음: 적대적 검증. 진행할까요?`

**Phase 4 — 적대적 검증 (생성 산출물 대상 · 채팅만, 파일 저장 안 함)**

```
요구↔스토리 추적: FR-1 ✅US-1 / FR-2 ✅US-1 / NFR-1 ❌(미커버)
결함: NFR-1(응답속도)을 검증할 스토리 없음 → additive로 US-3 추가 제안
```
`[Phase 4] 적대적 검증 완료(결함 1건 / 추적 커버율 90%) — 마무리할까요?`
→ additive-first로 US-3을 추가하고 `PRD.md`·`user-stories.md`만 갱신(검증 리포트 자체는 저장 안 함).

**마무리 — DoR 평가 결과 저장 (PRD·스토리 생성 후 · opt-in)**

`[저장 확인] Phase 1의 기획안 DoR 평가 결과(# DoR Review 결과 + 보강 체크리스트)를 product-spec-review.md로 저장할까요? (예/아니오)`

"예"를 선택한 경우에만 저장합니다. 최종 산출물:

```
_workspace/
  PRD.md
  user-stories.md
  product-spec-review.md   # 저장을 선택한 경우에만
```

> **대화형 인터뷰 모드(B)** 는 입력 기획안이 없으므로 Phase 1을 건너뛰고, 한 번에 한 질문으로 카드를 채운 뒤 PRD·스토리를 만듭니다. 필요하면 마무리에서 *생성된* PRD·스토리를 대상으로 DoR 평가·저장을 선택할 수 있습니다.

## 산출물

- **제품 기획문서(PRD)** — 배경·목표·범위·요구사항·리스크·마일스톤을 담은 한국어 문서 (`PRD.md`)
- **사용자 스토리 세트** — 관찰형 수용기준(Given/When/Then)과 INVEST 점검을 포함 (`user-stories.md`)
- **기획안 DoR 평가 리포트** — `# DoR Review 결과` + 보강 체크리스트(미체크 = 보강 대상). Phase 1 평가를 진행하고 **마무리에서 저장을 선택한 경우에만** `product-spec-review.md`로 저장 (그 전엔 채팅 제시).
- **적대적 검증 리포트** — 완전성·일관성·관찰성·모호성 점검 결과. **파일로 저장하지 않고 채팅으로만** 제시하며, 결함은 additive-first로 PRD·스토리에 반영됩니다.

## evals

`evals/trigger-eval.json`은 이 하네스가 발동해야 하는 경우(should-trigger)와 발동하면 안 되는 경우(should-not-trigger, 예: 프론트엔드 개발 PRD·코드·커밋)를 정의해 트리거 정확도를 점검합니다.
