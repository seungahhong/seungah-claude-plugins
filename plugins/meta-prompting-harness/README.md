# Meta-Prompting Harness

> 다른 도구에 넣을 **프롬프트를 AI가 최적화해 만들게** 하고, 그 프롬프트를 새 컨텍스트에 주입해 실제 작업을 돌리는 **메타 프롬프팅**을 4단계 인터랙티브로 수행하는 하네스.

## 메타 프롬프팅이란

"X 만들어줘"라고 직접 지시하는 대신 → **"X를 만들 좋은 프롬프트를 최적화해 만들어줘"** 로 AI에게 프롬프트를 짓게 하고, 그 결과 프롬프트를 **새 창(fresh context)에 주입**해 실제 작업을 돌린다. 며칠 도는 Goal/loop, 많은 토큰을 태우는 긴 작업에서 **처음부터 명확한 프롬프트**를 넣어 시간·토큰 낭비를 막는 게 목적이다.

이 하네스는 널리 쓰이는 메타 프롬프팅 실천 방법론을, 각 기법을 `/deep-research`로 2023~2026 1차 문헌과 대조해 **근거 등급**을 붙여 재구성했다.

## 언제 쓰나

- "이거 넣을 프롬프트 좋게 깎아줘", "메타 프롬프트 만들어줘"
- Goal/loop·ultracode·Claude Code·이미지 생성·리서치에 넣을 **지시 프롬프트**를 최적화하고 싶을 때
- 긴 작업을 처음부터 명확한 프롬프트로 시작하고 싶을 때

## 어떻게 동작하나 (4 Phase, 매 단계 승인)

| Phase | 하는 일 | 에이전트 |
|-------|---------|----------|
| **0 인테이크** | 목표 + 실행 환경을 **한 번에 한 질문씩** 확정. 회의록·메모 있으면 정제 없이 붙여넣기(선택·강요 안 함) | (스킬 본문) |
| **1 질문 유도** | 빠진 요소(타겟·기능·완성도)를 AI가 **한 번에 하나씩** 되묻기(번호 선택지·답변 분기·한꺼번에 안 쏟음), 전부 위임 방지 | context-elicitor |
| **2 성공·검증 조건** | specific 성공 + **외부 기준 대비 검증**("모든 버튼·폼 동작 확인") | success-definer |
| **3 환경 맞춤 깎기** | 환경 제약·길이 상한 반영, 큰그림→축소로 **최종 프롬프트** 산출 | prompt-carver |
| **4 점검·주입** | 모호·과대·미검증 적대 점검 → 사용자 도메인 수정 → **fresh 창 주입 안내** | prompt-critic |

산출물(최종 프롬프트)은 기본적으로 채팅에 제시되며, 원하면 `.claude/_docs/<슬러그>/`에 저장한다. **이 하네스는 프롬프트를 만들 뿐, 실행(새 창 주입·`/goal`·이미지 생성)은 사용자가 한다.**

## 근거 등급 (정직성)

각 기법을 실증 강도로 정직하게 구분한다(상세: [evidence.md](skills/meta-prompting/references/evidence.md)).

- **STRONG (1차 근거·단 조건부)**
  - 메타프롬프팅이 직접 프롬프팅 상회 — Suzgun&Kalai(2401.12954)·OPRO(2309.03409)·PromptAgent(2310.16427). *단, 모델·과제 조건부이고 전이 불안정.*
  - 생성 전 명확화 질문이 정확도 상승 — ClarifyGPT(FSE 2024)·Orchid(2604.21505). *단, 명확화는 모델 의존 역량이라 명시 유도 필요.*
  - 성공+검증 조건 명시 — Anthropic 공식 가이드. *외부 기준 대비 검증(순진한 자기수정 아님).*
  - fresh 컨텍스트 주입 — Lost in the Middle(2307.03172)·Chroma Context Rot. *방향성 추론.*
  - 은탄환 아님 — Anthropic 공식.
- **report-only (folklore·근거 약함)** — 실행 환경 맞춤 변환·큰그림→축소. 실무 관행으로만 제시, **"실증됐다"고 말하지 않음**.
- **REFUTED (미사용)** — "메타프롬프팅=지휘자가 전문가 인스턴스 생성" 프레이밍·"LLM 스스로 명확화 불가".

**절대 규칙**: 최적화 프롬프트를 보편 법칙으로 말하지 않음 · **전부 위임 금지** · AI가 깎은 프롬프트는 **사용자 도메인 검증 필수** · 프롬프트 품질을 점수화하지 않음 · 커밋 안 함.

## 경계

다음은 산출물이 **프롬프트가 아니므로** 범위 밖이다:

- **컨텍스트 페이로드 조립·압축·리트리벌** — payload vs 지시 프롬프트.
- **하네스/스킬 자체 생성·진단·개선** — 하네스 자산 vs 프롬프트.
- **PRD·사용자 스토리 작성** — 문서 vs 프롬프트.
- **계획/방법론 정렬 인터뷰** — 최종 산출물이 프롬프트가 아님.
- **자율 반복 루프 실행** — 이 하네스는 그 루프에 넣을 프롬프트만 만들고 돌리지 않음.
- **코드 구현·리뷰·커밋** — 프롬프트만 만들고 실행하지 않음.

고유 판별자: *다른 도구에 주입할 지시 프롬프트 자체*를 질문 유도·성공조건·환경 맞춤으로 깎아냄. 다른 플러그인을 참조하거나 의존하지 않고 자족하는 독립 플러그인.

## 예시

```
> CRM 만들 프롬프트를 Goal에 넣게 메타프롬프팅으로 깎아줘

[Phase 0] 무엇을 만들 프롬프트인가요? → (답변 뒤) 어느 환경에 넣나요? → Goal/loop 확정. 메모 있으면 붙여넣어도 되고, 없으면 다음 단계에서 하나씩 여쭤볼게요.
[Phase 1] (context-elicitor) 한 번에 한 질문씩 — 타겟 업종? → (답변) → 목적? → (답변) → 사용자? … 직전 답이 다음 질문을 결정. (일부는 "네가 정해줘"로 넘겨도 됩니다)
[Phase 2] (success-definer) 성공 조건 + "모든 폼·버튼 동작 확인" 검증.
[Phase 3] (prompt-carver) Goal 제약(중지 조건·길이 상한) 반영, 큰그림→축소 → 최종 프롬프트.
[Phase 4] (prompt-critic) 점검 → 리센드 말고 다른 도구로? 다크모드로? 수정 → 새 창에 /goal 로 주입.
```
