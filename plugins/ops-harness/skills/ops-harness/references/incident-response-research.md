## 1. AIOpsLab — 인시던트 생애주기 4단계 + 텔레메트리 substrate

**주장**: 인시던트 대응은 4단계로 분해된다 — **Detection(L1) → Localization(L2) → Root Cause Analysis(L3) → Mitigation(L4)**.
평가 환경은 **fault 주입 오케스트레이터**가 실제 장애를 주입하고, 에이전트는 **중재 API 액션**(get logs / get metrics / get traces / exec shell)으로만 시스템에 접근한다.
텔레메트리 substrate는 **traces(Jaeger) + logs(Filebeat/Logstash) + metrics(Prometheus)** 3종이다.

**정량(등급·CAVEAT 동반 인용)**: SOTA 에이전트도 약하다 — 최고 에이전트(Flash)의 전체 정확도 **59.32%**, mitigation 성공률은 **0%(GPT-3.5)~54.55%(Flash)** 범위.

- 출처: **arXiv:2501.06706 (AIOpsLab)** + Microsoft Research 블로그(2024-12 ~ 2025-01).
- 등급: **[GOLD/SILVER]** (논문=GOLD 성격, 블로그 해설=SILVER).
- **CAVEAT**: 수치는 특정 벤치마크·모델 세대 기준의 스냅샷이다. "59.32%/0~54.55%"는 *운영 자율화가 미성숙·고난도 영역*임을 보이는 존재 증거로 인용하며, 다른 환경의 성능 보장이 아니다.
- **설계 함의**: 4단계 파이프라인을 오케스트레이션 골격으로 차용. 텔레메트리 3종 + 중재 읽기 액션을 substrate로 가정. SOTA가 약하므로 *완전 자율이 아니라 휴먼-인-더-루프*를 유지한다(완화는 제안·사람 집행).

## 2. 인시던트 멀티에이전트 오케스트레이션 — 역할 분리 vs 단일 에이전트

**주장**: 인시던트 대응을 **진단/조치/위험으로 역할 분리**한 멀티에이전트 오케스트레이션이, 단일 에이전트보다 *실행 가능한* 결정론적 권고를 낸다.
완화·권고 품질은 **DQ = 0.40·타당성(Validity) + 0.30·구체성(Specificity) + 0.30·정확성(Correctness)** 루브릭으로 평가한다.

**정량(강한 CAVEAT 동반 인용)**: 단일 에이전트 실행가능 권고율 **1.7%** vs 역할 분리 **100%**. 두 구조의 지연은 모두 **~40초**(→ 가치는 속도가 아니라 결정론적 품질).

- 출처: **arXiv:2511.15755 (2025-11)**.
- 등급: **[SILVER]** (방법론 공개 preprint).
- **CAVEAT(필수 표기)**: 100% / 1.7%는 **단일 인증서비스(auth) 시나리오**의, **human-validation이 없는 알고리즘 산출치**다. → **일반화 금지.** 두 수치는 반드시 이 한계와 함께만 인용하며, "역할 분리가 N배 낫다"는 보편 주장으로 쓰지 않는다. 채택하는 것은 *정성적 방향*(역할 분리 → 더 실행 가능·결정론적)과 *DQ 루브릭*이다.
- **설계 함의**: 진단(L1–L3)과 조치계획·위험평가(L4)를 다른 에이전트로 분리. DQ 루브릭을 완화 품질 게이트로 차용.

## 3. 클라우드 RCA 추론 실패모드 — anti-anchoring & Straight-Shot 폴백

**주장**: 클라우드 RCA에서 네 가지 추론 실패모드가 정확도 저하와 **연관**된다 — **anchoring(첫 가설 고착) · 반복 정체(stalled) · 임의적 증거 선택 · 신념 미갱신.**
또한 **agentic 다단계 추론(ReAct / Plan-Execute)이 RCA에서 수확체감·음효과를 자주 보이며**, 비-agentic **Straight-Shot(단일 직답)**이 동등 이상인 경우가 잦다 — *특히 작은 모델에서*.

**정량(연관≠인과, 등급·CAVEAT 동반 인용)**: 네 실패모드는 각각 정확도 **최소 15%p 하락 / 정답 확률 최소 45% 감소**와 *연관*. 작은 모델(Llama 3.2)에서 agentic은 LA@3 **0.31 → 0.05**로 붕괴.

- 출처: **arXiv:2601.22208 (2026)**.
- 등급: **[SILVER]** (방법론 공개 preprint).
- **CAVEAT**: "15%p / 45%"는 **연관(correlation) 측정치이지 인과(causal)가 아니다** — "이 실패모드를 없애면 정확도가 그만큼 오른다"로 읽지 않는다. Llama 3.2의 0.31→0.05는 *작은 모델* 한정 관찰이다. 채택하는 것은 *방향*(이 실패모드를 경계하라)과 *Straight-Shot 폴백의 정당성*이다.
- **설계 함의**: RCA 에이전트에 anti-anchoring·반증 우선·정체 감지·신념 갱신 가드레일을 1급 원칙으로 내장. 단순 케이스/작은 모델엔 다단계를 강요하지 않는 **Straight-Shot 폴백**을 둔다(over-agentic anti-pattern 차단).

## 5. caveats (신뢰도 한계 — 본문에 함께 인용)

1. **AIOpsLab 수치는 모델 세대 스냅샷** — 59.32%/0~54.55%는 *미성숙 영역* 존재 증거이지 성능 보장이 아니다.
2. **역할분리 100%/1.7%는 단일 시나리오·human-validation 부재 알고리즘 산출치** — 일반화 금지, 한계 동반 인용 필수. 채택은 정성 방향 + DQ 루브릭.
3. **RCA 실패모드 15%p/45%는 연관이지 인과 아님** — "제거하면 그만큼 개선"으로 읽지 않는다. Llama 3.2 붕괴는 작은 모델 한정.
4. **세 근거 모두 특정 벤치마크/시나리오 기반** — 운영 환경마다 신호 가용성·규모·도메인이 다르다. 본 하네스는 substrate 가용성(특히 traces 부재)을 Phase 0에서 명시 점검하고 그에 따라 신뢰도를 낮춘다.
5. **'개선 N%' 약속 없음** — 본 하네스는 *진단·완화 제안의 구조와 정직성*을 제공하며, 인시던트 해결률 향상을 수치로 약속하지 않는다(baseline-before-target).

## 6. open questions (후속)

- traces가 없는(로그만) 환경에서 국소화 정밀도 저하를 보완할 검증된 대체 전략.
- Straight-Shot vs 다단계 RCA를 가르는 운용 결정규칙(케이스 단순성의 조작적 정의) — 현 근거는 방향만 제시.
- mitigation DQ 임계값의 캘리브레이션(어느 점수 미만을 보강 트리거로 둘지) — 근거 미제시, 운영 중 튜닝 대상.
