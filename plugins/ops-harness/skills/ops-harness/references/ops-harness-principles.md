# Ops Harness — 원리 · 단계 분해 · DQ/가드레일 설계 · anti-pattern

> 이 문서는 ops-harness 하네스가 따르는 설계 원리의 근거다. 오케스트레이터(SKILL.md)와 각 에이전트가 참조한다.
> 정량 수치는 [incident-response-research.md](./incident-response-research.md)의 등급·출처·CAVEAT와 함께만 인용한다.

## 1. 무엇이 ops-harness인가

개발 워크플로우는 보통 *코드를 배포하면* 끝나지만, 시스템의 수명 대부분은 그 *이후*의 운영이다.
ops-harness는 개발 하류의 **'운영' 단계** — 이미 배포돼 돌아가는 프로덕션 시스템의 이상을 **탐지·국소화·원인분석·완화**하는 것을 담당한다.

핵심 전제는 두 가지다.

1. **운영은 텔레메트리에서 시작한다** — 코드 diff가 아니라 *런타임 신호*(traces+logs+metrics)가 1차 증거다.
2. **운영 자율화는 아직 미성숙·고난도 영역이다** — SOTA 에이전트조차 인시던트 전 과정을 자율로 끝내지 못한다(§research). 따라서 이 하네스는 *완전 자율*이 아니라 *휴먼-인-더-루프 보조*로 설계한다 — 관측·진단·완화 *제안*까지가 에이전트의 일이고, 인프라 변경 집행은 사람이 한다.

## 2. 텔레메트리 substrate — 세 종류의 신호

로그만으로는 분산 시스템의 인시던트를 국소화하기 어렵다. 이 하네스는 **traces+logs+metrics 3종**을 가정한다.

- **traces**(예: Jaeger) — 요청이 서비스 경계를 넘는 경로. 오류/지연이 *어느 hop에서 발원*하는지 → 국소화의 1차 축.
- **logs**(예: Filebeat/Logstash) — 구조화 로그. 에러 메시지·스택·이벤트 디테일.
- **metrics**(예: Prometheus) — 시계열 지표. RED(Rate·Errors·Duration)·USE(Utilization·Saturation·Errors)로 증상을 정량화.

도구는 **중재된 읽기 액션**으로만 접근한다 — `get_logs`/`get_metrics`/`get_traces`/`exec_shell`(읽기 진단용).
이는 AIOpsLab의 중재 API 모델(§research)을 차용한 것으로, 에이전트가 인프라를 *관측*하되 *변경*하지 못하게 하는 안전 경계다.

## 3. 인시던트 생애주기 4단계 (AIOpsLab L1–L4)

이 하네스의 오케스트레이션 골격은 AIOpsLab의 4단계 파이프라인이다(§research).

1. **L1 Detection** — 정상에서 벗어났는가? 무엇이·언제부터·얼마나? (트리아지 — 범인 지목 아님)
2. **L2 Localization** — 어디서? traces 우선으로 범인 서비스/컴포넌트로 좁힘. (원인 단정 아님)
3. **L3 RCA** — 왜? 국소화된 표적의 root cause를 인과사슬로 확정. (anti-anchoring 가드레일)
4. **L4 Mitigation** — 어떻게 멈추나? 완화안 + 위험평가 (제안 형태, 사람 집행).

각 단계는 직전 단계의 *증거*만 입력으로 받는다(단계를 건너뛰지 않음 — 단, Phase 0의 Straight-Shot 판정 예외).

## 4. 역할 분리 오케스트레이션 (진단 ≠ 조치 ≠ 위험)

단일 에이전트가 탐지·국소화·원인·완화·위험을 다 하면, 자기 진단을 자기가 정당화하는 편향과 행동 가능성 낮은 권고가 생긴다.
이 하네스는 **진단(L1–L3)과 조치계획·위험평가(L4)를 다른 에이전트로 분리**한다. 역할 분리 오케스트레이션이
*실행 가능한* 결정론적 권고를 낸다는 근거가 있으나, 그 정량치(역할분리 실행가능 권고율 vs 단일에이전트)는 **단일 시나리오·human-validation 부재의 알고리즘 산출치이므로 일반화 금지**다(§research CAVEAT).
가치의 핵심은 *속도*가 아니라(두 구조 지연 유사) *결정론적 품질*이다.

> 본 저장소 컨벤션상 '에이전트'='전문화 프롬프트를 가진 호출'이어도 되지만, 4개 협업 하네스이므로 `agents/*.md` 정의 + `model:"opus"`를 둔다.

## 5. DQ 품질 게이트 (완화·권고)

완화안의 품질을 다음 루브릭으로 자가 채점한다(§research에서 차용).

```
DQ = 0.40·타당성(Validity) + 0.30·구체성(Specificity) + 0.30·정확성(Correctness)
```

- **타당성(0.40)** — 그 조치가 *실제로 root cause에 작용*하는가(증상만 가리지 않는가).
- **구체성(0.30)** — 사람이 *그대로 집행 가능*한가(대상·명령·순서·롤백이 명시됐는가).
- **정확성(0.30)** — 사실·시스템 상태와 어긋나지 않는가.

mitigation-planner는 각 안을 채점하고 임계 미달이면 보강·재채점한다(저품질 안 그대로 권고 금지).

## 6. RCA 가드레일 — 추론 실패모드 차단 (필수)

클라우드 RCA에서 다음 추론 실패모드가 정확도 하락과 *연관*된다(인과 아님, §research). RCA 에이전트는 이를 1급 작업 원칙으로 막는다.

| 실패모드 | 증상 | 가드레일 |
|----------|------|----------|
| anchoring bias | 첫 가설에 고착 | 경쟁 가설 ≥2개 동등 검토 |
| 반복 정체(stalled) | 같은 증거 재해석, 무진전 | 정체 감지 → 접근 전환/에스컬레이트 |
| 임의적 증거 선택 | 가설 지지 증거만 채택 | 반증 우선(falsify-first) |
| 신념 미갱신 | 새 증거 무시 | belief update — 충돌 시 가설 변경 |

### Straight-Shot 폴백 (anti-overengineering)
**agentic 다단계 추론이 항상 낫다는 가정은 위험하다.** 단순/명백 케이스나 작은 모델에선 다단계 RCA가
*수확체감·음효과*를 보이고, 비-agentic 단일 직답(Straight-Shot)이 동등 이상인 경우가 잦다(§research).
따라서 Phase 0에서 케이스가 단순하면(증거가 단일 원인을 명백히 지목) 다단계를 건너뛰고 Straight-Shot을 제안한다.

## 7. anti-pattern (피해야 할 운영 행동)

1. **인프라 직접 변경** — 에이전트가 스케일/롤백/플래그를 자동 집행. → 읽기 전용 + 사람 집행 제안으로 차단.
2. **로그만 보기** — traces 없이 분산 인시던트를 국소화하려다 전파된 증상을 발원지로 오인. → traces 1차 축.
3. **첫 가설 고착(anchoring)** — 가장 흔한 RCA 실패. → 경쟁 가설·반증 우선.
4. **단계 건너뛰기** — 증상에서 곧장 완화로 점프(국소화·RCA 생략)해 잘못된 곳을 고침. → L1→L4 순서(단순 케이스만 Straight-Shot).
5. **다단계 과잉(over-agentic)** — 단순 케이스에 불필요한 다단계로 정확도를 떨어뜨림. → Straight-Shot 폴백.
6. **완전 자율 신뢰** — 미성숙한 운영 자율화를 사람 검토 없이 신뢰. → 휴먼-인-더-루프(완화는 제안·사람 집행).
7. **저품질 완화 권고** — 추상적·집행 불가·고위험 안을 그대로 제시. → DQ 게이트.

## 8. 정직성 가드레일 (전 산출물 공통)

- 정량 수치는 출처 등급([GOLD]/[SILVER]/[BRONZE])·출처명·날짜와 함께만 인용한다.
- '개선 N%' 같은 약속을 하지 않는다. baseline을 먼저 제시하고 목표를 말한다.
- 반증·미검증 수치는 '신화'로만 표기하고 사실로 쓰지 않는다.
- §research의 CAVEAT(특히 역할분리 100%/1.7%의 단일 시나리오 한계, RCA 실패모드의 연관≠인과)를 본문에서 그대로 동반 인용한다.
